## 🧠 Память — общие понятия

- **Бит** — наименьшая единица информации: 0 или 1.
- **Байт** — 8 бит; минимальная адресуемая ячейка памяти. → [Как устроена память (C)](C/00-setup/00-how-computer-works.md)
- **Адрес** — номер ячейки памяти, по которому к ней обращаются.
- **Переменная** — именованная область памяти с значением определённого типа.
- **Стек (stack)** — область памяти для локальных переменных и вызовов функций; быстрый, автоматический, LIFO. → [Модель памяти (C)](C/02-memory/08-memory-model.md)
- **Куча (heap)** — область для динамической памяти, которой управляют вручную или через сборщик. → [Динамическая память (C)](C/02-memory/11-dynamic-memory.md)
- **Указатель (pointer)** — переменная, хранящая адрес другой переменной. → [Указатели (C)](C/02-memory/09-pointers.md)
- **Ссылка (reference)** — псевдоним/безопасная отсылка к объекту (в разных языках по-разному).
- **Утечка памяти (memory leak)** — выделенная память, которую забыли освободить. → [Ошибки памяти (C)](C/02-memory/12-memory-errors-tools.md)
- **Segfault** — аварийное обращение к чужой/недоступной памяти.
- **UB (undefined behavior)** — неопределённое поведение: программа имеет право на что угодно. → [Выравнивание и UB (C)](C/04-senior/20-alignment-ub.md)
- **Выравнивание (alignment) / padding** — размещение данных по «удобным» адресам; пустоты между полями. → [Выравнивание (C)](C/04-senior/20-alignment-ub.md)
- **Кэш / cache-line** — быстрая память между CPU и RAM; данные грузятся строками. → [Кэш и оптимизация (C)](C/04-senior/22-cache-optimization.md)

## ⚙️ Память по языкам

- **malloc / free** — ручное выделение и освобождение памяти в C. → [Динамическая память (C)](C/02-memory/11-dynamic-memory.md)
- **RAII** — приём C++: ресурс освобождается автоматически в деструкторе при выходе из области. → [RAII (C++)](Cpp/02-memory/10-raii.md)
- **Умный указатель** — обёртка C++ (`unique_ptr`/`shared_ptr`/`weak_ptr`), сама управляющая памятью. → [Умные указатели (C++)](Cpp/02-memory/11-smart-pointers.md)
- **Move-семантика** — передача владения ресурсом без копирования (C++ и Rust). → [Move (C++)](Cpp/02-memory/12-move-semantics.md)
- **Имя-ярлык (Python)** — переменная в Python — это ссылка на объект, а не «коробка». → [Переменные как ссылки (Python)](Python/01-basics/03-variables-references.md)
- **Подсчёт ссылок (refcount)** — Python считает ссылки на объект; ноль → удаление. → [refcount и GC (Python)](Python/02-memory/11-refcount-gc.md)
- **Сборщик мусора (GC)** — автоматически освобождает недостижимую память. → [refcount и GC (Python)](Python/02-memory/11-refcount-gc.md)
- **Изменяемые/неизменяемые (mutable/immutable)** — можно ли менять объект после создания. → [mutable/immutable (Python)](Python/02-memory/10-mutable-immutable.md)
- **Хеш-таблица** — структура «ключ→значение» с быстрым доступом (dict/set, HashMap). → [dict и хеши (Python)](Python/03-middle/14-dict-set-hashing.md)
- **Владение (ownership)** — правило Rust: у значения один владелец; вышел из области — освобождено. → [Владение (Rust)](Rust/02-memory/08-ownership.md)
- **Заимствование (borrowing)** — временный доступ по ссылке без передачи владения (Rust). → [Заимствование (Rust)](Rust/02-memory/09-borrowing.md)
- **Borrow checker** — компилятор Rust, проверяющий правила ссылок (один `&mut` ИЛИ много `&`). → [Borrow checker (Rust)](Rust/02-memory/10-borrow-checker.md)
- **Время жизни (lifetime)** — как долго ссылка действительна (Rust). → [Времена жизни (Rust)](Rust/02-memory/12-lifetimes.md)

## 🏛️ ООП и проектирование

- **Класс / объект** — чертёж и его экземпляр (объект — класс, «оживший» в памяти). → [Класс и объект](OOP/00-setup/01-class-object.md)
- **Инкапсуляция** — объединение данных с поведением + сокрытие внутренностей за интерфейсом. → [Инкапсуляция](OOP/02-pillars/09-encapsulation-deep.md)
- **Абстракция** — выделение существенного под задачу, сокрытие деталей («что», а не «как»). → [Абстракция](OOP/02-pillars/08-abstraction.md)
- **Наследование** — создание класса на основе другого («является»); легко злоупотребить. → [Наследование](OOP/02-pillars/10-inheritance.md)
- **Полиморфизм** — один интерфейс, разное поведение; убирает if-по-типу. → [Полиморфизм](OOP/02-pillars/11-polymorphism.md)

- **Композиция / агрегация** — объект содержит другие объекты («имеет»); композиция > наследование. → [Композиция](OOP/01-classes/06-composition.md)
- **SOLID** — пять принципов проектирования (SRP, OCP, LSP, ISP, DIP). → [SOLID](OOP/03-design/13-solid-intro.md)
- **DIP / Dependency Injection** — зависеть от абстракций; внедрять зависимости снаружи. → [DIP](OOP/03-design/15-lsp-isp-dip.md)
- **DRY / KISS / YAGNI** — не повторяйся / делай проще / не делай впрок. → [Принципы](OOP/03-design/17-principles.md)
- **Связность / зацепление** — сильная связность внутри модуля, слабое зацепление между. → [Принципы](OOP/03-design/17-principles.md)
- **Паттерн проектирования** — типовое решение задачи (Factory, Strategy, Observer…). → [Паттерны](OOP/04-patterns/19-creational.md)
- **Запах кода / антипаттерн** — признак плохого дизайна; лечится рефакторингом. → [Антипаттерны](OOP/04-patterns/22-antipatterns.md)
- **Чистая архитектура** — слои + зависимости, направленные к бизнес-логике (порты и адаптеры). → [Архитектура](OOP/04-patterns/23-architecture.md)

## 🖥️ Операционные системы

- **Ядро (kernel)** — сердце ОС: управляет процессами, памятью, устройствами, имеет полные права. → [Ядро и syscalls](OS/00-setup/01-kernel-syscalls.md)
- **User / kernel space** — пользовательский (ограниченный) и привилегированный (полный доступ) режимы. → [Ядро и syscalls](OS/00-setup/01-kernel-syscalls.md)
- **Системный вызов (syscall)** — единственная легальная «дверь» из программы в ядро (open/read/fork…). → [Ядро и syscalls](OS/00-setup/01-kernel-syscalls.md)
- **Процесс** — запущенная программа со своим адресным пространством, изолированная от других. → [Процесс](OS/01-processes/03-process.md)
- **Поток (thread)** — линия выполнения внутри процесса; потоки делят память процесса. → [Потоки](OS/01-processes/05-threads.md)
- **fork / exec** — создать копию процесса / заменить процесс другой программой. → [fork и exec](OS/01-processes/04-lifecycle.md)
- **Планировщик (scheduler)** — делит CPU между процессами, переключая их. → [Планировщик](OS/01-processes/06-scheduler.md)
- **Контекстное переключение** — сохранение состояния одного процесса и восстановление другого. → [Планировщик](OS/01-processes/06-scheduler.md)
- **Виртуальная память** — иллюзия собственной памяти для каждого процесса; перевод вирт.→физ. адреса. → [Зачем виртуальная память](OS/02-memory/08-why-virtual-memory.md)
- **Страница / таблица страниц / MMU** — блок памяти, карта перевода адресов, аппаратный переводчик. → [Страницы](OS/02-memory/09-paging.md)
- **Page fault** — обращение к странице, которой нет в RAM; ОС подгружает её (штатно). → [Page fault и swap](OS/02-memory/10-page-fault-swap.md)
- **Swap** — выгрузка редких страниц на диск, чтобы дать «памяти больше, чем RAM». → [Page fault и swap](OS/02-memory/10-page-fault-swap.md)
- **Мьютекс / семафор** — примитивы синхронизации: один ключ / N пропусков к общим данным. → [Мьютексы и семафоры](OS/03-sync/14-mutex-semaphore.md)
- **Гонка данных (race condition)** — баг от одновременного доступа к общим данным без синхронизации. → [Гонки данных](OS/03-sync/13-race-conditions.md)
- **Deadlock** — взаимная блокировка: потоки ждут друг друга навсегда. → [Deadlock](OS/03-sync/15-deadlock.md)
- **inode** — «паспорт файла»: метаданные и указатели на блоки (имя хранится отдельно). → [Файловые системы](OS/03-sync/16-filesystems.md)
- **namespaces / cgroups** — изоляция видимости и лимиты ресурсов; основа контейнеров. → [Виртуализация и контейнеры](OS/04-advanced/19-virtualization.md)

- **Динамический массив** — массив, удваивающий ёмкость при заполнении; доступ по индексу O(1). → [Массивы](Algorithms/01-structures/03-arrays.md)
- **Связный список** — узлы со ссылками; вставка O(1), доступ по индексу O(n). → [Связные списки](Algorithms/01-structures/04-linked-lists.md)
- **Стек / очередь** — LIFO / FIFO структуры доступа. → [Стек и очередь](Algorithms/01-structures/05-stack-queue.md)
- **Хеш-таблица** — отображение ключ→значение со средним доступом O(1). → [Хеш-таблицы](Algorithms/01-structures/06-hash-tables.md)
- **Бинарный поиск** — поиск в отсортированном массиве делением пополам, O(log n). → [Поиск](Algorithms/03-algorithms/13-search.md)
- **Сортировка слиянием / быстрая** — O(n log n) алгоритмы упорядочивания. → [Сортировки](Algorithms/03-algorithms/14-sorting.md)
- **Рекурсия** — функция, вызывающая себя; основа divide-and-conquer. → [Рекурсия](Algorithms/03-algorithms/15-recursion.md)
- **Дерево / BST** — иерархия узлов; в BST поиск/вставка O(log n) при балансе. → [Деревья](Algorithms/03-algorithms/16-trees.md)
- **Граф (BFS/DFS)** — вершины и рёбра; обход в ширину/глубину, оба O(V+E). → [Графы](Algorithms/03-algorithms/17-graphs.md)
- **Жадный алгоритм** — локально оптимальный выбор на каждом шаге. → [Жадные алгоритмы](Algorithms/03-algorithms/18-greedy.md)
- **Динамическое программирование (DP)** — разбиение на подзадачи с запоминанием результатов. → [DP](Algorithms/04-advanced/19-dynamic-programming.md)

## 🧭 Senior-мышление и инженерные практики

- **Trade-off (компромисс)** — у любого решения есть цена; «лучшего» нет, есть «лучшее для условий». Ядро senior-мышления. → [Trade-offs](Senior/02-decisions/08-tradeoffs.md)

- **Технический долг** — срезанные углы, за которые «платишь проценты» при будущих правках. → [Техдолг](Senior/01-craft/07-tech-debt.md)
- **Рефакторинг** — меняем структуру кода, не меняя поведение; малыми проверяемыми шагами. → [Рефакторинг](Senior/01-craft/05-refactoring.md)
- **ADR** — короткая запись значимого решения: контекст, выбор, отвергнутые альтернативы. → [ADR](Senior/02-decisions/11-adr.md)

- **Код-ревью** — совместная ответственность за код: критикуй код, не человека; blocker vs nit. → [Код-ревью](Senior/01-craft/06-code-review.md)
- **Bus factor** — сколько людей «под автобусом» обрушат проект; знание в одной голове — риск. → [Документация](Senior/03-practices/18-documentation.md)
- **Build vs Buy** — писать самим или взять готовое; код — пассив, а не актив. → [Build vs Buy](Senior/02-decisions/10-build-vs-buy.md)
- **Оценка (estimation)** — вероятностный диапазон с допущениями, не обязательство. → [Оценка](Senior/02-decisions/09-estimation.md)
- **Обратимость решения** — «дверь в одну/две стороны»: необратимое решай медленно, обратимое — быстро. → [Риски](Senior/02-decisions/12-risk-uncertainty.md)
- **Зацепление / связность** — слабое зацепление + высокая связность = код, который легко менять. → [Проектирование под изменения](Senior/03-practices/15-design-for-change.md)
- **STAR** — структура ответа на поведенческие вопросы: Situation–Task–Action–Result. → [Собеседования](Senior/04-leadership/23-interviews.md)

## 🎭 Социальная инженерия (защита)

> ⚠️ Защитно-образовательные термины. Применение приёмов к людям без согласия незаконно.

- **Социальная инженерия** — атака на человека (обман), а не на технику; причина большинства взломов. → [Что это](SocialEng/00-intro/00-what-is-se.md)
- **Фишинг** — массовая рассылка писем под доверенный источник ради данных/доступа. → [Фишинг](SocialEng/01-vectors/03-phishing.md)
- **Вишинг / смишинг** — то же по телефону (голос) / через SMS и мессенджеры. → [Вишинг](SocialEng/01-vectors/04-vishing-smishing.md)
- **Претекстинг** — выдуманный правдоподобный сценарий и роль для обмана. → [Претекстинг](SocialEng/01-vectors/05-pretexting.md)
- **Бейтинг** — «наживка» (найденная флешка, халява), играющая на любопытстве/жадности. → [Бейтинг](SocialEng/01-vectors/06-baiting.md)
- **Tailgating** — проход в закрытую зону «на хвосте» у того, у кого есть доступ. → [Физические приёмы](SocialEng/01-vectors/07-physical.md)
- **Принципы влияния** — авторитет, срочность, соц. доказательство, симпатия, взаимность, обязательство; рычаги атак. → [Шесть принципов](SocialEng/02-psychology/08-influence-principles.md)
- **Спуфинг** — подмена отправителя/номера, чтобы выглядеть доверенным («номер настоящий» ≠ доказательство). → [Вишинг](SocialEng/01-vectors/04-vishing-smishing.md)
- **OSINT** — сбор информации из открытых источников для подготовки атаки (и для защиты своего следа). → [OSINT](SocialEng/03-recon/13-osint.md)
- **Spear phishing / whaling** — целевой фишинг под конкретного человека / на руководителей. → [Целевой фишинг](SocialEng/03-recon/15-spear-phishing.md)
- **BEC** — компрометация деловой переписки ради перевода денег («письмо от директора»). → [Целевой фишинг](SocialEng/03-recon/15-spear-phishing.md)
- **Credential stuffing** — подбор по утёкшим парам email+пароль на других сервисах (отсюда — уникальные пароли). → [Утечки](SocialEng/03-recon/17-data-breaches.md)
- **MFA** — многофакторная аутентификация; главный барьер: даже зная пароль, без второго фактора не войти. → [Гигиена](SocialEng/04-defense/18-personal-hygiene.md)
- **Независимая верификация** — проверка просьбы через канал, который не контролирует атакующий; ловит почти всё. → [Проверка](SocialEng/04-defense/19-verification.md)
- **Защита в глубину** — несколько рубежей (след → распознавание → MFA → мониторинг → реакция). → [Цепочка атаки](SocialEng/03-recon/16-attack-chain.md)

## 🛡️ Этичный хакинг и AppSec

> ⚠️ Образовательные термины. Наступательные техники — только в легальной среде/с авторизацией.

- **White / grey / black hat** — этичный (с разрешения) / «серый» (без разрешения, но без умысла — тоже незаконно) / преступник. → [Этичный хакер](Security/00-foundations/00-ethical-hacker.md)
- **Пентест** — авторизованная проверка защиты по договору; продукт — отчёт с уязвимостями и как их закрыть. → [Пентест и bug bounty](Security/04-blue-team/25-pentest-bugbounty.md)
- **Bug bounty** — поиск уязвимостей в рамках публичной программы (scope+правила) за награду. → [Пентест и bug bounty](Security/04-blue-team/25-pentest-bugbounty.md)
- **Scope** — границы разрешённого в пентесте/программе; выход за него — преступление. → [Закон и авторизация](Security/00-foundations/02-law-authorization.md)
- **Модель угроз** — что/от кого/как защищаем + приоритеты по риску; STRIDE, границы доверия. → [Модель угроз](Security/01-recon/04-threat-modeling.md)
- **Поверхность атаки** — все точки входа; защита — минимизировать. → [Модель угроз](Security/01-recon/04-threat-modeling.md)
- **OWASP Top-10** — отраслевой список 10 критичных веб-рисков; обязательный минимум AppSec. → [OWASP Top-10](Security/02-vulnerabilities/14-owasp-top10.md)
- **Инъекция / SQLi** — ввод стал кодом/командой; защита — параметризация (разделять код и данные). → [Инъекции](Security/02-vulnerabilities/09-injection-sqli.md)
- **XSS** — внедрение скрипта в страницу; защита — экранирование вывода + CSP. → [XSS](Security/02-vulnerabilities/10-xss.md)
- **IDOR / broken access control** — доступ к чужому по id без проверки прав; №1 OWASP. → [Контроль доступа](Security/02-vulnerabilities/11-access-control.md)
- **CSRF / SSRF** — подделка запроса от лица пользователя / от лица сервера. → [CSRF и SSRF](Security/02-vulnerabilities/12-csrf-ssrf.md)
- **Аутентификация vs авторизация** — «кто ты» vs «что тебе можно»; проверять authz на каждое действие. → [Auth](Security/01-recon/07-auth.md)
- **CIA** — конфиденциальность / целостность / доступность — три цели безопасности. → [CIA и крипта](Security/01-recon/08-crypto-basics.md)
- **Хеширование паролей** — bcrypt/argon2 с солью; никогда не открытый текст/MD5. → [Хранение паролей](Security/03-defensive-code/17-password-storage.md)
- **Allow-list** — проверять ввод по «разрешённому», а не перечислять «запрещённое». → [Валидация](Security/03-defensive-code/16-input-validation.md)
- **Наименьшие привилегии** — минимум прав каждому; ограничивает ущерб при взломе. → [Думай как атакующий](Security/00-foundations/01-attacker-mindset.md)
- **Защита в глубину** — несколько рубежей; один пробьют — другой держит. → [Hardening](Security/04-blue-team/22-hardening.md)
- **Hardening** — укрепление: убрать лишнее + настроить безопасно + ограничить. → [Hardening](Security/04-blue-team/22-hardening.md)
- **DevSecOps / сдвиг влево** — встроить безопасность во все этапы разработки и в CI. → [DevSecOps](Security/04-blue-team/24-devsecops.md)
- **Responsible disclosure** — приватно сообщить об уязвимости, дать время исправить, не публиковать/эксплуатировать. → [Закон и авторизация](Security/00-foundations/02-law-authorization.md)

## ⚙️ Как работает компьютер (Computer Science)

- **ISA** — набор команд процессора (x86-64/ARM); «контракт» между компилятором и CPU. → [Машинный код](ComputerScience/01-hardware/05-machine-code.md)
- **Регистры** — крошечная сверхбыстрая память в CPU, где идут вычисления. → [Регистры](ComputerScience/01-hardware/04-registers.md)
- **Конвейер (pipeline)** — CPU обрабатывает части нескольких инструкций параллельно. → [Конвейер](ComputerScience/01-hardware/06-pipeline.md)
- **Предсказание ветвлений** — CPU угадывает исход if; ошибка = штраф (сброс конвейера). → [Конвейер](ComputerScience/01-hardware/06-pipeline.md)
- **Кэш L1/L2/L3** — быстрые буферы перед медленной RAM; промах = ~100× дороже. → [Кэш](ComputerScience/01-hardware/08-cache.md)
- **Кэш-линия** — блок (~64 байта), которым кэш грузит данные; основа cache-friendly кода. → [Кэш](ComputerScience/01-hardware/08-cache.md)
- **Локальность** — временна́я/пространственная; почему последовательный доступ быстр. → [Иерархия памяти](ComputerScience/01-hardware/07-ram-hierarchy.md)
- **Этапы сборки** — препроцессор → компилятор → ассемблер → линковщик. → [Этапы сборки](ComputerScience/02-toolchain/09-build-stages.md)
- **Ассемблер** — читаемая запись машинного кода (1:1); язык, на котором «говорит» CPU. → [Ассемблер](ComputerScience/02-toolchain/10-assembly.md)
- **Объектный файл / символы** — машинный код + таблица символов; основа линковки. → [Объектные файлы](ComputerScience/02-toolchain/12-object-files.md)
- **Линковка (статич./динамич.)** — соединение .o и библиотек; .a (вшить) vs .so/.dll (подгрузить). → [Линковка](ComputerScience/02-toolchain/13-linking.md)
- **ELF / PE** — формат исполняемого файла (Linux / Windows): код, данные, точка входа. → [ELF и PE](ComputerScience/02-toolchain/14-elf-pe.md)
- **Загрузчик** — часть ОС, что грузит ELF в память и запускает процесс. → [Загрузчик](ComputerScience/03-execution/15-loader.md)
- **Стек вызовов / кадр** — как функции «помнят» возврат; адрес возврата + локальные. → [Стек вызовов](ComputerScience/03-execution/17-call-stack.md)
- **Calling convention** — где передаются аргументы/результат функции (регистры/стек). → [Calling convention](ComputerScience/03-execution/18-calling-convention.md)
- **ABI** — бинарный контракт платформы (вызовы, раскладка типов, формат); C ABI — лингва франка FFI. → [Calling convention и ABI](ComputerScience/03-execution/18-calling-convention.md)
- **Виртуальная память / MMU** — виртуальные адреса процесса → физ. RAM; изоляция, segfault. → [Виртуальная память](ComputerScience/03-execution/19-virtual-memory.md)
- **PLT / GOT** — таблицы для вызова функций из динамических библиотек; ленивое связывание. → [Динамические библиотеки](ComputerScience/03-execution/20-dynamic-libs.md)
- **memory/compute/branch-bound** — чем ограничен код; определяет, как оптимизировать. → [Что мешает CPU](ComputerScience/04-performance/22-cpu-bottlenecks.md)
- **SIMD / автовекторизация** — одна инструкция над многими данными; обычно через компилятор. → [SIMD](ComputerScience/04-performance/24-simd.md)

## 🗄️ Базы данных

- **Реляционная модель** — данные как таблицы (отношения): строки (записи) + столбцы (атрибуты). → [Реляционная модель](Database/01-modeling/03-relational-model.md)
- **Первичный ключ (PK)** — столбец, уникально идентифицирующий строку. → [Ключи и связи](Database/01-modeling/04-keys-relations.md)
- **Внешний ключ (FK)** — ссылка на PK другой таблицы; задаёт связи между таблицами. → [Ключи и связи](Database/01-modeling/04-keys-relations.md)
- **Нормализация** — устранение дублирования разбиением на таблицы (1НФ→2НФ→3НФ). → [Нормализация](Database/01-modeling/05-normalization.md)
- **SQL** — декларативный язык запросов к реляционным БД (что нужно, не как достать). → [SELECT](Database/02-sql/08-select-basics.md)
- **JOIN** — соединение строк нескольких таблиц по связи (ON-условие); ядро SQL. → [JOIN-ы](Database/02-sql/09-joins.md)
- **INNER / LEFT JOIN** — только совпадения / все левые + совпавшие (NULL для несовпавших). → [JOIN-ы](Database/02-sql/09-joins.md)
- **GROUP BY / агрегаты** — группировка строк + COUNT/SUM/AVG; HAVING фильтрует группы. → [Агрегация](Database/02-sql/10-aggregation.md)
- **CTE (WITH)** — именованный временный результат для читаемых сложных запросов. → [Подзапросы и CTE](Database/02-sql/11-subqueries-cte.md)
- **Индекс** — B-дерево над столбцом: поиск O(log n) вместо перебора O(n). Цена: место + замедление записи. → [Индексы](Database/03-performance/13-indexes.md)
- **EXPLAIN** — план выполнения запроса: seq scan vs index scan, стоимость, порядок join. → [EXPLAIN](Database/03-performance/14-explain.md)
- **Seq scan / Index scan** — полный перебор таблицы / поиск по индексу. → [EXPLAIN](Database/03-performance/14-explain.md)
- **N+1 запрос** — 1 запрос списка + N запросов по элементам; антипаттерн ORM. → [Оптимизация](Database/03-performance/15-query-optimization.md)
- **Транзакция** — группа операций как неделимая единица (всё или ничего); BEGIN/COMMIT/ROLLBACK. → [Транзакции](Database/03-performance/16-transactions-acid.md)
- **ACID** — atomicity/consistency/isolation/durability — гарантии надёжности транзакций. → [Транзакции](Database/03-performance/16-transactions-acid.md)
- **Уровни изоляции** — read committed / repeatable read / serializable; ↔ аномалии параллельности. → [Изоляция](Database/03-performance/17-isolation-locks.md)
- **Deadlock** — взаимная блокировка двух транзакций; СУБД убивает одну. → [Изоляция и блокировки](Database/03-performance/17-isolation-locks.md)
- **NoSQL** — нереляционные БД: документная / key-value / колоночная / графовая. → [NoSQL](Database/04-scale/18-nosql.md)
- **Репликация** — копии БД (primary пишет, реплики читают); масштаб чтения + надёжность. → [Репликация](Database/04-scale/19-replication-sharding.md)
- **Replication lag** — отставание реплик от primary → можно прочитать устаревшее. → [Репликация](Database/04-scale/19-replication-sharding.md)
- **Шардинг** — разбиение данных по серверам (shard key); масштаб записи/объёма; усложняет всё. → [Шардинг](Database/04-scale/19-replication-sharding.md)
- **CAP** — при сетевом разделении выбираешь Consistency или Availability (не оба). → [CAP](Database/04-scale/20-cap.md)
- **Eventual consistency** — узлы сойдутся «в конце концов»; терпимо для лайков, нет для баланса. → [CAP](Database/04-scale/20-cap.md)
- **Кэш / cache-aside** — частые данные в быстром хранилище (Redis); miss→БД→положить с TTL. → [Кэширование](Database/04-scale/21-caching.md)
- **Инвалидация кэша** — сброс устаревших данных (TTL / при записи); главная сложность кэша. → [Кэширование](Database/04-scale/21-caching.md)
- **Миграция** — версионируемое обратимое изменение схемы (UP/DOWN); обратно совместимо. → [Миграции](Database/04-scale/22-migrations.md)
- **SQL-инъекция** — внедрение SQL через ввод; защита — параметризованные запросы. → [Безопасность БД](Database/04-scale/23-security.md)

## 🌳 Git и код-ревью

- **Контроль версий (VCS)** — система, хранящая историю изменений проекта (кто/что/когда/зачем). → [Зачем VCS](Git/00-intro/00-why-version-control.md)
- **Коммит** — снимок всего проекта + автор/дата/сообщение + ссылка на родителя. → [Что такое Git](Git/00-intro/01-what-is-git.md)
- **Снимок vs дельта** — Git хранит снимки состояния, а не пошаговые изменения. → [Что такое Git](Git/00-intro/01-what-is-git.md)
- **Staging (индекс)** — зона «что войдёт в следующий коммит»; собирается через `git add`. → [Staging и commit](Git/01-basics/03-repo-staging-commit.md)
- **HEAD** — указатель на «где ты сейчас» (обычно текущая ветка → её последний коммит). → [Что такое Git](Git/00-intro/01-what-is-git.md)
- **git status** — компас: что изменено, в staging, на какой ветке; подсказывает команды. → [Staging и commit](Git/01-basics/03-repo-staging-commit.md)
- **diff / blame** — разница между версиями / кто написал каждую строку и в каком коммите. → [История](Git/01-basics/04-history-log-diff.md)
- **reset / restore / revert** — отменить коммит (двигает ветку) / откатить файл / коммит-отмена (для опубликованного). → [Отмена изменений](Git/01-basics/05-undoing-changes.md)
- **reflog** — журнал всех перемещений HEAD; страховка для восстановления «потерянного». → [Восстановление](Git/04-pro/23-recovery.md)
- **.gitignore** — список путей, которые Git не отслеживает (артефакты, зависимости, секреты). → [.gitignore](Git/01-basics/06-gitignore.md)
- **Атомарный коммит** — одно логическое, обратимое изменение с ясным сообщением (императив, «почему»). → [Хорошие коммиты](Git/01-basics/07-good-commits.md)
- **Ветка** — лёгкий подвижный указатель на коммит (не копия файлов); создаётся мгновенно. → [Ветки](Git/02-branching/08-branches.md)
- **Merge** — объединение веток; fast-forward (перемотка) или merge-коммит (2 родителя). → [Слияние](Git/02-branching/09-merge.md)
- **Конфликт слияния** — обе ветки правили одно место; решает человек (маркеры `<<< === >>>`). → [Конфликты](Git/02-branching/10-conflicts.md)
- **Rebase** — «пересадка» коммитов на новую базу → линейная история; переписывает хеши. → [Rebase](Git/02-branching/11-rebase.md)
- **Золотое правило rebase** — не переписывать опубликованную/общую историю. → [Rebase](Git/02-branching/11-rebase.md)
- **Remote / origin** — ссылка на копию репозитория на сервере; origin — главный по умолчанию. → [Удалённые репозитории](Git/03-collaboration/13-remotes.md)
- **fetch / pull / push** — забрать в origin/* / забрать+слить / отдать свои коммиты. → [Fetch/pull/push](Git/03-collaboration/14-fetch-pull-push.md)
- **Pull request (PR)** — предложение влить ветку с диффом, ревью и CI; ворота качества перед main. → [Pull request](Git/03-collaboration/15-pull-requests.md)
- **Код-ревью** — проверка изменений другим (качество + знание + обучение); критикуй код, не человека. → [Код-ревью](Git/03-collaboration/16-code-review.md)
- **Workflow (GitHub Flow / trunk-based / Git Flow)** — модель ветвления команды. → [Рабочие процессы](Git/03-collaboration/17-workflows.md)
- **Интерактивный rebase** — причесать историю до публикации (squash/reword/drop/reorder). → [Interactive rebase](Git/04-pro/18-interactive-rebase.md)
- **stash / cherry-pick** — отложить незакоммиченное / перенести один коммит между ветками. → [stash/cherry/reflog](Git/04-pro/19-stash-cherry-reflog.md)
- **Тег / semver** — метка версии на коммите / MAJOR.MINOR.PATCH как контракт совместимости. → [Теги и релизы](Git/04-pro/20-tags-releases.md)
- **Git-хук / CI** — скрипт на событие (pre-commit) / серверные проверки на PR (ворота качества). → [Хуки и CI](Git/04-pro/21-hooks-ci.md)
- **git bisect** — двоичный поиск коммита, внёсшего баг (log₂ проверок вместо перебора). → [Bisect](Git/04-pro/22-large-repos-bisect.md)
