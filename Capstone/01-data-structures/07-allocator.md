## 📖 Что строим и зачем

```
   АЛЛОКАТОР — управляет куском памяти, выдавая и забирая блоки (как malloc/free, но твой).
   зачем свой: malloc универсален, но медленен и фрагментирует. для частых мелких выделений
   (узлы, объекты в игре/парсере) кастомный аллокатор в РАЗЫ быстрее.

   реализовав, ты ПОЙМЁШЬ, что делает malloc, и научишься управлять памятью на уровне Senior.

   два простых, но мощных типа:
   • ARENA (bump/linear) — выделяет последовательно сдвигом указателя; освобождает ВСЁ сразу.
   • POOL — заранее нарезанные блоки фиксированного размера; выдаёт/возвращает за O(1).
```

🖼️
```
   ARENA:  [████░░░░░░░░░░░░]  alloc(n): вернуть текущую позицию, сдвинуть на n. O(1)!
            ↑позиция          reset(): позиция = начало (всё «освобождено» разом). НЕТ пер-объектного free.

   POOL (блоки фикс. размера): [#][#][#][ ][#][ ] + список свободных.
            alloc: взять из списка свободных. free: вернуть в список. оба O(1), без фрагментации.
```

## ⭐ Milestones

```
   === ARENA аллокатор ===
   1. возьми большой буфер (malloc один раз или статический). храни указатель «текущая позиция».
   2. alloc(size): вернуть позицию, сдвинуть позицию на size (с ВЫРАВНИВАНИЕМ!). если не влезает — fail/новый блок.
   3. reset(): позиция = начало (массовое освобождение). деструктор освобождает буфер.
   4. ВЫРАВНИВАНИЕ: округляй позицию вверх до alignment (8/16 байт) — иначе UB/медленно.

   === POOL аллокатор ===
   5. нарежь буфер на блоки фикс. размера, свяжи в список свободных (free list).
   6. alloc(): снять блок с free list (O(1)). free(ptr): вернуть блок в free list (O(1)).
   7. (⭐) обработка исчерпания, несколько чанков.

   === применение ===
   8. ПОДКЛЮЧИ к прошлому проекту: например, узлы списка/хеш-таблицы выделяй из своего pool.
      замерь ускорение vs malloc.
   готово: arena и pool работают, корректны (выравнивание!), дают ускорение на мелких выделениях.
```

## ⭐⭐ Ключевые решения и уроки

```
   ⚡ ПОЧЕМУ БЫСТРЕЕ malloc:
      • malloc — общего назначения: ищет подходящий блок, ведёт метаданные, потокобезопасен → дорого.
      • arena: alloc = сдвиг указателя (одна инструкция!). pool: снять с list. → в разы быстрее.
      • + ЛОКАЛЬНОСТЬ: объекты из arena/pool лежат рядом → кэш-дружелюбнее, чем разбросанный malloc.

   📐 ВЫРАВНИВАНИЕ (alignment) — данные должны лежать по «круглым» адресам (int по кратному 4, и т.д.).
      невыровненный доступ = UB / замедление / крэш на некоторых архитектурах. округляй позицию вверх.
      (связь: [выравнивание/padding в C](../../C/04-senior/20-alignment-ub.md), [⚙️ ABI](../../ComputerScience/03-execution/18-calling-convention.md))

   🗑️ FRAGMENTATION (фрагментация) — почему универсальный malloc «дырявит» память:
      разные размеры выделяются/освобождаются → дыры → память есть, но непрерывного блока нет.
      arena (всё сразу) и pool (фикс. размер) ЛИШЕНЫ фрагментации — в этом их сила.

   ⚖️ TRADE-OFF: arena — мгновенно, но нет пер-объектного освобождения (только всё). pool — есть free,
      но фикс. размер. выбираешь под паттерн использования. malloc — универсален, но медленнее.
```

💡 ⭐⭐ Главный урок: **аллокатор — это trade-off скорость/гибкость/фрагментация**. Универсальный
malloc платит за универсальность; зная свой паттерн выделений, ты делаешь специализированный
аллокатор в разы быстрее. Это инструмент Senior для горячих участков (игры, парсеры, серверы). И
ты наконец понимаешь, что malloc делает «под капотом».

## 📖 Как проверить

```
   ✅ arena: выделяй блоки → они не пересекаются, выровнены, данные целы; reset переиспользует.
   ✅ pool: alloc/free много раз → корректно переиспользует блоки, нет пересечений.
   ✅ выравнивание: проверь, что адреса кратны alignment.
   ✅ БЕНЧМАРК: 1млн мелких выделений через malloc vs твой pool/arena. во сколько раз быстрее?
   ✅ память: общий буфер освобождается; нет выхода за границы (ASan).
```

## 🌟 Расширения

```
   🥚 stack allocator (LIFO освобождение — освобождай в обратном порядке).
   🐣 free-list аллокатор переменного размера (ближе к malloc) — coalescing соседних дыр.
   🦅 STL-совместимый аллокатор (подключить к std::vector через Allocator-интерфейс).
   🚀 потокобезопасный/per-thread аллокатор (мост к Ур.2).
```

## ⚠️ Ловушки

- ❌ Забыть выравнивание → UB/крэш/медленно.
- ❌ Выход за границы буфера (не проверять, влезает ли).
- ❌ Pool: вернуть блок не того размера / двойной free.
- ❌ Arena: пытаться освобождать по одному (она не для этого — только reset).
- ❌ Не замерить выигрыш (а он — смысл проекта).
- ❌ Считать, что свой аллокатор всегда лучше (malloc универсальнее; это trade-off).
