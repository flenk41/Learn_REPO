# 01 · Методология больших проектов 🖼️⭐⭐

> 🎯 **Цель блока:** освоить, КАК доводить большой проект до результата, не утонув, — итеративно,
> от минимального работающего к полному.

---

## 📖 Большой проект пугает — режь на маленькие шаги

```
   «написать БД» — парализует (с чего начать? как всё охватить?). секрет: НЕ строй всё сразу.
   строй МИНИМАЛЬНУЮ работающую версию, потом наращивай по одной фиче. всегда имей нечто РАБОТАЮЩЕЕ.
```

💡 ⭐⭐ Главный принцип: **итеративность от MVP**. Минимальный жизнеспособный продукт (MVP) — самая
простая версия, которая уже что-то делает. Сервер, отвечающий «hello» одному клиенту. БД,
хранящая строки в памяти. Интерпретатор, считающий `2+2`. От работающего малого — наращиваешь.

🖼️
```
   ❌ строить всё сразу:  [............ 3 месяца ............] → «о боже, ничего не работает»
   ✅ итеративно:  [MVP✅]→[+фича✅]→[+фича✅]→[+фича✅]→ полный
                    всегда РАБОТАЕТ, виден прогресс, можно остановиться в любой точке
```

---

## ⭐ Цикл итерации

```
   для каждой фичи — короткий цикл (как в [рефакторинге Senior]):
   1. ВЫБЕРИ следующую маленькую фичу/milestone (что добавит ценность).
   2. ПОСТРОЙ её просто (не идеально — работающе).
   3. ПРОВЕРЬ (тест/ручная проверка) — работает?
   4. ЗАКОММИТЬ (git — точка возврата).
   5. (по необходимости) ОТРЕФАКТОРЬ — почисти, раз работает.
   → повтори. всегда «зелёный» рабочий проект между итерациями.
```

💡 ⭐ Маленькие коммиты + всегда рабочее состояние = ты не теряешься и не боишься сломать. Сломал —
откатился на коммит. Это [рефакторинг малыми шагами](../../Senior/01-craft/05-refactoring.md) на
уровне проекта.

---

## ⭐⭐ Тесты с самого начала

```
   на больших проектах БЕЗ тестов ты тонешь: добавил фичу → сломал старое → не заметил.
   ✅ пиши тесты ПО ХОДУ: для каждой фичи — проверка, что она работает И что старое не сломалось.
   • свой vector → тесты push/get/grow/edge-cases.
   • сервер → тест: подключился клиент, получил ответ.
   • интерпретатор → тесты: «2+2»=4, «(1+2)*3»=9.
   тесты = страховка, позволяющая смело наращивать и рефакторить.
```

💡 ⭐⭐ Тесты — не «потом», а инструмент скорости: они ловят регрессии мгновенно, и ты строишь
смело. Особенно для проектов с краевыми случаями (структуры данных, парсеры) тесты экономят часы
отладки. Минимум — набор проверок «вход → ожидаемый выход».

---

## 📖 Инструменты дисциплины

```
   ✅ GIT — коммить часто; ветки для экспериментов; история = можно откатиться/понять.
   ✅ СИСТЕМА СБОРКИ — Makefile/CMake/Cargo с самого начала (не «соберу руками»).
   ✅ ОТЛАДЧИК (gdb) и САНИТАЙЗЕРЫ (ASan/UBSan для C/C++) — ловят память/UB рано.
   ✅ ПРОФАЙЛЕР — когда дойдёшь до «сделать быстро» (не преждевременно).
   ✅ README/заметки — что строишь, как запустить, какие решения (для себя через месяц).
```

> 🧭 Это [Senior-практики](../../Senior/03-practices/13-debugging-method.md) (отладка, git, тесты,
> доведение) в действии — капстоун-проект тренирует их по-настоящему.

---

## ⭐ Антипаттерны больших проектов

```
   ❌ «СНАЧАЛА ИДЕАЛЬНАЯ АРХИТЕКТУРА» — застрянешь в проектировании, не написав строки. Начни просто,
      архитектура вырастет из работающего кода (но думай на шаг вперёд).
   ❌ ПРЕЖДЕВРЕМЕННАЯ ОПТИМИЗАЦИЯ — сначала «работает», потом «быстро» (измеряя).
   ❌ ОХВАТИТЬ ВСЁ СРАЗУ — режь на milestones.
   ❌ БЕЗ ТЕСТОВ/GIT — тонешь в регрессиях, не можешь откатиться.
   ❌ ПЕРФЕКЦИОНИЗМ — «работает и понятно» > «идеально, но не закончено».
```

---

## ⚠️ Ловушки

- ❌ Строить всё сразу вместо MVP + итерации.
- ❌ Откладывать тесты и git «на потом» — утонешь в регрессиях.
- ❌ Застрять в проектировании идеальной архитектуры до первой строки.
- ❌ Оптимизировать до того, как заработало.
- ❌ Бросать на 80% из-за перфекционизма.

---

## ✅ Упражнения на размышление

1. **MVP.** Для проекта, который выберешь, сформулируй MVP — самую простую версию, которая уже
   что-то делает. Что это?
2. **Milestones.** Разбей этот проект на 5-7 маленьких milestones от MVP до полного.
3. **Тесты.** Для первого milestone — какие 3 теста докажут, что он работает?
4. **Дисциплина.** Какие инструменты (git/сборка/санитайзеры/тесты) настроишь до начала?

---

## ❓ Проверь себя

1. Что такое MVP и почему строить от него?
2. Опиши цикл итерации (выбери→построй→проверь→коммить→рефактор).
3. Почему тесты нужны с самого начала?
4. Назови антипаттерны больших проектов.

---

## ✅ Чек-лист

- [ ] Строю от MVP, итеративно, всегда имея рабочее
- [ ] Веду git с частыми коммитами
- [ ] Пишу тесты по ходу, не «потом»
- [ ] Настраиваю сборку/санитайзеры/отладчик заранее
- [ ] Избегаю перфекционизма и преждевременной оптимизации
