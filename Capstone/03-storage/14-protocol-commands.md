# 14 · Протокол и команды 🖼️⭐

> 🎯 **Проект:** спроектируй протокол общения клиента с хранилищем — как сериализовать команды и
> ответы поверх TCP. Учит дизайну протоколов и парсингу.

> 🛠️ Применяешь: [свой сервер](../02-server/09-tcp-echo.md), парсинг (мостик к [интерпретатору](../04-language/18-lexer.md)).

---

## 📖 Зачем протокол

```
   клиент и хранилище общаются по сети байтами (TCP — поток, модуль 09). нужен ПРОТОКОЛ — соглашение,
   КАК закодировать команду (SET name Аня) в байты и КАК распознать ГРАНИЦЫ команд/ответов в потоке.
   без протокола сервер не знает, где кончается одна команда и начинается следующая.
```

🖼️
```
   клиент: команду "SET name Аня" → СЕРИАЛИЗОВАТЬ в байты по протоколу → отправить.
   сервер: байты → РАЗОБРАТЬ (где границы? какая команда? аргументы?) → выполнить → ответ по протоколу.
   ключевая проблема (снова): ГРАНИЦЫ сообщений в потоке TCP.
```

---

## ⭐ Дизайн протокола: текстовый vs бинарный

```
   ТЕКСТОВЫЙ (как Redis RESP, HTTP):
   ✅ человекочитаемый (легко отлаживать через nc/telnet), простой парсинг.
   ❌ чуть больше байт, парсинг строк.
   пример (с длиной/разделителями):  *3\r\n$3\r\nSET\r\n$4\r\nname\r\n$3\r\nАня\r\n
   или проще:  SET name "Аня"\n   (команда — строка до \n, аргументы по пробелам/кавычкам)

   БИНАРНЫЙ:
   ✅ компактнее, быстрее парсить.
   ❌ нечитаемый, сложнее отлаживать, надо думать про endianness/выравнивание.

   рекомендация для учебного: НАЧНИ С ТЕКСТОВОГО (проще отлаживать), при желании оптимизируй в бинарный.
```

💡 ⭐ Это [trade-off](../../Senior/02-decisions/08-tradeoffs.md): читаемость/простота (текст) vs
компактность/скорость (бинарь). Redis выбрал текстовый RESP (легко отлаживать) — хороший ориентир.
HTTP (модуль 12) — тоже текстовый протокол; ты уже его парсил.

## ⭐⭐ Главная проблема: framing (границы сообщений)

```
   TCP — поток. как понять, где кончается команда? три способа (FRAMING):
   1. РАЗДЕЛИТЕЛЬ — команда до \n (или \r\n). просто, но что если данные содержат \n? (экранирование/кавычки)
   2. ПРЕФИКС ДЛИНЫ — сначала число байт, потом столько байт данных. надёжно для бинарных данных.
   3. КОМБИНАЦИЯ (как RESP) — тип + длина + данные.

   сервер должен: копить байты в буфере → проверять, собралась ли ПОЛНАЯ команда → если да, парсить и
   выполнять → остаток оставить в буфере для следующей. (это машина состояний из event loop, модуль 10!)
```

💡 ⭐⭐ Framing — фундаментальная проблема ЛЮБОГО протокола поверх TCP. Ты уже встречал её в HTTP
(границы по \r\n\r\n + Content-Length). Здесь решаешь явно: разделитель или префикс длины. Это и
есть «дизайн протокола» — то, что делает сетевые приложения возможными.

## 📖 Milestones

```
   1. ВЫБЕРИ протокол (рекомендую текстовый: команда — строка, аргументы по пробелам, кавычки для пробелов).
   2. ПАРСЕР КОМАНД: байты из сокета → буфер → выдели полную команду (до \n) → разбери на токены (команда + аргументы).
   3. ДИСПЕТЧЕР: команда → вызов соответствующей операции хранилища (SET/GET/DEL...).
   4. ФОРМАТ ОТВЕТА: успех/значение/ошибка/nil — закодируй по протоколу (например, +OK\n, $значение\n, -ERR\n).
   5. ОБРАБОТКА ОШИБОК: неизвестная команда, неверные аргументы → понятный ответ-ошибка, не падение.
   6. ИНТЕГРАЦИЯ с буфером соединения (event loop) — частичные команды, несколько команд за раз (pipelining).
   готово: можно через nc/свой клиент слать команды и получать ответы; протокол устойчив.
```

---

## ⚠️ Ловушки

- ❌ Игнорировать framing (где границы команд в потоке) → склейка/разрыв команд.
- ❌ Парсить, не дождавшись полной команды (частичные данные из TCP).
- ❌ Не обрабатывать данные со спецсимволами (\n, пробелы) — нужны кавычки/экранирование/длина.
- ❌ Падать на кривой команде вместо ответа-ошибки.
- ❌ Не поддержать несколько команд в одном буфере (pipelining).

---

## ✅ Задачи

1. **Текстовый протокол.** Спроектируй формат команд и ответов. Опиши (мини-спека).
2. **Парсер.** Из буфера выделяй полные команды (framing по \n), разбирай на токены. Тесты на
   частичные/склеенные.
3. **Диспетчер.** Команды → операции хранилища; ответы по протоколу. Неизвестные → ошибка.
4. ⭐ **Кавычки/спецсимволы.** Поддержи аргументы с пробелами (кавычки) или префикс длины.
5. ⭐ **Pipelining.** Несколько команд в одном буфере обрабатываются по порядку.
