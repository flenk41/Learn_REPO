# 16 · Индексы и storage engine 🖼️⭐⭐

> 🎯 **Проект:** шагни от in-memory к настоящей БД — храни данные на диске с индексами, чтобы
> искать быстро в данных больше памяти. Учит B-tree/LSM, дисковому I-O, индексам.

> 🛠️ Применяешь: [Алгоритмы/деревья](../../Algorithms/03-algorithms/16-trees.md), файлы/mmap, предыдущие модули.

---

## 📖 От «всё в памяти» к «данные на диске»

```
   in-memory хранилище упирается в RAM. настоящая БД хранит данные на ДИСКЕ (могут быть больше RAM)
   и быстро ищет через ИНДЕКСЫ. ключевая разница — диск медленный (модуль 15), поэтому структуры
   данных оптимизированы под МИНИМУМ дисковых обращений.

   STORAGE ENGINE — компонент БД, отвечающий за то, КАК данные лежат на диске и как их искать.
   две доминирующие архитектуры: B-tree и LSM-tree.
```

🖼️
```
   in-memory:  hash map в RAM (O(1), но всё в памяти).
   на диске:   индекс (B-tree/LSM) → находит, ГДЕ на диске лежит значение → читает блок.
   цель индекса: найти данные за МИНИМУМ чтений с диска (диск ~в миллион раз медленнее RAM!).
```

---

## ⭐ B-tree — индекс классических БД

```
   B-TREE (B+ tree) — сбалансированное дерево с ШИРОКИМИ узлами (много ключей в узле). почему широкие:
   узел = дисковый блок (~4КБ); читаешь блок целиком → клади в него много ключей → меньше уровней →
   меньше дисковых чтений. поиск/вставка — O(log n), но с очень малым основанием (мало обращений к диску).

   ✅ отличен для ЧТЕНИЯ и диапазонных запросов (ключи упорядочены).
   используется: PostgreSQL, MySQL (InnoDB), SQLite.
```

🖼️
```
   B-tree (широкие узлы под дисковый блок):
            [ 10 | 20 | 30 ]              ← один узел = один блок (~4КБ, десятки-сотни ключей)
           /     |    |     \
        [..]   [..]  [..]   [..]          ← мало уровней → 3-4 чтения с диска даже для млрд записей
   ключ → спуск по дереву → лист с указателем на данные.
```

## ⭐⭐ LSM-tree — индекс для записи (Redis/Cassandra/RocksDB)

```
   LSM-TREE (Log-Structured Merge) — оптимизирован под ЗАПИСЬ:
   • записи идут в память (memtable, сортированная структура) + WAL (надёжность, модуль 15).
   • memtable заполнилась → СБРОС на диск как отсортированный файл (SSTable), неизменяемый.
   • со временем — много SSTable; периодически СЛИВАЮТСЯ (compaction) в большие, удаляя устаревшее.
   • чтение: ищем в memtable → в SSTable от новых к старым (+ Bloom-фильтры, чтобы не читать зря).

   ✅ быстрая ЗАПИСЬ (последовательная на диск — диск любит последовательное!).
   ❌ чтение сложнее (несколько мест), compaction — фоновая работа.
   используется: LevelDB, RocksDB, Cassandra, ScyllaDB.
```

💡 ⭐⭐ Главный урок: **B-tree vs LSM — это trade-off чтение vs запись**, продиктованный физикой
диска (последовательный доступ намного быстрее случайного). B-tree — читай-оптимизирован, LSM —
пиши-оптимизирован. Выбор движка под нагрузку — фундаментальное решение БД. Реализовав упрощённую
версию, ты понимаешь, как устроены настоящие БД изнутри.

## 📖 Milestones (упрощённо)

```
   выбери ОДИН движок (для учёбы — оба сложны; начни проще):
   ВАРИАНТ A (проще, LSM-подобный):
   1. memtable (сортированная map в памяти) + WAL.
   2. сброс на диск как отсортированный файл (SSTable) при заполнении.
   3. чтение: memtable → SSTable'ы (от новых к старым).
   4. (⭐) compaction: слить SSTable, убрать устаревшее/удалённое.
   5. (⭐) Bloom-фильтр: быстро отсекать «точно нет в этом SSTable».

   ВАРИАНТ B (B-tree):
   1. узлы — блоки на диске; поиск/вставка с балансировкой.
   2. кэш страниц (часто читаемые узлы в памяти).
   готово: данные на диске, поиск через индекс за мало чтений, работает на данных больше RAM.
```

> (диск в миллионы раз медленнее → минимизируй обращения).

## ⚠️ Ловушки

- ❌ Игнорировать, что диск медленный → дизайн под минимум обращений (узел = блок).
- ❌ Случайные записи на диск (медленно) вместо последовательных (LSM любит последовательное).
- ❌ B-tree с узкими узлами (как в памяти) → много уровней → много чтений.
- ❌ LSM без compaction → бесконечно растут SSTable, чтение деградирует.
- ❌ Браться за полноценный B-tree/LSM сразу — упрости (одна структура, happy path).
