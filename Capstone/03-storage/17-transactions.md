# 17 · Транзакции и конкурентный доступ 🖼️⭐

> 🎯 **Проект:** добавь хранилищу транзакции и корректный параллельный доступ — учит ACID,
> блокировкам, изоляции. Венец проекта «БД».

> 🛠️ Применяешь: [конкурентность из Ур.2](../02-server/11-concurrency.md), [ОС/синхронизация](../../OS/03-sync/15-deadlock.md).

---

## 📖 Проблема: много клиентов, одни данные

```
   к хранилищу обращаются МНОГО клиентов одновременно (через твой сервер). без защиты:
   • гонки: два клиента меняют один ключ → потеря обновления / порча.
   • частичные операции: перевод денег (списать у A, добавить B) прервался на середине → деньги пропали.
   ТРАНЗАКЦИИ решают это: группа операций выполняется как ЕДИНОЕ ЦЕЛОЕ, корректно при параллелизме.
```

---

## ⭐ ACID — гарантии транзакций

```
   A — ATOMICITY (атомарность): транзакция выполняется ПОЛНОСТЬЮ или НИКАК. сбой на середине →
       откат (rollback), как будто ничего не было. (перевод денег: оба шага или ни одного.)
   C — CONSISTENCY (согласованность): транзакция переводит БД из одного валидного состояния в другое
       (инварианты соблюдены).
   I — ISOLATION (изоляция): параллельные транзакции не видят «грязных» промежуточных состояний
       друг друга (как будто выполнялись по очереди).
   D — DURABILITY (долговечность): зафиксированная (commit) транзакция переживёт сбой (это твой WAL+fsync, модуль 15!).
```

💡 ⭐ ACID — «контракт надёжности» БД. Atomicity ты обеспечиваешь через журнал (записать намерение,
применить, при сбое откатить/доиграть). Durability — это WAL+fsync (модуль 15). Isolation —
блокировки/версионирование (ниже). Реализовав хотя бы частично, ты понимаешь, что значит «настоящая БД».

## ⭐⭐ Изоляция: блокировки и MVCC

```
   как обеспечить ИЗОЛЯЦИЮ параллельных транзакций:

   1. БЛОКИРОВКИ (locking):
      • транзакция блокирует данные, которые читает/пишет; другие ждут.
      • read-write lock: много читателей ИЛИ один писатель.
      ❌ снижает параллелизм; ⚠️ DEADLOCK (две транзакции ждут блокировки друг друга).

   2. MVCC (Multi-Version Concurrency Control) — как в PostgreSQL:
      • хранить НЕСКОЛЬКО версий данных; читатели видят согласованный СНИМОК, не блокируя писателей.
      • читатели не ждут писателей, писатели не ждут читателей → высокий параллелизм.
      ✅ масштабируемо; ❌ сложнее (версии, сборка мусора старых версий).

   УРОВНИ ИЗОЛЯЦИИ — trade-off строгость ↔ параллелизм: от Read Uncommitted (быстро, грязно) до
   Serializable (строго, как по очереди, дороже). выбираешь под нужды.
```

💡 ⭐⭐ Главный урок: **изоляция — это trade-off корректность ↔ параллелизм**. Строгая изоляция
(Serializable) безопасна, но сериализует; слабая — быстра, но допускает аномалии. Блокировки проще,
MVCC масштабируемее. Это глубокая тема настоящих БД; реализовав даже простую блокировочную версию,
ты понимаешь, почему БД так устроены и что значат «уровни изоляции».

## 📖 Milestones (упрощённо)

```
   MVP: MULTI/EXEC — группа команд выполняется атомарно (как Redis-транзакции).
   1. потокобезопасный доступ к хранилищу (mutex/шардирование — из Ур.2 модуль 11).
   2. ТРАНЗАКЦИЯ: BEGIN → набор операций → COMMIT (применить все) / ROLLBACK (откатить).
      простейшая атомарность: выполнять под блокировкой; при ошибке — откатить по журналу изменений.
   3. ИЗОЛЯЦИЯ: начни с грубой (глобальная блокировка на транзакцию) → корректно, но медленно.
   4. (⭐) гранулярные блокировки (на ключ/шард) — больше параллелизма; следи за deadlock.
   5. (⭐) простой MVCC: версии значений, читатели видят снимок.
   готово: параллельные клиенты не портят данные; транзакции атомарны; понятны уровни изоляции.
```

## ⚠️ Ловушки

- ❌ Игнорировать гонки при параллельном доступе → порча данных.
- ❌ Транзакция без атомарности → частичные изменения при сбое.
- ❌ Deadlock при гранулярных блокировках (единый порядок захвата!).
- ❌ Слишком грубая блокировка (всё под одним локом) → нет параллелизма.
- ❌ Забыть, что durability — это fsync (commit должен пережить краш).
- ❌ Браться за полный MVCC сразу — начни с блокировок.

## ✅ Задачи

1. **Потокобезопасность.** Защити хранилище от гонок при параллельном доступе. Проверь под TSan/нагрузкой.
2. **Транзакции.** MULTI/EXEC: группа операций атомарна (всё или ничего). Тест: ошибка в середине → откат.
3. ⭐ **Изоляция.** Покажи проблему без изоляции (грязное чтение/потеря обновления) и реши блокировками.
4. ⭐ **Гранулярные блокировки.** Блокируй по ключу, а не глобально. Избегай deadlock. Замерь параллелизм.
5. ⭐⭐ **Простой MVCC.** Версии значений; читатель видит согласованный снимок без блокировки писателя.
