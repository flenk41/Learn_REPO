# 13 · Key-value in-memory (Redis-подобное) 🖼️⭐⭐

> 🎯 **Проект:** построй in-memory key-value хранилище (как упрощённый Redis) — учит структурам
> данных на масштабе, проектированию хранилища, работе под нагрузкой.

> 🛠️ Применяешь: свою [hash map](../01-data-structures/05-hash-map.md), [свой сервер](../02-server/12-http-server.md)
> (для сетевого доступа), [⚙️ память/локальность](../../ComputerScience/01-hardware/08-cache.md).

---

## 📖 Что строим

```
   KEY-VALUE ХРАНИЛИЩЕ — хранит пары ключ→значение в памяти, доступ за O(1). сердце — хеш-таблица
   (твоя из Ур.1!). операции: SET key value, GET key, DEL key, EXISTS key.
   расширения Redis: разные ТИПЫ значений (строки, списки, множества, хеши), TTL (истечение).

   это «база данных в памяти» — быстрая, простая, и отличный проект, чтобы соединить структуры
   данных + сервер + персистентность.
```

🖼️
```
   клиент ──SET name "Аня"──► [хранилище: hash map { name → "Аня", age → "25", ... }]
   клиент ──GET name───────► "Аня"
   в основе — твоя хеш-таблица; обёрнутая в команды и (далее) сетевой доступ + персистентность.
```

---

## ⭐ Milestones

```
   MVP: in-memory store с SET/GET/DEL через локальный интерфейс (без сети).
   1. ЯДРО ХРАНИЛИЩА: хеш-таблица string→value. SET/GET/DEL/EXISTS.
   2. ТИПЫ ЗНАЧЕНИЙ: начни со строк. потом добавь список (LPUSH/LRANGE), множество, хеш, счётчики (INCR).
   3. TTL / ИСТЕЧЕНИЕ: ключ с временем жизни; удаление по истечении (ленивое при доступе + периодическое).
   4. СЕТЕВОЙ ДОСТУП: оберни в свой TCP-сервер (Ур.2) — команды по сети (модуль 14 — протокол).
   5. КОНКУРЕНТНЫЙ ДОСТУП: много клиентов → синхронизация хранилища (модуль 17).
   6. ПЕРСИСТЕНТНОСТЬ: сохранять на диск, чтобы пережить перезапуск (модуль 15).
   готово: сетевое key-value хранилище с несколькими типами, TTL, доступное по протоколу, переживает рестарт.
```

## ⭐⭐ Ключевые решения и уроки

```
   🗃️ ВЫБОР СТРУКТУР ДАННЫХ под операции:
      • строки/общее → хеш-таблица (O(1) доступ).
      • списки → связный список или дек.
      • множества → хеш-множество.
      • отсортированные множества (как Redis ZSET) → дерево/skip list (продвинутое).
      каждый тип = своя структура данных под свои операции. твои из Ур.1 пригодятся!

   ⏱️ TTL И ИСТЕЧЕНИЕ — два подхода (Redis использует оба):
      • ЛЕНИВОЕ: при GET проверяй, не истёк ли → удали. дёшево, но истёкшие занимают память.
      • АКТИВНОЕ: периодически сканируй и удаляй истёкшие. освобождает память.

   🧠 ПАМЯТЬ: in-memory = быстро, но ограничено RAM. думай о расходе памяти (компактность значений,
      эвикция при переполнении — как LRU-кэш). (связь: [⚙️ память](../../ComputerScience/03-execution/16-process-layout.md))

   🌐 РАЗДЕЛЕНИЕ: ядро (структуры данных) отдели от транспорта (сервер) и персистентности — чистая
      архитектура ([слои](../../OOP/04-patterns/23-architecture.md)), легче тестировать и менять.
```

💡 ⭐⭐ Главный урок: **разные операции требуют разных структур данных** — Redis быстр, потому что
под каждый тип данных подобрана оптимальная структура. И ты применяешь СВОИ структуры из Ур.1 в
реальном проекте. Плюс архитектура: ядро / сеть / диск — отдельные слои.

## 📖 Как проверить

```
   ✅ SET/GET/DEL/EXISTS работают; перезапись существующего ключа.
   ✅ типы: список (push/range), множество, счётчик (INCR) — корректны.
   ✅ TTL: ключ с истечением исчезает после времени; без TTL — живёт.
   ✅ по сети (после модуля 14): клиент шлёт команды, получает ответы.
   ✅ нагрузка: тысячи SET/GET — производительность, корректность.
   ✅ память: нет утечек; истёкшие ключи освобождаются.
```

---

## 🌟 Расширения

```
   🥚 INCR/DECR, APPEND, TTL-команды (EXPIRE/PERSIST).
   🐣 LRU-эвикция при достижении лимита памяти (как кэш).
   🦅 отсортированные множества (skip list) — как Redis ZSET.
   🚀 pub/sub (публикация-подписка) поверх своего сервера.
   🚀 сравни производительность (GET/SET) с настоящим Redis (redis-benchmark-подобно).
```

## ⚠️ Ловушки

- ❌ Одна структура под все типы (под списки/множества нужны свои).
- ❌ Не чистить истёкшие ключи → утечка памяти.
- ❌ Смешать ядро/сеть/диск в один клубок — не протестировать, не менять.
- ❌ Игнорировать лимит памяти (in-memory упрётся в RAM).
- ❌ Не синхронизировать при конкурентном доступе (гонки — модуль 17).
