## 📖 Зачем конкурентность серверу

```
   event loop (модуль 10) масштабирует ОДИН поток, но использует ОДНО ядро. современные CPU —
   многоядерные. чтобы задействовать все ядра и/или делать «тяжёлую» работу не блокируя приём:
   • ПОТОКИ — параллельная обработка на разных ядрах.
   • ПУЛ ПОТОКОВ — переиспользуемый набор потоков (не создавать на каждого).
   • КОМБИНАЦИЯ с event loop — несколько event loop'ов (по одному на ядро).
```

🖼️
```
   1 event loop, 1 ядро:     [ядро0: loop держит 10к соединений]  ← но 7 ядер простаивают.
   N event loop, N ядер:     [ядро0: loop][ядро1: loop]...[ядроN] ← полная утилизация.
   пул потоков для тяжёлого: event loop принимает → тяжёлую задачу отдаёт в пул → не блокирует приём.
```

💡 ⭐ Архитектурный выбор: thread-per-connection (просто, но не масштабно), пул потоков (баланс),
event-loop-per-core (масштаб + многоядерность, как nginx). Понимание этих моделей — senior-навык.

## ⭐ Модели и milestones

```
   === модель 1: пул потоков (проще) ===
   1. создай ФИКСИРОВАННЫЙ набор потоков (= числу ядер обычно).
   2. ОЧЕРЕДЬ ЗАДАЧ (потокобезопасная): принятые соединения/запросы кладутся в очередь.
   3. потоки берут из очереди и обрабатывают. синхронизация очереди (mutex + condition_variable).

   === модель 2: event-loop-per-core (мощнее) ===
   4. запусти по event loop на каждом ядре; распределяй принятые соединения между ними.

   === синхронизация общего состояния ===
   5. если потоки делят данные (кэш, счётчики, key-value хранилище — Ур.3):
      • MUTEX — защитить критическую секцию (но это сериализация — узкое место).
      • ATOMIC — для простых счётчиков без блокировки.
      • READ-WRITE lock — много читателей / один писатель.
      • шардирование — разбить данные, у каждого потока своя часть (меньше контеншена).
   готово: сервер использует несколько ядер, корректен при параллельном доступе, нет гонок.
```

## ⭐⭐ Главная опасность: гонки данных

```
   🏁 ГОНКА ДАННЫХ (data race) — два потока обращаются к одним данным, хотя бы один пишет, без
      синхронизации → НЕОПРЕДЕЛЁННОЕ поведение (порча данных, крэши, «иногда работает»).
      пример: два потока делают counter++ без защиты → потеряются инкременты.

   🔒 СИНХРОНИЗАЦИЯ — защита общих данных:
      • mutex: только один поток в критической секции. но СЕРИАЛИЗУЕТ → меньше параллелизма.
      • atomic: для простых операций (счётчики) без полноценной блокировки.
      ⚠️ DEADLOCK — потоки ждут блокировки друг друга вечно. избегай: единый порядок захвата, меньше блокировок.

   ⚖️ TRADE-OFF: больше синхронизации = меньше гонок, но и меньше параллелизма (узкие места на
      блокировках). цель — минимизировать общее изменяемое состояние (шардирование, неизменяемость,
      message passing — как в [Rust/каналы](../../Rust/04-senior/20-concurrency.md)).
```

💡 ⭐⭐ Конкурентность — мощно, но опасно: гонки дают «хайзенбаги» (то есть, то нет). Главный
принцип: **минимизируй общее изменяемое состояние**. Что не делится между потоками — не требует
синхронизации. Шардируй данные, используй неизменяемость, передавай сообщения вместо общей памяти.
Rust здесь учит лучше всех — он не даёт скомпилировать гонку. (False sharing из [⚙️ кэша](../../ComputerScience/04-performance/23-cache-friendly.md) — тоже здесь.)

## 📖 Как проверить

```
   ✅ нагрузка с многих клиентов → задействованы несколько ядер (htop/top).
   ✅ ПРАВИЛЬНОСТЬ под параллелизмом: общий счётчик/хранилище — итог корректен после N параллельных операций.
   ✅ ThreadSanitizer (TSan) — детектор гонок: прогони под ним, исправь все гонки.
   ✅ нет deadlock: сервер не «зависает» под нагрузкой.
   ✅ масштаб: производительность растёт с числом ядер (хотя бы частично).
```

---

## 🌟 Расширения

```
   🥚 graceful shutdown: дождаться завершения активных запросов.
   🐣 work stealing между потоками пула (балансировка).
   🦅 lock-free структуры (очередь) — продвинутое.
   🚀 сравни модели (поток-на-клиента / пул / event-loop-per-core) по throughput под нагрузкой.
```

---

## ⚠️ Ловушки

- ❌ Гонки данных (общее изменяемое состояние без синхронизации) → порча/крэши.
- ❌ Deadlock (взаимная блокировка) → зависание.
- ❌ Слишком грубая блокировка (один mutex на всё) → нет параллелизма.
- ❌ Создавать поток на каждого клиента (дорого, не масштабно).
- ❌ Блокирующие операции под глобальным локом.
- ❌ Не проверять под TSan (гонки незаметны без него).
