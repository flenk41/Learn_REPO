# 12 · HTTP-сервер 🖼️⭐⭐

> 🎯 **Проект:** надстрой над своим TCP-сервером разбор HTTP — получи настоящий веб-сервер,
> отвечающий браузеру. Финал уровня «сервер».

> 🛠️ Применяешь: предыдущие модули (TCP/event loop/конкурентность) + [Сети/HTTP](../../Network/03-application/13-http.md).

---

## 📖 HTTP поверх TCP

```
   HTTP — текстовый протокол ПОВЕРХ TCP. твой TCP-сервер уже умеет принимать байты; теперь нужно
   РАЗОБРАТЬ их как HTTP-запрос и составить HTTP-ответ.

   запрос:                          ответ:
   GET /index.html HTTP/1.1         HTTP/1.1 200 OK
   Host: localhost                  Content-Type: text/html
   User-Agent: ...                  Content-Length: 137
   (пустая строка)                  (пустая строка)
   (тело для POST)                  <html>...</html>
```

🖼️
```
   браузер ──TCP──► [твой сервер]
   сервер: прочитать байты → РАЗОБРАТЬ HTTP (метод, путь, заголовки) →
           обработать (роут/файл) → составить HTTP-ОТВЕТ (статус+заголовки+тело) → отправить.
```

💡 ⭐⭐ Это синтез: твой сетевой движок (Ур.2) + парсинг протокола. После этого браузер реально
открывает страницу с твоего сервера — мощное ощущение «я построил веб-сервер».

---

## ⭐ Milestones

```
   MVP: отвечает «200 OK + Hello» на ЛЮБОЙ запрос.
   1. прочитай запрос (до пустой строки \r\n\r\n — конец заголовков). помни: TCP поток → читай в цикле.
   2. отправь фиксированный валидный HTTP-ответ. проверь В БРАУЗЕРЕ (http://localhost:ПОРТ).

   наращивание:
   3. РАЗБОР ЗАПРОСА: первая строка → метод (GET/POST), путь, версия. заголовки → пары ключ:значение.
   4. РОУТИНГ: разные пути → разные ответы (/ → главная, /about → ...). 404 для неизвестных.
   5. СТАТИЧЕСКИЕ ФАЙЛЫ: путь → файл на диске → отдать с правильным Content-Type. ⚠️ защита от path
      traversal (../../etc/passwd)! (связь: [безопасность](../../Security/02-vulnerabilities/11-access-control.md))
   6. КОРРЕКТНЫЕ ЗАГОЛОВКИ: Content-Length (обязателен!), Content-Type, статусы (200/404/400/500).
   7. POST: чтение тела по Content-Length; обработка данных.
   8. KEEP-ALIVE: не закрывать соединение после ответа (несколько запросов по одному соединению).
   готово: браузер открывает страницы, отдаются файлы, роуты работают, корректные статусы.
```

---

## ⭐⭐ Ключевые уроки и подводные камни

```
   📏 ГРАНИЦЫ СООБЩЕНИЙ — где кончается запрос? заголовки кончаются пустой строкой (\r\n\r\n);
      тело — по Content-Length. это решает «TCP — поток» проблему (модуль 09): протокол задаёт границы.

   🧾 ПАРСИНГ — аккуратно: разные клиенты шлют по-разному; обрабатывай кривые запросы (400 Bad Request),
      не падай на них. (мини-парсер — мостик к интерпретатору, Ур.4)

   🔒 БЕЗОПАСНОСТЬ (важно даже в учебном):
      • PATH TRAVERSAL: путь /../../etc/passwd не должен отдать системный файл. нормализуй/проверяй путь.
      • не доверяй заголовкам/телу (валидируй размеры — защита от переполнения/DoS).
      (связь: [AppSec](../../Security/03-defensive-code/16-input-validation.md))

   ⚙️ ПРОИЗВОДИТЕЛЬНОСТЬ: keep-alive (не открывать TCP на каждый запрос), отдавать файлы эффективно,
      работать поверх своего event loop (Ур.2) для многих клиентов.
```

💡 ⭐⭐ Уроки: **протокол задаёт границы сообщений** поверх потока TCP; **парсинг недоверенного ввода**
требует устойчивости и безопасности (path traversal — классика). Это связывает сети, парсинг и
безопасность в одном проекте — очень senior.

## 📖 Как проверить

```
   ✅ БРАУЗЕР: http://localhost:ПОРТ → видишь страницу. отдаются файлы (html/css/картинки).
   ✅ curl -v: посмотри сырой запрос/ответ. статусы, заголовки корректны?
   ✅ роутинг: разные пути → разные ответы; неизвестный → 404.
   ✅ кривой запрос: отправь мусор (nc) → 400, сервер не падает.
   ✅ path traversal: запрос /../секрет → НЕ отдаётся. (защита работает?)
   ✅ нагрузка: ab/wrk (бенчмарк) → сколько запросов/сек держит? (поверх event loop)
```

## 🌟 Расширения

```
   🥚 простой шаблонизатор / динамические страницы.
   🐣 REST-эндпоинты (JSON) — мост к key-value хранилищу (Ур.3): GET/POST /data.

   🚀 HTTPS (TLS поверх) — через библиотеку (не свою крипту! — [безопасность](../../Security/01-recon/08-crypto-basics.md)).
   🚀 бенчмарк vs nginx на статике — насколько близко?
```

## ⚠️ Ловушки

- ❌ Не находить границу запроса (читать «до пустой строки» + тело по Content-Length).
- ❌ Забыть Content-Length в ответе → браузер «висит».
- ❌ Path traversal (отдать системные файлы) — критическая дыра.
- ❌ Падать на кривых запросах вместо 400.
- ❌ Открывать новое TCP на каждый запрос (нет keep-alive) — медленно.
- ❌ Не валидировать размеры тела/заголовков (DoS).

---

## ✅ Задачи

1. **MVP.** Отвечай валидным HTTP на любой запрос. Открой в браузере.
2. **Парсинг + роутинг.** Разбери метод/путь/заголовки. Разные пути → разные ответы, 404.
3. **Статика.** Отдавай файлы с правильным Content-Type. Защити от path traversal.
4. ⭐ **Устойчивость + безопасность.** Кривые запросы → 400, без падений. Валидация размеров.
5. ⭐ **Поверх event loop + бенчмарк.** Запусти на своём event loop, замерь req/sec через ab/wrk.
