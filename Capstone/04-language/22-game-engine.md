# 22 · Игровой движок: цикл и ECS 🖼️⭐

> 🎯 **Проект:** построй основу игрового движка — игровой цикл, рендеринг, ECS-архитектура.
> Учит производительности, архитектуре и реальному применению cache-friendly данных.

> 🛠️ C++ (классика игр) или Rust. Применяешь: [⚙️ производительность/кэш](../../ComputerScience/04-performance/23-cache-friendly.md),

## 📖 Игровой движок: что это и почему учит

```
   ИГРОВОЙ ДВИЖОК — фреймворк, прогоняющий мир игры много раз в секунду: обновить состояние →
   отрисовать → повторить (60+ раз/сек). это жёсткий бюджет времени на кадр (~16мс при 60 FPS) →
   движок — отличный тренажёр ПРОИЗВОДИТЕЛЬНОСТИ и АРХИТЕКТУРЫ.

   зачем строить: синтезирует cache-friendly данные (Ур.⚙️), архитектуру, математику, реальные
   ограничения скорости. и просто увлекательно (видишь результат на экране).
```

---

## ⭐ Игровой цикл

```
   ГЛАВНЫЙ ЦИКЛ движка:
   while (running):
       1. ВВОД — обработать клавиши/мышь.
       2. ОБНОВИТЬ (update) — продвинуть физику/логику на delta time (Δt с прошлого кадра).
       3. ОТРИСОВАТЬ (render) — нарисовать текущее состояние.
       4. (тайминг) удержать стабильный FPS.

   ⏱️ DELTA TIME — обновляй по реальному прошедшему времени (Δt), а не по кадрам → плавность не
      зависит от FPS. фиксированный шаг физики (fixed timestep) — для стабильной симуляции.
```

🖼️
```
   [ввод] → [update(Δt): физика, логика, AI] → [render: нарисовать кадр] → повторить 60×/сек
   бюджет ~16мс на ВСЁ при 60 FPS → каждая миллисекунда на счету → производительность критична.
```

💡 ⭐ Игровой цикл — жёсткий реальный дедлайн (кадр за 16мс). Здесь знание [⚙️ производительности](../../ComputerScience/04-performance/22-cpu-bottlenecks.md)
(кэш, ветвления, профилирование) применяется по-настоящему: тормозящий кадр виден глазом.

## ⭐⭐ ECS — архитектура, дружащая с кэшем

```
   ECS (Entity-Component-System) — популярная архитектура движков, КЭШ-ОПТИМИЗИРОВАННАЯ:
   • ENTITY — просто id (объект игры: игрок, враг, пуля).
   • COMPONENT — данные (Position, Velocity, Health) — БЕЗ логики.
   • SYSTEM — логика, обрабатывающая все entity с нужными компонентами (MovementSystem обновляет
     Position по Velocity у всех).

   ПОЧЕМУ ECS быстр: компоненты хранятся МАССИВАМИ ОДНОГО ТИПА (SoA — структура массивов!).
   система проходит массив Position+Velocity ПОДРЯД → кэш-дружелюбно → быстро.
   ↕ сравни с ООП-подходом (массив разнородных объектов, виртуальные вызовы) — прыжки по памяти, медленнее.
```

💡 ⭐⭐ Главный урок: **ECS — это [data-oriented design](../../ComputerScience/04-performance/23-cache-friendly.md)
на практике**. Данные разложены под кэш (SoA, массивы компонентов), обработка последовательна →
огромная разница в скорости на тысячах объектов. Это прямое применение «расположение данных важнее
числа операций» из ⚙️ CS — и почему движки уходят от классического ООП к ECS.

## 📖 Milestones (упрощённо)

```
   MVP: окно + игровой цикл + движущийся объект (с использованием графической библиотеки — SDL/SFML/raylib,
        свою графику с нуля не пиши, это отдельная огромная тема).
   1. окно и цикл: ввод → update → render с delta time.
   2. простой объект (позиция/скорость), движется, отскакивает от краёв.
   3. ECS: entity-id, компоненты (массивы), системы (movement, render).
   4. много объектов (тысячи) → проверь, что ECS держит FPS (профилируй!).
   5. (⭐) простая физика (коллизии), сцены, ресурсы.
   готово: окно с циклом, ECS-архитектура, тысячи объектов на стабильном FPS.
```

> 🧭 Графику (OpenGL/Vulkan) и физику с нуля — это отдельные огромные темы; для капстоуна используй
> библиотеку для окна/рендера, а строй САМ архитектуру (цикл, ECS) — там и есть обучение.

## ⚠️ Ловушки

- ❌ Обновлять по кадрам, а не по delta time → скорость зависит от FPS.
- ❌ ООП-архитектура с виртуальными вызовами в горячем цикле на тысячах объектов (медленно).
- ❌ Игнорировать кэш (разрозненные объекты) вместо ECS/SoA.
- ❌ Писать графику/физику с нуля для первого движка (используй библиотеку).
- ❌ Не профилировать (кадр-бюджет жёсткий — найди, что тормозит).
- ❌ Преждевременная сложность — начни с цикла + одного объекта.
