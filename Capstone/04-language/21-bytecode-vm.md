# 21 · Байткод-VM и компиляция 🖼️⭐

> 🎯 **Проект:** ускорь свой язык — скомпилируй AST в байткод и исполняй на стековой виртуальной
> машине. Учит, как работают Python/JVM/реальные языки.

> 🛠️ Применяешь: [⚙️ машинный код/стек](../../ComputerScience/03-execution/17-call-stack.md),
> свой стек (Ур.1), предыдущие модули. (Продвинутое — опционально, но очень показательно.)

## 📖 Зачем байткод

```
   обход AST (модуль 20) прост, но медленен (рекурсия по дереву каждый раз). быстрее — СКОМПИЛИРОВАТЬ
   AST в БАЙТКОД (плоский список простых инструкций) ОДИН раз, и исполнять его на ВИРТУАЛЬНОЙ МАШИНЕ.

   так работают Python (.pyc), Java (JVM bytecode), C#, Lua, WebAssembly. это «золотая середина»
   между интерпретацией (просто, медленно) и компиляцией в машинный код (быстро, сложно/непереносимо).
```

🖼️
```
   AST  ──КОМПИЛЯТОР──►  БАЙТКОД (плоские инструкции)  ──VM──►  результат
   2+3*4:                PUSH 2; PUSH 3; PUSH 4; MUL; ADD       исполняется на стековой VM
   компилируем РАЗ, исполняем БЫСТРО (плоский цикл вместо обхода дерева).
```

---

## ⭐ Стековая виртуальная машина

```
   СТЕКОВАЯ VM — исполняет байткод, используя СТЕК для операндов (твой стек из Ур.1!):
   PUSH n   — положить число на стек.
   ADD      — снять два, сложить, положить результат.
   MUL/SUB/DIV — аналогично.
   LOAD x / STORE x — переменные (из окружения).
   JMP / JMP_IF — переходы (для if/while — как условные переходы в [⚙️ CS](../../ComputerScience/01-hardware/05-machine-code.md)!).
   CALL / RET — функции.

   "2 + 3 * 4" → PUSH 2, PUSH 3, PUSH 4, MUL, ADD:
   стек: [2] → [2,3] → [2,3,4] → MUL → [2,12] → ADD → [14]. результат 14 на вершине.
```

💡 ⭐ Стековая VM красива простотой: каждая инструкция работает со стеком. Это в точности механика
вычисления выражений + та же идея, что [стек вызовов и инструкции CPU из ⚙️ CS](../../ComputerScience/03-execution/17-call-stack.md).
Ты буквально строишь маленький «процессор» для своего языка.

## ⭐⭐ Milestones

```
   1. ОПРЕДЕЛИ набор байткод-инструкций (опкоды): PUSH, ADD/SUB/MUL/DIV, LOAD/STORE, JMP/JMP_IF, CALL/RET, PRINT.
   2. КОМПИЛЯТОР: обход AST → генерация байткода (вместо вычисления — эмиссия инструкций).
      число → PUSH; бинарная операция → код left, код right, опкод операции (как в примере).
   3. VM: цикл по байткоду (instruction pointer), switch по опкоду, операции над стеком.
      это твой fetch-decode-execute (привет, [⚙️ цикл CPU](../../ComputerScience/01-hardware/03-cpu-cycle.md))!
   4. ПЕРЕМЕННЫЕ: LOAD/STORE в окружение/слоты.
   5. УПРАВЛЕНИЕ: if/while → JMP/JMP_IF с метками (вычисли смещения переходов).
   6. ФУНКЦИИ: CALL/RET, кадры на стеке вызовов (как [⚙️ calling convention](../../ComputerScience/03-execution/18-calling-convention.md)).
   7. БЕНЧМАРК: сравни скорость байткод-VM и tree-walking (модуль 20) на цикле/рекурсии. насколько быстрее?
   готово: язык компилируется в байткод и исполняется на VM, заметно быстрее обхода AST.
```

## 📖 Спектр исполнения

```
   как языки исполняются — спектр от простого/медленного к сложному/быстрому:
   1. ОБХОД AST (модуль 20) — просто, медленно.
   2. БАЙТКОД + VM (этот) — баланс; переносимо (Python, JVM, Lua, WASM).
   3. JIT (just-in-time) — компилировать горячий байткод в машинный код на лету (V8, JVM HotSpot, PyPy).
   4. AOT-компиляция в машинный код — быстро, но непереносимо ([⚙️ CS](../../ComputerScience/02-toolchain/09-build-stages.md): C/C++/Rust/Go).
   твой путь: AST → байткод. дальше (по желанию) — JIT/нативная компиляция = вершина.
```

## ⚠️ Ловушки

- ❌ Неправильный порядок эмиссии (для стековой VM: операнды ДО операции).
- ❌ Переходы (JMP) без корректных смещений/меток для if/while.
- ❌ Не управлять стеком аккуратно (лишнее/недостающее на стеке → крэш VM).
- ❌ Функции без кадров на стеке вызовов → порча при рекурсии.
- ❌ Браться за JIT/нативную компиляцию сразу — байткод+VM достаточно показателен.
- ❌ Не замерить выигрыш над tree-walking (смысл — скорость).
