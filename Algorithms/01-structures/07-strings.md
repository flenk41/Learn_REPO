> ловушки работы с ними (особенно неизменяемость).

## ⭐ Строка — это массив символов

**Строка** — последовательность символов. По сути — **массив** (модуль 03), где элементы —
символы. Поэтому у строк похожий профиль: быстрый доступ по индексу, и важный нюанс —
**неизменяемость** во многих языках.

```
   "hello"
    индекс: 0   1   2   3   4
          [ h ][ e ][ l ][ l ][ o ]   ← как массив символов
```

💡 ⭐ Доступ к символу по индексу — O(1) (как массив). Но строки часто **неизменяемы** (immutable)
— в Python, Java, JS. Это значит: «изменить» строку = создать **новую**. Связь с
[mutable/immutable в Python](../../Python/02-memory/10-mutable-immutable.md) и
[String vs &str в Rust](../../Rust/02-memory/11-slices-strings.md).

## ⭐⭐ Главная ловушка: конкатенация в цикле

```
   ❌ result = ""
      for word in words:
          result += word      ← каждый += создаёт НОВУЮ строку (копирует всё!)
      → O(n²) суммарно (на больших данных катастрофа)

   ✅ result = "".join(words)  ← один проход, O(n)
   ✅ или собирать в список, потом join
```

💡 ⭐⭐ Это классическая ошибка: строки неизменяемы, поэтому `+=` в цикле копирует всю накопленную
строку каждый раз → квадратичная сложность. Решение — **накапливать в список и `join`** (или
использовать «строитель строк»/StringBuilder в Java). Помнишь задачу «`join` vs `+=`» из
[Python senior](../../Python/04-senior/PROJECT.md)? Вот её теория.

## 📖 Частые операции и их сложность

```
   доступ s[i]                 → O(1)
   длина len(s)                → O(1) (обычно хранится)
   срез s[a:b]                 → O(k), где k — длина среза (копирует)
   поиск подстроки s.find(t)   → O(n·m) наивно (есть быстрые алгоритмы — KMP и др.)
   конкатенация s + t          → O(n+m) (создаёт новую)
   сравнение s == t            → O(n)
```

💡 Многие строковые задачи сводятся к: пройти по символам, использовать хеш-таблицу (частоты,
анаграммы), два указателя (палиндром, модуль 22). Строка + хеш-таблица — мощная комбинация.

## ⭐ Типичные приёмы со строками

```
   📊 частота символов     → хеш-таблица (анаграммы, уникальность)
   ↔️ два указателя         → палиндром, реверс (модуль 22)
   🪟 скользящее окно       → подстроки, последовательности (модуль 22)
   🔤 сортировка символов   → ключ для группировки анаграмм
```

💡 ⭐ Строки — частая тема собеседований. Большинство задач решаются комбинацией: проход +
хеш-таблица / два указателя / окно. Знание этих приёмов (уровень 4) закрывает большинство
строковых задач.

## 📖 Юникод и символы ≠ байты

```
   ⚠️ символ ≠ байт: в UTF-8 один символ может занимать несколько байт (кириллица, эмодзи)
   len() символов ≠ len() байт
```

💡 Помнишь «байты vs символы» из [Rust](../../Rust/02-memory/11-slices-strings.md) и
[Интеграции](../../Interop/02-boundary/10-strings.md)? При работе со строками на разных языках/
данных учитывай кодировку: индексация по «символам» и по «байтам» может различаться.

## ⚠️ Ловушки

- ❌ Конкатенация `+=` в цикле — O(n²); используй `join`/builder.
- ❌ Считать символ равным байту (проблемы с юникодом).
- ❌ Делать срезы в цикле без нужды — каждый срез копирует (O(k)).
- ❌ Забывать про неизменяемость — «изменение» строки создаёт новую.
