# 11 · Как анализировать свой код 🖼️⭐

> 🎯 **Цель блока:** выработать практический навык — быстро определять Big-O любого куска кода,
> включая хитрые случаи.

---

## ⭐ Пошаговый метод анализа

```
   1. найди ОСНОВНУЮ работу, зависящую от n (циклы, рекурсия)
   2. посчитай, сколько РАЗ она выполняется в зависимости от n
   3. учти СКРЫТУЮ сложность вызываемых операций/функций
   4. перемножь вложенное, сложи последовательное
   5. упрости (отбрось константы, оставь старший член)
```

💡 ⭐ Это рецепт. Большинство ошибок — на шаге 3 (скрытая сложность) и 4 (путают умножение и
сложение). Разберём типичные случаи.

---

## ⭐ Циклы: умножение vs сложение

```python
# ПОСЛЕДОВАТЕЛЬНО → складываем → O(n)
for x in arr: ...      # O(n)
for y in arr: ...      # O(n)   → O(n) + O(n) = O(n)

# ВЛОЖЕННО → умножаем → O(n²)
for x in arr:          # n
    for y in arr:      #   × n
        ...            # = O(n²)

# разные размеры → O(n·m), НЕ O(n²)
for x in a:            # n
    for y in b:        #   × m
        ...            # = O(n·m)
```

💡 ⭐ Вложенные циклы по одному размеру → O(n²). По разным → O(n·m). Последовательные блоки →
складываются, берём старший. Цикл, где счётчик **удваивается** (`i *= 2`) → O(log n).

---

## ⭐ Скрытая сложность — главная ловушка

```python
# выглядит как O(n), но на деле O(n²)!
for x in arr:               # n итераций
    if x in seen_list:      # ← `in list` это O(n)!  → n × n = O(n²)
        ...

# исправление: используем set (поиск O(1))
seen = set()
for x in arr:               # n
    if x in seen:           # O(1) → O(n) total ✅
        ...
```

💡 ⭐⭐ Это самая частая невидимая O(n²): операция внутри цикла сама по себе O(n). Запомни скрытые
стоимости:

```
   x in list / list.index()    → O(n)   (а x in set → O(1)!)
   list.insert(0, x) / pop(0)   → O(n)   (сдвиг)
   срез list[a:b] / строки      → O(k)
   sorted(...) / .sort()        → O(n log n)
   копирование структуры        → O(n)
```

⚠️ Эти операции **внутри цикла** умножаются на число итераций. Замена `list` на `set`/`dict` —
главный приём убрать скрытую O(n²).

---

## 📖 Анализ рекурсии (кратко)

```
   считаем: сколько вызовов × работа на вызов

   factorial(n): n вызовов × O(1) = O(n) время, O(n) память (стек)
   наивный fib(n): ~2ⁿ вызовов (двойная рекурсия без кэша) → O(2ⁿ) 😱
   с мемоизацией fib(n): n уникальных → O(n) (модуль 19)
```

💡 Рекурсия с **двумя** рекурсивными вызовами без сохранения результатов часто даёт
**экспоненту** O(2ⁿ) — пересчитывает одно и то же много раз. Это лечится мемоизацией/динамическим
программированием (уровень 4). Анализ рекурсии — навык, который оттачивается с практикой.

---

## ⚠️ Ловушки

- ❌ Не замечать скрытую O(n) внутри цикла (`in list`, `pop(0)`, срезы) → невидимая O(n²).
- ❌ Путать вложенные (×) и последовательные (+) циклы.
- ❌ Считать рекурсию дешёвой (бывает O(2ⁿ)).
- ❌ Анализировать только время, забывая память.
