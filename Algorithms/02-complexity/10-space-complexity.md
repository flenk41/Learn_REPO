## ⭐⭐ Память — вторая ось эффективности

**Пространственная сложность** = сколько **дополнительной памяти** растёт с размером входа n.
Алгоритм может быть быстрым, но «прожорливым» по памяти — и наоборот.

```
   ⏱️ временная сложность  — сколько ОПЕРАЦИЙ (модуль 09)
   💾 пространственная     — сколько ПАМЯТИ (этот модуль)
```

💡 ⭐⭐ Вот где трек встречается со сквозной темой курса: память — не бесплатна. Каждая структура
данных, которую ты создаёшь, занимает место в той самой памяти, что разбирали в
[C](../../C/02-memory/11-dynamic-memory.md), [ОС](../../OS/02-memory/08-why-virtual-memory.md).
Анализ памяти алгоритма — это анализ того, сколько RAM он «съест».

---

## ⭐⭐ Как считать память

Считаем **дополнительную** память (помимо самого входа), растущую с n:

```python
# O(1) памяти — несколько переменных, не зависит от n
def sum_all(arr):
    s = 0                    # одна переменная
    for x in arr: s += x     # цикл памяти не добавляет
    return s                 # O(1) доп. памяти

# O(n) памяти — создаём новую структуру размера n
def doubled(arr):
    result = []              # новый список
    for x in arr:
        result.append(x * 2) # вырастет до n элементов
    return result            # O(n) доп. памяти

# O(n) памяти от рекурсии — n кадров на стеке вызовов!
def factorial(n):
    if n <= 1: return 1
    return n * factorial(n - 1)   # глубина рекурсии n → n кадров стека
```

💡 ⭐⭐ Источники памяти: **новые структуры** (списки, словари, копии) и **стек вызовов** при
рекурсии (каждый вызов — кадр в памяти, модуль 15). Переменные-счётчики — O(1). Если создаёшь
копию входа или структуру размера n — это O(n) памяти.

⚠️ **Рекурсия незаметно ест память**: глубина n = n кадров на стеке. Глубокая рекурсия может
переполнить стек (stack overflow!) — связь со [стеком вызовов](../../OS/01-processes/03-process.md).

## ⭐ Компромисс время ↔ память

```
   часто можно УСКОРИТЬ ценой ПАМЯТИ (и наоборот):

   поиск пары с суммой:
     O(n²) время, O(1) память — двойной цикл (медленно, но без доп. памяти)
     O(n)  время, O(n) память — хеш-множество (быстро, но память на хеш)

   мемоизация (модуль 19): хранить результаты → быстрее, но память O(n)
```

💡 ⭐ Это фундаментальный компромисс: **время vs память**. Нет «лучшего» ответа — зависит от
ограничений. Памяти много, скорость критична → трать память на ускорение (хеши, кэш). Память
ограничена (embedded, огромные данные) → экономь, жертвуя скоростью. Senior выбирает осознанно.

## 📖 In-place алгоритмы — O(1) памяти

```
   IN-PLACE — алгоритм работает в исходной структуре, не создавая копию
   → O(1) дополнительной памяти

   пример: разворот массива перестановкой элементов на месте (O(1) памяти)
   против: создать новый перевёрнутый массив (O(n) памяти)
```

💡 In-place — ценно, когда памяти мало или данные огромные. Но иногда усложняет код или невозможно
(нужна копия). Указывая сложность, всегда уточняй: «O(n) время, O(1) **дополнительной** памяти».

---

## ⚠️ Ловушки

- ❌ Считать только время, игнорируя память (обе оси важны).
- ❌ Забывать про память стека при рекурсии (глубина n = O(n) памяти, риск переполнения).
- ❌ Создавать лишние копии данных в цикле (раздувает память).
- ❌ Не видеть компромисс: «ускорил» хешем — но добавил O(n) памяти.
