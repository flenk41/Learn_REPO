# 🚀 Пет-проект уровня 2 — «Аналитик сложности»

> 🎯 **Цель:** натренировать главный навык трека — определять и улучшать сложность кода — на
> наборе реальных примеров с замерами, подтверждающими теорию.

---

## 📋 Что делаем

Собираем «лабораторию сложности»: берём задачи, анализируем Big-O решений, **подтверждаем
замерами**, и улучшаем класс сложности.

### Часть A. Анализ (теория)

1. Возьми 6–8 фрагментов кода разной сложности (O(1), O(log n), O(n), O(n log n), O(n²),
   и один со **скрытой** O(n²) через `in list`). Для каждого определи время И память.
2. Оформи таблицу: фрагмент → временная сложность → пространственная → пояснение.

### Часть B. Подтверждение замерами (практика)

3. Для 3 фрагментов разной сложности замерь реальное время на растущих n (например, n = 100, 1k,
   10k, 100k). Построй таблицу/график.
4. Проверь, что рост времени **соответствует** теоретической Big-O (O(n) — линейный рост, O(n²) —
   квадратичный, O(log n) — почти не растёт).

### Часть C. Улучшение сложности

5. Возьми 2 задачи с наивным решением O(n²) и улучши до O(n) или O(n log n):
   - например, «есть ли дубликаты» O(n²) → O(n) через set;
   - «пара с суммой target» O(n²) → O(n) через хеш.
   Замерь «до» и «после» на больших данных — увидь разницу вживую.

### Часть D. Компромисс

6. Для одной задачи покажи два решения: экономное по памяти (медленнее) и быстрое (память).
   Сформулируй, в каких условиях какое предпочесть.

### Отчёт

`README.md`: таблица анализа, графики замеров (теория vs практика), примеры улучшения O(n²)→O(n) с
числами, разбор компромисса время/память.

- [ ] Улучшил 2 задачи O(n²)→O(n)/O(n log n) с доказательством
