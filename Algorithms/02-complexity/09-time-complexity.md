# 09 · Временная сложность 🖼️⭐⭐

> 🎯 **Цель блока (ЯДРО трека):** научиться определять временную сложность кода — считать, как
> число операций растёт с размером данных — на конкретных примерах.

---

## ⭐⭐ Считаем операции в зависимости от n

Временная сложность = как растёт **число операций** с размером входа **n**. Разберём по примерам.

```python
# O(1) — константа: не зависит от n
def first(arr):
    return arr[0]            # одна операция, всегда

# O(n) — линейно: один проход по n элементам
def sum_all(arr):
    s = 0
    for x in arr:           # n итераций
        s += x
    return s

# O(n²) — квадрат: вложенный цикл
def all_pairs(arr):
    for i in arr:           # n раз
        for j in arr:       #   × n раз = n²
            print(i, j)
```

💡 ⭐⭐ Главное правило: **один цикл по n → O(n); цикл в цикле по n → O(n²)**. Считай, сколько раз
выполняется самая «внутренняя» работа в зависимости от n. Вложенность циклов **перемножается**,
последовательные блоки — **складываются** (и берём старший).

---

## ⭐⭐ O(log n) — деление пополам

```python
# O(log n) — на каждом шаге ОТБРАСЫВАЕМ половину
def binary_search(arr, target):   # arr отсортирован
    lo, hi = 0, len(arr) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if arr[mid] == target: return mid
        elif arr[mid] < target: lo = mid + 1   # отбросили левую половину
        else: hi = mid - 1                       # отбросили правую половину
    return -1
```

💡 ⭐⭐ **O(log n)** возникает, когда задача на каждом шаге **уменьшается в разы** (обычно вдвое).
Из миллиона элементов бинарный поиск найдёт за ~20 шагов (log₂(1 000 000) ≈ 20)! Логарифм — это
«сколько раз можно поделить n пополам до 1». Очень эффективно. Деревья и бинарный поиск (уровень
3) дают O(log n).

---

## ⭐ O(n log n) — лучшее для сортировки

```
   O(n log n) = делаем log n уровней работы, на каждом обрабатываем все n элементов
   это сложность ХОРОШИХ сортировок (быстрая, слиянием — модуль 14)
   практически это «почти линейно» — намного лучше O(n²)
```

💡 ⭐ O(n log n) — «золотой стандарт» для задач, где нужно отсортировать или обработать с
делением. Заметно быстрее O(n²): для миллиона элементов ~20 млн операций против триллиона.

---

## 📖 Как быстро оценить код

```
   нет циклов по данным / прямой доступ     → O(1)
   один цикл по n                           → O(n)
   два вложенных цикла по n                 → O(n²)
   деление задачи пополам каждый шаг        → O(log n)
   сортировка + проход                      → O(n log n)
   рекурсия с двумя ветвями без мемоизации  → часто O(2ⁿ) (осторожно!)
```

💡 ⭐ Быстрый метод: найди **самую вложенную/повторяемую** часть и посчитай, сколько раз она
выполняется в зависимости от n. Вызовы функций считай по их сложности. `arr in list` внутри
цикла — это скрытый O(n) → даёт O(n²)!

⚠️ Скрытые сложности: операции вроде `x in list` (O(n)), срез строки (O(k)), сортировка (O(n log
n)) внутри цикла **умножаются** на число итераций. Их легко не заметить.

## ⚠️ Ловушки

- ❌ Не замечать скрытую сложность операций внутри цикла (`in list`, срезы, сортировка).
- ❌ Считать рекурсию «бесплатной» — она может быть O(2ⁿ) (модуль 15, 19).
- ❌ Путать вложенные циклы (умножение, O(n²)) и последовательные (сложение, O(n)).
- ❌ Забывать, что два цикла по разным размерам — O(n·m), а не O(n²).
