# 12 · Компромиссы и амортизация 🖼️⭐

> 🎯 **Цель блока:** понять ключевые идеи зрелого анализа — компромисс время/память,
> амортизированная сложность и почему константы иногда всё же важны.

---

## ⭐ Компромисс время ↔ память (углублённо)

Из модуля 10: часто можно ускорить ценой памяти. Это **главный рычаг** оптимизации алгоритмов:

```
   КЭШИРОВАНИЕ/МЕМОИЗАЦИЯ — храним вычисленное → не пересчитываем (память ↑, время ↓)
   ХЕШ-ТАБЛИЦА вместо перебора — O(n²)→O(n) ценой O(n) памяти
   ПРЕДВЫЧИСЛЕНИЕ — посчитать заранее (префиксные суммы) → запросы за O(1)
```

🖼️
```
   медленно, мало памяти  ◄──────────────────►  быстро, много памяти
   двойной цикл O(n²)/O(1)     хеш O(n)/O(n)      таблица предвычислений
```

💡 ⭐ Нет универсально «лучшего» — есть подходящее под ограничения. Памяти вдоволь и нужна
скорость → меняй память на время. Память дорога (embedded, гигантские данные) → экономь её. Умение
**сознательно выбирать** точку на этой шкале — признак зрелого инженера.

## ⭐⭐ Амортизированная сложность

**Амортизированная** сложность — средняя стоимость операции, когда **большинство** операций
дёшевы, а редкие — дороги.

```
   динамический массив, push (модуль 03):
   обычно O(1) (есть место)
   изредка O(n) (заполнился → удвоение + копирование)
   НО: удвоение происходит так редко, что В СРЕДНЕМ на операцию → O(1) амортизированно
```

🖼️
```
   push push push push [ДОРОГО:копирование] push push push push [ДОРОГО] ...
   ←─── много дешёвых ──→         ←─── много дешёвых ──→
   редкие дорогие «размазываются» на множество дешёвых → амортизированно O(1)
```

💡 ⭐⭐ Амортизация отвечает: «дорого ли это **в среднем за серию операций**?». Хоть отдельный push
иногда O(n), за миллион push'ей суммарная работа ~O(n) → на операцию O(1). Так же хеш-таблица: в
среднем O(1), хотя отдельная операция при коллизиях дороже. Не путай амортизированную (средняя по
серии) со средним случаем (по входным данным) — но идея похожа: «обычно дёшево».

## ⭐ Когда константы всё же важны

```
   Big-O отбрасывает константы — но в РЕАЛЬНОСТИ они иногда решают:

   - на МАЛЫХ n: O(n²) с маленькой константой может быть быстрее O(n log n) с большой
     (поэтому быстрые сортировки переключаются на простые для маленьких массивов)
   - КЭШ процессора: массив (непрерывный) часто быстрее списка той же сложности
     (константа из-за промахов кэша — привет, [иерархия памяти ОС](../../OS/02-memory/12-tlb-hierarchy.md))
```

💡 ⭐ Big-O — про **асимптотику** (большие n). На реальных данных и железе константы и кэш тоже
влияют. Поэтому: сначала выбери правильный **класс** сложности (это важнее всего), потом, если
нужно, оптимизируй константы (кэш-дружелюбность, меньше аллокаций). Не путай порядок: класс
сложности первичен.

## 📖 Преждевременная оптимизация

```
   «Преждевременная оптимизация — корень всех зол» (Кнут)
   сначала: правильный АЛГОРИТМ (класс сложности) — это даёт основной выигрыш
   потом (если нужно): профилируй, оптимизируй УЗКОЕ место константами
   НЕ: микрооптимизировать всё подряд вслепую
```

💡 ⚠️ Выбор алгоритма (O(n²)→O(n log n)) даёт несравнимо больше, чем микрооптимизации констант.
Сначала правильный класс сложности, потом — измерять (профилировать, см.
[C](../../C/04-senior/24-debugging-profiling.md)/[Python](../../Python/04-senior/22-profiling.md))
и точечно ускорять реальные узкие места. Оптимизировать вслепую — пустая трата времени.

## ⚠️ Ловушки

- ❌ Считать, что «лучше по Big-O» = «всегда быстрее» (на малых n константы решают).
- ❌ Путать амортизированную сложность со «всегда O(1)» (отдельная операция бывает дорогой).
- ❌ Преждевременно оптимизировать константы вместо выбора правильного алгоритма.
- ❌ Игнорировать память/кэш, глядя только на временную сложность.

## 🛠️ Практика

1. Возьми задачу и реши её «экономно по памяти, но медленно» и «быстро, но с памятью». Сравни.
