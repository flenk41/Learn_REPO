# 12 · Времена жизни (Lifetimes) 🖼️⭐

> 🎯 **Цель блока:** понять времена жизни — как Rust гарантирует, что ссылки никогда не
> переживают данные. Это последний кусочек системы владения. Тема кажется сложной, но идея
> простая.

---

## 📖 Зачем нужны времена жизни

Времена жизни (lifetimes) — это способ компилятора убедиться, что **ссылка не живёт дольше,
чем данные, на которые она указывает**. Это формализация защиты от висячих ссылок (модуль 10).

🖼️
```
   данные:  [======жизнь данных======]
   ссылка:     [==жизнь ссылки==]          ✅ ссылка внутри жизни данных — ок

   данные:  [==жизнь данных==]
   ссылка:     [======жизнь ссылки======]  ❌ ссылка переживёт данные — висячая!
```

💡 Компилятор отслеживает эти «времена жизни» и не даёт ссылке пережить источник. В
большинстве случаев он выводит их **сам** — ты их даже не пишешь.

---

## 📖 Чаще всего времена жизни невидимы

```rust
fn first_word(s: &str) -> &str {       // времена жизни выведены автоматически
    s.split(' ').next().unwrap_or("")
}
```

Здесь компилятор сам понимает: возвращаемый `&str` живёт столько же, сколько входной `s`.
Ты не пишешь ничего особенного. Это называется **элизия времён жизни** (lifetime elision).

---

## ⭐ Когда нужно указывать явно

Иногда компилятор не может вывести связь — например, функция возвращает одну из **двух**
ссылок:

```rust
// ❌ компилятор не знает: возвращаемая ссылка связана с x или с y?
fn longest(x: &str, y: &str) -> &str {
    if x.len() > y.len() { x } else { y }
}
```

Нужно **аннотировать** время жизни — сказать «результат живёт столько же, сколько входные»:

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

🖼️ Разбор синтаксиса:
```
   fn longest<'a>(x: &'a str, y: &'a str) -> &'a str
              │        └─ x живёт время 'a   └─ результат живёт время 'a
           объявляем параметр времени жизни 'a
```

💡 `'a` — это **имя времени жизни** (читается «тик a»). Аннотация говорит: «результат
живёт не дольше, чем самый короткоживущий из `x` и `y`». Это **не меняет** время жизни —
лишь **описывает связь** для компилятора, чтобы он проверил безопасность.

---

## 🧪 Зачем это защищает

```rust
let result;
{
    let s1 = String::from("длинная строка");
    let s2 = String::from("короткая");
    result = longest(&s1, &s2);     // result ссылается на s1 или s2
    println!("{}", result);         // ✅ ок — пока s1, s2 живы
}
// println!("{}", result);          // ❌ ОШИБКА! s1, s2 уничтожены → result повис бы
```

💡 Благодаря `'a` компилятор знает: `result` не может пережить `s1`/`s2`. Попытка
использовать его после их уничтожения — ошибка компиляции. Висячая ссылка предотвращена.

---

## 📖 Времена жизни в структурах

Если структура **хранит ссылку**, нужно указать время жизни — структура не должна пережить
данные, на которые ссылается:

```rust
struct Excerpt<'a> {
    text: &'a str,           // структура хранит ссылку, живущую время 'a
}

let novel = String::from("Война и мир. Том первый.");
let first = novel.split('.').next().unwrap();
let e = Excerpt { text: first };   // e не может пережить novel
println!("{}", e.text);
```

💡 Это гарантирует: структура с ссылкой внутри не переживёт источник данных. Снова — защита
от висячих ссылок, проверенная компилятором.

---

## 📖 'static — живёт всю программу

```rust
let s: &'static str = "я живу всю программу";   // строковые литералы — 'static
```

`'static` — особое время жизни: данные живут всё время работы программы. Все строковые
литералы имеют его, потому что «вшиты» в исполняемый файл.

---

## 💡 Главное про времена жизни

- Это **не** управление памятью вручную — лишь **описание связей** между ссылками для
  компилятора.
- В 90% кода они **выводятся автоматически**, ты их не пишешь.
- Аннотации нужны, когда компилятор не может сам понять связь (несколько ссылок, ссылки в
  структурах).
- Они существуют **только при компиляции** — в рантайме их нет, ноль накладных расходов.

> 💡 Времена жизни поначалу пугают, но это просто способ компилятора доказать, что твои
> ссылки безопасны. Не зубри синтаксис — понимай идею: «ссылка не должна пережить данные».
