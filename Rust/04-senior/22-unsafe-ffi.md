# 22 · Unsafe и FFI 🖼️

> 🎯 **Цель блока:** заглянуть «под капот» — где Rust позволяет обойти проверки (`unsafe`)
> и как он связывается с кодом на C (FFI). Это мост к твоему C-курсу.

---

## 📖 Зачем нужен unsafe

Borrow checker гарантирует безопасность, но иногда он **слишком строг** — например, при
работе с железом, сырыми указателями или вызове C-кода. Для таких случаев есть **`unsafe`** —
блок, где ты берёшь ответственность на себя.

```rust
unsafe {
    // здесь разрешены операции, которые компилятор не может проверить
}
```

> ⚠️ `unsafe` **не отключает** borrow checker полностью. Он лишь разрешает 5 дополнительных
> действий (разыменование сырых указателей, вызов unsafe-функций и т.п.). Обычные правила
> внутри `unsafe` всё ещё действуют.

---

## ⭐ Сырые указатели

Внутри `unsafe` можно работать с сырыми указателями (как в C):

```rust
let mut num = 5;
let r1 = &num as *const i32;       // сырой указатель (только чтение)
let r2 = &mut num as *mut i32;     // сырой изменяемый указатель

unsafe {
    println!("{}", *r1);           // разыменование — только в unsafe
    *r2 = 10;
}
```

🖼️
```
   Безопасный Rust: &T, &mut T — проверены borrow checker
   unsafe:          *const T, *mut T — как указатели C, БЕЗ проверок
                    ответственность на программисте
```

💡 Сырые указатели (`*const`/`*mut`) — это указатели C. Они могут быть невалидными, null,
висячими — компилятор их не проверяет. Используй с осторожностью.

---

## 📖 Когда оправдан unsafe

- реализация структур данных, которые borrow checker не выражает (например двусвязный
  список);
- работа с железом (embedded, драйверы);
- вызов кода на C (FFI);
- крайняя оптимизация.

> 💡 **Правило:** держи `unsafe` минимальным и **оборачивай его в безопасный API**. Стандартная
> библиотека полна `unsafe` внутри (`Vec`, `String`), но снаружи даёт безопасный интерфейс.
> Большинству прикладных программистов `unsafe` почти не нужен.

---

## ⭐ FFI — вызов кода на C

**FFI** (Foreign Function Interface) позволяет Rust вызывать функции C (и наоборот). Это
прямая связь с твоим [C-курсом](../../C/README.md)!

### Вызов C из Rust
```rust
extern "C" {
    fn abs(input: i32) -> i32;     // объявляем C-функцию из libc
}

fn main() {
    unsafe {
        println!("{}", abs(-5));   // вызов C-функции — только в unsafe
    }
}
```

🖼️
```
   Rust код  ──extern "C"──►  функция на C (libc, своя .c библиотека)
   вызов через unsafe (Rust не может проверить безопасность чужого C-кода)
```

### Экспорт Rust для вызова из C
```rust
#[no_mangle]                        // не искажать имя функции
pub extern "C" fn add(a: i32, b: i32) -> i32 {
    a + b
}
// теперь эту функцию можно вызвать из C-программы
```

💡 FFI делает Rust отличной заменой/дополнением C: можно постепенно переписывать C-проект на
Rust по функции, или использовать готовые C-библиотеки. Многие Rust-крейты — это безопасные
обёртки над C-библиотеками.

---

## 📖 Почему это важно понимать

Даже если сам не пишешь `unsafe`:
- ты понимаешь, что безопасность Rust держится на правилах, а `unsafe` — обдуманное
  исключение;
- умеешь читать `unsafe`-код в библиотеках;
- можешь интегрировать Rust с существующим C/C++ кодом;
- видишь связь всех языков курса: под безопасным Rust лежат те же сырые указатели, что ты
  изучал в C.

## ✅ Задачи

1. **Сырые указатели.** Создай `*const`/`*mut` на переменную, разыменуй в `unsafe`, измени
   значение.
2. **unsafe-функция.** Напиши `unsafe fn`, вызови её в `unsafe`-блоке.
3. **FFI — вызов C.** Вызови `abs` или `sqrt` из libc через `extern "C"`.
4. **Безопасная обёртка.** Оберни `unsafe`-операцию в безопасную функцию с проверками.
5. ⭐ **Экспорт в C.** Напиши `#[no_mangle] extern "C"` функцию, подумай, как вызвать её из C.
6. ⭐ **Связь с C-курсом.** Сравни сырой указатель Rust и указатель из C — что общего, в чём
   разница в безопасности.
