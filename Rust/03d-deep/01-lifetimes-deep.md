# 1 · Лайфтаймы глубоко: elision, HRTB, variance 🖼️⭐⭐

> 🎯 **Цель блока:** перестать «бороться» с лайфтаймами и понять их как систему — правила вывода
> (elision), высокоранговые лайфтаймы (HRTB) и вариантность.

---

## 📖 Лайфтайм = срок валидности ссылки

```
   ЛАЙФТАЙМ — это не «время жизни значения», а ОБЛАСТЬ, в которой ССЫЛКА гарантированно валидна.
   borrow checker проверяет: ссылка не переживёт данные, на которые указывает.
   аннотации 'a — это НЕ управление памятью, а СВЯЗЫВАНИЕ сроков ссылок для проверки компилятором.

   fn longest<'a>(x: &'a str, y: &'a str) -> &'a str
   ↑ «возвращаемая ссылка живёт не дольше, чем ОБЕ входные» — контракт для проверки.
```

💡 ⭐⭐ Ключевое осознание: лайфтайм-аннотации ничего не меняют в рантайме — это **информация для
borrow checker**, как связаны сроки ссылок. Ты не «продлеваешь» жизнь, а **описываешь
отношения**, чтобы компилятор доказал безопасность. Поняв это, перестаёшь воспринимать их как магию.

---

## ⭐ Lifetime elision: почему часто не пишешь 'a

```
   в большинстве функций лайфтаймы НЕ пишут — компилятор выводит их по ПРАВИЛАМ ELISION:
   1. каждая входная ссылка получает свой лайфтайм.
   2. если входная ссылка ОДНА → её лайфтайм назначается всем выходным.
   3. если есть &self/&mut self → лайфтайм self назначается выходным.

   fn first(s: &str) -> &str          // выводится как fn first<'a>(s: &'a str) -> &'a str (правило 2)
   fn get(&self) -> &Field            // выход привязан к &self (правило 3)

   аннотации НУЖНЫ, когда правила НЕ ХВАТАЕТ (несколько входных ссылок, неоднозначность).
```

💡 ⭐ Большая часть «не пишу лайфтаймы» — это работа elision. Понимая правила, ты знаешь, КОГДА
аннотации обязательны (несколько ссылок на входе, возврат ссылки) и почему компилятор иногда
требует их явно.

## ⭐⭐ Структуры с ссылками и 'static

```rust
struct Parser<'a> {
    input: &'a str,        // структура ХРАНИТ ссылку → нужен лайфтайм-параметр
}
// Parser не может пережить input, на который ссылается. компилятор это проверит.

// 'static — ссылка валидна ВСЮ программу:
let s: &'static str = "literal";   // строковые литералы — 'static (в бинарнике)
// 'static как ОГРАНИЧЕНИЕ (T: 'static) = «тип не содержит не-'static ссылок» (часто нужно для потоков).
```

💡 ⭐ Структура со ссылкой требует лайфтайм-параметр — она «заимствует» и не может пережить
источник. `'static` — особый лайфтайм «вся программа» (литералы, или владеющие данные без коротких
ссылок). `T: 'static` часто всплывает в многопоточности/async (данные должны жить достаточно долго).

## 📖 HRTB и вариантность (продвинутое)

```
   HRTB (Higher-Ranked Trait Bounds) — «для ЛЮБОГО лайфтайма»:
   for<'a> Fn(&'a str) -> &'a str
   нужно для замыканий/функций, работающих со ссылками ЛЮБОГО срока (например, параметр-функция,
   которую вызовут с разными лайфтаймами). встречается в обобщённых API с колбэками.

   ВАРИАНТНОСТЬ (variance) — как лайфтаймы соотносятся при подтипировании:
   • &'long T можно использовать там, где ждут &'short T (longer ↦ shorter) — ковариантность по лайфтайму.
   • &mut T инвариантен (нельзя подменять лайфтайм) — иначе можно нарушить безопасность.
   обычно компилятор выводит это сам; знать полезно для понимания «странных» ошибок лайфтаймов.
```

💡 ⭐⭐ HRTB и вариантность — то, что обычно «просто работает», но всплывает в сложных обобщённых
API и при ошибках лайфтаймов, которые кажутся непонятными. Понимание: ковариантность позволяет
использовать «более долгую» ссылку как «более короткую» (безопасно); `&mut` инвариантен (подмена
сломала бы гарантии). Не нужно писать руками, но понимание убирает «магию» сложных ошибок.

## ⚠️ Ловушки

- ❌ Думать, что лайфтаймы влияют на рантайм (это инфо для проверки).
- ❌ Аннотировать лайфтаймы там, где elision справляется (лишний шум).
- ❌ Возвращать ссылку на локальную переменную (не переживёт функцию) — частая ошибка новичка.
- ❌ Лепить `'static` «чтобы заткнуть компилятор» (часто означает «возьми владение» вместо ссылки).
- ❌ Бороться с лайфтаймами вместо рефакторинга на владение (иногда проще clone/owned).
