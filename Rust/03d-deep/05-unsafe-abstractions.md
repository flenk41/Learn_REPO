# 5 · Unsafe и безопасные абстракции 🖼️⭐⭐

> 🎯 **Цель блока:** понять unsafe не как «выключить безопасность», а как инструмент построения
> **безопасных абстракций** — то, на чём держится вся стандартная библиотека Rust.

---

## 📖 Что unsafe на самом деле даёт

```
   unsafe НЕ отключает borrow checker и не делает «всё можно». он разрешает РОВНО 5 вещей,
   которые компилятор не может проверить:
   1. разыменовать сырой указатель (*const T / *mut T).
   2. вызвать unsafe-функцию (включая FFI — C-функции).
   3. обратиться/изменить mutable static переменную.
   4. реализовать unsafe-трейт (Send/Sync вручную).
   5. обратиться к полю union.
   ВСЁ остальное (borrow checking, типы) работает как обычно ВНУТРИ unsafe.
```

💡 ⭐⭐ Ключевое осознание: `unsafe` — это **«я, программист, гарантирую инварианты, которые
компилятор не может проверить»**, а не «безопасность выключена». Это договор: ты берёшь на себя
доказательство корректности этих 5 операций. Неправильно — UB (как в C). Правильно — безопасно.

---

## ⭐ Безопасная абстракция над unsafe

```
   главный паттерн Rust: МАЛЕНЬКИЙ unsafe внутри, обёрнутый в БЕЗОПАСНЫЙ API, который НЕВОЗМОЖНО
   использовать неправильно. снаружи — обычный безопасный Rust; внутри — выверенный unsafe.

   пример: Vec<T> внутри использует сырые указатели и unsafe (управление буфером), но снаружи —
   полностью безопасный API (push/get/...). ты пользуешься Vec безопасно, не зная про unsafe внутри.
```

🖼️
```
   [ безопасный публичный API ]   ← пользователь видит только это, не может сломать
        │ обёртывает
   [ маленький выверенный unsafe ]  ← сырые указатели, ручная память; инварианты доказаны автором
   принцип: unsafe ЛОКАЛИЗОВАН и СПРЯТАН за безопасной границей.
```

💡 ⭐⭐ Так устроена ВСЯ stdlib: `Vec`, `Box`, `Rc`, `String`, каналы — внутри unsafe (иначе не
реализовать), снаружи — безопасны. Это и есть «zero-cost безопасность»: производительность C
(unsafe внутри) + гарантии Rust (безопасный API снаружи). Твоя задача при unsafe — **сделать так,
чтобы безопасный API нельзя было использовать во вред**, даже если внутри сырые операции.

---

## ⭐ Инварианты и обязанности

```
   используя unsafe, ты ОБЯЗАН поддержать инварианты, которые иначе гарантирует компилятор:
   • сырой указатель валиден, выровнен, не висячий, когда разыменовываешь.
   • не создаёшь два &mut на одни данные (даже через сырые указатели → UB).
   • Send/Sync реализуешь вручную ТОЛЬКО если тип реально потокобезопасен.
   • FFI: соблюдаешь контракт C-функции (владение, время жизни, null).
   нарушил → UB (как C): «иногда работает», крэши, дыры безопасности.
```

> 🧭 Это [граница памяти из Interop](../../Interop/02-boundary/09-memory-ownership.md) и
> [UB из C](../../C/04-senior/20-alignment-ub.md): unsafe Rust = ответственность уровня C, но
> локализованная и явная (видно `unsafe`, легко аудировать).

---

## ⭐⭐ Когда (НЕ) использовать unsafe

```
   ✅ ОПРАВДАННО:
   • реализация структуры данных, невыразимой в safe (intrusive, lock-free, свой Vec/аллокатор).
   • FFI — вызов C-библиотек.
   • экстремальная оптимизация горячего участка (доказанно, после профилирования).
   • работа с железом/embedded (регистры, MMIO).

   ❌ НЕ НАДО:
   • «чтобы заткнуть borrow checker» — почти всегда есть safe-решение (Rc/RefCell, рефакторинг).
   • без крайней необходимости — safe Rust покрывает 99% задач.
   правило: unsafe — последнее средство, локализованное, задокументированное (// SAFETY: почему безопасно),
   и обёрнутое в безопасный API. минимизируй его поверхность.
```

💡 ⭐⭐ Главное правило: **минимизируй unsafe, локализуй, документируй инвариант (`// SAFETY:`),
оберни в безопасный API**. Новички тянутся к unsafe «чтобы компилятор не мешал» — это почти всегда
ошибка (есть safe-путь). Опытные используют unsafe точечно, для того, что иначе невыразимо, и
доказывают корректность. Это и есть мастерство Rust: safe по умолчанию, unsafe осознанно и спрятанно.

---

## ⚠️ Ловушки

- ❌ Думать, что unsafe отключает borrow checker (он разрешает лишь 5 операций).
- ❌ Использовать unsafe «чтобы обойти компилятор» вместо safe-решения.
- ❌ Не локализовать/не документировать unsafe (`// SAFETY:`).
- ❌ Создать два `&mut` через сырые указатели (UB, даже в unsafe).
- ❌ Реализовать Send/Sync для непотокобезопасного типа (гонки).
- ❌ Дырявый безопасный API над unsafe (можно сломать снаружи) — главная ошибка.

## ✅ Задачи

1. **5 операций.** Для каждой из 5 unsafe-операций приведи минимальный пример. Что компилятор не может проверить?
2. **Сырые указатели.** Создай `*mut T`, разыменуй в unsafe. Покажи, что safe-код снаружи не пострадал.
3. ⭐ **Безопасная обёртка.** Сделай маленькую структуру с unsafe внутри (например, split на две
   непересекающиеся части среза) и безопасным API. Докажи (// SAFETY), что использовать во вред нельзя.
4. ⭐ **FFI.** Вызови простую C-функцию через unsafe extern. Оберни в безопасную Rust-функцию.
5. **Аудит.** Возьми unsafe-блок (свой/из stdlib) и напиши комментарий `// SAFETY:` — почему он корректен.
