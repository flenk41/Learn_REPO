# 2 · Концепты и ограничения (C++20) 🖼️⭐⭐

> 🎯 **Цель блока:** освоить концепты (concepts) — способ задать **требования к шаблонным типам**,
> сделать ошибки шаблонов понятными и код выразительным.

---

## 📖 Проблема: шаблоны принимают «что угодно»

```cpp
template <typename T>
T add(T a, T b) { return a + b; }

add(std::string{}, 5);   // ошибка где-то ГЛУБОКО внутри, на 50 строк, нечитаемая.
```

```
   до C++20: шаблон не говорит, ЧТО он требует от T. подставишь неподходящий тип → ошибка
   возникает ВНУТРИ шаблона (при использовании +), длинная и непонятная.
   КОНЦЕПТЫ: задают требования ЯВНО → ошибка на ГРАНИЦЕ («T должен поддерживать +»), понятная.
```

💡 ⭐⭐ Концепты — это **именованные ограничения на типы**. Они делают шаблоны самодокументируемыми
(видно, что требуется), дают **понятные ошибки** (нарушил требование — ясное сообщение), и
позволяют перегрузку по свойствам типа. Одно из главных улучшений C++20.

---

## ⭐ Определение и использование концепта

```cpp
#include <concepts>

// свой концепт: тип, поддерживающий сложение
template <typename T>
concept Addable = requires(T a, T b) {
    { a + b } -> std::same_as<T>;     // выражение a+b должно компилироваться и возвращать T
};

// использование (несколько синтаксисов):
template <Addable T>                   // 1) как «уточнённый typename»
T add(T a, T b) { return a + b; }

template <typename T> requires Addable<T>   // 2) requires-clause
T add2(T a, T b) { return a + b; }

auto add3(Addable auto a, Addable auto b) { return a + b; }  // 3) сокращённый синтаксис
```

```
   add("hi", 5) теперь → ясная ошибка: «не удовлетворяет Addable», а не дебри внутри шаблона.
```

## ⭐⭐ requires-выражения: что можно требовать

```cpp
template <typename T>
concept Container = requires(T c) {
    c.size();                          // есть метод size()
    c.begin();                         // есть begin()
    c.end();                           // есть end()
    typename T::value_type;            // есть вложенный тип value_type
    { c.empty() } -> std::convertible_to<bool>;   // empty() возвращает нечто, конвертируемое в bool
};
```

```
   в requires{ ... } перечисляешь, что должно быть ВАЛИДНО для типа:
   • вызовы методов/выражения (просто компилируемость).
   • вложенные типы (typename T::...).
   • ограничения на тип результата ({ выражение } -> Концепт).
   • вложенные требования (requires другое-условие).
```

💡 ⭐⭐ Концепт — это «утиная типизация на этапе компиляции с проверкой»: тип подходит, если
удовлетворяет требованиям (как структурная типизация). Стандартная библиотека даёт готовые
(`std::integral`, `std::floating_point`, `std::sortable`, `std::ranges::range`...) — используй их,
прежде чем писать свои.

## 📖 Перегрузка по концептам и subsumption

```cpp
// разные реализации для разных «категорий» типов:
template <std::integral T>           T process(T x) { /* для целых */ }
template <std::floating_point T>     T process(T x) { /* для дробных */ }

process(5);     // → целочисленная версия
process(3.14);  // → дробная версия
```

💡 Концепты позволяют выбирать перегрузку по **свойствам типа** — чище и понятнее, чем
SFINAE/`enable_if` (модуль 3). Компилятор выбирает наиболее **специфичный** удовлетворённый
концепт (subsumption). Это заменяет многословные SFINAE-трюки выразительными ограничениями.

> 🧭 Концепты — «человеческое лицо» метапрограммирования (модуль 3): то же управление шаблонами по
> свойствам типов, но читаемо и с понятными ошибками.

## ⚠️ Ловушки

- ❌ Писать свои концепты, когда есть стандартные (`std::integral` и т.д.).
- ❌ Слишком слабые ограничения (концепт ничего реального не требует) → бесполезен.
- ❌ Слишком сильные (требуешь лишнего) → отвергаешь валидные типы.
- ❌ Путать `requires(параметры){...}` (requires-выражение) и `requires условие` (requires-clause).
- ❌ Ожидать рантайм-проверки — концепты проверяются на КОМПИЛЯЦИИ.
