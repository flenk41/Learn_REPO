# 07c · Макросы в C++: когда нужны, когда избегать 🖼️⭐

> 🎯 **Цель блока:** понять макросы препроцессора (наследие из C) — где они в C++ всё ещё нужны, а
> где их заменяют безопасными `constexpr`/`inline`/`template`. В C++ макросы — «крайнее средство».

> 🧭 Механизм тот же, что в [C-препроцессоре](../../C/03-middle/16-preprocessor-modularity.md). Здесь —
> акцент на C++: почему мы их в основном НЕ используем.

---

## 📖 Что такое макрос (напоминание)

```
   МАКРОС обрабатывает ПРЕПРОЦЕССОР — ДО компилятора, простой подстановкой ТЕКСТА.
   • object-like:   #define PI 3.14159          → везде PI заменяется на 3.14159
   • function-like: #define SQUARE(x) ((x)*(x)) → SQUARE(5) заменяется на ((5)*(5))

   препроцессор НЕ знает ни типов, ни области видимости, ни namespace — он просто меняет текст.
   именно поэтому в C++ макросы опасны (см. ниже) и их по возможности избегают.
```

💡 ⭐ Ты уже видел макрос в [07b](07b-multiple-files.md): include guard (`#ifndef/#define/#endif`) и
`#pragma once` — это препроцессор. Макросы в C++ остались, но их роль сильно сузилась по сравнению с C.

---

## ⭐⭐ Почему C++ избегает макросов

```
   у макросов нет того, что есть у нормального кода C++:
   ❌ НЕТ ТИПОВ — макрос не проверяет типы (ошибки не ловятся компилятором).
   ❌ НЕТ ОБЛАСТИ ВИДИМОСТИ / namespace — макрос виден ВЕЗДЕ после #define (конфликты имён).
   ❌ НЕВИДИМ ОТЛАДЧИКУ — в дебаггере макрос уже «развёрнут», не пошагать.
   ❌ ЛОВУШКИ ПОДСТАНОВКИ:
      #define SQUARE(x) x*x      →  SQUARE(2+3) = 2+3*2+3 = 11 (а не 25!)  ← забыл скобки
      #define SQUARE(x) ((x)*(x)) →  SQUARE(i++) = ((i++)*(i++))  ← ДВОЙНОЕ вычисление i++ (UB!)
      #define MAX(a,b) ((a)>(b)?(a):(b)) → MAX(f(), g()) вызовет функцию ДВАЖДЫ.

   в C это терпели (не было альтернатив). в C++ есть constexpr/inline/template — БЕЗОПАСНЫЕ замены.
```

💡 ⭐⭐ Главная мысль: **макрос — это слепая текстовая замена без типов и области видимости**, отсюда
коварные баги (двойное вычисление, конфликты имён, забытые скобки). В C++ почти всегда есть
типобезопасная альтернатива — её и предпочитай.

## ⭐⭐ Чем заменять макросы в C++

```
   БЫЛО (макрос, C-стиль)              СТАЛО (C++ — типобезопасно)
   ──────────────────────             ────────────────────────────
   #define PI 3.14159          →       constexpr double PI = 3.14159;        // типизированная константа
   #define MAX_USERS 100        →       constexpr int MaxUsers = 100;
   #define SQUARE(x) ((x)*(x))  →       constexpr int square(int x){return x*x;}  // или шаблон:
                                        template<class T> constexpr T square(T x){return x*x;}
   #define MAX(a,b) (...)        →       std::max(a, b);                       // из <algorithm>
   #define RED 0\n#define GRN 1  →       enum class Color { Red, Green };       // типобезопасный enum

   плюсы замен: проверка типов, область видимости/namespace, видны отладчику, без двойного вычисления.
```

💡 ⭐⭐ Запомни связку замен: **константы → `constexpr`**, **макро-функции → `constexpr`/`inline`
функции или шаблоны**, **наборы констант → `enum class`**, **MIN/MAX → `std::min`/`std::max`**. Это
правило современного C++ ([модуль 24](../04-senior/24-modern-cpp.md)): макрос ради значения или
«функции» — почти всегда ошибка дизайна.

## ⭐ Где макросы в C++ всё ещё НУЖНЫ

```
   препроцессор незаменим там, где работаешь ДО/ВНЕ системы типов:
   ✅ INCLUDE GUARDS — #ifndef/#define/#endif или #pragma once (защита заголовков, модуль 07b).
   ✅ УСЛОВНАЯ КОМПИЛЯЦИЯ — разный код под платформу/конфигурацию:
        #ifdef _WIN32 ... #elif __linux__ ... #endif
        #ifdef DEBUG ... #endif        (отладочный код только в debug-сборке)
   ✅ __FILE__, __LINE__, __func__ — для логов/ассертов (макрос видит, ОТКУДА его позвали):
        #define LOG(msg) std::cerr << __FILE__ << ":" << __LINE__ << " " << msg << "\n"
   ✅ assert(...) — стандартный макрос (отключается через NDEBUG).
   ✅ X-MACROS и генерация (редко) — когда нужно «сгенерить» повторяющийся код (deep-dive).

   общее: это задачи МЕТА-уровня (сборка, платформа, место в коде), где типов ещё/уже нет.
```

💡 ⭐ Законные применения макросов в C++ — про **сборку и окружение**, а не про значения/логику:
include guards, `#ifdef` под платформу, `__FILE__/__LINE__` для логов, `assert`. Если макрос
заменяет КОНСТАНТУ или ФУНКЦИЮ — замени на `constexpr`/`inline`/`template`.

## 📖 `constexpr` vs макрос vs `const`

```
   const int n = 5;        // константа времени выполнения (обычно), типизирована.
   constexpr int n = 5;    // константа времени КОМПИЛЯЦИИ, типизирована — лучшая замена #define-значения.
   #define n 5             // текстовая подстановка, без типа/области — избегай для значений.

   constexpr идёт дальше: constexpr-функции вычисляются на этапе компиляции (можно как замену
   макро-функций и для compile-time вычислений). подробнее — модуль 22 (constexpr/шаблоны).
```

## ⚠️ Ловушки

- ❌ `#define` для констант/функций вместо `constexpr`/`inline`/`template` (теряешь типы/область/отладку).
- ❌ Макрос-функция без скобок вокруг аргументов и всего тела (`SQUARE(2+3)` = 11).
- ❌ Передача выражений с побочными эффектами в макрос (`MAX(i++, j)` — двойное вычисление, UB).
- ❌ Макросы с обычными (не ВЕРХНИМ_РЕГИСТРОМ) именами → конфликты (соглашение: МАКРОСЫ_КАПСОМ).
- ❌ `#define min/max` (особенно на Windows) — ломает `std::min/std::max` (определи `NOMINMAX`).
- ❌ Считать, что макросы «быстрее» функций — `inline`/`constexpr` так же быстры и безопасны.
