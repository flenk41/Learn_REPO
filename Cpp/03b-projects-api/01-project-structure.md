# 1 · Структура проекта и сборка (CMake) 🖼️

> 🎯 **Цель блока:** научиться раскладывать C++ проект по файлам и собирать его через
> **CMake** — стандарт индустрии. Это то, как выглядят реальные C++ проекты.

---

## 📖 Зачем разделять

Один файл на 2000 строк — это боль: не найти ничего, страшно менять, нельзя
переиспользовать, невозможно работать в команде. Решение — **разделение
ответственности**: каждый файл отвечает за одну тему.

---

## ⭐ Пара заголовок + исходник

В C++ код модуля делят на два файла:

- **`.hpp` / `.h` (заголовок)** — *что* умеет класс/модуль: объявления классов, функций,
  публичный интерфейс. Это **API**.
- **`.cpp` (исходник)** — *как* это сделано: реализация методов.

```cpp
// stack.hpp — ЧТО (интерфейс)
#pragma once
#include <cstddef>

class Stack {
public:
    void push(int value);
    int  pop();
    std::size_t size() const;
private:
    std::vector<int> data_;     // детали можно держать здесь...
};
```
```cpp
// stack.cpp — КАК (реализация)
#include "stack.hpp"

void Stack::push(int value) { data_.push_back(value); }
int  Stack::pop()           { int v = data_.back(); data_.pop_back(); return v; }
std::size_t Stack::size() const { return data_.size(); }
```

💡 `#pragma once` — современная замена include guard (`#ifndef/#define/#endif`), защищает
от двойного включения. Кто подключает `stack.hpp`, видит **интерфейс**, а реализация — в
`.cpp`.

## 🖼️ Структура настоящего проекта

```
myproject/
├── include/myproject/    ← публичные заголовки (.hpp) — API
│   └── stack.hpp
├── src/                  ← реализация (.cpp)
│   ├── stack.cpp
│   └── main.cpp
├── tests/                ← тесты (Catch2 / GoogleTest)
│   └── test_stack.cpp
├── CMakeLists.txt        ← как собирать
├── build/                ← артефакты сборки (в .gitignore!)
└── README.md
```

| Папка | Что лежит |
|-------|-----------|
| `include/` | заголовки `.hpp` — публичный API |
| `src/` | исходники `.cpp` |
| `tests/` | тесты |
| `build/` | результаты сборки (не в git) |

## ⭐ CMake — стандартная сборка C++

Компилировать вручную `g++ a.cpp b.cpp ...` неудобно. **CMake** — стандартный инструмент:
он описывает проект кроссплатформенно и сам зовёт компилятор.

`CMakeLists.txt`:
```cmake
cmake_minimum_required(VERSION 3.15)
project(MyProject CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# собрать программу из исходников
add_executable(myapp
    src/main.cpp
    src/stack.cpp
)
# где искать заголовки
target_include_directories(myapp PRIVATE include)
```

Сборка:
```bash
cmake -B build           # сконфигурировать (создаёт build/)
cmake --build build      # собрать
./build/myapp            # запустить
```

🖼️
```
   CMakeLists.txt ──cmake──► системные файлы сборки ──► компилятор ──► myapp
   меняешь файлы → cmake --build пересобирает только нужное
```

💡 CMake кроссплатформенный: одна конфигурация работает и на Windows (MSVC/MinGW), и на
Linux, и на macOS. Все серьёзные C++ проекты используют его (или аналог).

## ⭐ Библиотека + программа

Часто код оформляют как **библиотеку**, которую подключают и тесты, и программа:

```cmake
# библиотека из реализации
add_library(mylib src/stack.cpp)
target_include_directories(mylib PUBLIC include)

# программа использует библиотеку
add_executable(myapp src/main.cpp)
target_link_libraries(myapp PRIVATE mylib)

# тесты тоже используют библиотеку
add_executable(tests tests/test_stack.cpp)
target_link_libraries(tests PRIVATE mylib)
```

💡 `PUBLIC`/`PRIVATE` управляют тем, кто видит заголовки и зависимости — это и есть
**направление зависимостей** на уровне сборки.

---

## ⭐ Направление зависимостей и пространства имён

🖼️ Модули должны зависеть **в одну сторону**, без циклов:
```
   main.cpp ──► логика ──► утилиты (низкоуровневые, ни от кого не зависят)
   ✅ верхний слой знает про нижний, не наоборот
```

Код библиотеки оборачивают в **namespace**, чтобы имена не конфликтовали:
```cpp
namespace myproject {
    class Stack { ... };
}
// использование: myproject::Stack s;
```

> ⚠️ Внутренние («приватные») функции, не входящие в API, прячь в анонимный namespace в
> `.cpp` — они не будут видны снаружи:
> ```cpp
> namespace { void helper() { ... } }   // видна только в этом .cpp
> ```

## 📖 Когда делить

| Ситуация | Решение |
|----------|---------|
| Файл за ~300–500 строк | пора делить |
| «Две темы» в файле | раздели по темам |
| Кусок переиспользуется | вынеси в модуль/библиотеку |
| Программа на 50 строк | не дроби |

> ⚠️ Шаблоны (модуль 15) — исключение: их реализацию держат **в заголовке**, а не в `.cpp`.

## ✅ Задачи

1. **Раздели монолит** на `main.cpp` + класс в `.hpp`/`.cpp`.
2. **Раскладка + CMake.** Перенеси проект в `include/ src/`, напиши `CMakeLists.txt`,
   собери через `cmake -B build && cmake --build build`.
3. **Библиотека.** Оформи логику как `add_library`, подключи к программе через
   `target_link_libraries`.
4. **namespace.** Оберни свой код в namespace, спрячь внутренние функции в анонимный
   namespace.
5. ⭐ **Три модуля** с правильным направлением зависимостей + CMake.
