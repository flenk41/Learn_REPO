# 09 · new / delete и ручная память 🖼️⭐

> 🎯 **Цель блока:** понять ручное управление кучей в C++ (`new`/`delete`) — и **почему
> его опасно использовать напрямую**. Это подготовка к RAII, который решит все проблемы.

---

## 📖 new / delete — это malloc/free для C++

В C ты выделял память через `malloc`/`free`. В C++ для этого есть `new`/`delete` — они
делают то же, **плюс** вызывают конструкторы и деструкторы объектов.

```cpp
int* p = new int(42);    // выделить в куче int со значением 42
std::cout << *p << "\n"; // 42
delete p;                // освободить
p = nullptr;

// массив
int* arr = new int[100]; // выделить массив из 100 int
arr[0] = 1;
delete[] arr;            // ⚠️ для массива — delete[], не delete!
```

🖼️
```
   STACK            HEAP (куча)
  ┌──────┐        ┌────────────┐
  │ p ●──┼───────►│  42 (int)  │   new вернул адрес
  └──────┘        └────────────┘
   delete p освобождает блок в куче (как free)
```

🆚 Сравни с C:
```c
// C:    int* p = malloc(sizeof(int)); *p = 42; free(p);
// C++:  int* p = new int(42);                  delete p;
```

| | `malloc`/`free` | `new`/`delete` |
|--|-----------------|----------------|
| Вызывает конструктор/деструктор | нет | **да** |
| Возвращает | `void*` (нужно приведение) | типизированный указатель |
| Размер | считаешь сам (`sizeof`) | компилятор знает |

---

## ⚠️ Те же смертные грехи, что в C

`new`/`delete` так же опасны, как `malloc`/`free`. Все ошибки памяти из
[C-курса](../../C/02-memory/11-dynamic-memory.md) здесь повторяются:

```cpp
// 1. 💧 Утечка — забыл delete
int* p = new int(5);
// ... return; ← забыли delete p → память потеряна

// 2. 💀 Use-after-free
int* p = new int(5);
delete p;
*p = 10;             // ⚠️ запись в освобождённую память

// 3. ☠️ Double free
delete p;
delete p;            // ⚠️ двойное освобождение

// 4. ⚠️ Неправильный delete для массива
int* arr = new int[10];
delete arr;          // ⚠️ должно быть delete[] arr
```

---

## ⚠️⚠️ Главная проблема: исключения ломают ручной delete

В C++ есть исключения (ошибки), и они делают ручной `delete` **очень** опасным:

```cpp
void bad() {
    int* p = new int[1000];
    doSomething();       // ⚠️ если здесь выбросится исключение...
    delete[] p;          // ...эта строка НЕ выполнится → утечка!
}
```

🖼️
```
   new[]  ──►  doSomething() бросает исключение  ──►  ✗ delete[] пропущен
                                                       💧 УТЕЧКА
```

> ⚠️ Даже аккуратный программист не может гарантировать `delete` при каждом пути выхода
> (исключения, ранние `return`, `break`). Ручное управление памятью в C++ **принципиально
> ненадёжно**.

---

## 💡 Вывод: НЕ используй new/delete напрямую

Это ключевая мысль курса C++:

> **В современном C++ ты почти никогда не пишешь `new`/`delete` вручную.** Вместо этого
> используешь объекты, которые сами управляют памятью — через **RAII** (модуль 10) и
> **умные указатели** (модуль 11).

```cpp
// ❌ старый стиль (опасно)
int* arr = new int[1000];
// ... не забудь delete[] ...

// ✅ современный стиль — память управляется сама
std::vector<int> arr(1000);   // вектор сам выделит и освободит память
// никаких delete — всё автоматически и безопасно
```

💡 Понимать `new`/`delete` нужно (чтобы читать чужой код и понимать, что внутри), но
**писать** их вручную — почти всегда ошибка. Дальше узнаешь, чем заменить.

---

## 🛠️ Инструменты остаются те же

Как и в C, ошибки памяти ловит AddressSanitizer:
```bash
g++ -std=c++17 -fsanitize=address -g program.cpp -o program
./program
```
Он поймает утечки, use-after-free, double-free. На этапе обучения памяти **всегда**
компилируй с ним.

## ✅ Задачи

1. **new/delete.** Выдели `int` через `new`, используй, освободи. То же для массива
   (`new[]`/`delete[]`).
2. **Поймай утечку.** Напиши код с забытым `delete`, найди утечку через ASan.
3. **Use-after-free.** Воспроизведи и поймай ASan-ом.
4. **Исключение ломает delete.** Напиши функцию, где между `new` и `delete` бросается
   исключение (через `throw`). Убедись через ASan, что память утекла. Подумай, как
   починить (ответ — RAII, следующий модуль).
5. **Замени на vector.** Перепиши задачу с `new int[]` на `std::vector<int>` — убедись,
   что `delete` больше не нужен и ASan чист.
