# 09 · Слияние (merge) ⭐⭐ 🖼️

> 🎯 **Цель блока:** научиться сливать ветки обратно — объединять работу. Merge — то, ради чего
> ветвление существует: разошлись, поработали, соединили.

---

## ⭐⭐ Что такое merge

```
   MERGE объединяет изменения одной ветки в другую. обычно: влить feature обратно в main.

   git switch main          # перейти в ЦЕЛЕВУЮ ветку (куда вливаем)
   git merge feature        # влить feature в main

   Git берёт обе линии работы и объединяет их в одно состояние.
```

---

## ⭐⭐ Два вида слияния

```
   1. FAST-FORWARD (перемотка) — если main НЕ менялся с момента ответвления:
      main просто «догоняет» feature, сдвигая указатель вперёд. Нового коммита нет.

      до:   main→C        feature→E          после:   main→E (просто переместился)
            A─B─C─D─E                                  A─B─C─D─E

   2. MERGE-КОММИТ (трёхсторонний) — если ОБЕ ветки расходились (main тоже получил коммиты):
      Git создаёт новый «слиятельный» коммит с ДВУМЯ родителями.
```

🖼️
```
   merge-коммит (обе ветки развивались):

   main:     A ── B ── C ──────── M     ← M: merge-коммит (2 родителя: C и E)
                       \         /
   feature:             D ── E ─┘

   M объединяет C и E. История показывает, что была параллельная работа.
```

💡 ⭐⭐ Fast-forward — когда сливать нечего «поперёк» (main не двигался): просто перемотка указателя.
Merge-коммит — когда обе ветки развивались: Git делает трёхсторонний merge (общий предок + два конца)
и фиксирует слияние коммитом с двумя родителями. Merge **сохраняет историю как было** (видно, что
велась параллельная работа) — в отличие от rebase (модуль 11).

---

## ⭐ Управление слиянием

```
   git merge feature                # обычное слияние (ff, если возможно)
   git merge --no-ff feature        # ВСЕГДА merge-коммит (даже если можно ff) —
                                    #   сохраняет факт «здесь была фича» (частая командная практика)
   git merge --ff-only feature      # только если возможна перемотка (иначе откажет)
   git merge --abort                # отменить слияние при конфликте (модуль 10)

   после слияния feature в main:
   git branch -d feature            # удалить влитую ветку
```

💡 ⭐ `--no-ff` популярен в командах: создаёт явный merge-коммит на каждую фичу, и в истории видны
«пузыри» работы по задачам (легче читать/откатывать фичу целиком). Это вопрос стиля истории (модуль 12).

---

## 📖 Типичный цикл фичи

```
   1. git switch -c feature/login          # ветка под задачу
   2. ...работаешь, коммитишь...
   3. git switch main                       # вернуться в main
   4. git pull                              # подтянуть свежий main (модуль 14)
   5. git merge feature/login               # влить (или через pull request — модуль 15)
   6. git branch -d feature/login           # убрать ветку
```

---

## ⚠️ Ловушки

- ❌ Запускать `git merge` НЕ в той ветке (сливаешь не туда — merge вливает в ТЕКУЩУЮ).
- ❌ Сливать давно не обновлявшуюся ветку (много конфликтов — сливай/обновляй часто).
- ❌ Паниковать при конфликте (это нормально — модуль 10; есть `--abort`).
- ❌ Забыть удалить влитые ветки (копятся, засоряют).

## ✅ Задачи

1. Создай ветку, сделай коммит, слей в main fast-forward (main не трогал). Появился ли merge-коммит?
2. Сделай коммиты И в main, И в ветке → слей. Теперь появился merge-коммит? Посмотри `--graph`.
3. Слей с `--no-ff` — сравни историю с обычным ff.
4. ⭐ Пройди полный цикл: ветка → коммиты → switch main → merge → удалить ветку.
5. Нарисуй на бумаге fast-forward и трёхсторонний merge для своего примера.
