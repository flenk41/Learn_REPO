# 11 · Rebase 🖼️⭐⭐

> 🎯 **Цель блока:** понять rebase — «пересадку» ветки на новое основание. Мощный инструмент чистой
> истории, но с золотым правилом безопасности.

---

## ⭐⭐ Что такое rebase

```
   REBASE «пересаживает» твои коммиты на другую базу — как будто ты ответвился ПОЗЖЕ.
   Git берёт твои коммиты, временно откладывает, перематывает ветку на новую базу, заново
   накладывает твои коммиты поверх (с НОВЫМИ хешами — это переписанные копии!).

   git switch feature
   git rebase main          # пересадить feature поверх свежего main
```

🖼️
```
   было (main ушёл вперёд, пока ты делал feature):
   main:     A ── B ── C ── F        (F появился в main)
                       \
   feature:             D ── E

   после git rebase main:
   main:     A ── B ── C ── F
                            \
   feature:                  D' ── E'    ← D,E ПЕРЕНАЛОЖЕНЫ поверх F (новые хеши D', E')

   теперь история feature ЛИНЕЙНА относительно main (как будто начал после F).
```

💡 ⭐⭐ Rebase делает историю **линейной и чистой**: вместо merge-коммитов и «пузырей» — прямая
цепочка. Цена: коммиты **переписываются** (новые хеши). Поэтому rebase — переписывание истории, и к
нему применимо золотое правило (ниже).

---

## ⭐⭐ Золотое правило rebase

```
   ⚠️⚠️ НЕ делай rebase коммитов, которые УЖЕ ОПУБЛИКОВАНЫ (запушены и кто-то мог их забрать).

   почему: rebase создаёт НОВЫЕ коммиты (D'→ вместо D). если коллега уже взял старые D, E —
   у вас разойдутся истории, push потребует --force, ты перепишешь общую историю → хаос/потери.

   ✅ безопасно: rebase СВОИХ ЛОКАЛЬНЫХ, ещё не запушенных коммитов (причесать перед push).
   ❌ опасно: rebase общей ветки (main/develop) или уже расшаренных коммитов.
```

💡 ⭐⭐ Запомни: **rebase — для локальной, личной истории до публикации**. Опубликовал — не
переписывай (как и reset, модуль 05). Для общих веток — только merge. Нарушение этого правила — самая
частая «катастрофа» новичков с rebase.

---

## ⭐ Зачем rebase на практике

```
   1. АКТУАЛИЗИРОВАТЬ ветку: подтянуть свежий main под свою фичу без merge-коммита:
      git switch feature; git rebase main     # фича снова «поверх» актуального main

   2. ПРИЧЕСАТЬ историю перед push (интерактивный rebase — модуль 18):
      git rebase -i HEAD~5     # объединить/переименовать/переставить свои коммиты

   3. git pull --rebase        # подтянуть удалённые изменения, наложив свои поверх (линейно, модуль 14)

   при конфликте во время rebase:
   реши конфликт → git add → git rebase --continue   (или git rebase --abort, чтобы отменить всё)
```

---

## 📖 Rebase vs Merge (кратко)

```
   MERGE  — сохраняет историю КАК БЫЛО (с ветвлениями), безопасен для общих веток, добавляет merge-коммиты.
   REBASE — переписывает в ЛИНЕЙНУЮ историю, чище, но меняет хеши → только для неопубликованного.
   подробное сравнение и когда что — модуль 12.
```

---

## ⚠️ Ловушки

- ❌❌ Rebase опубликованных/общих коммитов → переписал общую историю, сломал коллегам.
- ❌ Думать, что rebase «безопаснее» merge — он мощнее, но опаснее (переписывает).
- ❌ Растеряться при конфликте в rebase (он бывает на КАЖДОМ коммите; `--continue`/`--abort`).
- ❌ `git push --force` после rebase в общую ветку (используй `--force-with-lease`, и лучше не надо).
