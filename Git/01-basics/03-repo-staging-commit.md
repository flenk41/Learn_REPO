# 03 · Репозиторий, staging, commit 🖼️⭐⭐

> 🎯 **Цель блока:** освоить базовый цикл Git — создать репозиторий, добавить изменения в staging,
> сделать коммит. Это операция, которую ты будешь повторять тысячи раз.

---

## 📖 Создание репозитория

```
   git init                  # превратить текущую папку в Git-репозиторий (создаётся .git/)
   git clone <url>           # склонировать существующий репозиторий с сервера

   .git/ — скрытая папка, где Git хранит ВСЮ историю и объекты. Удалишь её — потеряешь историю
   (но рабочие файлы останутся). Не трогай её содержимое руками.
```

---

## ⭐⭐ Базовый цикл: status → add → commit

```
   git status                # ГЛАВНАЯ команда: что изменено, что в staging, на какой ветке.
                             #   запускай постоянно — она подсказывает следующий шаг.

   git add file.txt          # добавить файл в staging (войдёт в следующий коммит)
   git add .                 # добавить ВСЕ изменения в текущей папке
   git add -p                # интерактивно: выбрать КУСКИ изменений (для аккуратных коммитов)

   git commit -m "Описание"  # сохранить staged-изменения в историю с сообщением
   git commit                # без -m: откроет редактор для подробного сообщения
```

🖼️
```
   рабочая папка          staging (индекс)        репозиторий (история)
   ┌───────────┐  add     ┌───────────┐  commit   ┌───────────┐
   │ изменил   │ ───────► │ отмечено  │ ────────► │  снимок   │
   │ file.txt  │          │ к коммиту │           │  в истории│
   └───────────┘          └───────────┘           └───────────┘
        ▲                                                │
        └──────────── git checkout / restore ◄───────────┘
```

💡 ⭐⭐ Цикл `status → add → commit` — сердцевина ежедневной работы. `git status` — твой компас:
показывает текущее состояние и подсказывает команды. Делай его рефлексом перед каждым add/commit.

---

## ⭐ Зачем staging (а не «закоммитить всё»)

```
   staging позволяет собрать коммит ОСМЫСЛЕННО:
   • изменил 3 файла по двум разным задачам → закоммить их ДВУМЯ коммитами
     (add первых двух → commit; add третьего → commit). История чище.
   • git add -p — добавить только часть изменений в файле (по кускам/hunks).

   плохо:  один гигантский коммит «много всего».
   хорошо: маленькие коммиты, каждый — одно логическое изменение (модуль 07).
```

💡 ⭐ Staging — это «корзина», куда ты складываешь именно то, что должно войти в следующий коммит.
Не всё, что изменено, обязано идти в один коммит. Это инструмент для чистой истории.

---

## 📖 Полезные варианты

```
   git commit -am "..."      # add (только отслеживаемых) + commit одной командой
   git diff                  # что изменено, но НЕ в staging
   git diff --staged         # что в staging (войдёт в коммит)
   git log --oneline         # краткая история (модуль 04)
```

---

## ⚠️ Ловушки

- ❌ `git add .` вслепую → закоммитил лишнее (временные файлы, секреты — см. .gitignore модуль 06).
- ❌ Огромные коммиты «всё подряд» вместо логических порций.
- ❌ Пустое/бессмысленное сообщение коммита («фикс», «update», «.») — модуль 07.
- ❌ Удалить папку `.git/` (потеряешь историю).
- ❌ Забыть `git add` перед commit → коммит без твоих изменений.

---

## ✅ Задачи

1. Создай репозиторий (`git init`), добавь файл, сделай первый коммит. Проверяй `git status` на каждом шаге.
2. Измени два файла под разные задачи. Закоммить их ДВУМЯ отдельными коммитами (через `git add`).
3. Поэкспериментируй с `git add -p` — добавь только часть изменений файла.
4. Сравни `git diff` и `git diff --staged` после изменений и после add.
