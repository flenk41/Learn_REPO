# 06 · .gitignore и что не коммитить 🖼️⭐

> 🎯 **Цель блока:** научиться исключать из репозитория то, что туда не должно попадать —
> артефакты сборки, зависимости, временные файлы и, главное, СЕКРЕТЫ.

---

## ⭐ Что НЕ коммитить

```
   ❌ артефакты сборки:      build/, dist/, *.o, *.exe, target/, __pycache__/
   ❌ зависимости:           node_modules/, venv/, vendor/  (ставятся из манифеста, не хранятся)
   ❌ временное/системное:   .DS_Store, Thumbs.db, *.log, *.tmp, *.swp
   ❌ настройки IDE:         .idea/, .vscode/ (часто; иногда часть коммитят)
   ❌❌ СЕКРЕТЫ:              .env, ключи API, пароли, токены, *.pem, приватные ключи

   ✅ коммитить:  исходный код, конфиги проекта, манифесты зависимостей (package.json,
                  requirements.txt, Cargo.toml), документацию, тесты.
```

💡 ⭐ Правило: в репозиторий идёт то, из чего проект **собирается**, а не то, что **получается**
(результаты сборки/зависимости — воспроизводимы). И никогда — секреты.

---

## 📖 Файл .gitignore

```
   создай файл .gitignore в корне репозитория. Git не будет отслеживать совпадающие пути:

   # комментарий
   node_modules/        # папка целиком
   *.log                # все .log файлы
   build/               # папка сборки
   .env                 # секреты окружения
   !important.log       # ! — исключение: ЭТОТ .log всё-таки отслеживать

   готовые шаблоны: github.com/github/gitignore (для каждого языка/инструмента).
```

💡 Подбери .gitignore под стек СРАЗУ при создании репозитория (на GitHub при создании можно выбрать
шаблон). Это избавляет от случайного коммита мусора с первого дня.

---

## ⭐⭐ Секреты: критично

```
   ⚠️ .gitignore работает ТОЛЬКО для НЕотслеживаемых файлов. Если файл УЖЕ закоммичен —
      добавление в .gitignore не уберёт его из истории!

   убрать уже отслеживаемый файл (оставив локально):
   git rm --cached .env
   echo ".env" >> .gitignore
   git commit -m "Убрать .env из репозитория"

   ⚠️⚠️ если СЕКРЕТ попал в коммит и был запушен — он КОМПРОМЕТИРОВАН. Удалить из истории мало:
   • НЕМЕДЛЕННО отзови/смени ключ/пароль (rotate) — считай его утёкшим.
   • затем чисти историю (git filter-repo / BFG) — но ротация важнее.
```

💡 ⭐⭐ Секрет, попавший в Git и отправленный на сервер, — **утёкший секрет**, даже если ты его потом
удалил. История Git хранит всё; боты сканируют публичные репо за секунды. Поэтому: храни секреты в
переменных окружения / менеджере секретов, а `.env` — в `.gitignore` с самого начала. Это пересечение
с [🛡️ безопасностью](../../Security/03-defensive-code/19-secrets-dependencies.md).

---

## ⚠️ Ловушки

- ❌ Закоммитить `node_modules/`/`venv/` (раздувает репо; должны ставиться из манифеста).
- ❌ Добавить файл в `.gitignore`, когда он УЖЕ отслеживается (нужен `git rm --cached`).
- ❌❌ Закоммитить `.env`/ключ → утечка. Сразу ротируй секрет, не только удаляй.
- ❌ Коммитить артефакты сборки (build/dist) — они воспроизводимы.

## ✅ Задачи

1. Создай `.gitignore`, добавь `*.log` и `build/`. Создай такие файлы — убедись, что Git их игнорирует.
2. Закоммить файл, потом «передумай»: убери его из отслеживания через `git rm --cached` + .gitignore.
3. Найди готовый шаблон .gitignore для своего языка (github/gitignore). Что он исключает и почему?
4. ⭐ Объясни, почему добавление секрета в .gitignore ПОСЛЕ коммита не решает проблему утечки.

## ✅ Чек-лист

- [ ] Создаю .gitignore под стек с самого начала
- [ ] Не коммичу артефакты, зависимости, временное
- [ ] Никогда не коммичу секреты (.env, ключи)
- [ ] Знаю про `git rm --cached` и важность ротации утёкшего секрета
