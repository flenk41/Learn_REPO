# 05 · Отмена изменений 🖼️⭐⭐

> 🎯 **Цель блока:** научиться безопасно откатывать — от «выкинуть правки в файле» до «отменить
> опубликованный коммит». Это спасает каждый день; путаница здесь — частая боль новичков.

---

## ⭐⭐ Карта отмены: что именно откатываешь

```
   зависит от того, в какой ЗОНЕ изменение:

   1. РАБОЧАЯ ПАПКА (изменил, не добавил) — выкинуть правки:
      git restore file.txt           # вернуть файл к последнему коммиту (правки ПРОПАДУТ)
      git restore .                  # все файлы

   2. STAGING (добавил, не закоммитил) — убрать из staging (правки сохранятся в файле):
      git restore --staged file.txt  # «развыделить», вернуть в рабочую папку

   3. ПОСЛЕДНИЙ КОММИТ — переделать:
      git commit --amend             # изменить сообщение/добавить файлы в последний коммит
      git reset --soft HEAD~1        # отменить коммит, изменения вернуть в staging
      git reset HEAD~1               # отменить коммит, изменения в рабочей папке (не staged)
      git reset --hard HEAD~1        # ⚠️ отменить коммит И стереть изменения (опасно!)
```

🖼️
```
   --soft   : коммит ↩  →  изменения остаются в STAGING        (мягко)
   --mixed  : коммит ↩  →  изменения в РАБОЧЕЙ ПАПКЕ (по умолч.)
   --hard   : коммит ↩  →  изменения УНИЧТОЖЕНЫ ⚠️              (жёстко, осторожно)
```

💡 ⭐⭐ Ключ — понять, в какой зоне правка (рабочая / staging / коммит) и хочешь ли СОХРАНИТЬ
изменения. `--soft`/`--mixed` сохраняют, `--hard` — стирает. `git status` всегда подсказывает нужную
команду restore прямо в выводе.

---

## ⭐ revert: безопасная отмена опубликованного

```
   git revert <хеш>     # создаёт НОВЫЙ коммит, отменяющий изменения указанного — НЕ переписывает историю

   reset  vs  revert:
   • reset  — «как будто коммита не было» (двигает ветку назад). Для ЛОКАЛЬНЫХ, неопубликованных.
   • revert — «отменяю, но честно записываю это в историю». Для ОПУБЛИКОВАННЫХ (запушенных) коммитов.
```

💡 ⭐ Золотое правило: **не переписывай опубликованную историю** (reset/rebase на запушенных
коммитах ломает работу коллег — модуль 12). Чтобы откатить то, что уже на сервере — `git revert`
(новый коммит-отмена, история не переписывается).

---

## 📖 Спасение «потерянного»

```
   git reflog            # журнал ВСЕХ перемещений HEAD (даже «удалённые» коммиты)
                        # → почти всё восстановимо! найди хеш в reflog → git reset/checkout на него.
   git stash             # временно убрать незакоммиченные правки «в карман» (модуль 19)
```

💡 Почти ничто в Git не теряется навсегда: коммиты живут в reflog ~30+ дней, даже после `reset --hard`.
Запаниковал? Сначала `git reflog` (подробно — модуль 23). Это снимает страх «сломать».

---

## ⚠️ Ловушки

- ❌ `git reset --hard` без понимания → безвозвратно стёр незакоммиченную работу.
- ❌ Делать `reset`/`--amend` на УЖЕ запушенных коммитах (ломает историю команды) → используй `revert`.
- ❌ Путать `restore` (рабочая папка/staging) и `reset` (коммиты).
- ❌ Паниковать после ошибки, не зная про `git reflog`.

---

## ✅ Задачи

1. Измени файл и выкинь правку через `git restore`. Убедись, что вернулось к коммиту.
2. Добавь файл в staging и убери через `git restore --staged`.
3. Сделай коммит и отмени через `git reset --soft HEAD~1` — куда делись изменения?
4. Сделай коммит, запушь (учебный репо), отмени через `git revert`. Чем отличается история от reset?
5. Сделай `git reset --hard`, потом найди «потерянный» коммит через `git reflog` и восстанови.

---

## ❓ Проверь себя

1. Чем `restore` отличается от `reset`?
2. `--soft`, `--mixed`, `--hard` — что с изменениями в каждом?
3. Когда `revert`, а когда `reset`? (опубликовано или нет)
4. Что такое `git reflog` и зачем он?
