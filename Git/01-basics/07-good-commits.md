# 07 · Анатомия хорошего коммита 🖼️⭐⭐

> 🎯 **Цель блока:** научиться делать коммиты, которые помогают (а не мешают) — атомарные, с ясными
> сообщениями. История — это документация проекта, и ты её автор.

---

## ⭐⭐ Атомарный коммит: одно логическое изменение

```
   АТОМАРНЫЙ КОММИТ = одна завершённая, логически цельная единица работы.

   ❌ плохо: «Добавил фичу профиля, починил баг с датой, обновил зависимости, переименовал переменные»
      → нельзя откатить только баг; ревьюеру невозможно понять; blame бесполезен.

   ✅ хорошо: три коммита:
      1. «Добавить страницу профиля пользователя»
      2. «Исправить парсинг даты в локали ru»
      3. «Обновить lodash до 4.17.21»
      → каждый можно откатить/перенести/отревьюить отдельно; история читается как журнал.
```

💡 ⭐⭐ Атомарность — главное качество коммита. Каждый коммит должен быть **обратимым и осмысленным
сам по себе**: проект после него собирается, изменение — одно. Это окупается при откате, ревью,
`git bisect` (поиск бага, модуль 22) и `revert`.

---

## ⭐⭐ Сообщение коммита

```
   формат (общепринятый):

   Краткое резюме в повелительном наклонении, до ~50 символов

   Подробное объяснение (при необходимости): ЗАЧЕМ это изменение, какой
   проблемой вызвано, какие есть нюансы. Перенос строки ~72 символа.
   Код объясняет КАК; сообщение — ПОЧЕМУ.

   Fixes #123        ← ссылка на issue (необязательно)

   правила:
   • заголовок в ИМПЕРАТИВЕ: «Добавить», «Исправить», «Убрать» (не «Добавил»/«Добавление»).
     мнемоника: «Если применить этот коммит, он [заголовок]».
   • заголовок коротко и по сути; без точки в конце.
   • пустая строка между заголовком и телом.
   • в теле — ПОЧЕМУ, а не пересказ диффа.
```

💡 ⭐⭐ Сообщение отвечает на вопрос будущего разработчика (часто тебя через полгода): «зачем это
сделали?». Дифф показывает ЧТО изменилось — это видно. Ценность сообщения — объяснить **почему**.
Плохие сообщения («фикс», «update», «.», «asdf») превращают историю в мусор.

## 📖 Conventional Commits (опционально)

```
   популярная схема префиксов (для автогенерации changelog, semver):
   feat:     новая функциональность       feat: добавить экспорт в CSV
   fix:      исправление бага              fix: убрать падение на пустом вводе
   docs:     документация                  docs: дополнить README про установку
   refactor: рефакторинг без смены поведения
   test:     тесты         chore: рутина/сборка        perf: производительность

   feat!: или BREAKING CHANGE — ломающее изменение (→ мажорная версия, semver модуль 20).
```

💡 Conventional Commits удобны в командах и для инструментов (автоматический changelog/версия). Не
обязательны, но дисциплинируют. Главное — последовательность внутри команды.

---

## 📖 Гигиена истории

```
   • коммить часто, маленькими порциями (легче откатывать/ревьюить).
   • не коммить «сломанное» состояние (проект должен собираться на каждом коммите — в идеале).
   • не мешай форматирование/переименования с логическими правками (отдельными коммитами —
     иначе дифф нечитаем).
   • перед push можно причесать локальную историю (squash мелочи — модуль 18).
```

> 🧭 Чистая история — часть [Senior-ремесла](../../Senior/01-craft/03-clean-code.md): код-ревью
> (модуль 16) идёт по коммитам и диффам; аккуратная история ускоряет ревью и отладку.

## ⚠️ Ловушки

- ❌ Коммиты-«свалки» (много несвязанных изменений вместе).
- ❌ Бессмысленные сообщения («фикс», «wip», «.», «изменения»).
- ❌ Описывать в сообщении КАК (видно в диффе) вместо ПОЧЕМУ.
- ❌ Прошедшее время/существительные вместо императива в заголовке.
- ❌ Мешать массовое форматирование с логикой (дифф невозможно ревьюить).

2. Напиши коммит с заголовком в императиве + телом, объясняющим ПОЧЕМУ.
3. Перепиши 3 плохих сообщения («фикс», «update», «asdf») в хорошие.
4. ⭐ Попробуй Conventional Commits на 5 коммитах (feat/fix/docs/...).
5. Открой историю крупного opensource-проекта на GitHub — оцени их сообщения. Что перенять?

## ❓ Проверь себя

1. Что такое атомарный коммит и зачем он?
2. Почему сообщение должно объяснять «почему», а не «что»?
3. Почему заголовок пишут в повелительном наклонении?
