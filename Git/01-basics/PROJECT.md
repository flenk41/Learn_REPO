# 🏗️ Проект уровня 1 — Веди реальный проект под Git с чистой историей

> 🎯 **Цель:** превратить любой свой проект (или один из курса) в Git-репозиторий и вести его так,
> чтобы история была образцовой. Привычка, а не разовое действие.

---

## Что делаем

Возьми существующий проект (например, [свой Vector из C](../../C/02-memory/PROJECT.md) или любой код)
и поставь его под версионный контроль с нуля, аккуратно.

---

## Этапы

- [ ] **Инициализация.** `git init`, добавь подходящий `.gitignore` (по шаблону для языка).
- [ ] **Первый коммит.** Только исходники — никаких артефактов/зависимостей/секретов.
- [ ] **Развитие по коммитам.** Вноси изменения порциями: каждая фича/фикс — отдельный атомарный
      коммит с осмысленным сообщением (императив + «почему» в теле, где нужно).
- [ ] **Минимум 10 коммитов**, отражающих реальный ход работы.
- [ ] **Демонстрация отмены.** Намеренно сделай «ошибочный» коммит и откати его двумя способами:
      `reset` (локально) и `revert` (как будто опубликовано). Опиши разницу.
- [ ] **Чистый .gitignore.** Убедись, что в репозитории нет мусора (`git status` чист, `git ls-files`
      показывает только нужное).
- [ ] **README.** Добавь README с описанием проекта и инструкцией запуска.

---

## 📦 Результат

1. Репозиторий с **читаемой историей** (`git log --oneline --graph` рассказывает историю проекта).
2. Все коммиты атомарны, сообщения осмысленны.
3. `.gitignore` корректен (ни артефактов, ни секретов).
4. Краткий отчёт: разница reset vs revert на твоём примере.

---

## 🌟 Усложнения

- Используй Conventional Commits (feat/fix/docs/...) последовательно.
- Настрой алиасы Git (`git config --global alias.lg "log --oneline --graph --all"`).
- Попробуй `git add -p` для сборки аккуратных коммитов из смешанных правок.

---

## ✅ Критерии

- [ ] История читается как журнал работы (атомарные коммиты, ясные сообщения)
- [ ] Нет лишних/опасных файлов (рабочий .gitignore)
- [ ] Понимаю и показал разницу reset/revert/restore
- [ ] Репозиторий готов к публикации на GitHub (уровень 3)
