# 00 · Зачем нужен контроль версий 🖼️

> 🎯 **Цель блока:** понять, какую боль решает система контроля версий (VCS) и почему Git — стандарт.

---

## 📖 Боль без контроля версий

```
   как люди «версионируют» файлы без VCS:
   проект_финал.zip
   проект_финал_v2.zip
   проект_финал_ИСПРАВЛЕНО.zip
   проект_финал_ИСПРАВЛЕНО_точно_последний.zip   ← знакомо?

   проблемы:
   • не видно, ЧТО изменилось между версиями и ПОЧЕМУ.
   • нельзя вернуться к конкретному состоянию точно.
   • двое правят один файл → как объединить? копипастой вручную?
   • сломал что-то 3 дня назад — а когда именно? кто? как откатить только это?
```

💡 Контроль версий (VCS) — система, которая **запоминает историю изменений** проекта: кто, что,
когда и зачем менял, и позволяет вернуться к любому состоянию, сравнить версии и объединить работу
нескольких людей. Git — самая популярная VCS в мире.

---

## ⭐ Что даёт контроль версий

```
   📜 ИСТОРИЯ — полный журнал: каждое изменение с автором, датой и описанием (зачем).
   ⏪ ОТКАТ — вернуться к любому прошлому состоянию (вчера, неделю назад, к релизу).
   🔀 ВЕТКИ — параллельные линии работы: фича, эксперимент, исправление — не мешая основному коду.
   👥 СОВМЕСТНАЯ РАБОТА — несколько людей правят проект, Git объединяет изменения (и подсвечивает конфликты).
   🔍 КТО И ПОЧЕМУ — `git blame`: кто написал эту строку и в каком коммите (с описанием причины).
   💾 РЕЗЕРВ — копия истории на сервере (GitHub) и у каждого разработчика.
```

💡 ⭐ Главное: VCS превращает хаос «копий файлов» в **управляемую историю**. Без него командная
разработка невозможна, а сольная — рискованна (нечем откатиться, не видно, что менял).

---

## 📖 Git vs другие VCS

```
   • Git — РАСПРЕДЕЛЁННАЯ VCS: у каждого полная копия истории (работаешь офлайн, быстро). Стандарт.
   • Прежние (SVN, CVS) — ЦЕНТРАЛИЗОВАННЫЕ: история только на сервере (нужна связь, медленнее).
   • Git создан Линусом Торвальдсом (2005) для разработки ядра Linux — быстрый, надёжный, для команд.

   GitHub / GitLab / Bitbucket — это НЕ Git. Это хостинг репозиториев Git + надстройки
   (pull request'ы, ревью, CI, issues). Git работает и без них, локально.
```

💡 Частая путаница: **Git ≠ GitHub**. Git — инструмент контроля версий (на твоём компьютере).
GitHub — сайт, где хранят Git-репозитории и сотрудничают. Можно пользоваться Git вообще без GitHub.
