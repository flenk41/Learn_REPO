# 19 · stash, cherry-pick, reflog 🖼️⭐

> 🎯 **Цель блока:** три точечных инструмента на каждый день — отложить правки (stash), перенести
> один коммит (cherry-pick), найти «потерянное» (reflog).

---

## ⭐ git stash — отложить незакоммиченное

```
   ситуация: работаешь, не готов коммитить, но НАДО срочно переключить ветку (хотфикс).
   git stash               # убрать незакоммиченные правки «в карман», рабочая папка чистая
   ...переключился, сделал срочное...
   git stash pop           # вернуть отложенное обратно (и убрать из стека)

   git stash list          # список заначек
   git stash apply         # вернуть, но ОСТАВИТЬ в стеке (pop = apply + drop)
   git stash -u            # включая неотслеживаемые файлы
   git stash drop          # выкинуть заначку
```

💡 ⭐ Stash — «пауза»: спрятал недоделанное, сделал срочное, вернулся. Альтернатива — просто
закоммитить черновик в свою ветку (часто чище: коммит виден, не забудется в стеке заначек). Не копи
много stash'ей — про них легко забыть.

---

## ⭐ git cherry-pick — перенести один коммит

```
   git cherry-pick <хеш>        # применить ОДИН коммит из другой ветки в текущую (копия с новым хешем)

   зачем:
   • хотфикс из main нужно занести и в release-ветку (перенести именно этот коммит).
   • случайно закоммитил в не ту ветку → cherry-pick в нужную + убрать из неверной.
   • вытащить полезный коммит из заброшенной ветки.

   конфликт при cherry-pick → реши → git cherry-pick --continue (или --abort).
```

💡 ⭐ Cherry-pick — «возьми вон тот коммит сюда». Полезно для точечного переноса (особенно хотфиксов
между релизными ветками). Не злоупотребляй для массового переноса (это работа merge/rebase) — иначе
получишь дубли коммитов в истории.

---

## ⭐⭐ git reflog — журнал спасения

```
   reflog — локальный журнал ВСЕХ перемещений HEAD: коммиты, переключения, reset, rebase, merge.
   даже «удалённое» (после reset --hard, обрыва rebase) живёт здесь ~30-90 дней.

   git reflog                       # история HEAD: a1b2c3d HEAD@{2}: commit: ...
   git reset --hard HEAD@{5}        # вернуть HEAD к состоянию «5 шагов назад»
   git switch -c rescue <хеш>       # создать ветку на «потерянном» коммите из reflog
```

🖼️
```
   «всё пропало» после git reset --hard:
   git reflog                       → находишь хеш коммита ДО reset
   git reset --hard <тот хеш>       → вернулся. ничего не потеряно.
```

💡 ⭐⭐ Reflog — **страховочный трос Git**: пока коммит был в HEAD, его можно вернуть, даже после
«разрушительных» команд. Главный вывод: **не паникуй** — почти всё восстановимо (подробнее про аварии
— модуль 23). Это то, что превращает страх «сломать Git» в спокойствие.

---

## 📖 Когда что

```
   stash       — временно отложить грязную рабочую папку.
   cherry-pick — перенести КОНКРЕТНЫЙ коммит между ветками.
   reflog      — найти/восстановить «потерянное» (после reset/rebase/удаления ветки).
```

---

## ⚠️ Ловушки

- ❌ Копить десятки stash'ей и забывать о них (лучше коммить черновик в ветку).
- ❌ Cherry-pick'ить пачками вместо merge/rebase (дубли коммитов, путаная история).
- ❌ Забывать про reflog и паниковать после reset --hard.
- ❌ Думать, что reflog бесконечен (записи истекают; восстанавливай вовремя).
- ❌ Stash pop при конфликте без понимания (реши конфликт, заначка останется до drop).

---

## ✅ Задачи

1. Начни правку, сделай `git stash`, переключись и сделай «хотфикс», вернись и `git stash pop`.
2. Сделай коммит в ветке A, перенеси его в ветку B через `cherry-pick`. Сравни хеши.
3. ⭐ Закоммить «не в ту ветку», исправь: cherry-pick в нужную + reset в неверной.
4. ⭐⭐ Сделай `git reset --hard`, потом восстанови «потерянный» коммит через `git reflog`.
5. Сформулируй, когда выбрать stash, cherry-pick, reflog.
