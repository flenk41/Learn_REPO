# 23 · Восстановление и аварии 🖼️⭐⭐

> 🎯 **Цель блока:** собрать «аптечку» Git — что делать в типичных «катастрофах». Главный вывод
> трека: **почти всё восстановимо, не паникуй**.

---

## ⭐⭐ Принцип: Git почти ничего не теряет

```
   коммит, который был достижим из HEAD/ветки, живёт в reflog ~30-90 дней даже после «удаления».
   объекты Git удаляются только сборкой мусора (gc), и то — недостижимые и старые.

   первая реакция на «всё пропало»:  🛑 не делай резких движений → git status, git reflog → найди хеш.
```

💡 ⭐⭐ Спокойствие — главный навык. Большинство «катастроф» — это «HEAD/ветка указывает не туда»,
а нужные коммиты целы и находятся в `git reflog`. Восстановление = «вернуть указатель на нужный коммит».

---

## ⭐⭐ Аптечка: типичные ситуации

```
   «сделал reset --hard, потерял коммиты»:
      git reflog → найди хеш до reset → git reset --hard <хеш>

   «удалил ветку с нужной работой»:
      git reflog → найди последний коммит ветки → git switch -c имя <хеш>

   «закоммитил не в ту ветку»:
      git switch нужная; git cherry-pick <хеш>;  затем в неверной: git reset --hard HEAD~1

   «нужно отменить ОПУБЛИКОВАННЫЙ коммит»:
      git revert <хеш>        # новый коммит-отмена (не переписывает историю)

   «испортил последний коммит / забыл файл»:
      git commit --amend       # (только если НЕ запушен)

   «застрял в середине merge/rebase»:
      git merge --abort   /   git rebase --abort      # вернуться как было

   «случайно удалил незакоммиченные правки файла» (git restore затёр):
      ⚠️ если НИКОГДА не было в коммите/staging — Git не поможет (нужен бэкап/IDE local history).
```

💡 ⭐⭐ Запомни связку: **`git status` (где я) → `git reflog` (где были коммиты) → вернуть указатель**.
Это решает большинство аварий. Важное исключение: **никогда не коммитившиеся** правки (выкинутые
`restore`/`reset --hard` из рабочей папки) Git восстановить НЕ может — тут спасает только локальная
история IDE или бэкап. Поэтому коммить чаще.

---

## ⭐ Опасные команды — осознанно

```
   осторожно (могут терять/переписывать):
   git reset --hard          # стирает незакоммиченное
   git clean -fd             # УДАЛЯЕТ неотслеживаемые файлы/папки (физически! сначала git clean -n — показать)
   git push --force          # затирает удалённую историю (используй --force-with-lease, не в общих ветках)
   git rebase / commit --amend на опубликованном → переписывает общую историю

   привычки безопасности:
   • git status / git clean -n (dry-run) ПЕРЕД разрушительным действием.
   • коммить часто (закоммиченное почти не теряется).
   • пуши важное (бэкап в облаке).
```

💡 ⭐ Перед «разрушительной» командой — пауза и `git status`/dry-run (`-n`). Самые болезненные потери —
от `git clean -fd` (удаляет файлы, которых Git вообще не знал) и `reset --hard` по незакоммиченному.
Их Git вернуть не сможет.

---

## 📖 Чистка истории от секрета/большого файла

```
   секрет/огромный файл попал в историю (не только последний коммит):
   • git filter-repo (рекомендуется) или BFG Repo-Cleaner — переписать историю, убрав файл.
   ⚠️ это переписывает ВСЮ историю (новые хеши) → согласуй с командой, всем нужно переклонировать.
   ⚠️ для СЕКРЕТА: сначала ОТЗОВИ/смени ключ (он утёк), потом чисти историю (модуль 06).
```

---

## ⚠️ Ловушки

- ❌ Паниковать и делать новые резкие команды (сначала `status` + `reflog`).
- ❌ `git clean -fd` без `-n` (dry-run) → удалил нужные неотслеживаемые файлы безвозвратно.
- ❌ Думать, что `restore`/`reset --hard` по незакоммиченному обратимы (нет — коммить чаще).
- ❌ Чистить секрет из истории, но не ротировать ключ (он всё равно утёк).
- ❌ `filter-repo`/`force-push` общей истории без согласования с командой.

2. Сделай `reset --hard`, потеряв коммиты, и восстанови через `reflog`.
3. Удали ветку с работой и восстанови её из `reflog`.
4. ⭐ Перед `git clean -fd` сделай `git clean -n` — что бы удалилось? Прочувствуй опасность.
5. ⭐⭐ Опиши план действий, если секрет попал в историю и был запушен (по шагам: ротация → чистка → команда).
