# 22 · Большие репозитории и bisect 🖼️⭐

> 🎯 **Цель блока:** инструменты для масштаба и расследований — найти коммит, внёсший баг (bisect),
> и приёмы для больших репозиториев (submodules, monorepo, частичные клоны).

---

## ⭐⭐ git bisect — двоичный поиск бага

```
   баг появился где-то на последних 200 коммитах, но КОГДА — неизвестно. bisect находит за ~log₂(200)≈8
   проверок (двоичный поиск по истории!).

   git bisect start
   git bisect bad                 # текущий коммит — БАГ есть
   git bisect good v1.2.0         # вот этот (старый) — баг ещё НЕ было
   → Git прыгает на СЕРЕДИНУ диапазона. ты проверяешь (запускаешь/тестишь):
   git bisect good                # на этом коммите бага нет → ищем дальше (новее)
   git bisect bad                 # на этом баг есть → ищем раньше (старее)
   ... повторяешь ~8 раз ...
   → Git называет ПЕРВЫЙ плохой коммит (где баг внесён).
   git bisect reset               # выйти, вернуться на исходную ветку
```

🖼️
```
   good ───────────────?───────────────── bad
   v1.2.0          ↑ проверяем середину        HEAD
   нет бага      good/bad → сужаем вдвое      есть баг
   200 коммитов → ~8 проверок вместо 200.
```

💡 ⭐⭐ Bisect — суперсила отладки регрессий: вместо «читать 200 коммитов» — 8 проверок «работает/нет».
А если есть автотест на баг — `git bisect run ./test.sh` найдёт виновный коммит **полностью
автоматически**. Это окупает аккуратные атомарные коммиты (модуль 07): чем мельче коммит, тем точнее
bisect укажет причину.

## ⭐ Submodules: репо внутри репо

```
   SUBMODULE — вложенный Git-репозиторий, закреплённый на конкретном коммите (зависимость как код).
   git submodule add <url> libs/foo     # добавить
   git clone --recurse-submodules <url> # клонировать вместе с субмодулями
   git submodule update --init --remote # подтянуть/обновить

   ⚠️ субмодули неудобны (легко забыть обновить, путаница с коммитами). часто предпочитают
   пакетные менеджеры или monorepo. используй осознанно.
```

---

## 📖 Monorepo и масштаб

```
   MONOREPO — много проектов/пакетов в ОДНОМ репозитории (так делают Google, Meta).
   + общий код/рефакторинг через границы, единая история, атомарные кросс-проектные изменения.
   − большой размер, нужны спец-инструменты (sparse-checkout, build-системы Bazel/Nx).

   приёмы для больших/тяжёлых репо:
   git clone --depth 1 <url>            # SHALLOW clone — только последний коммит (быстро, без истории)
   git clone --filter=blob:none <url>   # partial clone — история без всех файлов (подтянет по нужде)
   git sparse-checkout set dir/         # выгрузить только часть дерева
   git gc                               # сборка мусора, уплотнение репозитория
```

💡 ⭐ Для CI и быстрых проверок — `--depth 1` (shallow): не тянет всю историю, экономит время/трафик.
Для огромных монорепо — partial clone + sparse-checkout. Большинству проектов это не нужно, но знать
полезно, встретив гигантский репозиторий.

## ⚠️ Ловушки

- ❌ Искать баг чтением сотен коммитов вместо `git bisect`.
- ❌ `bisect` без чёткого критерия «good/bad» (нужен воспроизводимый тест/проверка).
- ❌ Забыть `git bisect reset` (останешься в detached HEAD посреди истории).
- ❌ Субмодули «забыл обновить» → собираешь старую версию зависимости.
- ❌ Клонировать гигантский монорепо целиком, когда хватит shallow/sparse.
