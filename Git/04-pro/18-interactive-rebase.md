# 18 · Интерактивный rebase 🖼️⭐⭐

> 🎯 **Цель блока:** причёсывать свою историю перед публикацией — объединять, переименовывать,
> переставлять и редактировать коммиты. Превратить черновую работу в чистую серию.

---

## ⭐⭐ Что это и зачем

```
   git rebase -i HEAD~5        # интерактивно переписать последние 5 коммитов

   откроется список коммитов с командами — ты решаешь, что с каждым сделать:

   pick   a1b2c3d Добавить парсер
   squash d4e5f6a исправить опечатку     ← влить в предыдущий (объединить)
   reword 9z8y7x6 Добавить тесты         ← изменить сообщение
   drop   1a2b3c4 отладочный print       ← выкинуть коммит

   команды: pick (оставить), reword (сменить сообщение), edit (остановиться и поправить),
            squash (слить в предыдущий + объединить сообщения), fixup (слить, отбросив сообщение),
            drop (удалить), а также ПЕРЕСТАВИТЬ строки = переставить коммиты.
```

💡 ⭐⭐ Интерактивный rebase — инструмент **чистой истории**: «накоммитил черновиков (wip, fix typo,
debug) → перед PR собрал в 2-3 осмысленных коммита». Ревьюеру видна аккуратная серия, а не твой
поток сознания. Это уважение к читателю истории.

---

## ⭐ Частые сценарии

```
   СЖАТЬ мелочь в один коммит (squash/fixup):
   pick   Добавить фичу
   fixup  исправить опечатку      → опечатка вливается в «Добавить фичу», её сообщение отброшено

   ПЕРЕИМЕНОВАТЬ плохое сообщение:
   reword коммит → откроется редактор → пиши нормальное сообщение

   ВЫКИНУТЬ отладочный коммит:
   drop коммит с debug-print

   ПЕРЕСТАВИТЬ порядок:
   просто поменяй строки местами.

   завершение: сохрани и закрой редактор → Git применит. конфликт → реши → git rebase --continue.
```

💡 ⭐ `git commit --fixup=<хеш>` + `git rebase -i --autosquash` — автоматизирует сжатие правок в
нужные коммиты (помечает fixup'ы и расставляет сам). Удобно для «допиливания» конкретного коммита.

---

## ⭐⭐ Золотое правило (снова)

```
   ⚠️⚠️ интерактивный rebase ПЕРЕПИСЫВАЕТ коммиты (новые хеши) → применяй ТОЛЬКО к локальной,
   ещё НЕ запушенной истории. (то же правило, что для обычного rebase — модуль 11.)

   если причесал УЖЕ запушенную свою ветку (только свою!):
   git push --force-with-lease     # безопаснее --force: откажет, если кто-то успел запушить
   ❌ никогда не делай это с общими ветками (main/develop).
```

💡 ⭐⭐ Идеальный момент для interactive rebase — **перед открытием PR**: причесал локально → запушил
чистую серию. Если ветка уже в PR и только твоя — `--force-with-lease` обновит её (ревью переедет на
новые коммиты). Общую историю — никогда.

---

## 📖 Альтернатива: --amend

```
   для правки ТОЛЬКО последнего коммита interactive rebase избыточен:
   git commit --amend                 # добавить staged-правки / изменить сообщение последнего коммита
   git commit --amend --no-edit       # дописать забытый файл в последний коммит, не меняя сообщение
```

---

## ⚠️ Ловушки

- ❌❌ Interactive rebase на опубликованной/общей истории (ломает коллегам).
- ❌ `git push --force` (без lease) — может затереть чужой push; используй `--force-with-lease`.
- ❌ Запутаться в большом списке (rebase -i на 50 коммитов) — делай малыми порциями.
- ❌ Бросить rebase на полпути в странном состоянии (`git rebase --abort` вернёт как было).
- ❌ Сквошить ВСЁ в один гигантский коммит (теряешь полезную гранулярность).

## ✅ Задачи

1. Сделай 5 черновых коммитов (включая «fix typo», «debug»). Через `rebase -i` собери их в 2 осмысленных.
2. Переименуй плохое сообщение через `reword`.
3. Выкинь отладочный коммит через `drop`.
4. ⭐ Используй `git commit --fixup` + `rebase -i --autosquash`.
5. ⭐ Допиши забытый файл в последний коммит через `git commit --amend --no-edit`.
