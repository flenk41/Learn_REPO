# 20 · Теги, релизы, semver 🖼️⭐

> 🎯 **Цель блока:** помечать версии тегами и понимать семантическое версионирование (semver) —
> язык, на котором софт сообщает о совместимости.

---

## ⭐ Теги: метки на коммитах

```
   ТЕГ — постоянная именованная метка на коммите (обычно версия релиза). В отличие от ветки, тег
   НЕ двигается — он навсегда указывает на этот коммит.

   git tag v1.2.0                       # лёгкий тег на текущий коммит
   git tag -a v1.2.0 -m "Релиз 1.2.0"   # аннотированный тег (с автором/датой/сообщением) — для релизов
   git tag                              # список тегов
   git show v1.2.0                      # что в теге/коммите
   git push origin v1.2.0               # отправить тег на сервер (обычные push НЕ шлют теги!)
   git push origin --tags               # отправить все теги
   git checkout v1.2.0                  # перейти к состоянию релиза (detached HEAD)
```

💡 ⭐ Для релизов используй **аннотированные теги** (`-a`): в них автор, дата, сообщение — это полноценный
объект. И помни: `git push` сам по себе **не отправляет теги** — нужен явный `push origin --tags` или
конкретный тег. Тег = «вот это состояние было версией X».

## ⭐⭐ Semantic Versioning (semver)

```
   версия MAJOR.MINOR.PATCH   (например 2.4.1):

   MAJOR (2) — ЛОМАЮЩИЕ изменения (несовместимо со старым API). 1.x.x → 2.0.0.
   MINOR (4) — новая функциональность, СОВМЕСТИМАЯ назад. 2.3.x → 2.4.0.
   PATCH (1) — исправления багов, без новых фич, совместимо. 2.4.0 → 2.4.1.

   правило: повышай ту часть, что отражает СУТЬ изменения для пользователя твоего кода.
   0.x.y — «всё ещё нестабильно», ломающие изменения допустимы в minor.
   суффиксы: 2.4.1-beta.1, 2.4.1-rc.2 (пред-релизы).
```

🖼️
```
   2  .  4  .  1
   │     │     └─ PATCH: багфикс, безопасно обновляться
   │     └─────── MINOR: новые фичи, безопасно (обратно совместимо)
   └───────────── MAJOR: ломающее, читай changelog перед обновлением!
```

💡 ⭐⭐ Semver — **контракт совместимости**: по номеру версии потребитель твоей библиотеки понимает,
безопасно ли обновляться. PATCH/MINOR — обычно безопасно; MAJOR — жди ломающих изменений, читай
migration guide. Поэтому ломающее изменение ОБЯЗАНО поднимать MAJOR — иначе ты подведёшь зависящих от
тебя (связь с [Conventional Commits](../01-basics/07-good-commits.md): `feat!`/`BREAKING CHANGE` → major).

## 📖 Релизы на GitHub

```
   GitHub Releases — поверх тегов: страница релиза с changelog, бинарниками, заметками.
   • создаёшь Release из тега → пишешь «что нового» → прикладываешь артефакты.
   • changelog часто генерируют автоматически из Conventional Commits / PR.
   • пользователи видят историю версий, скачивают сборки.

   CHANGELOG.md — человекочитаемый файл изменений по версиям (Keep a Changelog).
```

> 🧭 Автосборка артефактов и публикация релиза по тегу — задача CI/CD (будущий DevOps-трек): «push
> тега v* → собрать → выложить Release».

## ⚠️ Ловушки

- ❌ Думать, что `git push` отправляет теги (нет — нужен `--tags` или явный тег).
- ❌ Ломающее изменение без поднятия MAJOR (подставляешь зависящих от тебя).
- ❌ Лёгкие теги для релизов вместо аннотированных (теряешь автора/дату/сообщение).
- ❌ Двигать/переписывать опубликованный тег (он должен быть постоянным).
- ❌ Версии «от балды» без semver (пользователь не знает, безопасно ли обновляться).

## ✅ Задачи

1. Поставь аннотированный тег `v1.0.0` на коммит, отправь на сервер (`push origin v1.0.0`).
2. Для трёх изменений (багфикс, новая фича, ломающее API) скажи, какая часть semver растёт.
3. ⭐ Создай Release на GitHub из тега с заметками «что нового».
4. ⭐ Заведи `CHANGELOG.md` и опиши пару версий в формате Keep a Changelog.
5. Объясни, почему ломающее изменение обязано поднимать MAJOR.
