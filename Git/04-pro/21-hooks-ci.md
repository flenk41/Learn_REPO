# 21 · Хуки и связь с CI 🖼️⭐

> 🎯 **Цель блока:** автоматизировать проверки вокруг Git — локальные хуки (перед коммитом) и
> серверный CI (на каждый PR). Сделать так, чтобы «плохой» код было трудно влить.

> 🧭 Полноценный CI/CD — будущий трек DevOps; здесь — стык Git с автоматизацией.

---

## ⭐ Git-хуки: скрипты на события

```
   ХУК — скрипт, который Git запускает на определённое событие (в .git/hooks/). Если хук «провалился»
   (ненулевой код) — операция отменяется.

   частые ЛОКАЛЬНЫЕ (клиентские) хуки:
   • pre-commit   — перед созданием коммита: линтер, формат, быстрые тесты, поиск секретов.
                    провалился → коммит не создаётся (чинишь и повторяешь).
   • commit-msg   — проверить формат сообщения (например, Conventional Commits).
   • pre-push     — перед push: прогнать тесты/сборку.

   серверные (на хостинге) — обычно реализуют через CI, не raw-хуки.
```

💡 ⭐ Pre-commit хук ловит проблемы **до** их попадания в историю: «не дать закоммитить незаформатированный
код / упавший линтер / секрет». Но локальные хуки можно обойти (`--no-verify`), поэтому они — удобство,
а не гарантия. Гарантия — серверный CI (ниже).

---

## ⭐ pre-commit framework

```
   .git/hooks/ не версионируется и руками неудобен. Стандарт — инструмент pre-commit (pre-commit.com):

   • файл .pre-commit-config.yaml в репозитории (версионируется, общий для команды).
   • набор готовых проверок: форматтеры (black/prettier), линтеры (ruff/eslint),
     trailing-whitespace, проверка больших файлов, ДЕТЕКТ СЕКРЕТОВ (detect-secrets/gitleaks).
   • pre-commit install — ставит хук; теперь проверки бегут на каждый git commit.

   плюс: вся команда получает одинаковые проверки из репозитория.
```

💡 ⭐ Автоматический детект секретов в pre-commit — дешёвая защита от утечки (модуль 06): хук не даст
закоммитить то, что похоже на ключ/токен. Дополняет серверное сканирование.

---

## ⭐⭐ CI: проверки на сервере

```
   CI (Continuous Integration) — сервер автоматически собирает и проверяет код на каждый push/PR.
   на GitHub — GitHub Actions (workflow в .github/workflows/*.yml).

   типичный CI на PR:
   1. установить зависимости.
   2. собрать проект.
   3. прогнать тесты.
   4. линт/формат/типы.
   5. (опц.) сканирование безопасности/секретов.
   результат: ✅ зелёный / ❌ красный прямо в PR. красный → merge заблокирован (если настроено).
```

🖼️
```
   push в PR ──► GitHub Actions ──► [сборка][тесты][линт][безопасность] ──► ✅/❌ в PR
                                                                            │
                                              merge разрешён только при ✅ ─┘ (+ required review)
```

💡 ⭐⭐ CI — **настоящие ворота качества**: в отличие от локальных хуков, его нельзя обойти `--no-verify`.
«Required checks» + «required review» (модуль 15) = в main попадает только то, что собралось, прошло
тесты и одобрено человеком. Это автоматизированная половина код-ревью.

---

## 📖 Локальные хуки vs CI

```
   ЛОКАЛЬНЫЕ ХУКИ — быстрая обратная связь у разработчика (поймать до коммита/push).
                    удобно, но обходимо (--no-verify) → не гарантия.
   CI НА СЕРВЕРЕ   — авторитетная проверка, обойти нельзя, блокирует merge → ГАРАНТИЯ.
   вместе: хук = «не трать время ревьюера на ерунду»; CI = «в main только проверенное».
```

---

## ⚠️ Ловушки

- ❌ Полагаться ТОЛЬКО на локальные хуки (обходятся `--no-verify`) — нужен серверный CI.
- ❌ Тяжёлый pre-commit (полный прогон тестов на каждый коммит) → раздражает, его отключают. Держи быстрым.
- ❌ Хуки в `.git/hooks/` без версионирования (каждый настраивает руками) — используй pre-commit framework.
- ❌ Красный CI игнорируется/мержится силой (теряется весь смысл).
