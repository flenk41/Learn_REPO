# 17 · Рабочие процессы (workflow) 🖼️⭐

> 🎯 **Цель блока:** узнать распространённые модели ветвления — как команды организуют ветки и
> релизы. Выбор workflow — это договорённость, подходящая под продукт и команду.

---

## ⭐ Зачем нужны соглашения о ветвлении

```
   Git не навязывает процесс — команда ДОГОВАРИВАЕТСЯ, как использовать ветки:
   • где «истина» (main),
   • как заводить фичи,
   • как и когда релизить,
   • что блокирует слияние.

   единый workflow → предсказуемость, меньше хаоса и конфликтов. главное — ОДИН на команду.
```

---

## ⭐⭐ Три популярные модели

```
   1. GITHUB FLOW (простой, самый частый):
      • main всегда деплоится (рабочий).
      • на задачу → ветка от main → коммиты → PR → ревью → merge в main → деплой.
      • короткие ветки, частые слияния. подходит большинству (веб-сервисы, continuous delivery).

   2. TRUNK-BASED (для зрелых команд / CI-CD):
      • все коммитят в main (trunk) очень малыми порциями, минимум короткоживущих веток.
      • фиче-флаги прячут незаконченное. требует сильных автотестов.
      • минимум конфликтов, максимум скорости интеграции.

   3. GIT FLOW (сложный, для версионируемых релизов):
      • долгоживущие ветки: main (релизы) + develop (интеграция) + feature/* + release/* + hotfix/*.
      • строгий, тяжеловесный. оправдан для софта с явными версиями (десктоп, библиотеки, поставки),
        НО для веба/continuous delivery часто избыточен.
```

🖼️
```
   GitHub Flow (рекомендуемый по умолчанию):

   main  ●────●────────●────────●────►  (всегда деплоим)
          \           /        /
   feat    ●──●──●──PR        /
   fix              ●──●────PR
```

💡 ⭐⭐ Для большинства проектов — **GitHub Flow**: main всегда рабочий, фичи через короткие ветки и
PR. Trunk-based — эволюция для команд с сильным CI. Git Flow мощный, но тяжёлый — бери его только при
реальной необходимости версионных релизов (частая ошибка — тащить Git Flow в простой веб-проект).
Это [trade-off](../../Senior/02-decisions/08-tradeoffs.md): сложность процесса против потребностей релизов.

---

## ⭐ Общие практики поверх workflow

```
   • main ЗАЩИЩЁН — нельзя пушить напрямую, только через PR с ревью и зелёным CI.
   • ВЕТКИ КОРОТКОЖИВУЩИЕ — день-два, не недели (меньше расхождений/конфликтов).
   • ИМЕНОВАНИЕ ветвей: feature/login, fix/crash-on-empty, chore/bump-deps.
   • CI на каждый PR — тесты/линт/сборка автоматически (трек DevOps).
   • РЕЛИЗЫ — тегами и версиями (semver, модуль 20).
```

💡 ⭐ Независимо от модели: защищай main, держи ветки короткими, гоняй CI на PR. Эти практики важнее
выбора конкретного «флоу». Короткие ветки — главный приём против конфликтов (модуль 10).

---

## 📖 Связь с CI/CD и релизами

```
   workflow смыкается с доставкой (DevOps-трек):
   • merge в main → CI собирает/тестирует → CD деплоит (continuous delivery).
   • в Git Flow релиз — отдельная ветка/тег; в GitHub Flow — каждый merge в main потенциально релиз.
   • фиче-флаги (trunk-based) разделяют «влито в код» и «включено для пользователей».
```

> 🧭 Как именно автоматизируют сборку/тесты/деплой по этим веткам — будущий трек DevOps/CI-CD.

---

## ⚠️ Ловушки

- ❌ Тащить тяжёлый Git Flow в простой проект (оверинжиниринг процесса).
- ❌ Долгоживущие ветки (расходятся → ад слияния).
- ❌ Незащищённый main (любой пушит напрямую → нестабильность).
- ❌ Разные люди — разные неписаные правила (договоритесь об ОДНОМ workflow).
- ❌ Безымянные/хаотичные имена веток.
