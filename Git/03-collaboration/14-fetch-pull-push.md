# 14 · Fetch, pull, push, tracking 🖼️⭐⭐

> 🎯 **Цель блока:** освоить обмен с сервером — забрать чужие изменения (fetch/pull) и отдать свои
> (push). Понять разницу fetch и pull, чтобы не запутаться.

---

## ⭐⭐ Три команды обмена

```
   git fetch       # ЗАБРАТЬ изменения с сервера в origin/*, но НЕ трогать твои ветки/файлы.
                  #   безопасно: просто обновляет «знание о сервере». Потом сам решаешь, что делать.

   git pull        # fetch + слить (merge или rebase) в ТЕКУЩУЮ ветку = забрать И применить сразу.
                  #   pull = fetch + merge origin/main  (или + rebase, если настроено)

   git push        # ОТДАТЬ свои коммиты на сервер (в соответствующую ветку origin).
```

🖼️
```
   fetch:   сервер ──► origin/main (обновилось)     твой main НЕ тронут — смотри сам.
   pull:    сервер ──► origin/main ──► слито в твой main  (за один шаг)
   push:    твой main ──► сервер  (если сервер не ушёл вперёд)
```

💡 ⭐⭐ `fetch` безопасен и предсказуем: смотришь, что изменилось (`git log main..origin/main`), потом
сливаешь осознанно. `pull` — то же, но сразу применяет (быстрее, но может неожиданно слить/конфликтовать).
Senior-привычка: `fetch` + посмотреть + слить, особенно на важных ветках. Для рутины `pull` ок.

## ⭐ Push и tracking

```
   первый push новой ветки (связать локальную с удалённой):
   git push -u origin feature       # -u (=--set-upstream) запоминает связь
   далее достаточно: git push / git pull (Git знает, куда)

   git push                         # отдать текущую ветку (если upstream настроен)
   git push origin --delete feature # удалить ветку на сервере

   TRACKING (upstream) — связь «локальная ветка ↔ ветка на сервере». После -u Git знает пару,
   показывает «ahead 2, behind 1» (на сколько коммитов впереди/позади сервера).
```

💡 ⭐ `git push -u origin <branch>` при первой публикации ветки — потом просто `push`/`pull`. Статус
«ahead/behind» (в `git status`) подсказывает: ahead → надо push, behind → надо pull.

## ⭐⭐ Отклонённый push (non-fast-forward)

```
   ❌ ! [rejected] ... (fetch first)
   причина: сервер УШЁЛ ВПЕРЁД (коллега запушил, пока ты работал). Git не даёт затереть чужое.

   правильно:
   git pull            # забрать чужие изменения (слить/ребейзнуть) → решить конфликты, если есть
   git push            # теперь отдать

   pull со слиянием vs ребейзом:
   git pull            # по умолчанию merge (создаёт merge-коммит при расхождении)
   git pull --rebase   # наложить твои коммиты ПОВЕРХ чужих (линейно, без merge-коммита)

   ⚠️ НИКОГДА git push --force в общую ветку (затрёшь чужую работу!). 
      только --force-with-lease и только для СВОЕЙ ветки после rebase.
```

💡 ⭐⭐ Отклонённый push — норма в команде: значит, сервер обновился. **Никогда не «продавливай»
через `--force`** в общую ветку — ты сотрёшь чужие коммиты. Решение всегда: `pull` (забрать) → решить
конфликты → `push`. `--force` допустим только для своей личной ветки после rebase, и то с `--force-with-lease`.

## 📖 Типичный ежедневный цикл

```
   утром:    git pull                          # подтянуть свежий main
   работа:   git switch -c feature/x; ...коммиты...
   обновить: git fetch; git rebase origin/main # (опц.) актуализировать свою ветку
   отдать:   git push -u origin feature/x      # опубликовать
   далее:    Pull Request → ревью → merge (модуль 15)
```

---

## ⚠️ Ловушки

- ❌❌ `git push --force` в общую ветку (затирает чужую работу — почти всегда ошибка).
- ❌ Путать fetch (только забрать в origin/*) и pull (забрать + слить).
- ❌ Игнорировать «rejected push» и пытаться форсить вместо pull.
- ❌ Забыть `-u` при первом push (потом `push` не знает, куда).
- ❌ Делать pull с незакоммиченными правками и попасть в неожиданный merge/конфликт.
