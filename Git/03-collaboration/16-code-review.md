# 16 · Код-ревью: автор и ревьюер 🖼️⭐⭐

> 🎯 **Цель блока:** научиться проводить и проходить код-ревью — главную практику качества и обмена
> знаниями в команде. Это и техника, и культура общения.

> 🧭 Глубже про культуру ревью — [Senior-ремесло](../../Senior/01-craft/06-code-review.md); здесь —
> с фокусом на Git/PR-процесс.

---

## ⭐⭐ Зачем код-ревью

```
   КОД-РЕВЬЮ — проверка изменений другим человеком перед вливанием в main. даёт:
   • КАЧЕСТВО — ловит баги, упущения, плохие решения ДО продакшена (дёшево).
   • ЗНАНИЕ — распространяет понимание кода по команде (не один знает модуль).
   • СТАНДАРТЫ — единый стиль, подходы, договорённости.
   • НАСТАВНИЧЕСТВО — junior учится у senior и наоборот, через конкретный код.

   ревью — НЕ про «поймать и наказать», а про общее качество и обучение.
```

💡 ⭐⭐ Главная ценность ревью — не только баги, но **общее владение кодом и обучение**. Команда, где
ревьюят, знает свою кодовую базу коллективно (нет «только Вася понимает этот модуль» — bus factor).
Это инвестиция в качество и в людей, а не бюрократия.

---

## ⭐⭐ Как РЕВЬЮЕР: что и как смотреть

```
   ЧТО проверять (по приоритету):
   1. КОРРЕКТНОСТЬ — работает ли? есть ли баги, краевые случаи, гонки?
   2. БЕЗОПАСНОСТЬ — инъекции, секреты, валидация ввода (трек безопасности).
   3. ЯСНОСТЬ — поймёт ли это другой через полгода? имена, структура.
   4. ТЕСТЫ — покрыто ли изменение? проходят ли?
   5. ДИЗАЙН — вписывается в архитектуру? нет ли дублирования?
   ❌ НЕ придирайся к стилю, который проверяет линтер (автоматизируй формат — не трать ревью на это).

   КАК комментировать (тон):
   ✅ «Что если name == null? Похоже, тут упадёт» (конкретно, по делу).
   ✅ «Можно вынести в функцию — переиспользуется в X» (предложение с причиной).
   ✅ «nit: опечатка» (помечай мелочь как необязательную — nitpick).
   ❌ «Это ужасно», «ты не умеешь» (критикуй КОД, не человека).
```

💡 ⭐⭐ Правило ревьюера: **критикуй код, а не человека**, объясняй ПОЧЕМУ, предлагай решение, отделяй
важное (баг) от вкусовщины (помечай `nit:`). Одобряй, когда «достаточно хорошо», а не «идеально» —
перфекционизм блокирует поставку. Быстрое ревью важнее идеального: PR, висящий 3 дня, тормозит всех.

---

## ⭐⭐ Как АВТОР: принимать ревью

```
   ✅ не воспринимай правки как личную атаку — это про КОД, общая цель — качество.
   ✅ отвечай на каждый комментарий: исправил / объяснил, почему оставил / обсудим.
   ✅ не согласен — аргументируй спокойно (ревью = диалог, не приказ); договоритесь.
   ✅ облегчи ревьюеру: маленький PR, описание, сам прочитай свой дифф (модуль 15).
   ✅ спорные вопросы вкуса — уступай легко; принципиальные (баг/безопасность) — отстаивай по сути.
```

💡 ⭐⭐ Отделяй себя от кода: ревью твоего PR — не оценка тебя как человека. Это самый быстрый способ
вырасти — опытные люди указывают на то, чего ты не видел. Реагируй на содержание, не на тон.

---

## 📖 Механика на GitHub

```
   • комментарии К СТРОКАМ диффа (inline) — конкретно к месту.
   • Suggestion — предложить точную замену кода кнопкой (автор применяет в один клик).
   • Review итог: Approve (одобрить) / Request changes (нужны правки) / Comment (без вердикта).
   • Resolve conversation — пометить обсуждение решённым.
   • автор пушит новые коммиты в ту же ветку → PR и ревью обновляются.
```

---

## 📖 Здоровая культура ревью

```
   • ревьюй БЫСТРО (в течение дня) — не блокируй коллег.
   • предполагай добрые намерения с обеих сторон.
   • хвали хорошее, не только указывай на плохое.
   • большие споры — голосом/в чате, не в ветке из 40 комментариев.
   • договоритесь о правилах (размер PR, сколько ревьюеров, что блокирует merge).
```

> 🧭 Это пересекается с [Senior-коммуникацией](../../Senior/04-leadership/19-communication.md) и
> [влиянием без власти](../../Senior/04-leadership/22-influence.md): ревью — про людей не меньше, чем про код.

---

## ⚠️ Ловушки

- ❌ Переходить на личности («ты плохой код пишешь») вместо обсуждения кода.
- ❌ Блокировать PR из-за вкусовщины (отделяй `nit:` от важного; автоматизируй стиль).
- ❌ Затягивать ревью на дни (тормозит команду).
- ❌ Штамповать «LGTM» не глядя (ревью ради галочки бесполезно).
- ❌ Как автор — обижаться/защищаться вместо обсуждения по сути.
- ❌ Огромные PR (невозможно ревьюить вдумчиво — см. модуль 15).

---

## ✅ Задачи

1. Открой PR и сам себя «отревьюй»: пройди по диффу построчно, оставь комментарии, что улучшить.
2. Возьми чужой (или учебный) PR и напиши 3 комментария: один про возможный баг, один про ясность,
   один `nit:`. Соблюди тон.
3. Перепиши 3 «плохих» ревью-комментария (грубых/расплывчатых) в конструктивные.
4. ⭐ Используй GitHub Suggestion, чтобы предложить точную правку.
5. ⭐ Сформулируй чек-лист ревьюера для своей команды (что смотреть, что игнорировать).

---

## ❓ Проверь себя

1. Зачем нужно код-ревью (кроме поиска багов)?
2. Что проверять ревьюеру и в каком приоритете?
3. Как давать обратную связь конструктивно (код, не человек; почему; nit)?
4. Как автору правильно принимать ревью?
