# 24 · DevSecOps и безопасный SDLC 🖼️⭐

> 🎯 **Цель блока:** понять, как встроить безопасность во весь процесс разработки (а не «проверить
> в конце»), чтобы дыры ловились рано и дёшево.

---

## 📖 «Сдвиг влево»: безопасность раньше

```
   стоимость исправления уязвимости растёт по ходу:
   на этапе кода — дёшево; в тестировании — дороже; в проде после взлома — катастрофа.
   DEVSECOPS = встроить безопасность во ВСЕ этапы разработки («сдвиг влево»), а не «безопасники
   проверят в конце». безопасность — общая ответственность, часть конвейера.
```

🖼️
```
   стоимость фикса:
   [дизайн] < [код] < [тест] < [прод] < [после взлома]
      ¢         $       $$       $$$        💥
   → лови дыры КАК МОЖНО РАНЬШЕ. «сдвиг влево» = дешевле и безопаснее.
```

💡 ⭐ Это [Senior-принцип «дешевле исправить рано»](../../Senior/03-practices/16-requirements.md),
применённый к безопасности. И [insecure design из OWASP](../02-vulnerabilities/14-owasp-top10.md):
безопасность закладывается с проектирования, не прикручивается потом.

## ⭐ Безопасность на каждом этапе SDLC

```
   ПРОЕКТИРОВАНИЕ → модель угроз (модуль 04), безопасные требования, выбор безопасных подходов.
   КОД           → принципы безопасного кода (Уровень 3), безопасные шаблоны, обучение разработчиков.
   РЕВЬЮ         → ревью с фокусом на безопасность (OWASP-чек-лист), линтеры безопасности.
   СБОРКА/CI     → автоматические проверки (см. ниже).
   ТЕСТ          → тесты безопасности, сканирование, иногда пентест перед релизом.
   ДЕПЛОЙ        → hardening (модуль 22), секреты из менеджера, безопасная конфигурация.
   ЭКСПЛУАТАЦИЯ  → мониторинг (модуль 21), реагирование (модуль 23), обновления.
```

## ⭐⭐ Автоматизация в CI/CD (конвейер безопасности)

```
   встрой проверки безопасности в пайплайн — они гоняются на КАЖДЫЙ коммит/PR автоматически:

   ✅ SAST — статический анализ кода на уязвимости (линтеры безопасности).
   ✅ СКАНЕР СЕКРЕТОВ — ловит случайно закоммиченные ключи ДО пуша (модуль 19).
   ✅ SCA — аудит зависимостей на известные уязвимости (модуль 19).
   ✅ DAST — динамическое сканирование запущенного приложения (в тестовой среде).
   ✅ сканирование образов контейнеров; проверка IaC-конфигов.
   ✅ «ворота качества»: критичная уязвимость → сборка не проходит (как падающие тесты).
```

🖼️
```
   коммит → CI: [SAST][секреты][зависимости][тесты безопасности] → ❌ нашли критичное? сборка падает
                                                                  → ✅ чисто? деплой
   безопасность как тесты: автоматически, на каждое изменение, блокирует плохое.
```

💡 ⭐⭐ Ключевая идея: **безопасность как часть CI, как тесты**. Не «иногда проверим вручную», а
автоматически на каждое изменение, с блокировкой критичного. Это масштабируемо и не зависит от
«вспомнил/не вспомнил». Твои защитные инструменты (модуль 20) — кирпичи такого конвейера.

## 📖 Культура: безопасность — общая

```
   DevSecOps — не только инструменты, но и КУЛЬТУРА:
   • безопасность — ответственность КАЖДОГО (разработчик, не только «безопасник»).
   • обучение разработчиков (этот трек!) — чтобы дыры не появлялись.
   • security champions в командах — энтузиасты, помогающие коллегам.
   • без наказания за найденные уязвимости/ошибки → их сообщают, а не прячут.
   • безопасность не «тормоз», а встроенная часть качества (как тесты).
```

> 🧭 Та же [культура без наказания](../../SocialEng/04-defense/23-security-culture.md) и
> [влияние без власти](../../Senior/04-leadership/22-influence.md): безопасность растёт через
> людей и привычки, не только тулзы.

---

## ⚠️ Ловушки

- ❌ «Безопасники проверят в конце» → дыры находят поздно и дорого.
- ❌ Ручные проверки «когда вспомним» вместо автоматизации в CI.
- ❌ Не блокировать сборку на критичных находках (предупреждения игнорируют).
- ❌ Считать безопасность «тормозом» вместо части качества.
- ❌ Сваливать всё на отдельных «безопасников», а не на команду.
- ❌ Инструменты без культуры (или культура без инструментов) — нужны оба.

## ✅ Упражнения

1. **Сдвиг влево.** Для своего процесса: на каких этапах сейчас думаешь о безопасности? Где можно
   раньше?
2. **CI-проверки.** Встрой в свой пайплайн хотя бы 2: сканер секретов + аудит зависимостей. Падает
   ли сборка на находке?
3. **Ревью.** Сделай для команды/себя короткий чек-лист безопасности для код-ревью (по OWASP).
4. **Ворота.** Определи, какие находки должны блокировать релиз, а какие — предупреждать.
