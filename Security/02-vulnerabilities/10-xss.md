# 10 · XSS — межсайтовый скриптинг 🖼️⭐⭐

> 🎯 **Цель блока:** понять XSS (внедрение скрипта в страницу) и научиться защищаться —
> экранированием вывода, CSP и безопасной работой с DOM.

> ⚠️ Практика — только на своих/учебных приложениях. Цель — защита.

---

## 📖 Суть XSS: ввод стал скриптом в браузере

```
   XSS (Cross-Site Scripting) — атакующий внедряет свой JavaScript в страницу, которую видят
   ДРУГИЕ пользователи. Браузер жертвы выполняет чужой скрипт как «доверенный» код сайта.
   это инъекция (модуль 09), но в HTML/браузере: ввод попадает в страницу как КОД, а не ТЕКСТ.
```

```
   ❌ УЯЗВИМО: ввод вставляется в HTML без обработки
   комментарий пользователя:  <script>украсть_cookie()</script>
   страница показывает его как ЕСТЬ → у каждого посетителя выполнится этот скрипт.
```

💡 ⭐⭐ Последствия XSS: кража сессий/cookie (вход под жертвой), подмена страницы, действия от имени
жертвы, кейлоггер, фишинг внутри настоящего сайта. Опасность в том, что скрипт выполняется в
**контексте доверенного сайта** жертвы.

---

## ⭐ Виды XSS

```
   • ОТРАЖЁННЫЙ (reflected) — скрипт в параметре запроса/ссылке, отражается в ответе
     (жертва кликает вредоносную ссылку).
   • ХРАНИМЫЙ (stored) — скрипт сохраняется на сервере (комментарий, профиль) и бьёт ВСЕХ,
     кто откроет страницу. Самый опасный.
   • DOM-based — уязвимость в клиентском JS, который небезопасно вставляет данные в DOM.
```

🖼️
```
   хранимый XSS:
   атакующий → оставляет "комментарий" с <script> → сервер сохраняет →
   каждый посетитель страницы → браузер выполняет чужой скрипт → кража сессии/действия
   ❗ один вредоносный ввод поражает всех зрителей.
```

## ⭐⭐ Защита: экранируй вывод (главное)

```
   КОРЕНЬ XSS — вывод недоверенных данных в HTML без обработки. Защита — ЭКРАНИРОВАНИЕ ВЫВОДА
   под контекст, куда вставляешь:

   ✅ В HTML-ТЕКСТ — экранируй спецсимволы: < → &lt;  > → &gt;  & → &amp;  " → &quot;
      тогда <script> станет ТЕКСТОМ "&lt;script&gt;", а не кодом — отобразится, но не выполнится.
   ✅ ИСПОЛЬЗУЙ ШАБЛОНИЗАТОРЫ с автоэкранированием (большинство современных экранируют по умолчанию —
      не отключай это и не используй «raw/unsafe» вставку для недоверенного).
   ✅ КОНТЕКСТ ВАЖЕН: экранирование для HTML, для атрибута, для JS, для URL — РАЗНОЕ.
      не вставляй недоверенное в опасные контексты (внутрь <script>, в обработчики on*, в javascript:).
```

```html
<!-- ❌ опасно -->  <div>{{ user_comment | raw }}</div>
<!-- ✅ безопасно --> <div>{{ user_comment }}</div>   <!-- автоэкранирование -->
```

💡 ⭐⭐ Главное правило: **недоверенные данные выводи как ТЕКСТ, а не как разметку**. Современные
фреймворки экранируют автоматически — твоя задача не **отключать** это (`raw`, `dangerouslySetInnerHTML`,
`innerHTML`) для пользовательских данных. XSS почти всегда там, где разработчик обошёл автоэкранирование.

## ⭐ Дополнительные слои защиты

```
   ✅ CSP (Content-Security-Policy) — заголовок, ограничивающий, откуда грузятся/выполняются скрипты.
      даже если XSS просочился, CSP может заблокировать выполнение постороннего скрипта. Сильный второй рубеж.
   ✅ HttpOnly cookie — JS не может прочитать cookie сессии → украсть её скриптом сложнее.
   ✅ SANITIZE HTML — если по задаче нужно разрешить разметку (редактор), используй проверенную
      библиотеку-санитайзер (allow-list тегов), не пиши свою.
   ✅ для DOM-based — используй безопасные API (textContent, а не innerHTML; избегай eval).
   ✅ валидация ввода (модуль 16) — как доп. слой, но НЕ замена экранированию вывода.
```

💡 ⭐ Ключевой нюанс: защищает именно **экранирование на ВЫВОДЕ** (в момент вставки в страницу), а
не только проверка на входе. Одни и те же данные могут выводиться в разных местах — экранируй
под каждый контекст.

## ⚠️ Ловушки

- ❌ Отключать автоэкранирование (`raw`/`innerHTML`/`dangerouslySetInnerHTML`) для пользовательских данных.
- ❌ Считать, что валидация на входе заменяет экранирование на выводе (не заменяет).
- ❌ Игнорировать контекст (HTML/атрибут/JS/URL требуют разного экранирования).
- ❌ Писать свой «санитайзер» HTML вместо проверенной библиотеки.
- ❌ Вставлять недоверенное в опасные контексты (внутрь `<script>`, `on*`, `javascript:`).
- ❌ Не ставить CSP и не использовать HttpOnly для cookie сессии.

## ✅ Упражнения (в лабе/на своём)

1. **Поломай и почини.** В учебном приложении (Juice Shop/DVWA) найди XSS, пойми тип, затем в
   своём коде убедись, что вывод экранируется/не используешь raw-вставку.
2. **Аудит raw.** Поищи в своём фронтенде `innerHTML`/`raw`/`dangerouslySetInnerHTML` с
   пользовательскими данными. Замени на безопасный вывод.
3. **CSP.** Добавь заголовок Content-Security-Policy в своё приложение (хотя бы базовый) и проверь,
   что легитимное работает.
4. **Cookie.** Проверь: стоит ли HttpOnly на cookie сессии?
