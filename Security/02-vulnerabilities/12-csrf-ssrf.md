# 12 · CSRF и SSRF 🖼️⭐

> 🎯 **Цель блока:** разобрать две «подделки запросов» — CSRF (от лица пользователя) и SSRF (от
> лица сервера) — и научиться от них защищаться.

> ⚠️ Практика — только на своих/учебных приложениях. Цель — защита.

---

## 📖 CSRF — подделка запроса от лица пользователя

```
   CSRF (Cross-Site Request Forgery) — жертва, будучи залогиненной на сайте А, заходит на
   вредоносную страницу, которая ОТ ЕЁ ИМЕНИ шлёт запрос на сайт А (браузер приложит её cookie).
   → действие выполняется от лица жертвы без её ведома (перевод, смена пароля/почты, удаление).
```

```
   ❌ жертва залогинена в банке. Открывает вредоносный сайт, где скрытая форма/запрос:
      POST bank.com/transfer  amount=1000  to=attacker
      браузер автоматически добавит cookie сессии банка → банк думает, что это сама жертва.
```

🖼️
```
   [жертва залогинена в bankA] → заходит на [evil.com]
                                      ↓ evil.com шлёт запрос на bankA
   браузер прикладывает cookie сессии bankA  → bankA выполняет действие «от жертвы»
   ❗ атака использует, что cookie шлются автоматически на свой домен.
```

---

## ⭐ Защита от CSRF

```
   ✅ CSRF-ТОКЕН — на каждую изменяющую форму сервер кладёт случайный секрет, которого нет у
      стороннего сайта. Запрос без верного токена отклоняется. (большинство фреймворков — встроенно.)
   ✅ SAMESITE COOKIE — cookie с SameSite=Lax/Strict не отправляются на кросс-сайтовые запросы →
      ломает основу CSRF. Ставь SameSite на cookie сессии.
   ✅ ПРОВЕРКА происхождения (Origin/Referer) для чувствительных действий.
   ✅ изменяющие действия — только POST/PUT/DELETE (не GET); GET не должен менять состояние.
   ✅ для важного — повторная аутентификация/подтверждение (смена пароля, перевод).
```

💡 ⭐ Связка **CSRF-токен + SameSite-cookie** закрывает CSRF в подавляющем большинстве случаев.
Современные фреймворки дают это «из коробки» — не отключай. И помни: GET-запрос **не должен**
менять данные (иначе достаточно картинки `<img src=...>` для атаки).

---

## ⭐⭐ SSRF — заставить СЕРВЕР сделать запрос

```
   SSRF (Server-Side Request Forgery) — атакующий заставляет ТВОЙ СЕРВЕР отправить запрос туда,
   куда хочет атакующий. Возникает, когда сервер ходит по URL, который дал пользователь.

   ❌ УЯЗВИМО: сервис «загрузи картинку по ссылке» / «проверь URL» берёт URL от пользователя и
      делает запрос. атакующий даёт ВНУТРЕННИЙ адрес:
      url = http://169.254.169.254/...   (метаданные облака — секреты!)
      url = http://localhost:6379        (внутренняя БД/сервис, недоступный извне)
   → сервер (он ВНУТРИ периметра!) обращается к внутренним ресурсам и возвращает атакующему.
```

🖼️
```
   атакующий → "обработай этот URL" → [ТВОЙ СЕРВЕР] → запрос на ВНУТРЕННИЙ адрес
   (внешний атакующий не достал бы напрямую — но сервер внутри периметра достаёт)
   → утечка внутренних данных, облачных метаданных/ключей, доступ к внутренним сервисам.
```

💡 ⭐⭐ SSRF опасен тем, что **сервер доверенный и внутри сети** — через него атакующий дотягивается
до того, что снаружи закрыто (внутренние сервисы, облачные метаданные с ключами). Это растущая
категория из-за облаков и микросервисов.

---

## ⭐ Защита от SSRF

```
   ✅ ALLOW-LIST адресов/доменов, куда серверу разрешено ходить (а не «куда скажет пользователь»).
   ✅ ЗАПРЕТИ внутренние/служебные адреса: localhost, 127.0.0.1, приватные диапазоны,
      облачные метаданные (169.254.169.254). Проверяй ПОСЛЕ резолва DNS (обход через DNS).
   ✅ НЕ ВОЗВРАЩАЙ сырой ответ внутреннего запроса пользователю.
   ✅ СЕТЕВАЯ СЕГМЕНТАЦИЯ: сервис, ходящий по внешним URL, изолируй от внутренних ресурсов.
   ✅ наименьшие привилегии + защита облачных метаданных (IMDSv2 и т.п.).
```

> 🧭 SSRF и сегментация — это [сети](../../Network/04-advanced/19-nat-firewall-vpn.md) и
> границы доверия из [модели угроз (модуль 04)](../01-recon/04-threat-modeling.md).

## ⚠️ Ловушки

- ❌ Менять состояние через GET (открывает простой CSRF).
- ❌ Отключать встроенную CSRF-защиту фреймворка «чтобы не мешала».
- ❌ Не ставить SameSite на cookie сессии.
- ❌ Ходить сервером по любому URL от пользователя без allow-list (SSRF).
- ❌ Проверять URL до DNS-резолва (обходится доменом, резолвящимся во внутренний IP).
- ❌ Возвращать пользователю сырой ответ внутреннего запроса.

## ✅ Упражнения (в лабе/на своём)

1. **CSRF.** Проверь, есть ли в твоём приложении CSRF-токены на изменяющих формах и SameSite на
   cookie. Где нет — добавь.
2. **GET-аудит.** Найди GET-запросы, меняющие состояние. Переведи на POST/PUT/DELETE.
3. **SSRF.** Есть ли функции, где сервер ходит по URL от пользователя? Добавь allow-list и запрет
   внутренних адресов.
4. **Метаданные.** Если используешь облако — защищены ли метаданные от SSRF (IMDSv2/ограничения)?
