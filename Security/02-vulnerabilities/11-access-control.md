# 11 · Контроль доступа и IDOR 🖼️⭐⭐

> 🎯 **Цель блока:** разобрать самую частую категорию уязвимостей по версии OWASP — нарушение
> контроля доступа (включая IDOR) — и научиться правильно проверять права.

> ⚠️ Тесты — только на своих/учебных приложениях. Цель — защита.

---

## 📖 Broken Access Control — №1 в OWASP

```
   НАРУШЕНИЕ КОНТРОЛЯ ДОСТУПА — пользователь делает то, что ему НЕ должно быть позволено:
   • читает/меняет ЧУЖИЕ данные
   • попадает в админ-функции
   • выполняет действия выше своих прав
   причина: авторизация (модуль 07) проверена слабо или не проверена там, где нужно.
```

💡 ⭐⭐ Это лидер списка реальных уязвимостей, потому что про authz **легко забыть**: фича работает,
«счастливый путь» проходит, а проверку «а ИМЕННО ЭТОМУ пользователю можно?» пропустили. Помни:
**аутентификация ≠ авторизация** (модуль 07). Вошёл — не значит «можно всё».

## ⭐ IDOR — доступ к чужому по идентификатору

```
   IDOR (Insecure Direct Object Reference) — доступ к объекту по id БЕЗ проверки прав на него.

   ❌ УЯЗВИМО:
   GET /api/orders/777        →  сервер отдаёт заказ 777, не проверяя, ТВОЙ ли он
   пользователь меняет 777 на 778 в URL →  видит ЧУЖОЙ заказ. Это IDOR.
```

🖼️
```
   user42 запрашивает  /orders/777  (его заказ)   → ✅ ок
   user42 меняет на    /orders/778  (заказ user99) → ❌ сервер отдаёт! (не проверил владельца)
   ❗ id предсказуем + нет проверки прав = чтение/изменение чужого простым перебором.
```

```python
# ❌ УЯЗВИМО: достаём по id без проверки владельца
order = db.get_order(request_id)
return order

# ✅ БЕЗОПАСНО: проверяем, что объект принадлежит текущему пользователю
order = db.get_order(request_id)
if order is None or order.user_id != current_user.id:
    return forbidden(403)          # не твой — запрещено
return order
```

💡 ⭐⭐ Защита от IDOR: **на каждый доступ к объекту проверяй, что он принадлежит/разрешён текущему
пользователю** — на сервере. Не полагайся на «id никто не угадает» (это [security by obscurity];
id перебирают). Проверка владения/прав — обязательна, даже если ссылка «не публичная».

## ⭐⭐ Принципы правильного контроля доступа

```
   ✅ ПРОВЕРЯЙ НА СЕРВЕРЕ — клиентские проверки (скрытая кнопка) обходятся прямым запросом.
   ✅ DENY BY DEFAULT — по умолчанию запрещено; доступ — явно выданный (модуль 07).
   ✅ ПРОВЕРЯЙ ВЛАДЕНИЕ/ПРАВА на КАЖДЫЙ доступ к ресурсу (не только при входе).
   ✅ ЦЕНТРАЛИЗУЙ логику доступа — единый механизм проверки, а не «копипаст if-ов» по эндпоинтам
     (легче не забыть и не ошибиться).
   ✅ НЕ ДОВЕРЯЙ клиентским данным о роли/правах ("isAdmin=true" из запроса — игнорируй, бери с сервера).
   ✅ НАИМЕНЬШИЕ ПРИВИЛЕГИИ — минимум прав каждой роли.
```

💡 ⭐ Частая ошибка — проверять доступ «местами», по наитию. Лучше **централизованный** механизм
(middleware/декоратор/политики), который применяется ко всем защищённым ресурсам единообразно —
тогда «забыл проверить» становится труднее. Это [проектирование под изменения (Senior)](../../Senior/03-practices/15-design-for-change.md):
один механизм вместо разбросанных проверок.

## 📖 Эскалация привилегий

```
   ВЕРТИКАЛЬНАЯ — обычный пользователь получает права админа (доступ к /admin без проверки роли).
   ГОРИЗОНТАЛЬНАЯ — пользователь получает доступ к данным РАВНОГО ему (IDOR — её частный случай).
   защита та же: серверная проверка роли/владения на каждый чувствительный доступ; deny by default.
   ⚠️ особенно проверяй «скрытые» админ-эндпоинты — «не в меню» не значит «недоступно».
```

---

## ⚠️ Ловушки

- ❌ Проверять доступ только на клиенте (скрыл кнопку/пункт меню — но запрос проходит).
- ❌ Доверять id как «секрету» («не угадают») вместо проверки прав.
- ❌ Не проверять владение объектом (IDOR).
- ❌ Доверять роли/правам из клиентского запроса/токена без серверной проверки.
- ❌ Разбросанные проверки доступа вместо централизованного механизма (легко забыть).
- ❌ «Скрытые» эндпоинты без проверки роли («раз не в меню — не найдут»).

## ✅ Упражнения (в лабе/на своём)

1. **IDOR-тест.** В своём приложении поменяй id ресурса на чужой. Доступно? Добавь проверку
   владельца на сервере и проверь снова.
2. **Клиент vs сервер.** Найди действие, скрытое только на клиенте (кнопка по роли). Отправь
   запрос напрямую — проходит? Добавь серверную проверку.
3. **Централизация.** Посмотри, как у тебя проверяется доступ — единым механизмом или россыпью
   if-ов? Как централизовать?
4. **Админ-эндпоинты.** Проверь, что все админ-функции требуют серверной проверки роли, даже
   «непубличные».
