# 13 · Небезопасная конфигурация и секреты 🖼️⭐

> 🎯 **Цель блока:** научиться видеть уязвимости, которые возникают не в коде, а в **настройке** —
> часто самые лёгкие для атакующего и самые обидные.

> ⚠️ Проверки — только на своих/учебных системах.

---

## 📖 Дыра не в коде, а в настройке

```
   код может быть идеальным, но система взломана из-за КОНФИГУРАЦИИ:
   • дефолтные/слабые пароли (admin/admin), оставленные тестовые аккаунты
   • открытые наружу админки, БД, дашборды, отладочные эндпоинты
   • подробные ошибки/стектрейсы в проде (раскрывают код, пути, версии)
   • включённый debug-режим в проде
   • лишние сервисы/порты/функции (раздутая поверхность атаки)
   • отсутствие/неправильные заголовки безопасности
   • устаревшее ПО с известными уязвимостями
```

💡 ⭐ Это одна из самых частых и **самых дешёвых для атакующего** категорий: не нужно искать
хитрый баг — достаточно найти открытую админку с паролем по умолчанию (а боты ищут это
автоматически). Защита тут — дисциплина настройки, а не код.

🖼️
```
   идеальный код  +  [admin/admin на открытой админ-панели]  =  взломано за минуту
   безопасность системы = код И конфигурация. Слабое звено решает.
```

---

## ⭐ Чек-лист безопасной конфигурации

```
   ✅ СМЕНИ дефолты: пароли, ключи, имена; удали тестовые/демо аккаунты.
   ✅ ВЫКЛЮЧИ debug в проде; ошибки пользователю — общие («что-то пошло не так»), детали — в логи.
   ✅ ЗАКРОЙ лишнее: ненужные порты, сервисы, эндпоинты, админки — не наружу.
   ✅ ОБНОВЛЯЙ ПО: ОС, сервер, библиотеки, фреймворк (известные уязвимости в старых версиях).
   ✅ ЗАГОЛОВКИ безопасности: CSP, HSTS, X-Content-Type-Options, скрыть Server/X-Powered-By.
   ✅ HTTPS везде; правильные настройки TLS.
   ✅ НАИМЕНЬШИЕ ПРИВИЛЕГИИ для сервисов/аккаунтов/контейнеров.
   ✅ разделяй окружения (dev/stage/prod), не тащи тестовые настройки в прод.
```

💡 ⭐ Принцип: **безопасно по умолчанию + минимум открытого**. Каждая «временная» поблажка
(«открою пока порт», «потом сменю пароль», «debug на чуть-чуть») — это дыра, которую забывают
закрыть. Автоматизируй проверку конфигурации (твой сканер из Уровня 1).

## ⭐⭐ Секреты: главная боль

```
   СЕКРЕТЫ — пароли, API-ключи, токены, ключи шифрования. Утечка секрета = прямой доступ.

   ❌ ЧАСТЫЕ ОШИБКИ:
   • секреты в коде/репозитории (хардкод) → попадают в git-историю НАВСЕГДА, видны всем с доступом.
   • секреты в публичном репозитории/гисте → боты находят за МИНУТЫ, ключи угоняют.
   • секреты в логах, в клиентском коде, в URL.
   • один секрет на всё / никогда не меняются.

   ✅ ПРАВИЛЬНО:
   • секреты — в переменных окружения / менеджере секретов (Vault и т.п.), НЕ в коде.
   • .gitignore для .env и конфигов с секретами; НИКОГДА не коммить.
   • разные секреты на окружения; ротация; отзыв при подозрении.
   • утёк секрет → немедленно отозвать/сменить (он в истории — «удалить из git» не спасает).
   • сканеры секретов в CI (ловят случайный коммит ключа до пуша).
```

💡 ⭐⭐ Правило: **секрет НИКОГДА не в коде**. Однажды закоммиченный ключ нужно считать
скомпрометированным (он в истории git, в форках, в кэшах) — отзывай и меняй, а не «удаляй коммит».
Это [build vs buy / гигиена](../../SocialEng/04-defense/18-personal-hygiene.md): используй менеджер
секретов, не изобретай хранение.

## 📖 Связь с зависимостями (мост к модулю 19)

```
   «небезопасная конфигурация» включает и УСТАРЕВШИЕ/УЯЗВИМЫЕ зависимости:
   • старая библиотека с известной уязвимостью = открытая дверь, даже если твой код чист.
   • защита: регулярный аудит зависимостей, обновления, отслеживание уязвимостей (модуль 19).
```

---

## ⚠️ Ловушки

- ❌ Оставлять дефолтные пароли/тестовые аккаунты/демо-данные.
- ❌ Debug/подробные ошибки в проде (утечка деталей).
- ❌ Открытые наружу админки, БД, отладочные эндпоинты.
- ❌ Секреты в коде/репозитории (особенно публичном).
- ❌ «Удалить коммит» вместо отзыва утёкшего секрета.
- ❌ Не обновлять ПО и зависимости (известные уязвимости).

## ✅ Упражнения (на своём/в лабе)

1. **Конфиг-аудит.** Пройди чек-лист по своему приложению/серверу. Что открыто лишнее? Debug в
   проде? Дефолты?
2. **Поиск секретов.** Прогони поиск по репозиторию на паттерны секретов (`password=`, `api_key`,
   токены). Нашёл в коде/истории — отзови и вынеси в окружение.
3. **.gitignore.** Проверь, что `.env`/конфиги с секретами в `.gitignore` и не закоммичены.
4. **Заголовки.** Проверь заголовки безопасности своего приложения (CSP/HSTS/скрытие версий) и добавь.
