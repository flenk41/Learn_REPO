# 16 · Валидация и санитизация ввода 🖼️⭐⭐

> 🎯 **Цель блока:** научиться правильно проверять и обрабатывать внешний ввод — первый рубеж
> защиты от целого класса атак.

---

## 📖 Весь внешний ввод — враждебный

```
   источники недоверенного ввода (всё это проверяй):
   формы, параметры URL, заголовки, cookie, тело запроса, загружаемые файлы, ответы чужих API,
   данные из очередей, иногда — данные из своей же БД (вдруг туда попало вредоносное раньше).
   правило: «доверяй, но проверяй» здесь = «НЕ доверяй, проверяй ВСЁ».
```

💡 ⭐⭐ Валидация ввода — не «приятное дополнение», а первый рубеж. Но важно: она **дополняет**, а
не заменяет защиту в точке использования (параметризация для SQL, экранирование для HTML).
Валидация + защита на выводе/в запросе = защита в глубину.

---

## ⭐⭐ Allow-list лучше deny-list

```
   ✅ ALLOW-LIST (белый список) — описываешь, что РАЗРЕШЕНО, остальное отвергаешь.
      "имя пользователя: буквы/цифры/_, длина 3-20" → всё прочее отклонить.
      надёжно: не нужно предугадывать все плохие варианты.

   ❌ DENY-LIST (чёрный список) — пытаешься перечислить, что ЗАПРЕЩЕНО.
      "запретим '<script>'..." → атакующий найдёт обход (другой регистр, кодирование, новый вектор).
      хрупко: всегда найдётся то, что ты не предусмотрел.
```

🖼️
```
   allow-list: [ разрешено: A-Z 0-9 _ ]  → всё остальное ❌  (надёжно, ничего не «протечёт»)
   deny-list:  [ запрещено: <script> ]   → <SCRIPT>, %3Cscript%3E... ✅ протекли (хрупко)
   правило: описывай ДОПУСТИМОЕ, а не перечисляй плохое.
```

💡 ⭐⭐ Это ключевой принцип валидации: **описывай допустимое**. Длина, тип, формат, диапазон,
набор символов — что должно быть. Всё, что не подходит, — отвергай. Так ты не зависишь от
фантазии атакующего.

---

## ⭐ Что проверять

```python
# ✅ валидация: тип, диапазон, формат, длина, набор значений
def validate_age(raw):
    if not raw.isdigit():            # тип: только цифры
        raise ValueError("age must be a number")
    age = int(raw)
    if not (0 <= age <= 150):        # диапазон
        raise ValueError("age out of range")
    return age

# enum / набор допустимых значений
ALLOWED_SORT = {"date", "name", "price"}
sort = request.get("sort")
if sort not in ALLOWED_SORT:          # allow-list значений
    sort = "date"                     # безопасное значение по умолчанию
```

```
   проверяй: ТИП (число/строка/дата) · ДЛИНУ (защита от переполнения/DoS) · ФОРМАТ (email/телефон) ·
   ДИАПАЗОН (числа/даты) · НАБОР (одно из допустимых) · РАЗМЕР (файлы/тело запроса).
   валидируй на СЕРВЕРЕ. Клиентская валидация — для удобства, НЕ для безопасности (обходится).
```

---

## ⭐ Валидация vs санитизация vs экранирование

```
   ВАЛИДАЦИЯ — принять/отвергнуть («это допустимый ввод?»). Лучший выбор: отвергай плохое.
   САНИТИЗАЦИЯ — очистить/преобразовать ввод (убрать опасное). Сложнее, применяй осторожно
      (для случаев, где надо принять «грязное», напр. HTML из редактора — проверенным санитайзером).
   ЭКРАНИРОВАНИЕ — на ВЫВОДЕ/в точке использования, под контекст (HTML/SQL/команда). НЕ заменяется
      валидацией — это разные рубежи (модули 09, 10).
```

💡 ⭐ Частая путаница: «я же валидирую ввод, зачем экранировать вывод?». Затем, что одни и те же
данные используются в разных контекстах, и безопасность зависит от **места использования**.
Валидация — на входе, экранирование/параметризация — в точке использования. Нужны оба.

---

## 📖 Особые случаи

```
   • ЗАГРУЗКА ФАЙЛОВ: проверяй тип/размер/содержимое (не только расширение!), храни вне веб-корня,
     генерируй своё имя, не доверяй имени/типу от клиента, ограничивай размер.
   • ЧИСЛА/ПЕРЕПОЛНЕНИЕ: проверяй диапазон (особенно в C/C++ — переполнение = уязвимость).
   • КОДИРОВКИ: нормализуй (Unicode-обходы); проверяй ПОСЛЕ декодирования.
   • РАЗМЕР/КОЛИЧЕСТВО: лимиты на размер тела, число элементов (защита от DoS).
```

> 🧭 Переполнение и работа с памятью — это [C/C++](../../C/02-memory/12-memory-errors-tools.md): там
> невалидированный размер/индекс = классическая уязвимость памяти.

## ⚠️ Ловушки

- ❌ Deny-list вместо allow-list (всегда найдётся обход).
- ❌ Валидация только на клиенте (обходится прямым запросом).
- ❌ Считать, что валидация заменяет экранирование/параметризацию (не заменяет).
- ❌ Доверять расширению/типу файла от клиента.
- ❌ Не ограничивать длину/размер/количество (DoS, переполнение).
- ❌ Проверять до декодирования (обход через кодировки).

---

## ✅ Упражнения

1. **Allow-list.** Перепиши проверку какого-то поля с deny-list на allow-list (опиши допустимое).
2. **Серверная валидация.** Найди валидацию только на клиенте. Продублируй на сервере (источник
   истины — сервер).
3. **Параметры.** Для эндпоинта с параметрами (сортировка/фильтр) сделай allow-list допустимых
   значений + безопасное значение по умолчанию.
4. **Загрузка.** Если есть загрузка файлов — проверь тип/размер/хранение/имя. Усиль.
