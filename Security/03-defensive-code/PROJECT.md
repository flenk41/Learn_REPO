# 🚀 Проект уровня 3 · Безопасное приложение + набор защитных инструментов

> Соедини всё: напиши (или укрепи) приложение, безопасное по построению, и собери **набор
> собственных защитных инструментов**, которые его проверяют и охраняют.

> ⚠️ Только свой код/лаба. Инструменты — защитные (анализ/мониторинг своих систем), не атакующие.

---

## 🎯 Что делаем

Две части: (A) приложение, написанное по принципам безопасного кода; (B) **toolkit** —
несколько защитных программ, которые ты написал сам и которые работают вместе.

```
   безопасное приложение  +  свои инструменты (сканер · монитор · проверка целостности · CI)
```

---

## 📋 Часть A · Безопасное приложение

Небольшое веб-приложение/API (регистрация, вход, список записей пользователя), реализованное
правильно:
```
   ✅ ВВОД: валидация allow-list на сервере, лимиты размера.
   ✅ ХРАНЕНИЕ ПАРОЛЕЙ: bcrypt/argon2 (с солью), без открытого текста, без логирования.
   ✅ ДОСТУП: серверная авторизация, проверка владения (против IDOR), deny by default.
   ✅ ИНЪЕКЦИИ/XSS: параметризованные запросы, экранирование вывода, CSP.
   ✅ СЕССИИ: безопасные cookie (HttpOnly/Secure/SameSite), истечение; CSRF-защита.
   ✅ СЕКРЕТЫ: вне кода (окружение), .env в .gitignore.
   ✅ КРИПТА: криптостойкий ГСЧ для токенов, проверенные алгоритмы.
   ✅ ОШИБКИ: общие пользователю, детали в логи; fail securely.
```

## 📋 Часть B · Набор защитных инструментов (пишем код)

Собери 3+ собственных инструмента (расширяя сканер из Уровней 1–2):
```
   1. АУДИТОР проекта — права файлов, секреты в коде, уязвимые зависимости, заголовки. Отчёт с
      риском и способом исправления.
   2. МОНИТОР логов — обнаружение перебора/аномалий во входах, сводка/тревога.
   3. ПРОВЕРКА ЦЕЛОСТНОСТИ — baseline хешей важных файлов → сигнал об изменении.
   (+ по желанию: валидатор конфигов, проверка .env, health-чек безопасности.)
```
- Инструменты — маленькие, читаемые, объясняют находки, мало ложных срабатываний.
- Встрой хотя бы один в CI/pre-commit своего приложения.

## 📋 Часть C · Демонстрация «атака → защита»

- В лабе/на своём приложении покажи, что типовые атаки (SQLi, XSS, IDOR) **не проходят** благодаря
  защите. Зафиксируй before (уязвимая версия) / after (защищённая).

## ✅ Критерии готовности

- [ ] Приложение реализовано по принципам безопасного кода (вся часть A)
- [ ] Пароли через bcrypt/argon2, секреты вне кода, авторизация на сервере
- [ ] Написаны 3+ собственных защитных инструмента, объясняющих находки
- [ ] Хотя бы один инструмент встроен в CI/pre-commit
- [ ] Показано «атака не проходит» (before/after) для типовых уязвимостей
- [ ] Код приложения и инструментов читаемый и модульный
