# 15 · Принципы безопасного кода 🖼️⭐⭐

> 🎯 **Цель блока:** собрать принципы, по которым пишут код, устойчивый к атакам **с самого
> начала**, — чтобы безопасность была встроена, а не «прикручена потом».

---

## 📖 Безопасность встраивают, а не добавляют

```
   «допишем безопасность потом» не работает: дыры закладываются на этапе кода и проектирования.
   дешевле и надёжнее — писать безопасно СРАЗУ, по принципам, которые становятся привычкой.
   (как чистый код у Senior: не «отрефакторим потом», а пишем хорошо изначально.)
```

💡 ⭐⭐ Эти принципы — обобщение всего Уровня 2. Если держать их в голове, большинство уязвимостей
просто **не появляются**: ты не пишешь склейку запроса (нет SQLi), не отключаешь экранирование
(нет XSS), проверяешь доступ на сервере (нет IDOR).

---

## ⭐⭐ Восемь принципов безопасного кода

```
   1. НЕ ДОВЕРЯЙ ВВОДУ. Весь внешний вход враждебен, пока не проверен (модуль 16).
      источники: пользователь, API, файлы, заголовки, чужие сервисы, даже «своя» БД.

   2. РАЗДЕЛЯЙ КОД И ДАННЫЕ. Параметризация, экранирование вывода — никогда не склеивай
      ввод в команды/запросы/разметку (модули 09, 10).

   3. НАИМЕНЬШИЕ ПРИВИЛЕГИИ. Каждому (пользователю, сервису, процессу, токену) — минимум прав.
      взломали одно → ущерб ограничен.

   4. ЗАЩИТА В ГЛУБИНУ. Несколько рубежей, не один. Пробили валидацию → ловит параметризация →
      ловит мониторинг.

   5. БЕЗОПАСНО ПО УМОЛЧАНИЮ. Закрыто/запрещено, пока явно не разрешил (deny by default).

   6. FAIL SECURELY. При ошибке — в БЕЗОПАСНОЕ состояние (отказать в доступе, не «пропустить на
      всякий случай»). Ошибка авторизации = запретить, а не разрешить.

   7. НЕ ИЗОБРЕТАЙ БЕЗОПАСНОСТЬ. Крипта, хранение паролей, auth, сессии — бери проверенное
      (модули 17, 18). Самодельное почти всегда дырявое.

   8. МИНИМИЗИРУЙ И УПРОЩАЙ. Меньше кода/функций/зависимостей = меньше поверхность атаки.
      сложность — враг безопасности (как KISS у Senior).
```

🖼️
```
   вход → [ не доверяй: валидируй ] → [ разделяй код/данные ] → [ проверь доступ ] →
          [ fail securely при ошибке ] → [ минимум привилегий ] → действие
   каждый принцип = рубеж. Вместе = код, который трудно сломать.
```

## ⭐ Fail securely и безопасные значения по умолчанию

```python
# ❌ ОПАСНО: при ошибке/неясности — пропускаем
def can_access(user, resource):
    try:
        return check_permission(user, resource)
    except:
        return True            # ← «на всякий случай пропустим» = дыра!

# ✅ БЕЗОПАСНО: при ошибке — запрещаем (fail closed)
def can_access(user, resource):
    try:
        return check_permission(user, resource)
    except Exception:
        log.warning("permission check failed", ...)
        return False           # ← неясно → ЗАПРЕТИТЬ
```

💡 ⭐ Когда что-то пошло не так — выбирай **безопасный** исход. Сломалась проверка прав → запретить.
Не распарсился ввод → отклонить. «Пропустить, чтобы не мешать пользователю» — частый путь к
уязвимости. Безопасность важнее удобства в спорных случаях.

## 📖 Обработка ошибок и логирование без утечек

```
   ✅ ПОЛЬЗОВАТЕЛЮ — общие сообщения: «что-то пошло не так», «доступ запрещён».
   ✅ В ЛОГИ (не пользователю) — детали для отладки.
   ❌ НЕ показывай стектрейсы, SQL-ошибки, пути, версии, внутренние данные — это подсказки атакующему.
   ❌ НЕ логируй секреты/пароли/токены/персональные данные (логи тоже утекают).
   ✅ логируй СОБЫТИЯ безопасности (входы, отказы доступа, аномалии) — для мониторинга (модуль 21).
```

## ⚠️ Ловушки

- ❌ «Безопасность потом» — дыры уже заложены в коде.
- ❌ Fail open (при ошибке пропускать) вместо fail closed.
- ❌ Доверять «внутренним»/«своим» источникам данных без проверки.
- ❌ Показывать технические ошибки пользователю; логировать секреты.
- ❌ Изобретать своё (крипта/auth/хранение паролей).
- ❌ Усложнять без нужды — сложность плодит дыры.

## ✅ Упражнения

1. **Принципы на своём коде.** Для каждого из 8 принципов найди в своём проекте пример соблюдения
   или нарушения.
2. **Fail securely.** Найди обработку ошибок, где при сбое «пропускается» доступ/действие. Сделай
   fail closed.
3. **Ошибки наружу.** Проверь, не показывает ли приложение стектрейсы/детали ошибок пользователю.
   Скрой, оставь в логах.
4. **Логи.** Убедись, что не логируешь пароли/токены. Логируешь ли события безопасности?
