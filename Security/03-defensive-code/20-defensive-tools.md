# 20 · Пишем защитные инструменты 🖼️⭐⭐

> 🎯 **Цель блока:** научиться писать **программы для защиты** — небольшие инструменты, которые
> автоматизируют безопасность: проверяют, мониторят, обнаруживают.

> ⚠️ Защитные инструменты (анализ, мониторинг, проверка своих систем) — да. Вредоносное ПО / оружие
> для атаки чужого — нет ([модуль 02](../00-foundations/02-law-authorization.md)).

---

## 📖 Защита, которую можно запрограммировать

```
   ручная безопасность не масштабируется. КОД защищает постоянно и одинаково хорошо:
   • СКАНЕРЫ — находят слабости (открытые порты, плохие права, секреты в коде, уязвимые зависимости).
   • ВАЛИДАТОРЫ/ФИЛЬТРЫ — проверяют ввод/конфиги на безопасность.
   • МОНИТОРЫ — следят за логами/событиями, поднимают тревогу на аномалии (модуль 21).
   • ПРОВЕРКИ ЦЕЛОСТНОСТИ — замечают подмену важных файлов (по хешам).
   • АВТОМАТИЗАЦИЯ в CI — гоняет проверки безопасности на каждый коммит (модуль 24).
```

💡 ⭐⭐ Это прямой ответ на «писать программы для защиты»: защитный инструмент — это обычная
программа, делающая одно полезное дело хорошо. Навыки разработки + понимание угроз (этот трек) =
ты пишешь инструменты, усиливающие защиту. Большинство «инструментов безопасности» — именно такие
небольшие программы.

## ⭐ Пример 1: монитор логов (детектор перебора)

```python
# Защитный инструмент: следит за логом входов и поднимает тревогу при переборе пароля.
# (анализирует СВОЙ лог — это легально и полезно)
from collections import defaultdict

THRESHOLD = 5          # сколько неудач подряд считаем подозрительным

def scan_auth_log(path: str):
    fails = defaultdict(int)
    alerts = []
    with open(path) as f:
        for line in f:
            if "Failed password" in line:
                ip = extract_ip(line)          # вытащить IP из строки лога
                fails[ip] += 1
                if fails[ip] == THRESHOLD:
                    alerts.append(f"⚠️ Возможный перебор с {ip} ({fails[ip]}+ неудач)")
            elif "Accepted password" in line:
                fails[extract_ip(line)] = 0     # успешный вход сбрасывает счётчик
    return alerts
```

💡 ⭐ Тут видно суть: читаешь источник (лог) → ищешь признак угрозы (серия неудач) → реагируешь
(тревога). Так устроены детекторы. Дальше можно добавить уведомление, автоблокировку IP, экспорт
отчёта — но ядро простое.

## ⭐⭐ Пример 2: аудитор конфигурации (мини-сканер)

```python
# Защитный инструмент: проверяет проект на типовые проблемы и выдаёт отчёт.
import os, re, stat

SECRET_PATTERNS = [r"api[_-]?key\s*=\s*['\"]", r"password\s*=\s*['\"]", r"secret\s*=\s*['\"]"]

def audit(project_dir: str):
    findings = []
    for root, _, files in os.walk(project_dir):
        for name in files:
            path = os.path.join(root, name)
            # 1) слишком открытые права на файл
            mode = os.stat(path).st_mode
            if mode & stat.S_IWOTH:                      # доступен на запись «всем»
                findings.append(("🔴", path, "доступен на запись всем (world-writable)"))
            # 2) секреты в коде
            if name.endswith((".py", ".js", ".env", ".yml", ".json")):
                text = open(path, errors="ignore").read()
                for pat in SECRET_PATTERNS:
                    if re.search(pat, text, re.I):
                        findings.append(("🟡", path, "возможный секрет в коде"))
    return findings

# вывод: findings → понятный отчёт «риск · файл · проблема · как починить»
```

💡 ⭐⭐ Это расширение твоего сканера из Уровня 1. Принцип защитного инструмента: **проверка →
находка → объяснение риска → подсказка по исправлению**. Не просто «нашёл», а «нашёл, вот почему
опасно, вот как закрыть» — инструмент, который реально помогает (как хорошее [код-ревью у Senior](../../Senior/01-craft/06-code-review.md)).

## 📖 Принципы хорошего защитного инструмента

```
   ✅ ДЕЛАЕТ ОДНО ХОРОШО — маленький, понятный, надёжный (Unix-философия, чистый код).
   ✅ ОБЪЯСНЯЕТ находки — риск + способ исправить, а не сырой вывод.
   ✅ МАЛО ЛОЖНЫХ СРАБАТЫВАНИЙ — иначе его перестанут слушать (как с фишинг-симуляциями).
   ✅ ВСТРАИВАЕТСЯ в процесс — pre-commit, CI, cron, алерты (автоматизация > ручной запуск).
   ✅ САМ БЕЗОПАСЕН — не хранит секреты, не открывает новых дыр, работает с минимальными правами.
   ✅ ЧЕСТНО О ГРАНИЦАХ — инструмент находит ТИПОВОЕ, не заменяет мышление и ревью.
```

> 🧭 Это и [Senior-ремесло](../../Senior/01-craft/04-naming-simplicity.md) (простые надёжные
> инструменты), и автоматизация: защита, работающая постоянно, а не «когда вспомнил».

## ⚠️ Ловушки

- ❌ Писать «комбайн на всё» вместо маленьких надёжных инструментов.
- ❌ Выдавать сырые находки без объяснения риска/исправления.
- ❌ Много ложных срабатываний → инструмент игнорируют.
- ❌ Инструмент сам небезопасен (хранит секреты, требует root зря).
- ❌ Считать, что инструмент заменяет ревью/мышление (он ловит типовое).
- ❌ Писать «инструменты атаки» для чужих систем (это уже не защита — модуль 02).

## ✅ Упражнения (на своём/в лабе)

1. **Монитор логов.** Напиши инструмент, который ищет в логе входов признаки перебора и выдаёт
   сводку по IP. Протестируй на учебном логе.
2. **Аудитор.** Расширь мини-сканер: добавь проверку (например, открытые порты или устаревшие
   зависимости). Каждая находка — с объяснением и фиксом.
3. **Целостность.** Напиши инструмент, который хеширует важные файлы и замечает их изменение
   (baseline → проверка).
4. **В CI.** Подключи свой сканер/линтер безопасности так, чтобы он запускался автоматически
   (pre-commit/CI) на твоём проекте.
