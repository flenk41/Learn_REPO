# 19 · Секреты, зависимости, supply chain 🖼️⭐

> 🎯 **Цель блока:** защитить две частые точки утечки — секреты в коде и уязвимые/вредоносные
> зависимости (атаки на цепочку поставок).

---

## 📖 Секреты: не в коде, никогда

```
   СЕКРЕТЫ — пароли, API-ключи, токены, ключи шифрования, строки подключения к БД.
   ❌ хардкод в коде → попадает в git-историю НАВСЕГДА (даже если потом «удалишь строку»),
      виден всем с доступом к репозиторию, утекает с форками/бэкапами.
   ❌ публичный репозиторий с ключом → боты находят за МИНУТЫ, ключи угоняют и используют.
```

```python
# ❌ ОПАСНО
API_KEY = "sk_live_a1b2c3d4..."          # в коде = в истории git навсегда

# ✅ ПРАВИЛЬНО: из окружения / менеджера секретов
import os
API_KEY = os.environ["API_KEY"]          # значение вне кода
```

💡 ⭐ Правило: **секрет вне кода**. Переменные окружения (`.env` в `.gitignore`, не коммитить) для
простого; менеджер секретов (Vault, облачные secret-managers) для серьёзного. Утёкший секрет
считай скомпрометированным навсегда — **отзывай и меняй**, «удалить коммит» не спасает (он в
истории, форках, кэшах).

🖼️
```
   секрет в коммите → git-история → форки/клоны/бэкапы/CI-логи → НЕ вытравить «удалением»
   → единственная защита: ОТОЗВАТЬ ключ и выпустить новый. И не коммитить впредь.
```

## ⭐ Защита секретов: чек-лист

```
   ✅ секреты в переменных окружения / менеджере секретов, НЕ в коде.
   ✅ .gitignore для .env и конфигов с секретами; шаблон .env.example БЕЗ значений.
   ✅ СКАНЕР СЕКРЕТОВ в pre-commit/CI (ловит случайный коммит ключа ДО пуша).
   ✅ разные секреты на окружения (dev/stage/prod); ротация; немедленный отзыв при утечке.
   ✅ не логируй секреты, не клади в URL, не передавай на клиент.
   ✅ ограничивай права ключей (наименьшие привилегии: ключ только на нужное).
```

## ⭐⭐ Зависимости и цепочка поставок

```
   твой код может быть идеален, но ты тянешь десятки-сотни чужих библиотек (прямых и транзитивных).
   РИСКИ ЦЕПОЧКИ ПОСТАВОК (supply chain):
   • УЯЗВИМАЯ зависимость — известная дыра в старой версии библиотеки = дыра в тебе.
   • ВРЕДОНОСНАЯ зависимость — злоумышленник публикует/захватывает пакет с вредоносом.
   • TYPOSQUATTING — пакет с именем, похожим на популярный (reqeusts вместо requests) → ставишь по ошибке.
   • КОМПРОМЕТАЦИЯ пакета/мейнтейнера — в популярный пакет внедряют вредонос (реальные инциденты).
```

🖼️
```
   твой код (чистый)  +  [ 200 зависимостей, одна — уязвима/вредоносна ]  =  взломан
   безопасность = твой код И вся цепочка чужого кода, которому ты доверяешь.
```

💡 ⭐⭐ Большая часть кода в проекте — **чужой** (зависимости). Доверяя пакету, ты доверяешь его
авторам и всем ЕГО зависимостям. Это [build vs buy (Senior)](../../Senior/02-decisions/10-build-vs-buy.md):
у «взять готовое» есть цена — её надо контролировать.

## ⭐ Защита цепочки поставок

```
   ✅ АУДИТ ЗАВИСИМОСТЕЙ — регулярно (`pip-audit`, `npm audit`, `cargo audit`, и т.п.):
      находит известные уязвимости в твоих пакетах. Встрой в CI.
   ✅ ОБНОВЛЯЙ зависимости (но осознанно; следи за безопасностью обновлений).
   ✅ ФИКСИРУЙ версии (lock-файлы) + проверяй целостность (хеши) — воспроизводимость и защита от подмены.
   ✅ МИНИМИЗИРУЙ зависимости — меньше пакетов = меньше поверхность (нужна 1 функция? может, не тащить
      пакет с 40 транзитивными — модуль build vs buy).
   ✅ ПРОВЕРЯЙ ПАКЕТ перед добавлением: живой ли, популярный, кто мейнтейнер, нет ли typosquatting.
   ✅ SBOM (список компонентов) — знай, что у тебя стоит, чтобы быстро реагировать на новые CVE.
```

> 🧭 Это часть OWASP Top-10 (уязвимые компоненты, целостность ПО) и [DevSecOps (модуль 24)](../04-blue-team/24-devsecops.md):
> аудит зависимостей и сканеры секретов встраивают в пайплайн.

## ⚠️ Ловушки

- ❌ Секреты в коде/репозитории (особенно публичном).
- ❌ «Удалить коммит» вместо отзыва утёкшего ключа.
- ❌ Никогда не обновлять зависимости (копятся известные уязвимости).
- ❌ Тянуть пакеты бездумно (typosquatting, лишние/заброшенные).
- ❌ Нет lock-файлов и аудита → невоспроизводимо и слепо к уязвимостям.
- ❌ Ключи с избыточными правами (взлом одного = доступ ко всему).

---

## ✅ Упражнения

1. **Поиск секретов.** Прогони репозиторий на паттерны секретов (и историю). Нашёл — вынеси в
   окружение, отзови ключ.
2. **gitignore.** Проверь, что `.env`/секреты игнорируются; сделай `.env.example` без значений.
3. **Аудит зависимостей.** Запусти `audit` своего пакетного менеджера. Сколько уязвимостей?
   Обнови критичные.
4. **Минимизация.** Найди зависимость, используемую ради 1 функции. Стоит ли её держать?
