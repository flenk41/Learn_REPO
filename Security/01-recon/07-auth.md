# 07 · Аутентификация и авторизация 🖼️⭐⭐

> 🎯 **Цель блока:** различать аутентификацию («кто ты?») и авторизацию («что тебе можно?») и
> понимать частые ошибки в обеих — это одна из самых атакуемых областей.

---

## 📖 Два разных вопроса

```
   АУТЕНТИФИКАЦИЯ (authn) — «КТО ТЫ?» Подтверждение личности (логин/пароль, токен, MFA).
   АВТОРИЗАЦИЯ (authz)    — «ЧТО ТЕБЕ МОЖНО?» Проверка прав на действие/ресурс.

   их ПУТАЮТ, и это рождает дыры:
   • аутентифицировали (вошёл), но не проверили авторизацию (а можно ли ему ЭТО?) → IDOR, эскалация.
   • "раз залогинен — значит, всё можно" → доступ к чужому/админскому.
```

💡 ⭐⭐ Правило: **аутентификация — на входе, авторизация — на КАЖДОЕ действие**. «Вошёл» не значит
«можно всё». Перед каждым доступом к ресурсу проверяй: имеет ли ИМЕННО этот пользователь право
ИМЕННО на это. Пропуск этой проверки — классическая уязвимость (модуль 11).

🖼️
```
   запрос → [ AUTHN: кто ты? ] → вошёл как user42
          → [ AUTHZ: а тебе можно? ] → user42 запрашивает /orders/777 — это ЕГО заказ? нет → 403
   ❗ забыли второй шаг → user42 читает чужие заказы (IDOR). Authn без authz = дыра.
```

---

## ⭐ Атаки на аутентификацию (и защита)

```
   🔓 ПОДБОР ПАРОЛЯ (brute force / по словарю / credential stuffing из утечек):
      🛡️ лимиты попыток, задержки, блокировки, CAPTCHA, MFA, мониторинг неудачных входов.
   🔓 СЛАБЫЕ ПАРОЛИ:
      🛡️ требования к длине (длина важнее «спецсимволов»), проверка по базам утёкших, поощрять менеджеры.
   🔓 УТЕЧКА/ПЕРЕХВАТ:
      🛡️ только HTTPS; не логировать пароли; правильное хранение (хеш+соль, модуль 17).
   🔓 ОБХОД ВОССТАНОВЛЕНИЯ (слабые «секретные вопросы», предсказуемые токены сброса):
      🛡️ надёжные токены сброса (случайные, одноразовые, с истечением); MFA.
   🔓 КРАЖА/ФИКСАЦИЯ СЕССИИ:
      🛡️ безопасные cookie (HttpOnly, Secure, SameSite), смена id сессии после входа, истечение.
```

💡 ⭐ **MFA** — самая мощная защита аутентификации: даже зная пароль, без второго фактора не войти
(перекликается с [гигиеной из соц. инженерии](../../SocialEng/04-defense/18-personal-hygiene.md)).

## ⭐⭐ Авторизация: проверяй на сервере, всегда

```
   ГЛАВНЫЕ ОШИБКИ авторизации:
   ❌ проверка прав только на клиенте (скрыли кнопку — но запрос-то можно послать напрямую!).
   ❌ доверие к данным от клиента ("role=admin" в запросе/токене без проверки).
   ❌ нет проверки владения объектом (взял чужой id → получил чужое: IDOR, модуль 11).
   ❌ «всё или ничего» вместо ролей/прав на конкретные действия.

   🛡️ ПРАВИЛЬНО:
   ✅ авторизацию проверяй на СЕРВЕРЕ при КАЖДОМ доступе к ресурсу.
   ✅ проверяй, что объект принадлежит/доступен именно этому пользователю.
   ✅ роли/права — минимально необходимые (наименьшие привилегии).
   ✅ по умолчанию ЗАПРЕЩЕНО (deny by default), доступ — явно разрешённый.
```

💡 ⭐⭐ «Запрещено по умолчанию» — ключевой принцип: новый эндпоинт/ресурс закрыт, пока ты явно не
разрешил кому надо. Обратное («открыто, забыл закрыть») — источник утечек. Это [наименьшие
привилегии](01-attacker-mindset.md) на уровне доступа.

## 📖 Токены и сессии (коротко)

```
   • СЕССИЯ — сервер помнит вошедшего (id сессии в безопасной cookie).
   • ТОКЕН (например, JWT) — самодостаточный «пропуск»; ⚠️ его НЕЛЬЗЯ слепо доверять:
     проверяй подпись, срок, не клади секреты внутрь (он читаем), умей отзывать.
   • OAuth/OIDC — делегированный вход («войти через...») — не пиши своё, бери проверенное (модуль 10 трека Senior: build vs buy).
   общее: защищай токены/сессии как пароли (HTTPS, безопасные cookie, истечение, отзыв).
```

---

## ⚠️ Ловушки

- ❌ Путать «вошёл» (authn) и «можно» (authz) — проверять только вход.
- ❌ Проверять права только на клиенте (обходится прямым запросом).
- ❌ Доверять `role`/`user_id` из клиентского запроса без серверной проверки.
- ❌ Не проверять владение объектом (IDOR).
- ❌ Слабые токены сброса/«секретные вопросы».
- ❌ «Открыто по умолчанию» вместо «запрещено по умолчанию».

---

## ✅ Упражнения (на своём/в лабе)

1. **Authn vs authz.** Для своего приложения опиши, где проверяешь «кто ты» и где «что можно».
   Проверяется ли authz на каждое действие?
2. **IDOR-тест.** Поменяй id ресурса в запросе на чужой. Получаешь чужое? Добавь проверку владения.
3. **Клиент vs сервер.** Найди проверку прав только на клиенте. Что будет при прямом запросе?
   Перенеси на сервер.
4. **Deny by default.** Проверь: новый эндпоинт у тебя закрыт по умолчанию или открыт?
