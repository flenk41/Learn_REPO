# 08 · CIA и основы криптографии 🖼️⭐

> 🎯 **Цель блока:** усвоить триаду CIA (фундамент безопасности) и базовые понятия криптографии,
> чтобы применять их правильно — и не изобретать своё.

---

## 📖 Триада CIA — три цели безопасности

```
   C — CONFIDENTIALITY (конфиденциальность): данные видят только те, кому можно.
       угроза: утечка. защита: шифрование, контроль доступа.
   I — INTEGRITY (целостность): данные не изменены незаметно.
       угроза: подмена. защита: хеши, подписи, валидация.
   A — AVAILABILITY (доступность): система работает, когда нужна.
       угроза: отказ (DoS, сбой). защита: резерв, лимиты, отказоустойчивость.
```

💡 ⭐ Любая защитная мера служит одной или нескольким буквам CIA. Это «компас»: спрашивай — какую
из C/I/A защищает эта мера и какую угрозу закрывает? Часто между ними [trade-off](../../Senior/02-decisions/08-tradeoffs.md)
(например, шифрование/проверки vs скорость).

🖼️
```
              БЕЗОПАСНОСТЬ
            /      |       \
   Конфиден-   Целост-    Доступ-
   циальность   ность      ность
   (не утечёт) (не подменят) (работает)
   защита любого ресурса = баланс всех трёх под его модель угроз.
```

## ⭐ Криптография: три инструмента

```
   1. ХЕШИРОВАНИЕ — односторонняя «свёртка» данных в фиксированную строку.
      • необратимо (нельзя вернуть исходник), детерминировано.
      • для: проверки целостности, хранения паролей (со спец. алгоритмами — модуль 17).
      • ⚠️ MD5/SHA1 — сломаны для безопасности; используй SHA-256+ (и спец-алгоритмы для паролей).

   2. СИММЕТРИЧНОЕ ШИФРОВАНИЕ — один ключ шифрует и расшифровывает (например, AES).
      • быстро; проблема — как безопасно передать ключ.
      • для: шифрования данных «на диске», больших объёмов.

   3. АСИММЕТРИЧНОЕ ШИФРОВАНИЕ — пара ключей: публичный (шифрует) + приватный (расшифровывает).
      • решает обмен ключами; медленнее.
      • для: TLS/HTTPS, подписей, обмена ключами. (см. TLS в треке Сети)
```

💡 ⭐ Эти три — кирпичи. Подпись = хеш + асимметрия (целостность + подлинность). TLS = асимметрия
для обмена ключом + симметрия для скорости. Понимая кирпичи, понимаешь, как устроена защита канала.

> 🧭 Связь с курсом: [TLS/HTTPS в Сетях](../../Network/03-application/14-tls.md) — криптография в
> действии для защиты трафика.

## ⭐⭐ Золотое правило: НЕ изобретай свою криптографию

```
   ❌ САМОЕ ОПАСНОЕ в крипте — написать своё. Даже эксперты ошибаются; самодельное почти всегда
      ломается. «Выглядит зашифрованным» ≠ безопасно.
   ✅ ИСПОЛЬЗУЙ проверенные библиотеки и стандартные алгоритмы (AES, SHA-256, bcrypt/argon2, TLS).
   ✅ НЕ придумывай свой «шифр», свой генератор случайных, своё хранение паролей.
   ✅ бери криптографически стойкий ГСЧ (не обычный random!) для токенов/ключей/солей.
```

💡 ⭐⭐ Это частный случай [build vs buy (Senior)](../../Senior/02-decisions/10-build-vs-buy.md):
криптография — ровно та область, где «написать самим» = почти гарантированная дыра. Твоя задача —
**правильно применить** готовое, а не изобрести алгоритм. 90% реальных крипто-уязвимостей — это
неправильное использование (слабый алгоритм, переиспользование, плохая случайность), а не взлом
самого шифра.

## 📖 Частые ошибки применения

```
   ❌ хранить пароли хешем MD5/SHA без соли (или вообще в открытом виде) — модуль 17.
   ❌ свой «шифр»/XOR с фиксированным ключом вместо AES.
   ❌ обычный random() для токенов/паролей (предсказуем) вместо криптостойкого ГСЧ.
   ❌ хардкод ключей в коде/репозитории (модуль 19).
   ❌ переиспользование одного и того же nonce/IV.
   ❌ передача чувствительного без TLS (по HTTP).
```

## ⚠️ Ловушки

- ❌ Писать свою криптографию/«шифр» — почти всегда дыра.
- ❌ Путать кодирование (base64) и шифрование — base64 НЕ защищает, легко обратимо.
- ❌ Использовать MD5/SHA1 для безопасности (сломаны).
- ❌ Обычный `random` для секретов вместо криптостойкого.
- ❌ Думать, что «зашифровал» = «безопасно», не разобравшись в применении.
- ❌ Игнорировать CIA — защищать одно (конфиденциальность), забыв целостность/доступность.

## ✅ Упражнения

1. **CIA-разбор.** Для своего приложения: какие меры защищают C, какие I, какие A? Где пробел?
2. **Хеш vs шифр vs кодирование.** Объясни разницу на примере (пароль / данные на диске / base64
   в URL). Что для чего?
3. **Свой проект.** Найди, нет ли самодельной «крипты»/слабых хешей/хардкод-ключей. Замени на
   стандартное.
4. **Случайность.** Проверь, чем генерируешь токены/соли — обычный random или криптостойкий ГСЧ?
