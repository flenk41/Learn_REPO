# 05 · Командная строка и Linux для безопасности 🖼️

> 🎯 **Цель блока:** освоить базу командной строки и Linux, без которой не обойтись в безопасности —
> и для анализа, и для защиты, и для автоматизации.

> ⚠️ Все команды применяй к своим/лабораторным системам.

---

## 📖 Почему CLI и Linux

```
   • большинство серверов/инфраструктуры — Linux. Защищаешь то, что понимаешь.
   • инструменты безопасности — в основном CLI; автоматизация — скриптами.
   • анализ логов, прав, процессов, сети — через терминал быстрее и точнее.
   • «думать как атакующий» проще, понимая, как устроена ОС изнутри (привет, трек ОС).
```

💡 Это смыкается с треком [ОС](../../OS/00-setup/02-tools.md): процессы, права, файловая система,
сеть. Здесь — прикладной, защитный взгляд на те же вещи.

---

## ⭐ Базовый набор для безопасности

```
   ФАЙЛЫ И ПРАВА:
   ls -la         — файлы + права + владельцы (права — основа контроля доступа)
   chmod / chown  — менять права/владельца (наименьшие привилегии!)
   find           — искать файлы (например, с опасными правами 777, или secrets)

   ПРОЦЕССЫ И РЕСУРСЫ:
   ps aux / top   — что запущено (подозрительные процессы)
   kill           — остановить процесс

   СЕТЬ:
   ss -tulpn      — открытые порты и слушающие сервисы (поверхность атаки!)
   ip / ifconfig  — сетевые интерфейсы

   ЛОГИ И ТЕКСТ (анализ — суперсила защитника):
   grep           — поиск по логам ("найди все неудачные входы")
   cat/less/tail  — читать; tail -f — следить за логом в реальном времени
   journalctl     — системные логи (systemd)

   АВТОМАТИЗАЦИЯ:
   bash-скрипты, pipe (|), редиректы (>, >>) — связывать команды в инструменты.
```

🖼️
```
   пример защитного однострочника (анализ логов):
   grep "Failed password" /var/log/auth.log | wc -l   → сколько попыток подбора пароля
   grep "Failed password" auth.log | awk '{print $11}' | sort | uniq -c | sort -rn
       → с каких IP больше всего попыток (кандидаты на блокировку)
```

## ⭐ Права доступа — фундамент защиты в Linux

```
   права rwx (чтение/запись/выполнение) для владельца/группы/остальных:
   -rw-r--r--   файл: владелец читает+пишет, остальные только читают
   chmod 600 secret.key   → только владелец (наименьшие привилегии для секретов!)
   ❌ chmod 777 — доступ всем на всё: классическая уязвимость конфигурации.
```

💡 ⭐ Неправильные права — частая дыра: секреты/конфиги, доступные всем; запись туда, где не надо.
Правило: **минимально необходимые права**, особенно для ключей, конфигов, скриптов. `find / -perm
-0777` помогает найти слишком открытые файлы.

---

## 📖 Скрипты как защитные инструменты

```
   из простых команд собираются полезные инструменты защиты:
   • мониторинг логов на подозрительное (grep + уведомление).
   • проверка прав/конфигов на опасные значения.
   • резервное копирование по расписанию (cron).
   • базовая проверка целостности файлов (хеши: меняется ли важный файл).
   → автоматизация = защита не «вручную иногда», а постоянно. (Уровень 3 — пишем такие инструменты.)
```

> 🧭 Скрипты и автоматизация — это и [Senior-ремесло](../../Senior/01-craft/05-refactoring.md) на
> практике: маленькие надёжные инструменты, делающие одно дело хорошо.

## ⚠️ Ловушки

- ❌ `chmod 777` «чтоб заработало» — открываешь дыру.
- ❌ Запускать всё под root/с лишними правами (нарушение наименьших привилегий).
- ❌ Хранить секреты в открытых файлах с широкими правами.
- ❌ Копипастить команды из интернета не понимая (особенно с `sudo`/`rm`).
- ❌ Игнорировать логи — там видно атаки, если уметь искать.
- ❌ Практиковать «опасные» команды на рабочей системе вместо ВМ.

## ✅ Упражнения (в лабе/на своей ВМ)

1. **Порты.** Выполни `ss -tulpn` — какие сервисы слушают? Это твоя поверхность атаки. Что лишнее?
2. **Права.** Найди файлы с правами 777 (`find . -perm -0777`). Стоит ли их ужесточить?
3. **Логи.** В учебной системе найди неудачные попытки входа через `grep` и посчитай по IP.
4. **Скрипт.** Напиши маленький bash-скрипт, который ищет в логе подозрительные строки и выводит сводку.
