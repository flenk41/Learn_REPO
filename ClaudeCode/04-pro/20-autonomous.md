# 20 · Автономные циклы и фоновые задачи 🖼️⭐

> 🎯 **Цель блока:** понять, как Claude Code работает над долгими задачами автономно —
> длинные циклы, фоновые процессы — и как делать это **безопасно**.

---

## 📖 От «шаг-ответ» к долгой автономной работе

До сих пор мы вели агента вручную: задача → результат → следующая. Но Claude Code силён в
**долгой автономной работе**: дай ему ясную цель и критерий — он сам пройдёт много шагов до
результата (крупный рефактор, прогон до зелёных тестов, «ночной» прогон задачи).

🖼️
```
   ручной режим:  ты ─► шаг ─► смотришь ─► ты ─► шаг ─► смотришь ...
   автономный:    ты ─► ЦЕЛЬ+критерий ─► агент сам: шаг→проверка→шаг→...
                                          → пока критерий не выполнен
```

💡 Чтобы автономность работала хорошо — дай **полную спецификацию сразу** (принцип «контекст
вперёд», модуль 04) и **проверяемый критерий «готово»** (тесты зелёные, поведение такое-то).
Чем яснее цель, тем дальше агент пройдёт без сбоев.

---

## ⭐ Условие успеха — проверяемая цель

Автономный агент должен **сам понимать**, когда он закончил. Поэтому:

```
   ✅ «доведи, пока `pytest` не станет зелёным» — машинно-проверяемо
   ✅ «реализуй фичу так, чтобы сценарий X отрабатывал» — проверяемо
   ❌ «сделай красиво» — агент не знает, когда остановиться
```

💡 Дай агенту **обратную связь, которую он может сам прочитать** (тесты, линтер, сборка).
Тогда цикл «сделал → проверил себя → исправил» крутится сам. Без проверяемого критерия
автономность вырождается в блуждание.

---

## 📖 Фоновые задачи

Долгие команды (сборка, тесты, дев-сервер) можно запускать **в фоне**, чтобы агент не
блокировался и мог продолжать работу, поглядывая на результат. Это полезно, когда часть
работы долгая, а агенту есть чем заняться параллельно.

```
   агент: запускает долгий прогон в фоне
        │ продолжает другую работу
        ▼ периодически проверяет вывод фоновой задачи
   готово → реагирует на результат
```

---

## ⭐ Безопасность автономности

Чем самостоятельнее агент, тем важнее **ограничители**:

```
   🛟 разрешения: deny на деструктивное (rm -rf, force-push), ask на рискованное
   🛟 hooks: блокировка опасных команд, авто-тесты как контроль качества
   🛟 git: частые коммиты как чек-поинты — всегда есть откат
   🛟 песочница/ветка: пускай автономный прогон в отдельной ветке, не в main
   🛟 наблюдение: поглядывай, прерывай (Esc), если пошло не туда
```

⚠️ Автономность — не «запустил и ушёл навсегда». Это «дал ясную цель, поставил
ограничители, наблюдаешь, проверяешь итог». Чем выше ставки — тем плотнее контроль.

---

## ⚠️ Ловушки

- ❌ Автономный прогон без проверяемого критерия → агент «не знает», когда стоп.
- ❌ Запуск в `main`/проде без ветки и без частых коммитов.
- ❌ Широкие разрешения у автономного агента (может натворить дел).
- ❌ «Запустил и забыл» на важной задаче.

---

## 🛠️ Практика

1. Поставь автономную задачу с машинным критерием («доведи тесты до зелёных») в отдельной
   ветке. Понаблюдай цикл сделал→проверил→исправил.
2. Запусти долгую команду в фоне и дай агенту работать параллельно.
3. Включи ограничители: deny на деструктив, hook с авто-тестами, частые коммиты.
4. Намеренно дай размытую цель и заметь, как агент «плавает» — затем уточни критерий.
