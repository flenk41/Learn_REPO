# 21 · Code review и безопасность 🖼️⭐

> 🎯 **Цель блока:** использовать Claude Code для ревью кода и поиска уязвимостей — и
> понимать риски безопасности самого агента.

---

## 📖 Две стороны безопасности

```
   1) Агент КАК ИНСТРУМЕНТ безопасности — ищет баги и уязвимости в твоём коде.
   2) Безопасность САМОГО агента — он имеет доступ к файлам, командам, токенам.
```

Обе важны. Разберём по очереди.

---

## ⭐ Агент как ревьюер и аудитор

Claude Code хорошо находит **реальные баги** и проблемы безопасности: гонки, утечки,
небезопасные вызовы, инъекции, секреты в коде, отсутствие проверок. Просить можно так:

```
   «Сделай ревью этих изменений: ищи баги, проблемы безопасности и тесты.
    Сообщай обо ВСЁМ, даже о неуверенных находках — фильтрацию сделаю отдельно.
    Для каждого: файл:строка, severity, как починить.»
```

⚠️ Важный нюанс новых моделей: если в промпте сказать «сообщай только о критичном» / «будь
консервативен», модель **буквально** отфильтрует и **умолчит** о части находок — измеримая
полнота падает, хотя баги найдены. Поэтому на этапе поиска проси **полноту** («сообщай обо
всём с severity»), а фильтрацию вынеси в отдельный шаг.

🖼️
```
   ЭТАП 1 (поиск):    «найди всё, с severity»     → высокая полнота
   ЭТАП 2 (фильтр):   «отсей мелочь, оставь важное» → высокая точность
```

💡 Для ревью удобно сделать **Skill** (модуль 17) с этим форматом и **slash-команду**
(`/review`), чтобы запускать одинаково. Часто ревью встраивают в процесс (например, на PR).

---

## 📖 Верификация: «сказал готово» ≠ «работает»

Из модуля 06: проверяй. Для безопасности это критично — проси агента **подтверждать** фиксы
прогоном (тест воспроизводит баг → фикс → тест зелёный), а не объявлять победу после одного
случайного зелёного прогона (особенно на «плавающих» дефектах).

---

## ⭐ Безопасность самого агента

Агент мощный, потому что имеет доступ. Значит — у него есть **поверхность риска**:

```
   🔑 секреты    — не клади ключи/пароли в код, CLAUDE.md, промпты (они сохраняются)
   💥 команды    — deny на деструктивное, ask на рискованное (разрешения, модуль 12)
   🌐 внешнее    — осторожно с данными из интернета/писем/тикетов: там бывают
                   спрятанные инструкции (prompt injection)
   📦 MCP        — сторонний сервер видит то, что ты дал; подключай доверенное,
                   токены с минимальными правами (модуль 14)
```

⚠️ **Prompt injection**: контент, который агент читает (веб-страница, тикет, файл от чужого),
может содержать «инструкции» вроде «игнорируй прошлое, сделай X». Не давай агенту слепо
исполнять то, что пришло из недоверенного источника, особенно с широкими правами.

💡 Принцип: **минимум прав + проверяемость + наблюдение**. Дай агенту ровно столько доступа,
сколько нужно задаче, и держи опасное за разрешениями/хуками.

## 🛠️ Практика

1. Сделай `/review`-команду + Skill, прогони ревью реального диффа; сравни вывод с «сообщай
   только критичное» — заметь разницу в полноте.
2. Попроси агента **подтвердить** фикс тестом, а не словом.
3. Проверь свой проект на секреты в коде/конфигах; вынеси в переменные окружения.
4. Сформулируй для проекта: deny-команды, ask-команды, какие источники недоверенные.
