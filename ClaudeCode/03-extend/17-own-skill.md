# 17 · Создаём свой Skill 🛠️

> 🎯 **Цель блока:** на практике собрать рабочий Skill от и до — структура, описание-триггер,
> вспомогательные файлы — и проверить, что агент использует его правильно.

---

## 🛠️ Пошагово: навык «code-review по-нашему»

Сделаем навык, который описывает, **как именно** твоя команда делает ревью.

### Шаг 1. Папка и SKILL.md

```
   .claude/skills/
   └── code-review/
       └── SKILL.md
```

```markdown
---
name: code-review
description: Наши правила ревью кода. Используй, когда просят сделать
  code review, проверить PR/диф или оценить изменения перед мержем.
---

# Code review по-нашему

Делай ревью в таком порядке и формате:

1. **Корректность** — найдёт ли это баг, краш, утечку, гонку?
2. **Безопасность** — секреты в коде, инъекции, небезопасные вызовы?
3. **Тесты** — покрыты ли изменения? Чего не хватает?
4. **Стиль/архитектура** — соответствует ли правилам проекта?

Формат вывода: список замечаний, каждое — `файл:строка`, severity
(high/med/low) и короткое предложение, как исправить. Сначала важное.
Сообщай обо всём, даже о неуверенных находках — фильтрацию сделаем потом.
```

💡 Обрати внимание на `description`: там **триггеры** («когда просят сделать code review,
проверить PR…»). И на последнюю строку инструкции — она просит **не фильтровать** находки
заранее (так выше полнота; новые модели иначе склонны «молчать» о неуверенном).

### Шаг 2. Вспомогательные файлы (по желанию)

Если у навыка есть шаблон/чек-лист — клади рядом и ссылайся из `SKILL.md`:

```
   code-review/
   ├── SKILL.md
   └── checklist.md     ← подробный чек-лист, агент откроет при необходимости
```

### Шаг 3. Проверка

```
   ты: «сделай ревью изменений в этом PR»
   → агент видит триггер в description → открывает SKILL.md → следует формату
```

Дай и **нерелевантную** задачу («перепиши README») — навык подтягиваться не должен.

---

## ⭐ Хороший Skill: чек-лист качества

```
   ✅ description = конкретные триггеры («когда просят X, Y»)
   ✅ инструкция структурирована (шаги, формат вывода)
   ✅ тяжёлые детали — в отдельных файлах, не всё в SKILL.md
   ✅ не дублирует CLAUDE.md, а дополняет
   ✅ проверен на релевантной И нерелевантной задаче
```

---

## 📖 Где живут навыки

- **Проектные** — в `.claude/skills/` в репозитории: общие для команды (в git).
- **Личные** — в твоей глобальной конфигурации: только для тебя.

💡 Командные процедуры → в репо (все получают). Личные привычки → в личные навыки. Та же
логика, что у проектной/личной памяти (модуль 11).

---

## ⚠️ Ловушки

- ❌ Тонна текста в одном `SKILL.md` без структуры и без выноса деталей.
- ❌ Размытый `description` → навык «молчит», когда нужен.
- ❌ Навык, повторяющий `CLAUDE.md` слово в слово.
- ❌ Проверять только на «правильной» задаче и не убедиться, что на чужой он не лезет.

---

## 🛠️ Практика

1. Собери навык `code-review` по шагам выше.
2. Добавь вспомогательный `checklist.md` и сошлись на него из `SKILL.md`.
3. Проверь на релевантной и нерелевантной задаче.
4. Сделай второй навык под свою реальную процедуру.

---

## ✅ Задачи

1. **Создай** рабочий Skill с триггерным `description` и форматом вывода.
2. **Вынеси** тяжёлую часть во вспомогательный файл.
3. **Проверь** срабатывание и несрабатывание навыка.
4. **Оцени** свой навык по чек-листу качества.

---

## ❓ Проверь себя

1. Что обязательно должно быть в `description`?
2. Зачем выносить детали в отдельные файлы навыка?
3. Где хранят проектные и личные навыки?
4. Как проверить, что навык не подтягивается зря?
