# 16 · Skills — навыки агента 🖼️⭐

> 🎯 **Цель блока:** понять Skills — папки с инструкциями, которые агент **подгружает только
> когда они нужны**, не засоряя контекст постоянно.

---

## 📖 Проблема: куда девать «как мы это делаем»

У команды есть специфические процедуры: «как мы оформляем релиз», «как пишем миграции»,
«наш стиль отчётов». Засунуть всё в `CLAUDE.md`? Он **всегда** в контексте — раздуется и
будет жрать окно каждую сессию (модуль 11). Нужен способ держать инструкции «под рукой», но
грузить их **только по необходимости**.

Это и есть **Skills** (навыки).

🖼️
```
   CLAUDE.md   → всегда в контексте (общие правила, кратко)
   Skill       → лежит на диске; в контексте только короткое ОПИСАНИЕ.
                 Полную инструкцию агент читает, ТОЛЬКО когда задача подходит.
```

💡 Это называется **прогрессивное раскрытие** (progressive disclosure): по умолчанию агент
видит лишь «у меня есть навык X для таких-то случаев». Когда задача совпала — он открывает
файл навыка и читает детали. Контекст не засоряется заранее.

---

## ⭐ Как устроен Skill

Skill — это **папка** с файлом `SKILL.md` (и при необходимости вспомогательными файлами):

```
   .claude/skills/
   └── release-notes/
       ├── SKILL.md          ← описание + инструкция навыка
       └── template.md       ← вспомогательный файл (по желанию)
```

`SKILL.md` начинается с метаданных (имя + краткое описание — именно описание видно агенту
всегда), дальше — подробная инструкция:

```markdown
---
name: release-notes
description: Как мы оформляем заметки к релизу. Используй, когда просят
  подготовить release notes или changelog.
---

# Как мы делаем release notes

1. Собери merged PR с прошлого тега.
2. Сгруппируй по: Features / Fixes / Breaking.
3. Используй шаблон template.md, тон — деловой, без воды.
...
```

💡 Поле `description` — самое важное: по нему агент решает, **когда** подтянуть навык. Пиши
там конкретные триггеры («когда просят X»). Точный формат сверяй с документацией.

---

## 📖 Skills vs MCP vs команды — не путать

```
   MCP-сервер     → новые ИНСТРУМЕНТЫ (действия: сходить в API, в БД)
   Skill          → ЗНАНИЕ/ПРОЦЕДУРА (как правильно делать), грузится по нужде
   slash-команда  → готовый ПРОМПТ-ШАБЛОН, который ты запускаешь вручную (/review)
   CLAUDE.md      → постоянные правила проекта (всегда в контексте)
```

💡 Грубо: MCP даёт **руки**, Skill даёт **знание**, команда — **кнопку**, CLAUDE.md —
**постоянную памятку**. Часто их комбинируют.

---

## ⚠️ Ловушки

- ❌ Писать в `description` расплывчато → навык не подтянется в нужный момент.
- ❌ Дублировать в навыке то, что уже в `CLAUDE.md`.
- ❌ Раздувать `SKILL.md` без структуры — агенту тяжело извлечь суть.
- ❌ Путать навык (знание) с инструментом (действие).

---

## 🛠️ Практика

1. Найди в своей работе процедуру, которую часто объясняешь («как мы делаем X»).
2. Оформи её как Skill: папка + `SKILL.md` с чётким `description`-триггером.
3. Дай агенту задачу, подпадающую под навык, — проследи, что он его подтянул.
4. Дай задачу **не** по теме навыка — убедись, что навык не подгрузился (контекст чист).
