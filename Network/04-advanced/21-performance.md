# 21 · Производительность сети 🖼️⭐

> 🎯 **Цель блока:** понять, от чего зависит «быстро/медленно» в сети — latency, throughput,
> MTU, буферы — и почему «больше мегабит» не всегда значит «быстрее».

---

## ⭐ Два разных «быстро»: latency и throughput

```
   LATENCY (задержка)    — сколько ВРЕМЕНИ идёт один сигнал туда (и обратно, RTT).
                           Измеряется в мс. «Как быстро отзывается».
   THROUGHPUT (пропускная) — сколько ДАННЫХ проходит за секунду (Мбит/с).
                           «Сколько влезает в трубу».
```

🖼️
```
   аналогия — труба с водой:
   latency    = за сколько вода доберётся до конца трубы (длина трубы)
   throughput = сколько литров в секунду проходит (толщина трубы)
```

💡 Это **разные** вещи. Толстая, но длинная труба: много данных, но первый отклик медленный
(спутник: throughput большой, latency огромный). Для игр/звонков критична **latency**, для
скачивания — **throughput**. «100 Мбит/с» ничего не говорит о задержке.

---

## 📖 Из чего складывается задержка

```
   latency = распространение (скорость света в среде, расстояние)
           + сериализация (время «вытолкнуть» биты в канал)
           + обработка (роутеры думают)
           + очереди (пакет ждёт в буферах загруженных роутеров)
```

💡 Расстояние не обмануть: до сервера на другом континенте сигнал физически идёт десятки мс
(каждый. раз). Поэтому контент приближают к пользователю — CDN (модуль 22). Очереди в буферах
(**bufferbloat**) тоже добавляют задержку при загрузке канала.

---

## ⭐ RTT и почему «круги» дорогие

**RTT** (Round-Trip Time) — время туда-обратно. Многие протоколы тратят несколько RTT на старт:

```
   TCP handshake     — 1 RTT
   TLS handshake     — ещё 1–2 RTT
   потом уже запрос  — ещё RTT
   → на далёкий сервер «пустые» круги складываются в заметную задержку
```

💡 Поэтому современные протоколы **сокращают число RTT** на старте (TLS 1.3, QUIC/HTTP3 —
модуль 18). Меньше кругов = быстрее «первый байт». Для далёких серверов это решает.

---

## 📖 MTU, окно и потери

Связь с уровнем 2:

```
   MTU (модуль 07)   — слишком большие пакеты → фрагментация → тормоз
   окно TCP (модуль 12) — мало «в полёте» → канал недозагружен; контроль перегрузки регулирует
   потери пакетов    → переотправки (TCP) → проседание скорости; на длинных каналах особенно больно
```

💡 На «длинных толстых» каналах (большой RTT + большая пропускная) важен достаточный размер
**окна**, иначе данные ждут подтверждений и труба простаивает. Потери бьют сильнее при большом RTT.

---

## ⭐ Как улучшают производительность

```
   📍 приближают данные   — CDN, кэш, edge (модуль 22)
   🔁 меньше RTT          — TLS 1.3, HTTP/2/3, keep-alive (переиспользование соединений)
   🗜️ меньше данных        — сжатие, кэширование, оптимизация ассетов
   ⚙️ настройка             — размеры окон/буферов, борьба с bufferbloat
```

💡 Часто «ускорить сайт» = не «больше мегабит», а **меньше кругов и ближе сервер**. Понимание
latency vs throughput меняет подход к оптимизации.

---

## ⚠️ Ловушки

- ❌ Путать latency и throughput; считать «100 Мбит/с» = «низкая задержка».
- ❌ Думать, что расстояние не важно — физику не обмануть, RTT растёт с дистанцией.
- ❌ Игнорировать число RTT на старте (handshakes) — на далёких серверах это главное.
- ❌ Гнаться за пропускной для задач реального времени, где решает задержка.

---

## 🛠️ Практика

1. `ping` к местному и зарубежному серверу — сравни RTT (задержку), почувствуй роль расстояния.
2. Тест скорости (speedtest) покажет и throughput, и ping (latency) — разведи эти числа.
3. В DevTools (Network) посмотри «Time to First Byte» и общее время — увидь вклад задержки.
4. Сравни загрузку сайта с CDN и без (если найдёшь пример) — заметь разницу.
