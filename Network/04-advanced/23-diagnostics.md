# 23 · Диагностика сети 🛠️⭐

> 🎯 **Цель блока:** собрать арсенал диагностики и методику «почему не работает сеть» — от
> `ping` до Wireshark, по слоям снизу вверх.

---

## ⭐ Методика: диагностируй по слоям

Когда «не работает интернет/сайт», не гадай — иди **по слоям снизу вверх** (модуль 01):

```
   1. Физика/канал  — есть ли линк? Wi-Fi подключён? кабель воткнут?
   2. IP (сеть)      — есть ли адрес? пингуется ли шлюз? (ip addr, ping шлюз)
   3. Маршрут        — пингуется ли внешний IP? (ping 8.8.8.8) — есть ли выход наружу
   4. DNS            — резолвится ли имя? (ping 8.8.8.8 ок, а ping google.com — нет → DNS)
   5. Транспорт/прил.— открывается ли порт/сайт? (curl, telnet/nc на порт)
```

🖼️
```
   «сайт не грузится»
        ▼
   ping 8.8.8.8 работает?  ──нет──► проблема ниже (IP/маршрут/шлюз/физика)
        │да
   ping google.com работает? ──нет──► проблема DNS
        │да
   curl https://сайт работает? ──нет──► проблема на прикладном уровне/сервере/TLS
```

💡 Эта лестница за минуту локализует проблему: «у меня нет адреса», «нет выхода наружу», «DNS
сломан», «конкретный сайт лежит». Не перебирай наугад — двигайся по слоям.

---

## 📖 Инструменты и что они показывают

```
   ping            — доступность хоста + RTT (задержка)
   traceroute/tracert — путь по роутерам, где «обрывается» или растёт задержка
   ip/ifconfig     — есть ли у меня адрес, какой интерфейс, шлюз
   ip route/route  — таблица маршрутов, default gateway
   dig/nslookup    — работает ли DNS, какой IP у имени
   curl -v         — весь HTTP/TLS-обмен по слоям
   ss/netstat      — мои соединения и слушающие порты
   nc (netcat)     — открыть ли TCP/UDP-порт вручную (жив ли сервис на порту)
   tcpdump/Wireshark — реальные пакеты (последний рубеж: видно всё)
```

💡 Связка: `ping` (жив?) → `traceroute` (где обрыв?) → `dig` (DNS?) → `curl`/`nc` (порт/сервис?)
→ Wireshark (что в пакетах?). От простого к подробному.

---

## ⭐ Wireshark/tcpdump — когда ничего не ясно

Когда команды не дали ответа, смотри **реальные пакеты**:

```
   видишь свой SYN, но нет SYN+ACK?  → сервер/порт недоступен или firewall режет
   DNS-запрос ушёл, ответа нет?       → проблема DNS-сервера
   TLS handshake обрывается?          → проблема сертификата/версии TLS
   куча повторов (retransmission)?    → потери/перегрузка на пути
```

💡 Wireshark показывает **правду на проводе** — то, что реально ушло и пришло. Это финальный
аргумент в споре «у меня всё работает» vs «не работает».

⚠️ Этика (модуль 02): смотри **свой** трафик. Перехват чужого без разрешения — незаконно.

---

## 📖 Типовые диагнозы

```
   нет адреса (169.254.x.x / нет IP)  → DHCP/линк проблема
   ping шлюза не идёт                 → локальная сеть/Wi-Fi/кабель
   ping 8.8.8.8 идёт, имена нет       → DNS
   конкретный сайт не открывается, другие да → сервер/маршрут до него/блокировка
   медленно, но работает              → задержка/потери (traceroute, RTT, окно)
```

💡 Держи в голове эту таблицу «симптом → слой». Большинство бытовых проблем диагностируются
за пару команд.

## ⚠️ Ловушки

- ❌ Чинить наугад («перезагружу всё»). Иди по слоям — быстрее и понятнее.
- ❌ Делать вывод «хост мёртв» по одному `ping` — ICMP могут блокировать (модуль 02).
- ❌ Игнорировать DNS как причину — очень частый виновник «сайт не грузится».
- ❌ Лезть в Wireshark сразу — сначала простые команды, пакеты — когда нужно подробно.

---

## 🛠️ Практика

1. Пройди лестницу диагностики на рабочем соединении: `ping шлюз` → `ping 8.8.8.8` →
   `ping google.com` → `curl -v`. Зафиксируй, что на каждом шаге подтверждается.
2. Сломай DNS специально (укажи неверный DNS-сервер во временной конфигурации) и увидь:
   `ping 8.8.8.8` работает, а имена — нет. Верни обратно.
3. Проверь, «жив» ли конкретный порт сервиса: `nc -vz host 443`.
4. В Wireshark найди неудачное соединение (SYN без ответа) на каком-нибудь закрытом порту.
