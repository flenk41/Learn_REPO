# 24 · Облако и современная сетевая инфраструктура 🖼️⭐

> 🎯 **Цель блока:** увидеть, как сетевые основы работают в облаке и контейнерах — виртуальные
> сети, контейнеры, оркестрация — и собрать всё в инженерную картину.

---

## 📖 Облако = чужие компьютеры + виртуальные сети

**Облако** (AWS, GCP, Azure и др.) — это дата-центры, которые сдают вычисления и сеть «по
запросу». Вся сетевая теория трека остаётся, но строится **виртуально, программно**:

```
   VPC (виртуальная приватная сеть) — твоя изолированная сеть в облаке (свои подсети)
   subnet                           — подсети внутри VPC (модуль 03)
   security group / firewall        — правила доступа (модуль 19)
   load balancer                    — балансировщик как сервис (модуль 22)
   public/private IP, NAT gateway   — те же приватные/публичные адреса и NAT (модули 03,19)
```

💡 То есть в облаке ты **проектируешь сеть кодом/настройками**: подсети, маршруты, firewall,
балансировщики. Понимание основ из этого трека — фундамент для облака.

---

## ⭐ Контейнеры и их сеть

**Контейнер** (Docker) — изолированная упаковка приложения. У контейнеров — **своя сеть**:

```
   хост
   ├── контейнер A (свой IP в виртуальной сети контейнеров)
   ├── контейнер B
   └── виртуальный мост/сеть соединяет их + проброс портов наружу
        (-p 8080:80 → порт 80 контейнера виден как 8080 на хосте — это NAT!)
```

💡 Проброс портов контейнера (`-p`) — это тот же **NAT/перенаправление портов** (модуль 19).
Контейнеры общаются по обычным TCP/IP, просто в виртуальной сети. Сетевые основы работают и тут.

---

## ⭐ Оркестрация (Kubernetes) — сеть в масштабе

Когда контейнеров много, ими управляет **оркестратор** (Kubernetes). Он добавляет сетевые
абстракции поверх всего трека:

```
   service       — стабильный адрес для группы контейнеров (встроенная балансировка)
   ingress       — вход извне, маршрутизация по хостам/путям (как умный реверс-прокси)
   DNS внутри    — сервисы находят друг друга по именам (модуль 06 — внутри кластера)
   network policy — firewall между сервисами (модуль 19/20)
```

💡 Каждая «магия» Kubernetes — это знакомые кирпичики: DNS, балансировка, прокси, firewall,
подсети. Зная основы, ты понимаешь, что происходит «под капотом» облака, а не зубришь кнопки.

---

## 📖 Инженерная картина: всё вместе

Собери путь запроса к современному облачному сервису:

```
   пользователь
     → DNS (модуль 06) → IP балансировщика/CDN
     → CDN отдаёт статику из близкого узла (модули 21,22)
     → запрос к API: TCP+TLS (модули 09,14), HTTP/2-3 (модуль 18)
     → ingress/балансировщик (модуль 22) → один из контейнеров (stateless)
     → контейнер читает состояние из общей БД/кэша
     → ответ JSON (модуль 16) тем же путём обратно
```

💡 Это и есть итог трека: ты видишь **весь путь** и понимаешь каждый слой — от DNS и TCP до
CDN, контейнеров и масштабирования. Сеть перестала быть «магией».

## ⭐ Куда расти дальше

```
   • глубже в TCP/QUIC, тюнинг производительности
   • сетевая безопасность (pentest СВОИХ систем, сертификаты, zero-trust)
   • облачные сети (VPC, service mesh), Kubernetes networking
   • наблюдаемость: метрики, трассировка, анализ трафика
   • протоколы: gRPC, WebRTC, MQTT под конкретные задачи
```

💡 Фундамент у тебя есть: слои, адреса, маршрутизация, TCP/UDP, HTTP/TLS, диагностика. Дальше —
специализация под твою область (бэкенд, devops, безопасность, embedded/IoT).
