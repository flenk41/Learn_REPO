# 22 · Балансировка, CDN и масштабирование 🖼️⭐

> 🎯 **Цель блока:** понять, как сервисы выдерживают миллионы пользователей — балансировщики
> нагрузки, CDN, кэширование и горизонтальное масштабирование.

---

## 📖 Проблема: один сервер не справится

Когда пользователей много, один сервер перегружается: не хватает CPU, памяти, канала.
Решений два направления:

```
   вертикальное масштабирование  — сделать сервер мощнее (предел есть, дорого)
   горизонтальное масштабирование — поставить МНОГО серверов и распределять нагрузку ⭐
```

💡 В вебе побеждает **горизонтальное**: добавляешь серверы по мере роста. Но тогда нужен тот,
кто **распределит** запросы между ними — балансировщик.

---

## ⭐ Балансировщик нагрузки (load balancer)

**Load balancer** стоит перед группой серверов и раздаёт им входящие запросы:

🖼️
```
   пользователи ──► БАЛАНСИРОВЩИК ──► сервер 1
                                  ├─► сервер 2
                                  └─► сервер 3
        (распределяет: по очереди, по нагрузке, по «здоровью» серверов)
```

💡 Балансировщик: (1) **распределяет** нагрузку, (2) **проверяет здоровье** серверов и убирает
упавшие из ротации, (3) даёт **отказоустойчивость** (один сервер упал — другие работают).
Стратегии: round-robin, по наименьшей загрузке, «прилипание» сессии и др.

---

## ⭐ CDN — контент ближе к пользователю

Из модуля 21: расстояние = задержка. **CDN** (Content Delivery Network) — сеть серверов по
всему миру, которые **кэшируют** контент рядом с пользователями.

```
   без CDN: пользователь в Азии ──── далеко ────► сервер в США  (большой RTT)
   с CDN:   пользователь в Азии ─► ближайший узел CDN (копия рядом) (маленький RTT)
```

💡 CDN отдаёт статику (картинки, видео, скрипты) из ближайшей точки → быстрее и разгружает
основной сервер. Заодно поглощает всплески и помогает против DDoS (модуль 20).

---

## 📖 Кэширование на всех уровнях

```
   браузер     — кэширует ответы (Cache-Control, модуль 13)
   CDN/edge    — кэширует контент ближе к пользователю
   обратный прокси — кэширует ответы перед серверами
   приложение  — кэширует результаты (например, в памяти/Redis)
```

💡 Кэш = «не делать одну работу дважды». Самый быстрый запрос — тот, что не дошёл до сервера
(ответ из кэша). Но кэш надо **инвалидировать** вовремя (через TTL/версии), иначе пользователи
видят устаревшее.

## ⭐ Состояние и масштабирование

При многих серверах возникает вопрос: где хранить **состояние** (сессии, данные)?

```
   ❌ хранить сессию в памяти конкретного сервера → следующий запрос попал на другой → «разлогинило»
   ✅ выносить состояние в общее хранилище (БД, кэш) → серверы «без состояния» (stateless)
      → любой сервер обслужит любой запрос → легко добавлять/убирать серверы
```

💡 **Stateless-серверы** — ключ к лёгкому горизонтальному масштабированию. Состояние — в общих
хранилищах. (Помнишь, HTTP сам по себе stateless — модуль 13? Это удобно именно для масштаба.)

---

## ⚠️ Ловушки

- ❌ Полагаться только на вертикальное масштабирование — упрёшься в потолок и в цену.
- ❌ Хранить сессии в памяти отдельного сервера за балансировщиком — пользователей будет
  «выкидывать». Выноси состояние.
- ❌ Забывать про инвалидацию кэша — устаревшие данные у пользователей.
- ❌ Думать, что CDN ускоряет всё. Динамику (персональные ответы) он кэшировать не может.

---

## 🛠️ Практика

1. По заголовкам ответа крупного сайта (`curl -I`) поищи признаки CDN (`Server`, `Via`, `CF-...`,
   `X-Cache`) и `Cache-Control`.
2. `traceroute` к большому сайту из разных «точек» (или сравни с другом из другой страны) —
   часто упрёшься в близкий CDN-узел, а не в далёкий дата-центр.
3. Объясни, почему статика грузится быстро, а персональная страница (профиль) — медленнее.
