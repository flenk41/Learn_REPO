# 15 · Прикладные протоколы 🖼️

> 🎯 **Цель блока:** обзор главных прикладных протоколов поверх TCP/UDP — кроме HTTP: DNS, SSH,
> SMTP, FTP, WebSocket — чтобы видеть всю «экосистему» интернета.

---

## 📖 Каждой задаче — свой протокол

HTTP — самый известный, но интернет держится на множестве прикладных протоколов. У каждого
свой порт и своя задача:

```
   протокол   порт      задача                      транспорт
   ───────────────────────────────────────────────────────────
   HTTP/HTTPS 80/443    веб-страницы, API           TCP
   DNS        53        имена → адреса              UDP (часто), TCP (большие)
   SSH        22        безопасный удалённый доступ TCP
   SMTP       25/587    отправка почты              TCP
   IMAP/POP3  143/110   получение почты             TCP
   FTP        21        передача файлов (устаревает) TCP
   WebSocket  80/443    двусторонний канал в реальном времени TCP
   NTP        123       синхронизация времени       UDP
```

💡 Зная порт, часто можно угадать службу (443 → веб, 22 → SSH, 53 → DNS). Большинство «пишущих»
протоколов на TCP (нужна надёжность), «живые» запрос-ответы вроде DNS/NTP — на UDP.

---

## ⭐ SSH — безопасный доступ

**SSH** (Secure Shell) — зашифрованный удалённый доступ к серверу (терминал, передача файлов
через `scp`/`sftp`, проброс портов). Заменил небезопасный `telnet`.

```
   ssh user@server   → шифрованная сессия → ты в терминале удалённой машины
```

💡 SSH — рабочая лошадка администрирования. Аутентификация по паролю или (лучше) по **ключам**
(пара публичный/приватный, как у TLS). Всё шифровано.

---

## 📖 Почта: SMTP + IMAP/POP3

```
   отправка: твой клиент → SMTP → почтовый сервер → SMTP → сервер получателя
   получение: клиент получателя ← IMAP/POP3 ← его сервер
```

💡 SMTP **отправляет/пересылает** письма между серверами, IMAP/POP3 — **забирают** их клиенту.
Куда слать письмо для домена — подсказывает DNS-запись **MX** (модуль 06).

---

## ⭐ WebSocket — двусторонний реальный канал

Обычный HTTP: клиент спросил — сервер ответил, и всё. А если серверу нужно **сам** слать данные
клиенту (чат, уведомления, биржевые котировки)? Для этого — **WebSocket**:

```
   HTTP-запрос «давай перейдём на WebSocket» (upgrade)
        ▼
   постоянный двусторонний канал: и клиент, и сервер шлют, когда захотят
```

💡 WebSocket начинается как HTTP (порт 80/443), затем «апгрейдится» в постоянное соединение.
Идеален для реального времени поверх веба, в отличие от «запрос-ответ» HTTP.

---

## 📖 FTP и почему он устаревает

**FTP** передаёт файлы, но **без шифрования** (логин/данные идут открыто). Поэтому его заменяют
на **SFTP** (через SSH) или передачу файлов по HTTPS.

💡 Общий тренд: незашифрованные протоколы (FTP, telnet, обычный HTTP) уступают шифрованным
(SFTP, SSH, HTTPS). Безопасность стала нормой.

---

## ⚠️ Ловушки

- ❌ Думать, что «интернет = HTTP». HTTP — лишь один из многих протоколов.
- ❌ Использовать незашифрованные протоколы (FTP, telnet) там, где есть безопасные аналоги.
- ❌ Путать SMTP (отправка) и IMAP/POP3 (получение).
- ❌ Тянуть HTTP-«опрос» там, где нужен WebSocket (реальное время).

---

## 🛠️ Практика

1. `ssh` на любую доступную машину (или прочитай, как настроить ключи) — пойми принцип.
2. Найди MX-запись почтового домена: `dig gmail.com MX`.
3. Открой сайт с чатом/обновлениями в реальном времени и в DevTools (Network → WS) найди
   WebSocket-соединение.
4. По выводу `ss`/`netstat` определи службы по номерам портов.
