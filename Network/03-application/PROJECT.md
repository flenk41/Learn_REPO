# 🚀 Пет-проект уровня 3 — «Свой сетевой сервис + клиент API»

> 🎯 **Цель:** собрать практику прикладного уровня: написать свой TCP-сервис, сделать клиента к
> настоящему REST API и разобрать HTTP/TLS вживую.

---

## 📋 Что делаем

Две связанные части: (A) свой простой сетевой сервис на сокетах и (B) клиент к публичному
веб-API — плюс разбор HTTP/TLS.

### Часть A. Свой TCP-сервис

1. Напиши TCP-сервер (Python, модуль 17), который делает что-то осмысленное: например,
   мини-«база» — принимает команды `SET ключ значение` / `GET ключ` и хранит их в памяти.
2. Напиши клиента к нему.
3. Реши проблему **границ сообщений** (разделитель `\n` или длина впереди) — объясни в отчёте,
   почему это нужно (TCP — поток байт).
4. Поймай обмен в Wireshark: handshake, команды, закрытие.

### Часть B. Клиент к REST API

5. Выбери публичный API (например, GitHub, погода open-meteo, любой открытый). Сделай запросы
   через curl и/или код:
   - успешный `GET` → разбери JSON;
   - запрос несуществующего ресурса → получи `404`;
   - посмотри коды состояния и заголовки (`-i`).
6. Если API требует ключ — храни его в **переменной окружения**, не в коде.

### Часть C. HTTP/TLS-разбор

7. `curl -v https://<api>` → найди в выводе: DNS→IP, TCP-соединение, **TLS handshake**, HTTP-
   запрос и ответ. Открой сертификат сайта в браузере — опиши CA и срок.
8. В DevTools определи версию HTTP (h2/h3) и найди (если есть) WebSocket.

### Отчёт

`report.md`: описание своего сервиса и протокола сообщений, разбор обмена в Wireshark, примеры
запросов к API с кодами/JSON, разбор HTTP/TLS, выводы про слои.

---

## ✅ Критерии готовности

- [ ] Работает свой TCP сервер+клиент с осмысленным протоколом
- [ ] Решена проблема границ сообщений, обмен пойман в Wireshark
- [ ] Сделан клиент к реальному REST API (успех + ошибка + коды)
- [ ] Ключ (если нужен) хранится безопасно
- [ ] Разобран HTTP/TLS-обмен и версия HTTP
