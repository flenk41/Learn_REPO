# 12 · Надёжность TCP подробно: окна и контроль перегрузки 🖼️⭐

> 🎯 **Цель блока:** заглянуть глубже в механику TCP — скользящее окно, контроль потока и
> контроль перегрузки — чтобы понять, как TCP быстр и при этом не «топит» сеть.

---

## 📖 Проблема: слать по одному пакету — медленно

Если ждать ACK после **каждого** пакета — впустую тратится время на круги туда-обратно.
Решение — слать **много пакетов сразу**, не дожидаясь подтверждения каждого. Сколько именно —
определяет **окно**.

🖼️
```
   наивно (по одному):  шлю1 → жду ACK → шлю2 → жду ACK ...   медленно
   окно:                шлю 1,2,3,4,5 сразу → подтверждения идут пачкой  быстро
```

---

## ⭐ Скользящее окно (sliding window)

**Окно** — это сколько байт отправитель может послать «в полёте», не получив подтверждения.
По мере прихода ACK окно «скользит» вперёд.

```
   [отправлено и подтверждено][отправлено, ждём ACK][можно слать][пока нельзя]
                              └────── окно ──────┘
   пришёл ACK → окно сдвигается вправо → можно слать ещё
```

💡 Окно — баланс между скоростью (больше в полёте = быстрее) и аккуратностью (не завалить
получателя и сеть). Размер окна не фиксирован — он подстраивается (см. ниже).

---

## ⭐ Контроль потока (flow control) — не завалить получателя

Получатель имеет ограниченный **буфер** памяти. Он сообщает отправителю свой **receive window** —
«у меня свободно столько-то». Отправитель не шлёт больше, чем поместится.

```
   получатель: «моё окно приёма = 64 КБ» (есть место в буфере)
   ... буфер заполняется ...
   получатель: «окно = 8 КБ» → отправитель сбавляет
   получатель: «окно = 0»   → отправитель ждёт, пока освободится
```

💡 ⭐ Снова связь с памятью: контроль потока защищает **буфер памяти** получателя от
переполнения. Сеть быстрая, а получатель может не успевать обрабатывать — TCP это учитывает.

---

## ⭐ Контроль перегрузки (congestion control) — не завалить сеть

Отдельная задача — не перегрузить **саму сеть** (роутеры между вами). TCP **не знает** заранее,
сколько сеть выдержит, поэтому **прощупывает**:

```
   старт: шлю осторожно, наращиваю окно (slow start — рост скорости)
   потеря пакета? → это признак перегрузки → резко СБАВЛЯЮ окно
   снова наращиваю аккуратнее ...
```

🖼️
```
   скорость
     ▲        /\        /\
     │       /  \      /  \      «пила»: растём, при потере — спад, снова растём
     │      /    \    /
     └──────────────────────► время
            ↑ потеря   ↑ потеря
```

💡 Потеря пакета TCP трактует как «сеть перегружена» и **снижает темп**. Так миллионы TCP-
соединений делят общие каналы, не устраивая коллапс. Это саморегуляция всего интернета.

---

## 📖 Почему это объясняет поведение сети

```
   «скорость скачивания разгоняется не сразу» → slow start (окно растёт постепенно)
   «при плохой связи скорость падает рывками»  → потери → спад окна → восстановление
   «получатель тормозит — отправитель ждёт»     → flow control (окно приёма мало)
   «один потерянный пакет тормозит весь поток»  → head-of-line blocking в TCP
```

💡 Эти механизмы объясняют реальную скорость соединений лучше, чем «просто канал такой».
TCP постоянно подстраивается под получателя и сеть.

---

## ⚠️ Ловушки

- ❌ Путать **контроль потока** (защита получателя) и **контроль перегрузки** (защита сети) —
  это разные механизмы с похожими названиями.
- ❌ Думать, что TCP всегда жмёт на максимум. Он осторожен: растёт постепенно, при потерях сбавляет.
- ❌ Считать потерю пакета «просто ошибкой» — для TCP это **сигнал** снизить скорость.

## 🛠️ Практика

1. В Wireshark при загрузке файла найди поле **Window** в TCP-заголовках и проследи, как оно
   меняется.
2. Скачай большой файл и понаблюдай: скорость разгоняется не мгновенно (slow start).
3. Объясни на своём опыте «рывки» скорости при слабом Wi-Fi через контроль перегрузки.
