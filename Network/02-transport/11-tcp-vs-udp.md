# 11 · TCP vs UDP — когда что выбрать 🖼️⭐

> 🎯 **Цель блока:** научиться осознанно выбирать транспорт под задачу — свести воедино
> сильные и слабые стороны TCP и UDP.

---

## 📖 Один вопрос решает почти всё

```
   Тебе важнее, чтобы дошло ВСЁ и ПО ПОРЯДКУ — или чтобы дошло БЫСТРО и СВЕЖЕ?

        полнота/порядок важнее  →  TCP
        свежесть/скорость важнее →  UDP
```

💡 Это главный критерий. Файл, страница, письмо, платёж — нельзя терять байты → **TCP**.
Живое видео, голос, игра, телеметрия — устаревшие данные не нужны → **UDP**.

---

## ⭐ Сводная таблица выбора

| Критерий | TCP | UDP |
|---|---|---|
| Соединение | да (handshake) | нет |
| Надёжность доставки | гарантирует | нет |
| Порядок | сохраняет | нет |
| Скорость старта | медленнее | быстрее |
| Накладные расходы | выше | минимум |
| Модель данных | поток байт | сообщения (дейтаграммы) |
| Потеря пакета | переотправит (тормоз) | пропустит (двигаемся дальше) |
| Типичное применение | веб, файлы, почта, БД | видео, голос, игры, DNS, IoT |

🖼️
```
   надо НАДЁЖНО          надо БЫСТРО/ЖИВО
   ┌─────────────┐       ┌─────────────┐
   │  HTTP/HTTPS │       │  видеозвонок│
   │  файлы (FTP)│  TCP  │  онлайн-игра│  UDP
   │  почта SMTP │       │  DNS-запрос │
   │  SSH, БД    │       │  стриминг   │
   └─────────────┘       └─────────────┘
```

---

## 📖 Разбор примеров

```
   Скачивание файла   → TCP: потеряешь байт — файл битый. Полнота критична.
   Видеозвонок        → UDP: потерянный кадр устарел; ждать его = «лаги». Свежесть.
   Веб-страница        → TCP: HTML/JS должны прийти целиком и по порядку.
   DNS-запрос          → UDP: крошечный обмен, рукопожатие — лишняя задержка.
   Онлайн-шутер        → UDP: позиция игрока обновляется 60 раз/с; старая не нужна.
   Платёж / банк       → TCP: ни единого потерянного байта.
```

💡 Заметь HTTP/3 (модуль 18): веб исторически на TCP, но новый веб переезжает на **QUIC поверх
UDP**, добирая надёжность сверху — чтобы убрать минусы TCP, сохранив гарантии. Граница «веб=TCP»
размывается.

## ⭐ Чек-лист выбора транспорта

```
   1. Потеря данных недопустима?           да → TCP
   2. Порядок строго важен?                да → TCP
   3. Это файл/документ/транзакция?        да → TCP
   4. Это реальное время (голос/видео/игра)? да → UDP
   5. Крошечный запрос-ответ, важна задержка? да → UDP (как DNS)
   6. Нужна скорость UDP + частичная надёжность? → UDP + своя логика / QUIC
```

---

## ⚠️ Ловушки

- ❌ «TCP всегда безопаснее, бери его». Для реального времени TCP-переотправки = лаги; UDP лучше.
- ❌ «UDP быстрее, бери его везде». Для файлов/денег потеря байта = катастрофа.
- ❌ Забывать, что выбор можно гибридить (надёжность поверх UDP там, где надо).
- ❌ Путать «надёжность» (TCP) с «безопасностью» (шифрование — это TLS, отдельно).
