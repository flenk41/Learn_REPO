# 🚀 Пет-проект уровня 1 — «Корзина интернет-магазина»

> 🎯 **Цель:** спроектировать связку классов с правильной инкапсуляцией, композицией и
> инвариантами — на классической задаче, где всё это естественно.

---

## 📋 Что делаем

Моделируем корзину/заказ интернет-магазина — несколько классов, которые **общаются через
методы**, с защищёнными инвариантами и продуманными отношениями.

### Классы (минимум)

- **Товар** — название, цена (неизменяемый объект-значение). Инвариант: цена ≥ 0.
- **Позиция** — товар + количество. Метод `стоимость()`. Инвариант: количество > 0.
- **Корзина** — содержит позиции (композиция). Методы: `добавить(товар, кол-во)`,
  `убрать(товар)`, `итого()`, `количество_товаров()`.
- **Клиент** — имя, баланс (приватный!). Методы: `оплатить(сумма)` (защищает «баланс ≥ 0»).
- **Заказ** — корзина + клиент. Метод `оформить()`: считает итого, просит клиента оплатить
  (через **сообщение**, не трогая баланс напрямую), возвращает результат.

### Требования (важно!)

1. **Инкапсуляция:** все поля приватные, наружу — только осмысленные методы. Никакого
   `клиент.баланс -= ...` снаружи.
2. **Инварианты:** цена ≥ 0, количество > 0, баланс ≥ 0 — защищены в конструкторах/методах.
3. **Композиция:** корзина владеет позициями; заказ связывает корзину и клиента.
4. **Сообщения:** `оформить()` просит клиента оплатить, клиент сам решает (может отказать).
5. **Неизменяемость:** `Товар` — неизменяемое значение.

### Сценарий

Создай товары → собери корзину → оформи заказ при достаточном балансе (успех) и при
недостаточном (отказ). Проверь, что инварианты не дают создать «сломанные» объекты.

## ✅ Критерии готовности

- [ ] Все поля приватные, доступ только через методы
- [ ] Инварианты защищены (нельзя создать невалидный объект)
- [ ] Корзина владеет позициями (композиция)
- [ ] Оформление идёт через сообщения, а не прямой доступ
- [ ] Товар — неизменяемое значение
- [ ] Проверены сценарии успеха и отказа
