# 24 · Когда ООП не нужно 🖼️⭐

> 🎯 **Цель блока:** обрести зрелость Senior — понимать **границы** ООП и сочетать парадигмы.
> Лучший инженер выбирает инструмент под задачу, а не молится одной парадигме.

---

## 📖 ООП — не золотой молоток

ООП мощно для систем с **сущностями, состоянием и поведением** (бизнес-домен, GUI, игры). Но это
**не** универсальное решение. Для некоторых задач другие парадигмы чище.

```
   👍 ООП силён:                      👎 ООП избыточен/неудобен:
   - бизнес-домены (заказы, счета)     - чистые вычисления/обработка данных
   - GUI, игры (объекты с состоянием)  - простые скрипты и утилиты
   - системы с чёткими сущностями      - конвейеры преобразования данных
   - где важна расширяемость           - математика, статистика
```

💡 ⭐ Senior-зрелость: **не оборачивать всё в классы**. Если задача — «прочитать файл,
преобразовать, записать» — это конвейер функций, а не иерархия классов. Класс ради класса =
лишняя сложность.

---

## ⭐ Функциональный стиль — сильная альтернатива

```
   ФУНКЦИОНАЛЬНОЕ программирование:
   - функции БЕЗ состояния (чистые: тот же вход → тот же выход, без побочных эффектов)
   - неизменяемые данные (модуль 07)
   - композиция функций: data |> filter |> map |> reduce
```

💡 Для **обработки данных** функциональный стиль часто чище ООП: нет скрытого состояния, легко
тестировать (чистые функции), безопасно в потоках (нет общих изменяемых данных — вспомни гонки в
[треке ОС](../../OS/03-sync/13-race-conditions.md)). Вспомни итераторы/`map/filter/reduce` из
[Rust](../../Rust/03-middle/18-closures-iterators.md) и
[Python](../../Python/03-middle/15-generators-iterators.md) — это функциональный стиль.

## ⭐ Современные языки сочетают парадигмы

```
   почти все современные языки МУЛЬТИПАРАДИГМЕННЫ:
   Python, C++, Rust, Kotlin, Scala, JS — и ООП, и функциональный стиль
```

💡 ⭐ Реальный код — **смесь**: объекты для моделирования домена + чистые функции для обработки
данных + неизменяемые значения. Не «ООП vs ФП», а «ООП **и** ФП, каждое к месту». Например: домен
из объектов (Заказ, Клиент), но расчёт скидок — чистые функции; состояние в объектах, но
преобразования — функциональные конвейеры.

---

## ⭐ Принцип выбора: задача, а не мода

```
   спроси себя:
   • есть сущности с состоянием и поведением, которые меняются вместе? → ООП
   • это преобразование данных вход→выход без состояния? → функции
   • нужна расширяемость новыми вариантами поведения? → ООП (полиморфизм)
   • простой одноразовый скрипт? → просто функции, без церемоний
```

💡 Не существует «правильной парадигмы вообще» — есть подходящая под задачу (как нет «лучшего»
TCP/UDP или модели ИИ — выбор по контексту). Знание ООП **и** его границ — признак зрелости.

## 📖 Итог трека: что делает Senior

```
   1. видит ОБЪЕКТЫ в задаче и моделирует домен
   2. владеет четырьмя столпами (абстракция, инкапсуляция, наследование, полиморфизм)
   3. применяет SOLID и принципы ОСОЗНАННО, не догматично
   4. предпочитает композицию наследованию, программирует к интерфейсу
   5. знает паттерны как общий язык, но не лепит их везде
   6. распознаёт запахи и рефакторит (с тестами)
   7. думает об архитектуре: слои, зависимости к бизнес-логике
   8. ВЫБИРАЕТ парадигму под задачу, сочетает ООП и функциональный стиль
```

💡 Главный вывод: проектирование — это **управление сложностью и изменениями**. ООП — мощный
инструмент для этого, но именно **инструмент**. Senior владеет им и знает, когда взять другой.

## ⚠️ Ловушки

- ❌ «ООП-фундаментализм» — всё в классы, даже простой скрипт.
- ❌ Игнорировать функциональный стиль для обработки данных.
- ❌ Считать парадигмы враждующими (они дополняют друг друга).
- ❌ Выбирать парадигму по моде/привычке, а не по задаче.

---

## 🛠️ Практика

1. Возьми задачу обработки данных (фильтр-преобразование-агрегация) и реши её **функционально**
   (без классов). Сравни с ООП-версией — где чище?
2. Возьми задачу с сущностями (магазин) и реши **объектно**. Объясни, почему здесь ООП уместнее.
3. Найди в своём коде класс, который стоило бы заменить парой функций (и наоборот).
