# 🚀 Финальный проект — «Спроектируй систему по-настоящему»

> 🎯 **Цель:** собрать весь трек — от объектов и столпов до SOLID, паттернов и архитектуры — в
> одном осмысленном приложении, спроектированном как сделал бы Senior.

---

## 📋 Что делаем

Выбираем небольшую, но **настоящую** систему и проектируем её целиком: от модели домена до
архитектуры, осознанно применяя принципы и паттерны (и зная, где их **не** применять).

### Идеи систем (выбери одну)

- 🎬 Сервис проката (фильмы, абоненты, аренда, тарифы)
- 📦 Система заказов (товары, корзина, оплата, доставка, уведомления)
- 🎮 Простая игра (сущности, поведение, состояния)
- 📝 Менеджер задач (задачи, проекты, статусы, напоминания)
- 🏦 Мини-банк (счета, транзакции, типы операций)

### Что должно быть в проекте

**Моделирование (уровень 0–1):**
- [ ] Выделены объекты домена с состоянием и поведением.
- [ ] Инкапсуляция: приватные поля, защищённые инварианты, Tell-Don't-Ask.
- [ ] Композиция там, где «имеет»; наследование только при честном «является».

**Столпы и принципы (уровень 2–3):**
- [ ] Полиморфизм вместо `if по типу` (расширяемость новыми классами).
- [ ] Программирование к интерфейсам; зависимости инвертированы (DIP).
- [ ] Классы следуют SRP; система открыта к расширению (OCP).
- [ ] Баланс: где надо — гибко (SOLID), где просто — просто (KISS/YAGNI).

**Паттерны и архитектура (уровень 4):**
- [ ] Применены 2–3 уместных паттерна (например, Strategy для тарифов/скидок, Factory для
      создания, Observer для уведомлений) — **только там, где они решают реальную проблему**.
- [ ] Деление на слои; бизнес-логика отделена от деталей (БД/UI за интерфейсами).
- [ ] Хотя бы одну деталь (хранилище) можно сменить, не трогая домен.

**Зрелость (уровень 4):**
- [ ] Где-то осознанно использован **функциональный** стиль (обработка данных) — не всё в классы.
- [ ] Нет God Object, анемичной модели, лишних абстракций.

### Артефакты

1. **Код** работающей системы с понятной структурой.
2. **UML-схема** классов и зависимостей (mermaid/бумага).
3. **README** с обоснованием решений: какие принципы/паттерны и **почему**; где сознательно
   **не** применял (KISS/YAGNI); где взял функциональный стиль и почему.
4. (По желанию) пара тестов с заглушками — доказательство тестируемости (DIP).

## ✅ Критерии готовности (выпускные)

- [ ] Домен смоделирован объектами с инкапсуляцией и правильными отношениями
- [ ] Применены четыре столпа; полиморфизм вместо if-по-типу
- [ ] Соблюдён SOLID осознанно, с балансом KISS/YAGNI
- [ ] Уместно применены 2–3 паттерна, есть слои и отделение бизнес-логики
- [ ] Видна зрелость: нет переусложнения, парадигмы смешаны по делу
- [ ] Есть код, UML-схема и README с обоснованием решений

## 🌟 Усложнение (для мастерства)

- Проведи **код-ревью** своего проекта: найди оставшиеся запахи, отрефактори.
- Добавь новое требование (новый тариф, канал уведомления, формат экспорта) — и докажи, что
  меняешь систему **добавлением**, а не переписыванием.
- Сравни свой дизайн с тем, как ту же задачу решили бы **процедурно** и **функционально** — что
  где уместнее?
