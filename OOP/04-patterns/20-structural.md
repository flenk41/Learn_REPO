# 20 · Структурные паттерны 🖼️⭐

> 🎯 **Цель блока:** освоить структурные паттерны — как **компоновать** объекты в большие
> структуры: Adapter, Decorator, Facade, Composite.

---

## 📖 Структурные паттерны — про композицию объектов

Они отвечают на вопрос «как **собрать** объекты вместе, чтобы получить нужную структуру и
гибкость». Все построены на **композиции** (уровень 3) — объекты содержат/оборачивают другие.

---

## ⭐ Adapter (Адаптер) — «переходник»

**Проблема:** есть объект с одним интерфейсом, а нужен другой (несовместимы).

```
   твой код ждёт интерфейс "Логгер" { записать(msg) }
   а у сторонней библиотеки — "ThirdPartyLog" { log(level, text) }

   Adapter оборачивает чужой объект и выдаёт ОЖИДАЕМЫЙ интерфейс:
   class ЛоггерАдаптер(Логгер):
       def __init__(self, чужой): self._чужой = чужой
       def записать(self, msg): self._чужой.log("INFO", msg)   # переводит вызов
```

💡 Адаптер — «переходник» (как USB↔HDMI). Позволяет использовать несовместимый объект, не меняя
ни его, ни свой код. Частый случай — обёртка над сторонней библиотекой, чтобы изолировать её за
своим интерфейсом (заодно — DIP).

---

## ⭐ Decorator (Декоратор) — добавить поведение, обернув

**Проблема:** нужно добавить объекту поведение, не меняя его класс и не плодя наследников на
каждую комбинацию.

```
   базовый: Кофе { цена() = 100 }
   декораторы оборачивают и ДОБАВЛЯЮТ:
     СМолоком(кофе)  → цена() = кофе.цена() + 20
     СCиропом(СМолоком(кофе)) → +20 +30 = 150

   все реализуют ОДИН интерфейс Напиток → можно вкладывать друг в друга
```

🖼️
```
   [ Сироп [ Молоко [ Кофе ] ] ]   ← матрёшка декораторов, каждый добавляет своё
```

💡 ⭐ Декоратор заменяет **комбинаторный взрыв наследования** (КофеСМолокомИСиропом, КофеСМолоком,
…) на гибкое **оборачивание**. Добавляешь поведение слоями в рантайме. Это «композиция вместо
наследования» (модуль 16) в чистом виде.

---

## ⭐ Facade (Фасад) — простой вход в сложную систему

**Проблема:** подсистема сложная (много классов, шагов), а пользователю нужен простой вход.

```
   сложно: ВидеоКодек + АудиоКодек + Контейнер + Битрейт... (10 объектов, порядок вызовов)
   Facade: КонвертерВидео.конвертировать(файл, формат)   ← одна простая кнопка
           # внутри фасад дёргает все подсистемы в нужном порядке
```

💡 Фасад прячет сложность за простым интерфейсом (привет, абстракция и инкапсуляция). Снаружи —
одна понятная операция; внутри — оркестровка многих объектов. Не убирает сложность, а **скрывает**
её от тех, кому детали не нужны.

---

## ⭐ Composite (Компоновщик) — дерево «часть-целое»

**Проблема:** нужно одинаково работать с **отдельным** объектом и с **группой** объектов.

```
   Файл и Папка реализуют общий интерфейс "Элемент { размер() }"
   Папка СОДЕРЖИТ элементы (файлы И папки) → размер() = сумма размеров детей
   → рекурсивное дерево; клиент зовёт размер() одинаково для файла и папки
```

💡 Composite строит **деревья** (файловая система, меню, UI-компоненты) и даёт работать с веткой
и листом **единообразно** через общий интерфейс (полиморфизм!). Вспомни «составной способ оплаты»
из проекта уровня 2 — это Composite.

---

## ⚠️ Ловушки

- ❌ Путать Adapter (меняет интерфейс) и Decorator (добавляет поведение, интерфейс тот же).
- ❌ Фасад, который сам разрастается в «божественный объект» — следи за SRP.
- ❌ Декораторы без общего интерфейса (вложение сломается).
- ❌ Применять паттерн, не узнав проблему (переусложнение).

---

## 🛠️ Практика

1. Напиши Adapter над «чужим» логгером/библиотекой к своему интерфейсу.
2. Реализуй Decorator: базовый напиток + декораторы-добавки, вкладываемые друг в друга.
3. Сделай Composite: файлы и папки с общим `размер()`, посчитай размер дерева.
