# 09 · Инкапсуляция (глубоко) 🖼️⭐⭐

> 🎯 **Цель блока (ЯДРО трека):** углубить инкапсуляцию — второй столп — до уровня
> проектирования: где провести границу объекта и почему это решает судьбу системы.

---

## ⭐⭐ Инкапсуляция = граница ответственности

В уровне 1 (модуль 05) инкапсуляция = «спрятать поля». Глубже это — **проведение границы**:
что объект держит **внутри** (его дело) и что показывает **наружу** (его контракт).

🖼️
```
   ┌──── граница объекта ────┐
   │  ВНУТРИ (private):       │
   │   состояние + логика     │  ← объект сам отвечает за свою корректность
   │  ─────────────────────── │
   │  СНАРУЖИ (public):       │
   │   интерфейс — контракт   │  ← обещание, как с ним работать
   └──────────────────────────┘
        другие объекты видят только контракт
```

💡 ⭐⭐ Объект сам **отвечает** за свои данные и инварианты. Никто снаружи не может привести его в
невалидное состояние, потому что не имеет доступа к внутренностям — только к методам, которые
сами следят за правилами. Это «объект — хозяин своих данных» из модуля 04, доведённое до
принципа проектирования.

---

## ⭐⭐ Tell, Don't Ask («говори, не спрашивай»)

Ключевая практика инкапсуляции:

```
   ❌ ASK:  if счёт.баланс >= сумма:        ← вытащили данные, решаем СНАРУЖИ
              счёт.баланс -= сумма
   ✅ TELL: счёт.снять(сумма)                ← сказали объекту, он решает САМ
              # внутри снять(): проверка + изменение
```

💡 ⭐⭐ Не **спрашивай** у объекта данные, чтобы принять решение за него — **скажи** ему, что
сделать, и пусть решает сам. Логика живёт там же, где данные. Иначе правила «расползаются» по
коду (в каждом месте, где кто-то спросил баланс, своя проверка) — и при изменении правила надо
искать все эти места. «Tell, Don't Ask» держит логику в одном месте — внутри объекта.

---

## ⭐ Закон Деметры («не разговаривай с незнакомцами»)

```
   ❌ заказ.клиент.адрес.город.индекс        ← цепочка вглубь чужих объектов
   ✅ заказ.индекс_доставки()                ← спроси заказ, он сам разберётся
```

💡 **Закон Деметры**: объект должен общаться только со своими **прямыми соседями**, а не лазить
по цепочке `a.b.c.d`. Длинные цепочки вызовов — признак, что ты знаешь слишком много о чужих
внутренностях (нарушение инкапсуляции). Они хрупкие: поменялась структура где-то в середине —
сломалось всё. Прячь навигацию за методами.

---

## 📖 Что значит «хорошая граница»

```
   ✅ узкий публичный интерфейс — минимум методов наружу
   ✅ объект нельзя сломать снаружи (инварианты защищены изнутри)
   ✅ внутреннее устройство можно менять, не трогая пользователей
   ✅ объект не отдаёт наружу изменяемые внутренности (отдаёт копии/неизменяемое)
```

💡 ⭐⭐ Чем меньше объект показывает наружу, тем легче его понимать, менять и тестировать. Хорошая
инкапсуляция — это про **зависимости**: чем уже контракт, тем меньше связей с остальной системой
(низкое зацепление, модуль 17). Это напрямую определяет, насколько система гибкая.

## ⚠️ Ловушки

- ❌ ASK-стиль: вытаскивать данные и решать снаружи (логика расползается).
- ❌ Длинные цепочки `a.b.c.d` (нарушение закона Деметры).
- ❌ Геттеры на всё — фактически открытые поля (мнимая инкапсуляция).
- ❌ Отдавать наружу внутренний изменяемый список/объект (его испортят снаружи).

---

## 🛠️ Практика

1. Найди в коде ASK-паттерн (вытащил данные → проверил снаружи) и перепиши в TELL (метод объекта).
2. Найди цепочку `a.b.c.d` и спрячь её за методом ближайшего объекта.
3. Проверь свой класс: можно ли сломать его инварианты снаружи? Закрой дыры.
