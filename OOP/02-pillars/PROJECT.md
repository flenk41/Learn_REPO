# 🚀 Пет-проект уровня 2 — «Система оплаты с разными способами»

> 🎯 **Цель:** применить все четыре столпа на классической задаче, где полиморфизм и интерфейсы
> сияют — обработка платежей разными способами.

---

## 📋 Что делаем

Проектируем систему, которая принимает оплату **разными способами** (карта, крипта, наличные,
бонусы), и которую легко расширять новыми способами — **не меняя** существующий код.

### Архитектура

1. **Интерфейс/абстрактный класс `СпособОплаты`** с методом `оплатить(сумма) -> Результат`
   (контракт — абстракция).
2. **Реализации:** `ОплатаКартой`, `ОплатаКриптой`, `ОплатаНаличными` — каждая реализует
   `оплатить` **по-своему** (полиморфизм), скрывая детали (инкапсуляция).
3. **Класс `Касса`** принимает **любой** `СпособОплаты` (программирование к интерфейсу) и
   обрабатывает платёж, **не зная** конкретный способ:
   ```
   касса.провести(способ, сумма)   # внутри: способ.оплатить(сумма)
   ```
4. **Результат** — неизменяемый объект (успех/отказ + сообщение).

### Требования (по столпам)

- 🎭 **Абстракция:** `Касса` работает с понятием «способ оплаты», не зная деталей.
- 🔒 **Инкапсуляция:** детали каждого способа (ключи, балансы, API) скрыты; наружу — `оплатить`.
  Применяй Tell-Don't-Ask, никаких `способ.внутренний_баланс` снаружи.
- 🧬 **Наследование:** общий контракт через абстрактный класс/интерфейс (честное «является
  способом оплаты»).
- 🔄 **Полиморфизм:** `Касса` вызывает `оплатить()` единообразно — каждый способ ведёт себя
  по-своему. **Никаких `if тип_способа == ...`**.

### Главная проверка

Добавь **новый** способ оплаты (например, `ОплатаБонусами`) — убедись, что `Касса` и остальной
код **не меняются** ни на строку. Если пришлось менять — дизайн нарушает полиморфизм/OCP.

### Отчёт

`README.md`: схема классов, где какой столп проявился, и демонстрация «добавил способ — старый код
цел».

---

## ✅ Критерии готовности

- [ ] Есть общий контракт `СпособОплаты` (абстракция/интерфейс)
- [ ] Несколько реализаций со скрытыми деталями (инкапсуляция)
- [ ] `Касса` работает полиморфно, без проверок типа
- [ ] Новый способ добавляется без правки существующего кода
- [ ] Результат — неизменяемый объект

---

## 🌟 Усложнение (по желанию)

- Добавь **скидки/комиссии** как отдельную абстракцию, подставляемую в кассу (готовим Стратегию,
  уровень 4).
- Сделай «составной» способ (сначала бонусы, остаток — картой) через **композицию** способов —
  он сам реализует тот же интерфейс `СпособОплаты` (паттерн Composite, уровень 4).
