# 12 · Интерфейсы и абстрактные классы 🖼️⭐

> 🎯 **Цель блока:** понять интерфейсы и абстрактные классы — как задавать **контракт**
> поведения, который делает полиморфизм надёжным.

---

## 📖 Интерфейс — контракт поведения

**Интерфейс** описывает, **что** объект должен уметь (набор методов), но **не как** (без
реализации). Это чистый контракт.

```
   интерфейс "Сериализуемый":
       to_json()       ← обещание: «я умею превращаться в JSON»
   (как именно — решает каждый класс, который реализует интерфейс)
```

🖼️
```
   interface Платёж:           ← КОНТРАКТ (только подписи методов)
       оплатить(сумма)
        ▲          ▲          ▲
   реализуют:                         (каждый по-своему)
   КартойПлатёж   КриптоПлатёж   НаличныеПлатёж
```

💡 Интерфейс — это **обещание**. Класс, который его реализует, обязан предоставить все методы.
Код, работающий с интерфейсом, может подставить **любую** реализацию (полиморфизм, модуль 11). В
Java/C#/Go — ключевое слово `interface`, в Rust — `trait`, в Python — абстрактные классы или
утиная типизация.

---

## ⭐ Абстрактный класс — частичный контракт + общий код

**Абстрактный класс** — нечто среднее: часть методов реализованы (общий код), часть —
**абстрактные** (обязаны реализовать потомки). Создать объект самого абстрактного класса нельзя.

```python
from abc import ABC, abstractmethod

class Фигура(ABC):
    @abstractmethod
    def площадь(self): ...           # обязан реализовать потомок
    def описание(self):              # общий готовый код
        return f"Площадь = {self.площадь()}"
```

💡 Абстрактный класс говорит: «у всех фигур есть площадь (как — решит потомок), но описание для
всех одинаковое». Нельзя создать «просто Фигуру» — только конкретную (Круг, Квадрат).

---

## ⭐ Интерфейс vs абстрактный класс

```
   ИНТЕРФЕЙС                       АБСТРАКТНЫЙ КЛАСС
   ────────────────────────────────────────────────────────
   только контракт (что уметь)     контракт + общий код + поля
   класс реализует МНОГО интерфейсов класс наследует ОДИН (обычно)
   «может делать» (способность)     «является» (тип/категория)
   Платёж, Сравнимый, Итерируемый   Фигура, Животное, БазовыйОбработчик
```

💡 ⭐ Грубое правило:
- **интерфейс** — когда описываешь **способность** («умеет оплачиваться», «можно сравнить»),
  которую могут иметь несовместимые классы;
- **абстрактный класс** — когда есть **общий код** и иерархия «является» с настоящим родством.

Современная практика склоняется к **интерфейсам** (+ композиция) — они гибче, не навязывают
иерархию, и класс может реализовать несколько (а наследовать обычно один).

## ⭐ Зачем это для проектирования: программируй к интерфейсу

```
   ❌ функция знает конкретный класс:   def оформить(платёж: КартойПлатёж)
   ✅ функция знает интерфейс:          def оформить(платёж: Платёж)
        → подставишь ЛЮБУЮ реализацию (карта, крипта, наличные)
        → добавишь новую — функция не меняется
```

💡 ⭐ Принцип «**программируй к интерфейсу, а не к реализации**» — один из главных в дизайне.
Зависимость от абстракции (интерфейса), а не от конкретики, делает код гибким и тестируемым
(можно подставить заглушку — мост к тестированию и к DIP, модуль 15).

---

## ⚠️ Ловушки

- ❌ Толстые интерфейсы (десятки методов) — реализациям приходится поддерживать лишнее (ISP,
  модуль 15).
- ❌ Абстрактный класс там, где хватило бы интерфейса (навязал иерархию зря).
- ❌ Зависеть от конкретных классов вместо интерфейсов (жёсткая связь).
- ❌ Создавать интерфейс с одной реализацией «на всякий случай» (YAGNI) — добавишь, когда понадобится.

---

## 🛠️ Практика

1. Опиши интерфейс `Хранилище` (`сохранить`, `загрузить`) и две реализации (в памяти, в файле).
2. Напиши функцию, принимающую `Хранилище` (интерфейс) — подставь обе реализации, не меняя функцию.
3. Сделай абстрактный класс `Фигура` с абстрактным `площадь()` и общим `описание()`.
