# 10 · Наследование 🖼️⭐⭐

> 🎯 **Цель блока (ЯДРО трека):** понять наследование — третий столп — как механизм
> переиспользования через «является», и почему им **легко злоупотребить**.

---

## ⭐⭐ Наследование — «является» (is-a)

**Наследование** позволяет создать класс на основе другого: потомок **получает** поля и методы
родителя и может добавлять/переопределять своё.

🖼️
```
   ┌─ Животное ──────────┐    ← РОДИТЕЛЬ (базовый класс)
   │ имя                  │
   │ есть()  спать()      │
   └──────────┬───────────┘
              │ наследуют (является)
      ┌───────┴────────┐
   ┌──▼───┐         ┌───▼───┐
   │ Собака│         │ Кошка │   ← ПОТОМКИ
   │ лаять()│        │мяукать()│  (+ своё поведение)
   └───────┘         └────────┘
```

```python
class Животное:
    def __init__(self, имя): self.имя = имя
    def есть(self): print(f"{self.имя} ест")

class Собака(Животное):          # Собака ЯВЛЯЕТСЯ Животным
    def лаять(self): print("Гав!")
```

💡 ⭐⭐ Тест: наследование уместно, только если **«потомок ЯВЛЯЕТСЯ родителем»** (Собака является
Животным). Это отношение **is-a** из модуля 06. Потомок получает поведение родителя «бесплатно»
и добавляет своё.

---

## ⭐⭐ Главная опасность: наследование злоупотребляют

Наследование выглядит удобно («переиспользую код!»), и поэтому им **часто злоупотребляют**. Это
самый проблемный из четырёх столпов.

```
   ❌ Стек наследует от Список («чтобы взять методы») → но Стек НЕ является Списком!
      теперь у Стека торчат лишние методы (вставка в середину), ломающие его суть

   ❌ глубокие иерархии (A→B→C→D→E) → изменение в A ломает всех потомков
```

💡 ⭐⭐ Проблема: наследование создаёт **жёсткую** связь. Потомок зависит от **внутренностей**
родителя; меняешь родителя — рискуешь сломать всех потомков (это называют «проблемой хрупкого
базового класса»). И если отношение не «является», а «имеет» — наследование тащит лишнее.

**Правило Senior:** «**предпочитай композицию наследованию**» (модуль 16). Наследуй только при
честном is-a и неглубоко.

## ⭐ Переопределение и вызов родителя

```python
class Собака(Животное):
    def есть(self):                 # ПЕРЕОПРЕДЕЛЯЕМ поведение родителя
        super().есть()              # можно вызвать родительское
        print(f"{self.имя} виляет хвостом")
```

💡 Потомок может **переопределить** метод родителя (изменить поведение) и при желании вызвать
родительскую версию через `super`. Это мост к полиморфизму (модуль 11): один вызов `есть()` ведёт
себя по-разному в зависимости от реального класса объекта.

⚠️ Множественное наследование (несколько родителей) есть в C++/Python, но усложняет («ромбовидная
проблема»). Java/C# его запрещают для классов, разрешая только для интерфейсов (модуль 12).

## 📖 Когда наследование оправдано

```
   ✅ честное «является» (Квадрат? — осторожно, см. LSP в модуле 15)
   ✅ неглубокая иерархия (1-2 уровня)
   ✅ родитель стабилен, потомки добавляют/уточняют поведение
   ✅ нужен полиморфизм через общий базовый тип (модуль 11)

   ❌ «просто чтобы переиспользовать код» → бери композицию
   ❌ отношение «имеет» → композиция
   ❌ глубокие деревья → пересмотри дизайн
```

---

## ⚠️ Ловушки

- ❌ Наследовать ради переиспользования кода (а не из-за «является»).
- ❌ Глубокие иерархии — хрупкие и непонятные.
- ❌ Потомок, ломающий ожидания от родителя (нарушение LSP, модуль 15).
- ❌ Считать наследование «главным» в ООП — важнее инкапсуляция и полиморфизм.

---

## 🛠️ Практика

1. Построй иерархию `Фигура → Круг/Прямоугольник` (честное «является») с методом `площадь()`.
2. Возьми пример злоупотребления (Стек наследует Список) и перепиши на **композицию** (Стек
   **содержит** список).
3. Переопредели метод в потомке с вызовом `super`.
