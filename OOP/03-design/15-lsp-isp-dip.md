# 15 · LSP, ISP, DIP 🖼️⭐

> 🎯 **Цель блока:** разобрать оставшиеся три принципа SOLID — подстановки Лисков (LSP),
> разделения интерфейсов (ISP) и инверсии зависимостей (DIP).

---

## ⭐ L — Liskov Substitution (подстановка Лисков)

**Объект потомка должен заменять объект родителя без сюрпризов.** Если код работает с `Птица`,
он должен работать с любым потомком — не ломаясь.

```
   ❌ классика нарушения: Квадрат наследует Прямоугольник
      у Прямоугольника: set_ширину(), set_высоту() независимо
      у Квадрата они должны быть равны → set_ширину(5) тайком меняет высоту
      → код, рассчитанный на Прямоугольник, ломается на Квадрате

   ❌ Пингвин наследует Птица с методом летать() → Пингвин.летать() падает
```

💡 ⭐ LSP — про **честность наследования**. Потомок не должен:
- усиливать требования (требовать больше, чем родитель);
- ослаблять обещания (давать меньше, чем обещал родитель);
- бросать неожиданные исключения там, где родитель работал.

Нарушение LSP = твой полиморфизм (модуль 11) **врёт**: вызвал метод через базовый тип, а потомок
повёл себя не так, как обещано. Лечение: пересмотреть иерархию (часто «является» оказывается
ложным — Квадрат не является заменяемым Прямоугольником в смысле поведения).

---

## ⭐ I — Interface Segregation (разделение интерфейсов)

**Лучше много маленьких специализированных интерфейсов, чем один толстый.** Класс не должен быть
вынужден реализовывать методы, которые ему не нужны.

```
   ❌ толстый интерфейс Работник:
       работать(), есть_обед(), получать_зарплату()
   → Робот реализует Работник → вынужден иметь есть_обед() (бессмыслица, заглушка)

   ✅ раздробить:
       Работающий { работать() }
       Питающийся { есть_обед() }
   → Робот реализует только Работающий
```

💡 ⭐ Толстые интерфейсы заставляют реализации тащить ненужное (пустые заглушки, `throw
NotImplemented`) и связывают всех, кто их использует, с лишними методами. Маленькие интерфейсы —
гибче: бери ровно то, что нужно. Связь с ISP: один класс может реализовать несколько мелких
интерфейсов (модуль 12).

---

## ⭐⭐ D — Dependency Inversion (инверсия зависимостей)

**Зависеть от абстракций, а не от конкретных реализаций.** И высокоуровневые модули, и
низкоуровневые должны зависеть от **интерфейса**, а не друг от друга.

🖼️
```
   ❌ ПРЯМАЯ зависимость:           ✅ ИНВЕРТИРОВАННАЯ:
   Заказ ──► MySQLБаза              Заказ ──► интерфейс Хранилище ◄── MySQLБаза
   (Заказ намертво привязан         (Заказ зависит от АБСТРАКЦИИ;             ◄── ФайлБаза
    к MySQL; сменить БД = боль)      подставишь любую реализацию)            ◄── ЗаглушкаБаза
```

```python
class Заказ:
    def __init__(self, хранилище: Хранилище):   # зависит от ИНТЕРФЕЙСА
        self._хранилище = хранилище              # внедрили снаружи (Dependency Injection)
```

💡 ⭐⭐ DIP — вершина SOLID и «программируй к интерфейсу» (модуль 12) на уровне архитектуры.
Высокоуровневая логика (Заказ) не должна знать про конкретную БД. Зависимость **внедряют снаружи**
(Dependency Injection) через конструктор. Выгоды:
- сменить реализацию (MySQL → Postgres) без правки логики;
- подставить **заглушку** для тестов (мост к тестированию!);
- слабая связанность всей системы.

---

## ⚠️ Ловушки

- ❌ Наследование, где потомок «не заменяет» родителя (нарушение LSP) — полиморфизм начнёт врать.
- ❌ Толстые интерфейсы с методами «на все случаи» (нарушение ISP).
- ❌ Высокоуровневый код, жёстко завязанный на конкретную БД/сервис (нарушение DIP).
- ❌ Применять DIP-абстракции там, где одна реализация и не предвидится другой (YAGNI).
