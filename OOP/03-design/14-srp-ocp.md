# 14 · SRP и OCP 🖼️⭐

> 🎯 **Цель блока:** разобрать первые два принципа SOLID — единственной ответственности (SRP) и
> открытости/закрытости (OCP).

---

## ⭐ S — Single Responsibility (одна ответственность)

**У класса должна быть одна причина для изменения** — одна ответственность.

```
   ❌ класс Отчёт:
       собрать_данные()    ← ответственность 1: данные
       посчитать()         ← ответственность 2: вычисления
       сохранить_в_PDF()   ← ответственность 3: форматирование/вывод
       отправить_email()   ← ответственность 4: рассылка
   → 4 причины меняться: поменялся формат PDF — лезешь в класс данных. Опасно.

   ✅ разделить:
       СборщикДанных · Калькулятор · PDFЭкспортёр · Рассыльщик
   → каждый меняется по своей причине, независимо
```

💡 ⭐ «Одна причина для изменения» — лучшая формулировка SRP. Если класс меняют **разные** люди
по **разным** поводам (дизайнер — формат, бухгалтер — расчёт) — у него слишком много
ответственностей. Раздели. Маленькие классы с одной ролью легче понять, тестировать и
переиспользовать.

⚠️ Не доводи до абсурда: класс на каждую строчку — это уже переусложнение. SRP про **связные**
обязанности, а не «один метод = один класс».

---

## ⭐ O — Open/Closed (открыт/закрыт)

**Код должен быть открыт для расширения, но закрыт для изменения.** Добавить новое поведение —
**не меняя** существующий код.

🖼️
```
   ❌ закрыт для расширения:           ✅ открыт для расширения:
   def площадь(ф):                     class Фигура: def площадь(self): ...
     if ф.тип=="круг": ...             class Круг(Фигура): ...
     elif ф.тип=="квадрат": ...        class Квадрат(Фигура): ...
   # новая фигура → ПРАВИШЬ функцию    # новая фигура → НОВЫЙ класс, функцию не трогаешь
```

💡 ⭐ Узнаёшь? Это **полиморфизм** из модуля 11! OCP — его принцип-формулировка. Признак нарушения
OCP — простыни `if/switch по типу`, которые приходится править при каждом новом варианте. Решение
— абстракция + полиморфизм: новый случай = новый класс, реализующий общий интерфейс.

```
   правило: «при добавлении функции ты ДОБАВЛЯЕШЬ код, а не МЕНЯЕШЬ существующий»
```

---

## 📖 Как они работают вместе

```
   SRP → каждый класс про одно → легко понять, что и где менять
   OCP → новые варианты добавляются классами → старое не трогаем → не ломаем
   вместе: система растёт ДОБАВЛЕНИЕМ, а не ПЕРЕПИСЫВАНИЕМ
```

💡 Это меняет мышление: вместо «открыть рабочий класс и впихнуть `if`» → «добавить новый класс,
реализующий интерфейс». Старый, оттестированный код остаётся нетронутым — меньше шанс что-то
сломать. Именно так растут хорошие системы.

---

## ⚠️ Ловушки

- ❌ Класс-«комбайн», делающий сбор + расчёт + вывод + отправку (нарушение SRP).
- ❌ Доводить SRP до классов-однострочников (другая крайность).
- ❌ `if/switch по типу`, растущие с каждым требованием (нарушение OCP).
- ❌ Делать всё «расширяемым» заранее на всякий случай (нарушение YAGNI — расширяемость стоит
  сложности; добавляй точку расширения, когда варианты реально появляются).

---

## 🛠️ Практика

1. Возьми класс-«комбайн» (данные+расчёт+вывод) и раздели по ответственностям (SRP).
2. Возьми функцию с `if по типу` и сделай её открытой к расширению через полиморфизм (OCP).
3. Добавь новый вариант и убедись, что старый код не изменился.
