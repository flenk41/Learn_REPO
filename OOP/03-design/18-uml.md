# 18 · UML и моделирование 🛠️

> 🎯 **Цель блока:** научиться рисовать дизайн до кода — диаграммы классов UML и отношения между
> ними — чтобы думать об архитектуре наглядно.

---

## 📖 Зачем рисовать перед тем, как писать

Дизайн в голове сложно удержать. **Диаграмма** делает структуру видимой: классы, их связи,
кто от кого зависит. Это дёшево менять (стёр стрелку) — в отличие от кода.

💡 **UML** (Unified Modeling Language) — стандартный язык таких схем. Тебе не нужен весь UML
(он огромен) — на практике хватает **диаграммы классов** и базовых стрелок. Цель — не «правильно
по UML», а **понять и обсудить** дизайн.

---

## ⭐ Диаграмма класса

```
   ┌─────────────────────┐
   │       Счёт          │   ← имя класса
   ├─────────────────────┤
   │ - баланс: int       │   ← поля (- приватное, + публичное)
   ├─────────────────────┤
   │ + пополнить(сумма)  │   ← методы
   │ + снять(сумма)      │
   └─────────────────────┘
```

💡 Три секции: имя, поля, методы. `-` приватное, `+` публичное, `#` protected. Этого достаточно,
чтобы показать **контракт** класса (что снаружи) и его состояние.

---

## ⭐ Стрелки отношений — главное

🖼️
```
   НАСЛЕДОВАНИЕ (является):     Собака ──▷ Животное      (пустой треугольник к родителю)
   РЕАЛИЗАЦИЯ интерфейса:       Карта ┄┄▷ СпособОплаты   (пунктир + треугольник)
   КОМПОЗИЦИЯ (часть умирает):  Дом ◆──── Комната        (закрашенный ромб у целого)
   АГРЕГАЦИЯ (часть живёт):     Команда ◇──── Игрок      (пустой ромб у целого)
   АССОЦИАЦИЯ/использует:       Заказ ────▶ Клиент       (обычная стрелка)
   ЗАВИСИМОСТЬ:                 Касса ┄┄▶ Логгер         (пунктирная стрелка)
```

💡 ⭐ Стрелки — самое полезное в UML: они показывают **зависимости** (а управление зависимостями
— суть всего проектирования, модуль 13). Глядя на схему, сразу видно: где наследование (а нет ли
злоупотребления?), где композиция, кто от кого зависит, нет ли циклов.

## 📖 Инструменты

```
   ✏️ бумага/доска   — быстрее всего для обдумывания (самый частый «инструмент» Senior)
   📝 PlantUML       — диаграммы текстом (удобно хранить в git рядом с кодом)
   🎨 draw.io / Excalidraw — рисование мышкой
   🧩 mermaid         — диаграммы в markdown (как в дорожных картах этого курса!)
```

Пример PlantUML/mermaid (текст → схема):
```
   class Счёт {
     - баланс: int
     + пополнить(сумма)
   }
   Заказ --> Клиент
   Карта ..|> СпособОплаты
```

💡 Для повседневной работы хватает наброска на бумаге. Текстовые форматы (PlantUML, mermaid)
хороши тем, что **версионируются** в git и не устаревают, как нарисованные картинки.

---

## ⭐ Когда моделировать (а когда нет)

```
   ✅ перед сложной фичей/системой — набросать классы и связи, обсудить с командой
   ✅ чтобы понять чужой код — нарисовать его структуру
   ✅ зафиксировать ключевые решения архитектуры
   ❌ не рисуй идеальный UML каждой строчки — это не самоцель
```

💡 Моделирование — это **думать руками**. Цель — лучше понять и обсудить дизайн **до** того, как
вложиться в код. Пара минут со стрелками экономит часы переделок (перекликается с планированием в
[Claude Code, Plan Mode](../../ClaudeCode/01-basics/05-plan-mode.md)).

## ⚠️ Ловушки

- ❌ Превращать UML в самоцель (идеальные диаграммы вместо рабочего дизайна).
- ❌ Рисовать после кода «для галочки» — теряется смысл (думать ДО).
- ❌ Хранить устаревшие картинки, не отражающие код. Лучше текстовые схемы рядом с кодом.
- ❌ Игнорировать моделирование на сложных задачах и сразу кодить «клубок».
