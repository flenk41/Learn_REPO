# 02 · Языки и инструменты 🛠️

> 🎯 **Цель блока:** выбрать, где практиковать ООП, и понять, что принципы одинаковы во всех
> объектных языках — меняется лишь синтаксис.

---

## 📖 ООП есть почти везде

Объектную модель поддерживают почти все популярные языки. Различается **степень** и синтаксис:

```
   Java, C#     — «чистые» объектные: почти всё — классы (классическое ООП)
   C++          — мультипарадигменный: ООП + процедурное + шаблоны
   Python       — всё объект, но ООП необязательно; гибко
   Kotlin, Swift — современное ООП с удобным синтаксисом
   JavaScript   — объекты и прототипы (+ классы как синтаксис)
   Go, Rust     — без классического наследования, но есть интерфейсы/трейты (композиция)
```

💡 ⭐ Главное: **принципы ООП (этот трек) переносимы**. Освоив абстракцию, инкапсуляцию,
полиморфизм и SOLID на одном языке, ты применишь их в любом другом — поменяется лишь, как
написать `class`, `extends`, `interface`.

---

## 🛠️ Где практиковать

Для изучения ООП удобнее всего язык с явными классами и читаемым синтаксисом:

```
   🐍 Python  — минимум синтаксиса, фокус на идеях (примеры в треке — в этом стиле)
   ☕ Java / C#  — строгое классическое ООП, явные интерфейсы/абстрактные классы
   ➕ C++  — если уже прошёл трек C++ (классы там разобраны подробно)
```

💡 Если ты уже проходил [трек C++](../../Cpp/02-memory/10-raii.md) или
[Python](../../Python/03-middle/16-oop-slots.md) — там ООП показано на конкретном языке. Этот
трек — про **проектирование** поверх синтаксиса: одинаково полезно с любым из них.

⚠️ Примеры в треке даны в **Python-стиле** (он самый читаемый для идей). Переписать на Java/C++ —
полезное упражнение: суть та же, синтаксис другой.

---

## 🛠️ Инструменты проектировщика

Помимо языка, пригодятся:

```
   📐 UML-схемы      — рисовать отношения классов (модуль 18); инструменты: PlantUML, draw.io
   🧪 IDE с рефакторингом — IntelliJ/PyCharm/VS Code (переименование, извлечение метода/класса)
   🔍 линтеры/анализаторы — подсказывают запахи кода и нарушения принципов
   📝 диаграммы на бумаге — самый быстрый способ продумать дизайн ДО кода
```

💡 Senior часто **рисует** дизайн до того, как писать код: пара квадратиков со стрелками на
бумаге экономит часы переделок. Проектирование — это в первую очередь **думать**, а потом
кодить (перекликается с Plan Mode из [трека Claude Code](../../ClaudeCode/01-basics/05-plan-mode.md)).

## ⚠️ Ловушки

- ❌ Думать, что ООП «привязано» к одному языку. Принципы переносимы.
- ❌ Гнаться за синтаксисом вместо понимания идей.
- ❌ Сразу кодить сложную систему без эскиза дизайна.
- ❌ Считать, что языки без классов (Go, Rust) «не ООП» — у них композиция и интерфейсы.

## 🛠️ Практика

1. Выбери язык для практики трека (рекомендуется Python — как в примерах).
2. Создай простой класс и объект в нём — убедись, что среда настроена.
3. Установи/попробуй PlantUML или draw.io — нарисуй класс с полями и методами.
