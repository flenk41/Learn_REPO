# 12 · Ошибки памяти и инструменты 🛠️⭐

> 🎯 **Цель блока:** научиться **находить** ошибки памяти автоматически. Профи не ищут
> утечки глазами — они используют инструменты. Освоишь их — будешь писать надёжный код.

---

## 📖 Почему ошибки памяти такие коварные

Ошибки памяти в C **часто не проявляются сразу**:
- программа может «работать» месяцами, а потом упасть;
- баг проявляется на чужой машине, но не на твоей;
- падает в случайном месте, не там, где настоящая ошибка.

Поэтому нужны **специальные инструменты**, которые следят за каждым обращением к памяти.

---

## 🛠️ Инструмент №1: AddressSanitizer (ASan) — встроен в GCC/Clang

Самый простой и мощный способ. Это «детектор», встроенный прямо в компилятор. Просто
добавь флаг `-fsanitize=address`:

```bash
gcc -fsanitize=address -g program.c -o program
./program
```

ASan следит за каждым доступом к памяти и при ошибке **печатает подробный отчёт**:
что произошло, в какой строке, какой адрес.

### 🧪 Пример: ловим выход за границы массива

```c
#include <stdlib.h>
int main(void) {
    int *a = malloc(3 * sizeof(int));
    a[5] = 10;          // ⚠️ выход за границы!
    free(a);
    return 0;
}
```

```bash
gcc -fsanitize=address -g bug.c -o bug
./bug
```

🖼️ ASan напечатает примерно:

```
==12345==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x...
WRITE of size 4 at 0x... thread T0
    #0 0x... in main bug.c:4          ← вот она, точная строка!
0x... is located 8 bytes to the right of 12-byte region
```

💡 ASan прямо говорит: **heap-buffer-overflow**, строка 4. Без него ты бы искал этот баг
часами.

### 🧪 Пример: ловим утечку памяти

```c
#include <stdlib.h>
int main(void) {
    int *p = malloc(100);   // забыли free!
    return 0;
}
```

```bash
gcc -fsanitize=address -g leak.c -o leak
./leak
```

ASan на выходе сообщит:
```
==12345==ERROR: LeakSanitizer: detected memory leaks
Direct leak of 100 byte(s) allocated from:
    #0 ... in main leak.c:3           ← где выделено и не освобождено
```

### Что ловит ASan
- ✅ выход за границы (buffer overflow)
- ✅ use-after-free (висячий указатель)
- ✅ double-free
- ✅ утечки памяти (LeakSanitizer)
- ✅ использование стека после выхода из функции

> 💡 **Привычка профи:** во время разработки **всегда** компилируй с
> `-fsanitize=address -g -Wall -Wextra`. Это поймает 90% ошибок памяти автоматически.

> ⚠️ На Windows с MinGW ASan может быть недоступен. Варианты: использовать **WSL**
> (Linux внутри Windows), Clang, или инструмент Dr. Memory (см. ниже). На Linux/macOS
> ASan работает «из коробки».

## 🛠️ Инструмент №2: Valgrind (Linux/macOS/WSL)

Классический инструмент. Не требует перекомпиляции с флагом — запускает программу в
«виртуальной машине» и отслеживает память.

```bash
gcc -g program.c -o program
valgrind --leak-check=full ./program
```

🖼️ Типичный вывод про утечку:

```
==12345== HEAP SUMMARY:
==12345==     in use at exit: 100 bytes in 1 blocks
==12345== LEAK SUMMARY:
==12345==    definitely lost: 100 bytes in 1 blocks   ← утечка!
==12345==    at malloc ... by main (program.c:3)
```

Идеальный результат:
```
All heap blocks were freed -- no leaks are possible
```

> ⚠️ Valgrind нет под нативный Windows. На Windows используй **WSL** или Dr. Memory.

---

## 🛠️ Инструмент №3: Dr. Memory (работает на Windows)

Аналог Valgrind, который **работает на Windows**:

1. Скачай с **https://drmemory.org**
2. Запусти:
   ```powershell
   drmemory -- .\program.exe
   ```
Он покажет утечки и ошибки доступа к памяти, как Valgrind.

---

## 🛠️ Инструмент №4: отладчик GDB — пошаговое выполнение

Когда нужно понять, **что происходит** в конкретный момент:

```bash
gcc -g program.c -o program      # -g обязателен для отладки!
gdb ./program
```

Базовые команды GDB:

| Команда | Действие |
|---------|----------|
| `break main` | поставить точку останова на функции `main` |
| `run` | запустить |
| `next` (`n`) | выполнить строку (не заходя в функции) |
| `step` (`s`) | выполнить строку (заходя внутрь функций) |
| `print x` (`p x`) | показать значение переменной `x` |
| `print *p` | показать значение по указателю |
| `print arr[2]` | показать элемент массива |
| `backtrace` (`bt`) | показать стек вызовов (где мы сейчас) |
| `continue` (`c`) | продолжить до следующей точки |
| `quit` (`q`) | выход |

💡 GDB особенно полезен при падении: запусти программу под GDB, и при краше команда `bt`
покажет **точную цепочку вызовов**, приведшую к ошибке.

> На Windows GDB ставится вместе с MinGW (`gdb` в `C:\msys64\ucrt64\bin`). В VS Code
> отладка работает через расширение C/C++ кнопкой **Run → Start Debugging (F5)**.

## 📋 Сводная таблица: чем что ловить

| Проблема | Инструмент | Команда |
|----------|-----------|---------|
| Утечки памяти | ASan / Valgrind / Dr.Memory | `-fsanitize=address` |
| Выход за границы | ASan | `-fsanitize=address` |
| Use-after-free | ASan / Valgrind | `-fsanitize=address` |
| Подозрительный код | компилятор | `-Wall -Wextra` |
| «Почему упало?» | GDB | `bt` после краша |
| Пошаговая отладка | GDB / VS Code | `step`, `print` |

## 🧪 Практика: поймай все баги

Скомпилируй с ASan и исправь каждую ошибку:

```c
#include <stdlib.h>
#include <stdio.h>

int main(void) {
    // Баг 1: выход за границы
    int arr[3] = {1, 2, 3};
    printf("%d\n", arr[3]);

    // Баг 2: утечка
    int *p = malloc(sizeof(int));
    *p = 10;

    // Баг 3: use-after-free
    int *q = malloc(sizeof(int));
    free(q);
    *q = 5;

    // Баг 4: double free
    int *r = malloc(sizeof(int));
    free(r);
    free(r);

    return 0;
}
```

Запусти под ASan, прочитай каждый отчёт, исправь все 4 бага. Цель — чистый вывод.
