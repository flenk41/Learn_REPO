# 4 · volatile, inline asm, барьеры памяти 🖼️⭐

> 🎯 **Цель блока:** понять низкоуровневые инструменты системного C — volatile, встроенный
> ассемблер, барьеры памяти — нужные для железа, драйверов, многопотока.

> ⚠️ Это инструменты «последней мили» — для embedded/драйверов/lock-free. В обычном коде не нужны.

---

## 📖 volatile: «не оптимизируй доступ»

```
   volatile говорит компилятору: «эта память может измениться ВНЕ видимости программы — НЕ
   оптимизируй обращения к ней (не кэшируй в регистре, не выбрасывай 'лишние' чтения)».

   зачем:
   • MEMORY-MAPPED I-O (регистры железа): чтение/запись по адресу = команда устройству. компилятор
     не должен «оптимизировать» повторные чтения (каждое — реальная операция с железом).
   • переменная, меняемая ОБРАБОТЧИКОМ СИГНАЛА / прерыванием.
   ❌ ЧАСТОЕ ЗАБЛУЖДЕНИЕ: volatile НЕ для многопоточности! он НЕ даёт атомарности и упорядочивания
      между потоками. для потоков — atomics (_Atomic / stdatomic.h), НЕ volatile.
```

```c
volatile uint32_t* reg = (uint32_t*)0x40000000;   // регистр устройства
*reg = 1;                                          // запись = команда железу (не выбросится)
while (*reg & BUSY) { }                            // каждое чтение реальное (не закэшируется)
```

💡 ⭐ volatile = «доступ к этой памяти имеет побочные эффекты, не трогай». Для железа и
сигналов — да. Для синхронизации потоков — НЕТ (это частая ошибка; нужны атомики и барьеры).

## ⭐ Inline assembly и интринсики

```
   когда нужны конкретные инструкции CPU, которых нет в C (специальные регистры, SIMD, барьеры,
   атомарные операции на низком уровне):

   ИНТРИНСИКИ (предпочтительно) — функции-обёртки над инструкциями (например, __builtin_*, <immintrin.h>
   для SIMD). компилятор понимает их, оптимизирует, переносимее inline asm.
   _mm_add_ps(...)        // SIMD-сложение (AVX) через интринсик

   INLINE ASM (крайнее средство) — вставка ассемблера прямо в C (gcc: __asm__):
   __asm__ volatile ("rdtsc" : "=a"(lo), "=d"(hi));   // прочитать счётчик тактов CPU
   ⚠️ платформозависимо, хрупко, мешает оптимизатору. используй ТОЛЬКО когда интринсика нет.
```

💡 ⭐ Правило: **интринсики > inline asm**. Интринсики переносимее и дружат с оптимизатором; inline
asm — для совсем специфичного (привилегированные инструкции, конкретное железо). Большинство задач
(SIMD, атомики, барьеры) решаются интринсиками/стандартными средствами, не сырым asm.

## ⭐⭐ Барьеры памяти и порядок

```
   ПРОБЛЕМА: и КОМПИЛЯТОР, и CPU могут ПЕРЕУПОРЯДОЧИВАТЬ операции с памятью ради скорости (пока
   результат для ОДНОГО потока тот же). но между ПОТОКАМИ/устройствами порядок виден → проблема.

   data = 42;        // (1)
   ready = 1;        // (2)
   другой поток видит ready==1, но data ещё НЕ 42 (переупорядочено/не доехало до его кэша)! → баг.

   БАРЬЕР ПАМЯТИ (memory barrier/fence) — запрещает переупорядочивание через него:
   • гарантирует, что (1) станет видна ДО (2) для других потоков.
   АТОМАРНЫЕ ОПЕРАЦИИ (C11 stdatomic.h) — несут барьеры с моделью памяти:
   atomic_store(&data, 42); atomic_store_explicit(&ready, 1, memory_order_release);
   memory_order (relaxed/acquire/release/seq_cst) — насколько строгий порядок (trade-off скорость/строгость).
```

💡 ⭐⭐ Это самый тонкий уровень: в многопотоке/lock-free **порядок операций с памятью не
гарантирован** без барьеров. `volatile` НЕ помогает (не даёт барьеров). Нужны **атомики с моделью
памяти** (acquire/release/seq_cst). Это [конкурентность из ОС/C++/Rust](../../OS/03-sync/13-race-conditions.md)
на уровне железа. Для большинства — используй мьютексы/высокоуровневые примитивы; барьеры/атомики
напрямую — для lock-free структур (экспертный уровень).

## ⚠️ Ловушки

- ❌ Использовать volatile для синхронизации потоков (не даёт атомарности/барьеров → нужны атомики).
- ❌ Inline asm там, где есть интринсик (хрупко, непереносимо).
- ❌ Lock-free код без понимания модели памяти/барьеров → тонкие гонки.
- ❌ Полагаться на порядок операций между потоками без барьеров.
- ❌ Излишний `memory_order_seq_cst` (медленно) или слишком слабый relaxed (баги) — без понимания.
- ❌ Считать, что «работает на моём CPU» = корректно (разные архитектуры переупорядочивают по-разному).
