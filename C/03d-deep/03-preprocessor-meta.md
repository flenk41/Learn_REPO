# 3 · Метапрограммирование препроцессором 🖼️⭐

> 🎯 **Цель блока:** освоить продвинутый препроцессор — X-macros, stringification, склейку токенов —
> мощные (и опасные) приёмы генерации кода в C.

---

## 📖 Препроцессор — текстовый генератор кода

```
   препроцессор работает с ТЕКСТОМ до компиляции (модуль из middle). продвинутые операторы:
   #  (stringify)    — превращает аргумент в строковый литерал: #x → "x".
   ## (token paste)  — склеивает токены: a ## b → ab.
   эти два + #define/#include дают примитивное метапрограммирование (генерацию повторяющегося кода).
   смотри результат: gcc -E file.c
```

```c
#define STR(x) #x                    // STR(hello) → "hello"
#define CONCAT(a, b) a ## b          // CONCAT(foo, bar) → foobar
#define MAKE_FUNC(name) void func_##name(void) { /* ... */ }
MAKE_FUNC(init)                      // → void func_init(void) { ... }
```

💡 ⭐ `#` и `##` — основа генерации имён и строк. `func_##name` строит уникальные имена функций;
`#x` превращает код в строку (для отладки/логов: `#define LOG(x) printf(#x " = %d\n", x)` печатает
имя переменной и значение). Это «магия» многих C-библиотек.

## ⭐⭐ X-macros — таблица данных в макросе

```
   X-MACROS — мощный паттерн: ОДИН список данных, используемый МНОГОКРАТНО по-разному. избавляет от
   дублирования (определить enum + строки + обработчики из одного списка).
```

```c
// 1. определяем СПИСОК (таблицу) один раз:
#define COLOR_LIST   \
    X(RED,   "red")   \
    X(GREEN, "green") \
    X(BLUE,  "blue")

// 2. используем его РАЗНЫМИ способами, переопределяя X:

// генерируем enum:
typedef enum {
    #define X(name, str) COLOR_##name,
    COLOR_LIST
    #undef X
} Color;

// генерируем массив строк-имён:
const char* color_names[] = {
    #define X(name, str) str,
    COLOR_LIST
    #undef X
};
// добавить новый цвет → одна строка в COLOR_LIST → и enum, и строки обновятся. DRY!
```

💡 ⭐⭐ X-macros — элегантное решение проблемы «синхронизировать enum и связанные таблицы». Список
определён ОДИН раз; каждое использование переопределяет `X`, чтобы извлечь нужное (имена для enum,
строки для массива, case'ы для switch). Добавил элемент в список → всё связанное обновилось
автоматически. Так делают парсеры, таблицы команд, сериализацию. Мощно, но требует привычки читать.

## ⭐ Полезные приёмы

```c
// counting / static assert:
#define ARRAY_SIZE(a) (sizeof(a) / sizeof((a)[0]))   // размер статического массива (НЕ для указателей!)

#define STATIC_ASSERT(cond, msg) _Static_assert(cond, msg)   // C11: проверка на компиляции

// безопасные макросы-функции (скобки!):
#define MAX(a, b) ((a) > (b) ? (a) : (b))            // скобки вокруг аргументов и всего выражения!
// do-while(0) для многооператорных макросов:
#define SWAP(a, b) do { int t = a; a = b; b = t; } while(0)   // ведёт себя как один оператор
```

💡 ⭐ Правила безопасных макросов: **скобки** вокруг каждого аргумента и всего выражения (иначе
приоритеты сломаются); `do { ... } while(0)` для многооператорных (чтобы вёл себя как один оператор
в if без скобок); избегай побочных эффектов в аргументах (`MAX(i++, j)` вычислит i++ дважды!).

## 📖 Когда препроцессор, а когда нет

```
   ✅ оправдано: X-macros (синхронизация таблиц), генерация повторяющегося шаблонного кода,
      условная компиляция (платформы), отладочные макросы (#x для имён).
   ❌ избегай: сложная «логика» в макросах (нечитаемо, не отлаживается), макросы вместо функций
      (inline-функции типобезопаснее и отлаживаемы), магия, которую никто не поймёт.
   правило: предпочитай функции/const/enum макросам, где можно. макросы — для того, что функции не могут
   (генерация типов/имён, условная компиляция, X-macros).
```

---

## ⚠️ Ловушки

- ❌ Макрос без скобок вокруг аргументов → сломанные приоритеты (`SQ(a+b)`).
- ❌ Побочные эффекты в аргументах (`MAX(i++, j)` → двойное вычисление).
- ❌ Многооператорный макрос без `do-while(0)` → ломается в if без скобок.
- ❌ `ARRAY_SIZE` на указателе (не массиве) → неверный результат.
- ❌ Сложная логика в макросах вместо функций (нечитаемо, не отлаживается).
- ❌ Забыть `#undef X` при X-macros → конфликты.
