# 2 · Обобщённое программирование в C 🖼️⭐

> 🎯 **Цель блока:** научиться писать обобщённый (работающий с разными типами) код в C, где нет
> шаблонов — через `void*`, `_Generic` и макросы.

---

## 📖 В C нет шаблонов — но обобщённость нужна

```
   C++ имеет шаблоны (vector<T>); C — нет. но обобщённые структуры (контейнеры на любой тип) нужны.
   три подхода в C, каждый со своими trade-off:
   1. void* — стирание типа (один код, любой тип, но теряется типобезопасность).
   2. МАКРОСЫ — генерация кода под тип (типобезопасно, но макро-проблемы).
   3. _Generic (C11) — выбор по типу при компиляции (типобезопасные «перегрузки»).
```

---

## ⭐ Подход 1: void* (стирание типа)

```c
// обобщённый контейнер через void* — хранит указатели на ЛЮБОЙ тип:
typedef struct {
    void** items;        // массив указателей на «что угодно»
    size_t size, cap;
} Vec;

void vec_push(Vec* v, void* item);   // не знает тип — просто хранит указатель
void* vec_get(Vec* v, size_t i);     // возвращает void*, caller кастует обратно

// так устроены qsort (принимает void* + размер элемента + функцию сравнения), generic-библиотеки.
```

```
   ✅ ОДИН код на все типы; гибко.
   ❌ ТЕРЯЕТСЯ типобезопасность (компилятор не проверит, что положил и достал тот же тип).
   ❌ оверхед (указатели, возможно лишние аллокации); ручное управление временем жизни.
   ❌ нет проверки размера — легко ошибиться (как в qsort: передал не тот размер элемента → UB).
```

💡 ⭐ `void*` — «стирание типа» в C: универсально, но небезопасно. Так работают `qsort`, `bsearch`,
generic-контейнеры. Цена — компилятор не помогает (касты, размеры — на тебе). Документируй и
оборачивай аккуратно.

## ⭐⭐ Подход 2: макросы (генерация кода под тип)

```c
// макрос ГЕНЕРИРУЕТ типизированную структуру и функции под конкретный тип:
#define DEFINE_VEC(T, name)                                  \
    typedef struct { T* items; size_t size, cap; } name;     \
    static void name##_push(name* v, T item) { /* ... */ }   \
    static T name##_get(name* v, size_t i) { return v->items[i]; }

DEFINE_VEC(int, IntVec)        // создаёт IntVec, IntVec_push, IntVec_get — типобезопасно для int
DEFINE_VEC(double, DblVec)     // и для double

IntVec v; IntVec_push(&v, 5);  // компилятор проверяет тип!
```

```
   ✅ ТИПОБЕЗОПАСНО (T реальный тип, проверка компилятором), без оверхеда (как шаблоны).
   ❌ макро-проблемы: нечитаемые ошибки, отладка тяжелее, дублирование кода на каждый тип (как
      мономорфизация шаблонов), `##` склейка имён, осторожность с аргументами.
   это «шаблоны бедняка» — так делают многие C-библиотеки (например, заголовочные generic-контейнеры).
```

💡 ⭐⭐ Макро-генерация — основной способ типобезопасной обобщённости в C: макрос — это «шаблон»,
порождающий типизированный код. `##` (склейка токенов) строит имена (`IntVec_push`). Цена —
макро-сложности (ошибки внутри макроса нечитаемы, отладка через `gcc -E`). Но это даёт скорость и
типобезопасность.

## 📖 Подход 3: _Generic (C11) — выбор по типу

```c
// _Generic выбирает выражение по ТИПУ аргумента при компиляции (типобезопасные «перегрузки»):
#define print_num(x) _Generic((x),       \
    int:    print_int,                    \
    double: print_double,                 \
    char*:  print_string                  \
)(x)

print_num(5);      // → print_int(5)
print_num(3.14);   // → print_double(3.14)
```

```
   _Generic — единственная «перегрузка по типу» в C (на этапе компиляции). используется, например,
   в <tgmath.h> (type-generic math: sqrt работает для float/double/long double).
   ограничен (только выбор выражения по типу), но полезен для type-generic макросов.
```

---

## ⚠️ Ловушки

- ❌ void*: потерять тип (положил int*, достал как double*) → UB.
- ❌ qsort/bsearch с неверным размером элемента → UB.
- ❌ Макросы: забыть скобки вокруг аргументов → неверный приоритет (`#define SQ(x) x*x` → `SQ(a+b)`).
- ❌ Макро-генерация без `static`/guard → конфликты при включении.
- ❌ Сложные ошибки внутри макросов (отлаживай через `gcc -E`).
- ❌ Использовать void* там, где макросы/_Generic дали бы типобезопасность.
