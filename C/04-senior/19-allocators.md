# 19 · Свои аллокаторы памяти 🖼️⭐

> 🎯 **Цель блока:** понять, как устроен `malloc` изнутри, и написать **свои**
> аллокаторы (arena, pool). Это то, что отличает системного программиста: ты управляешь
> памятью не на уровне «выделить/освободить», а на уровне стратегии.

---

## 📖 Почему `malloc` бывает недостаточно

`malloc`/`free` универсальны, но за универсальность платят:
- **медленно** — каждый вызов ищет подходящий блок, ведёт учёт, защищается от потоков;
- **фрагментация** — после многих malloc/free память «дырявится»;
- **накладные расходы** — на каждый блок хранится служебная информация (заголовок).

Если тебе нужно выделить **миллион мелких объектов** или ты пишешь игру/парсер/сервер,
свой аллокатор может быть в **10–100 раз быстрее**.

🖼️ Как примерно устроен `malloc` внутри (free-list):

```
Куча: [hdr|данные][hdr|свободно][hdr|данные][hdr|свободно...]
       │                                     
       свободные блоки связаны в список; malloc ищет подходящий,
       free возвращает блок в список и пытается слить с соседями
```

---

## ⭐ Аллокатор №1: Arena (bump allocator)

**Идея:** взять один большой буфер и выдавать куски, просто **сдвигая указатель вперёд**.
Освобождение поэлементно **не поддерживается** — освобождаем всё разом, обнулив указатель.

🖼️

```
буфер:  [################........................]
         │              │
         начало      offset (двигается вправо при каждом выделении)

arena_alloc(16): вернуть текущий offset, сдвинуть offset += 16
arena_reset():   offset = 0  → вся память "освобождена" мгновенно
```

```c
typedef struct {
    char  *buffer;     // большой блок памяти
    size_t capacity;
    size_t offset;     // сколько уже занято
} Arena;

void arena_init(Arena *a, size_t size) {
    a->buffer = malloc(size);
    a->capacity = size;
    a->offset = 0;
}

void* arena_alloc(Arena *a, size_t size) {
    if (a->offset + size > a->capacity) return NULL;   // место кончилось
    void *ptr = a->buffer + a->offset;                 // текущая позиция
    a->offset += size;                                 // сдвигаем вперёд
    return ptr;
}

void arena_reset(Arena *a) { a->offset = 0; }          // "освободить" всё
void arena_free(Arena *a)  { free(a->buffer); }
```

✅ **Плюсы:** выделение — это просто сложение, мгновенно. Ноль фрагментации.
⚠️ **Минусы:** нельзя освободить один объект, только всё сразу.

💡 **Где идеально:** обработка запроса (выделил всё под запрос → ответил → reset),
парсинг (выделил под дерево → обработал → освободил всё), кадр игры.

> ⚠️ В реальной arena нужно ещё **выравнивать** offset (см. модуль 20), иначе доступ к
> данным может быть медленным или невалидным. Об этом — в следующем модуле.

## ⭐ Аллокатор №2: Pool (пул фиксированных блоков)

**Идея:** все объекты одного размера. Разбиваем буфер на блоки и держим **список
свободных**. Выделение и освобождение — O(1), фрагментации нет.

🖼️

```
буфер разбит на блоки одного размера:
[блок0][блок1][блок2][блок3]...

free-list (список свободных) связывает пустые блоки:
free_head ──► блок2 ──► блок0 ──► блок3 ──► NULL

pool_alloc(): снять блок с вершины free-list
pool_free(b): вернуть блок b на вершину free-list
```

💡 Хитрость: пока блок свободен, **внутри него самого** хранится указатель на следующий
свободный блок. Память не простаивает.

```c
typedef struct FreeNode { struct FreeNode *next; } FreeNode;

typedef struct {
    char     *buffer;
    size_t    block_size;
    FreeNode *free_head;
} Pool;

void* pool_alloc(Pool *p) {
    if (p->free_head == NULL) return NULL;     // пул исчерпан
    FreeNode *block = p->free_head;
    p->free_head = block->next;                // снять с вершины
    return block;
}

void pool_free(Pool *p, void *ptr) {
    FreeNode *block = ptr;
    block->next = p->free_head;                // вернуть на вершину
    p->free_head = block;
}
```

✅ **Плюсы:** O(1) выделение И освобождение, ноль фрагментации, отличная локальность.
⚠️ **Минусы:** все блоки одного размера.

💡 **Где идеально:** узлы связных списков/деревьев, частицы, пули в игре, сетевые пакеты.

---

## 📖 Сравнение стратегий

| Аллокатор | Выделение | Освобождение | Фрагментация | Когда |
|-----------|-----------|--------------|--------------|-------|
| `malloc` | средне | средне (по одному) | есть | универсально |
| **Arena** | 🚀 O(1) | только всё сразу | нет | временные данные, запросы |
| **Pool** | 🚀 O(1) | 🚀 O(1) (по одному) | нет | много объектов одного размера |
| Stack alloc | 🚀 O(1) | LIFO-порядок | нет | вложенные области жизни |

## 📖 Зачем это знать, даже если пишешь на других языках

- Игровые движки (Unreal, Unity-нативная часть) повсеместно используют arena/pool.
- Базы данных и компиляторы управляют памятью пулами и аренами.
- В Rust/C++/Go есть библиотеки arena-аллокаторов — теперь ты понимаешь, как они работают.
- Понимание аллокаторов = понимание, **почему** одни программы быстрые, а другие нет.
