# 21 · Многопоточность и атомарность 🖼️⭐

> 🎯 **Цель блока:** запускать несколько потоков и понять, **почему общая память — это
> опасно**. Гонки данных (data races) — одна из самых сложных категорий багов, и она вся
> про память.

---

## 📖 Поток (thread) — это отдельная линия выполнения

Обычная программа выполняет инструкции одну за другой. **Потоки** позволяют выполнять
несколько последовательностей **одновременно** (на разных ядрах CPU).

🖼️ Что общее, а что своё у потоков одного процесса:

```
        ПРОЦЕСС
  ┌────────────────────────────────┐
  │  HEAP, DATA, код  ← ОБЩИЕ!      │   ← потоки делят кучу и глобальные данные
  ├──────────────┬──────────────────┤
  │  Поток 1     │   Поток 2        │
  │  свой стек   │   свой стек      │   ← у каждого потока СВОЙ стек
  │  свои регистры│  свои регистры  │
  └──────────────┴──────────────────┘
```

💡 Ключевое: потоки **делят кучу и глобальные переменные**, но у каждого **свой стек**.
Именно общая память порождает и силу, и опасность многопоточности.

---

## ⭐ Запуск потоков (C11 threads)

```c
#include <stdio.h>
#include <threads.h>

int worker(void *arg) {
    int id = *(int*)arg;
    printf("Поток %d работает\n", id);
    return 0;
}

int main(void) {
    thrd_t t1, t2;
    int a = 1, b = 2;

    thrd_create(&t1, worker, &a);   // запустить поток
    thrd_create(&t2, worker, &b);

    thrd_join(t1, NULL);            // дождаться завершения
    thrd_join(t2, NULL);
    return 0;
}
```

> 💡 На Linux/Mac часто используют **POSIX threads** (`pthread_create`, `-lpthread`).
> Идея та же. На Windows с MSYS2 работает и pthreads, и C11 threads.

---

## ⚠️ Главная проблема: ГОНКА ДАННЫХ (data race)

Когда два потока **одновременно** меняют одну переменную — результат непредсказуем.

```c
int counter = 0;                    // общая переменная

int increment(void *arg) {
    for (int i = 0; i < 1000000; i++)
        counter++;                  // ⚠️ ГОНКА! два потока портят друг друга
    return 0;
}
// Запусти 2 потока — counter будет НЕ 2000000, а случайное меньшее число!
```

🖼️ Почему `counter++` опасен — это **три** операции, не одна:

```
counter++  =  1. прочитать counter из памяти в регистр
              2. прибавить 1
              3. записать обратно в память

Поток A: читает 5 ──┐
Поток B: читает 5 ──┤  оба прочитали 5!
Поток A: пишет 6  ──┤
Поток B: пишет 6  ──┘  должно быть 7, а стало 6 — одно увеличение ПОТЕРЯНО
```

> ⚠️ Гонки данных — это **undefined behavior**. Они проявляются редко и случайно, ломаются
> на проде, исчезают под отладчиком. Самые ненавистные баги в индустрии.

---

## ⭐ Решение 1: Мьютекс (замок)

Мьютекс гарантирует, что в «защищённую» секцию входит **только один поток** за раз.

```c
#include <threads.h>

mtx_t lock;
int counter = 0;

int increment(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        mtx_lock(&lock);       // 🔒 захватить замок (другие ждут)
        counter++;             // критическая секция — безопасно
        mtx_unlock(&lock);     // 🔓 отпустить
    }
    return 0;
}
// main: mtx_init(&lock, mtx_plain); ... mtx_destroy(&lock);
```

🖼️

```
Поток A: 🔒──[counter++]──🔓
Поток B:    ⏳ ждёт...      🔒──[counter++]──🔓
            пока A держит замок, B стоит
```

✅ Теперь результат всегда 2000000.
⚠️ Цена: потоки ждут друг друга (теряется параллелизм), есть риск **deadlock** (взаимная
блокировка, когда A ждёт B, а B ждёт A).

---

## ⭐ Решение 2: Атомарные операции (atomics)

Для простых операций (счётчики, флаги) мьютекс — перебор. **Атомарная** операция
выполняется «неделимо» — её нельзя прервать на середине.

```c
#include <stdatomic.h>

atomic_int counter = 0;

int increment(void *arg) {
    for (int i = 0; i < 1000000; i++)
        atomic_fetch_add(&counter, 1);   // неделимое +1, без замка
    return 0;
}
```

✅ Быстрее мьютекса, нет deadlock.
⚠️ Подходит только для простых операций над одной переменной.

🖼️ Атомарность гарантирует, что «прочитать-изменить-записать» — это **один** неделимый
шаг:

```
atomic +1:  [read-modify-write]   ← никакой другой поток не вклинится
```

## 📖 Memory ordering (модель памяти) — продвинутая тема

Процессор и компилятор могут **переупорядочивать** обращения к памяти ради скорости. В
многопоточности это видно: один поток может увидеть записи другого «не в том порядке».
Атомики позволяют задать **порядок видимости** (`memory_order_seq_cst`, `acquire`,
`release`, `relaxed`).

> 💡 Это глубокая тема. На старте достаточно правила: **используй мьютекс или
> seq_cst-атомики**, и не угадаешь — но и не ошибёшься. Тонкие orderings — для оптимизации
> позже.

---

## 🛠️ Инструмент: ThreadSanitizer (TSan)

Гонки данных почти невозможно найти вручную. **TSan** находит их автоматически:

```bash
gcc -fsanitize=thread -g program.c -o program
./program
```

🖼️ TSan сообщит примерно:
```
WARNING: ThreadSanitizer: data race
  Write of size 4 by thread T2 ... counter++  program.c:8
  Previous write by thread T1 ... counter++  program.c:8
```

> 💡 Всегда прогоняй многопоточный код под TSan. Он покажет гонки, которые иначе ты бы
> ловил неделями.

## ✅ Задачи

1. **Два потока.** Запусти два потока, каждый печатает свой id 5 раз. Дождись обоих.
2. **Увидь гонку.** Реализуй счётчик без защиты, запусти 2–4 потока, убедись, что
   результат неверный и каждый раз разный. Подтверди под TSan.
3. **Почини мьютексом.** Добавь мьютекс, добейся правильного результата.
4. **Почини атомиком.** Перепиши через `atomic_int`, сравни скорость с мьютексом.
5. ⭐ **Параллельная сумма.** Раздели большой массив на N частей, посчитай сумму каждой
   части в своём потоке, сложи результаты. Сравни со скоростью однопоточной версии.
6. ⭐⭐ **Пул потоков.** Реализуй очередь задач и N рабочих потоков, которые их разбирают
   (producer-consumer с мьютексом и условной переменной `cnd_t`).
