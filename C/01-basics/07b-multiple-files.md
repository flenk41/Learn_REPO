# 07b · Разбиваем проект на файлы 🖼️⭐⭐

> 🎯 **Цель блока:** научиться выносить функции в отдельные файлы и вызывать их из других — основа
> любого проекта больше одного экрана кода. В C это связка **заголовок (.h) + реализация (.c)**.

> 🧭 Это базовый навык; организацию целого проекта (папки, сборка, API) углубляет
> [раздел «Проекты и API»](../03b-projects-api/01-project-structure.md).

---

## 📖 Зачем вообще разбивать

```
   весь код в одном main.c → при росте: не найти ничего, нельзя переиспользовать, тяжело собирать.
   РЕШЕНИЕ: группируй связанные функции в отдельные файлы (модули):
   • math_utils.c — математика,  string_utils.c — строки,  main.c — точка входа.
   каждый модуль = пара файлов: ЧТО он умеет (.h) + КАК это сделано (.c).
```

💡 ⭐ В C нет «импорта функции» как в Python. Чтобы вызвать функцию из другого файла, компилятору
нужно показать её **объявление** (заголовок), а линковщику — её **реализацию** (скомпилированный .c).
Разделение «объявление / определение» — ключ ко всему C-проекту.

---

## ⭐⭐ Объявление vs определение

```
   ОБЪЯВЛЕНИЕ (declaration) — «такая функция существует, вот её сигнатура» (без тела):
      int add(int a, int b);          // прототип, заканчивается ;

   ОПРЕДЕЛЕНИЕ (definition) — сама реализация (с телом):
      int add(int a, int b) { return a + b; }

   правило: чтобы ВЫЗВАТЬ функцию, компилятору в этом файле достаточно ОБЪЯВЛЕНИЯ.
   само ОПРЕДЕЛЕНИЕ может жить в другом .c — линковщик их соединит.
```

---

## ⭐⭐ Связка .h + .c (на примере)

🖼️
```
   math_utils.h  — ЗАГОЛОВОК: что модуль предлагает (объявления). его #include-ят другие.
   ─────────────────────────────────────────────
   #ifndef MATH_UTILS_H        // include guard — защита от двойного включения
   #define MATH_UTILS_H

   int add(int a, int b);      // объявления (прототипы), БЕЗ тел
   int mul(int a, int b);

   #endif

   math_utils.c  — РЕАЛИЗАЦИЯ: как это работает (определения).
   ─────────────────────────────────────────────
   #include "math_utils.h"     // свой же заголовок (проверка соответствия)

   int add(int a, int b) { return a + b; }
   int mul(int a, int b) { return a * b; }

   main.c  — ИСПОЛЬЗОВАНИЕ.
   ─────────────────────────────────────────────
   #include <stdio.h>
   #include "math_utils.h"      // подключаем ОБЪЯВЛЕНИЯ → можно вызывать add/mul

   int main(void) {
       printf("%d\n", add(2, 3));   // 5  — определение возьмётся из math_utils.c при линковке
       return 0;
   }
```

💡 ⭐⭐ Схема на каждый модуль: **в `.h` — объявления (что), в `.c` — определения (как), а другие файлы
`#include "модуль.h"`**. `#include "..."` (кавычки) — для своих файлов, `#include <...>` (угловые) — для
системных/библиотечных. `#include` буквально вставляет текст заголовка в твой файл перед компиляцией.

---

## ⭐ Include guard — обязательно

```
   если один .h включить дважды (напрямую и через другой заголовок) → двойные объявления → ошибка.
   ЗАЩИТА — include guard в каждом заголовке:

   #ifndef MATH_UTILS_H        // «если ещё не определено»
   #define MATH_UTILS_H        // определить (пометка «уже включали»)
   ... содержимое ...
   #endif

   современная альтернатива (короче, поддерживается везде):
   #pragma once
```

💡 ⭐ Каждый `.h` оборачивай в include guard (`#ifndef/#define/#endif`) или `#pragma once`. Без этого
большой проект развалится на «redefinition» при сложных включениях. Это не опционально — это привычка.

---

## ⭐⭐ Компиляция нескольких файлов

```
   КАЖДЫЙ .c компилируется ОТДЕЛЬНО в объектный файл (.o), потом ЛИНКОВЩИК соединяет их в программу.
   .h НЕ компилируются — они лишь вставляются (#include) в .c.

   разом:
   gcc main.c math_utils.c -o app          # скомпилировать оба .c и слинковать → app

   по шагам (как делают сборщики):
   gcc -c math_utils.c        # → math_utils.o (только компиляция, -c = не линковать)
   gcc -c main.c              # → main.o
   gcc main.o math_utils.o -o app           # линковка объектников в исполняемый файл
```

🖼️
```
   main.c ─┐ компиляция → main.o ──┐
           │                       ├─ ЛИНКОВКА → app (исполняемый)
   math_utils.c → math_utils.o ────┘
   (math_utils.h только #include-ится в .c — отдельно не компилируется)
```

💡 ⭐⭐ Запомни поток: **`.c` → `.o` (компиляция каждого по отдельности) → линковка всех `.o` в программу**.
Перечисляй ВСЕ свои `.c` при сборке (`gcc main.c math_utils.c ...`) — иначе линковщик скажет «undefined
reference to add» (объявление видел, а реализацию — нет). Как это автоматизировать через Makefile —
[модуль 16](../03-middle/16-preprocessor-modularity.md).

## ⚠️ Ловушки

- ❌ «undefined reference to `add`» — забыл указать `math_utils.c` при компиляции (нет определения для линковщика).
- ❌ Положить определение функции (с телом) в `.h` → при включении в несколько `.c` — «multiple definition».
  В `.h` — только объявления (прототипы)! (Исключение — `static inline`/макросы.)
- ❌ Забыть include guard → «redefinition» при двойном включении.
- ❌ `#include "file.c"` (включать .c) — почти всегда ошибка; включай только `.h`.
- ❌ Объявил функцию в `.h`, но сигнатура не совпала с определением в `.c` → ошибки/UB.
