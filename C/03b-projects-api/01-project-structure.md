# 1 · Структура проекта: разделяем на файлы 🖼️

> 🎯 **Цель блока:** научиться раскладывать C-проект по файлам и папкам так, как это
> делают в реальных командах. Опираемся на [16 · Препроцессор и модульность](../03-middle/16-preprocessor-modularity.md),
> но идём дальше — к архитектуре.

---

## 📖 Зачем вообще разделять

Когда программа — один файл на 2000 строк, в ней:
- невозможно ориентироваться;
- любая правка рискует сломать всё;
- нельзя переиспользовать кусок в другом проекте;
- невозможно работать вдвоём (постоянные конфликты).

🎯 Решение — **разделение ответственности** (separation of concerns): каждый файл
отвечает за **одну** тему. Парсер — отдельно, работа с сетью — отдельно, логика — отдельно.

---

## ⭐ Напоминание: пара .h + .c

- **`.h` (заголовок)** — *что* умеет модуль: объявления функций, типы, константы. Это его
  **публичное лицо** (API).
- **`.c` (реализация)** — *как* это сделано: тела функций. Это «внутренности».

```c
// stack.h — ЧТО (интерфейс)
#ifndef STACK_H
#define STACK_H
typedef struct Stack Stack;     // тип, детали скрыты
Stack *stack_create(void);
void   stack_push(Stack *s, int value);
int    stack_pop(Stack *s);
void   stack_destroy(Stack *s);
#endif
```
```c
// stack.c — КАК (реализация)
#include "stack.h"
#include <stdlib.h>
struct Stack { int *data; size_t size, cap; };   // детали тут, наружу не видны
Stack *stack_create(void) { ... }
// ...
```

💡 Кто подключает `stack.h`, видит только **функции**, но не знает, как устроен `Stack`
внутри. Это и есть инкапсуляция — подробно в [следующем модуле про API](02-designing-api.md).

## 🖼️ Структура настоящего проекта

Маленький проект может быть плоским, но как только он растёт — применяют стандартную
раскладку по папкам:

```
myproject/
├── include/              ← публичные заголовки (.h) — API проекта
│   └── stack.h
├── src/                  ← реализация (.c)
│   ├── stack.c
│   └── main.c
├── tests/                ← тесты
│   └── test_stack.c
├── build/                ← результаты сборки (.o, бинарь) — в .gitignore!
├── Makefile              ← как собирать
└── README.md             ← что это и как запустить
```

| Папка | Что лежит |
|-------|-----------|
| `include/` | заголовки `.h` — то, что видят другие (публичный API) |
| `src/` | исходники `.c` — реализация |
| `tests/` | тесты |
| `build/` | артефакты сборки (не хранятся в git) |

💡 Эта раскладка — негласный стандарт. Увидев её, любой C-разработчик сразу понимает,
где что искать.

## ⭐ Главное правило: направление зависимостей

🖼️ Модули должны зависеть **в одну сторону**, без циклов:

```
   main.c  ──►  логика  ──►  утилиты (stack, string, ...)
   (верх)       (середина)    (низ, ни от кого не зависит)

   ✅ верхний слой знает про нижний
   ❌ нижний слой НЕ должен знать про верхний (иначе клубок)
```

> ⚠️ Если `stack.c` начинает `#include "main.h"` — это тревожный сигнал: низкоуровневый
> модуль не должен зависеть от высокоуровневого. Утилита должна быть самодостаточной,
> чтобы её можно было вынуть и переиспользовать.

---

## 📖 Гигиена заголовков

```c
// ✅ В .h — только то, что нужно снаружи
#ifndef STACK_H          // include guard (или #pragma once)
#define STACK_H
#include <stddef.h>      // подключай в .h только то, что нужно ДЛЯ ЗАГОЛОВКА
typedef struct Stack Stack;
Stack *stack_create(void);
#endif
```

Правила хорошего тона:
- В `.h` — **минимум** `#include` (только нужное для объявлений). Остальное — в `.c`.
- Всегда **include guard** (`#ifndef`/`#define`/`#endif`) или `#pragma once`.
- Внутренние («приватные») функции делай `static` в `.c` — они не попадут в API.

```c
// stack.c
static void grow(Stack *s) { ... }   // static = видна только в этом файле (приватная)
```

## 🛠️ Сборка многофайлового проекта (Makefile)

🖼️ Каждый `.c` → `.o`, потом линковка (из модуля 16, но теперь с папками):

```makefile
CC = gcc
CFLAGS = -Wall -Wextra -g -fsanitize=address -Iinclude   # -I: где искать .h

build/app: build/main.o build/stack.o
	$(CC) $(CFLAGS) $^ -o $@

build/%.o: src/%.c
	@mkdir -p build
	$(CC) $(CFLAGS) -c $< -o $@

clean:
	rm -rf build
```

- `-Iinclude` говорит компилятору искать заголовки в папке `include/`.
- `make` пересоберёт только изменённые файлы.

> 💡 Для больших/кроссплатформенных проектов вместо Make берут **CMake** (см.
> [Senior · 23](../04-senior/23-systems.md)). Принцип тот же.

## 📖 Когда разделять, а когда нет

| Ситуация | Решение |
|----------|---------|
| Файл перевалил за ~300–500 строк | пора делить |
| В файле «две темы» (сеть + логика) | раздели по темам |
| Кусок хочется переиспользовать | вынеси в отдельный модуль |
| Программа на 50 строк | не дроби, не усложняй |

> ⚠️ Не впадай в крайность: 20 файлов по 10 строк — это тоже плохо. Дели по **смыслу**, а
> не по количеству строк. Один файл = одна понятная зона ответственности.

## ✅ Задачи

1. **Раздели монолит.** Возьми любую свою программу из 1 файла (например калькулятор) и
   раздели на `main.c` + модуль логики (`.h`/`.c`).
2. **Папки.** Перенеси проект в раскладку `include/ src/ build/`, напиши Makefile с
   `-Iinclude`. Собери.
3. **static.** Сделай внутренние функции модуля `static`, убедись, что они не видны из
   `main.c` (попробуй вызвать — получи ошибку линковки).
4. **Направление зависимостей.** Нарисуй на бумаге, какой модуль от какого зависит в
   твоём проекте. Есть ли циклы?
5. ⭐ **Три модуля.** Собери проект из трёх модулей (например `io`, `logic`, `utils`) +
   `main`, с правильным направлением зависимостей и одним Makefile.

## ❓ Проверь себя

1. Зачем разделять проект на файлы?
2. Что кладут в `.h`, а что в `.c`?
3. Что означает «направление зависимостей» и почему циклы — плохо?
4. Зачем функции делать `static` в `.c`?
5. Что делает флаг `-Iinclude`?
6. Когда НЕ стоит дробить код?
