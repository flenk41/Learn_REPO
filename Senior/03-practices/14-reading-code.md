# 14 · Чтение чужого большого кода 🖼️⭐

> 🎯 **Цель блока:** освоить навык, который Senior использует каждый день, — быстро въезжать в
> незнакомый большой код, не читая его весь подряд.

---

## 📖 Ты читаешь код больше, чем пишешь

```
   реальность Senior: 80% времени — в ЧУЖОМ коде (легаси, новый проект, опенсорс,
   код коллег на ревью). Писать с нуля «чистый лист» — редкая роскошь.
   → навык быстро понимать незнакомый код важнее навыка писать.
```

💡 ⭐ Junior боится большой незнакомой кодовой базы («ничего не понятно, всё сложно»). Senior
знает: **не нужно понимать всё**. Нужно понять ровно столько, сколько требует задача, и уметь
быстро это найти.

---

## ⭐ Стратегия: сверху вниз, а не подряд

```
   ❌ открыть файл №1 и читать каждую строку до конца проекта — утонешь.
   ✅ ВЕРХНЕУРОВНЕВО → ВГЛУБЬ ПО НУЖДЕ:

   1. КАРТА. README, структура папок, точки входа (main, маршруты, конфиг).
      "Где вход? Какие крупные части? Как они связаны?"
   2. ПОТОК. Проследи ОДИН сценарий end-to-end (например, "что происходит при логине"):
      от входа до БД и обратно. Один путь даёт каркас понимания.
   3. ВГЛУБЬ ТОЧЕЧНО. Ныряй в детали ТОЛЬКО там, где нужно для задачи.
```

🖼️
```
   понимание чужого кода = как карта города:
   сначала МЕТРО-схема (крупные районы и линии) → потом конкретная улица по нужде
   а не: выучить каждый дом подряд с окраины
```

---

## ⭐⭐ Инструменты понимания

```
   • "Иди за выполнением" — отладчик/breakpoint на входе сценария, шагай и смотри поток.
   • Поиск по проекту — grep/IDE: где определено? где вызывается? кто использует?
   • "Go to definition" / "Find usages" — навигация по связям, а не чтение подряд.
   • git log / git blame — ИСТОРИЯ: когда и зачем это появилось (контекст решения).
   • Тесты как документация — тест показывает, КАК код задуман к использованию.
   • Нарисуй схему — модули/потоки на бумаге; рисование = понимание.
```

💡 ⭐⭐ Тесты — недооценённый вход. Хочешь понять, что делает модуль? Прочти его тесты — там
примеры использования и ожидаемое поведение, без «воды». А `git blame` + сообщение коммита часто
отвечают на «почему здесь этот костыль» (если нет [ADR](../02-decisions/11-adr.md)).

---

## 📖 Как НЕ сломать незнакомое

```
   • маленькие изменения — большой непонятный код легче менять микрошагами.
   • опирайся на тесты — есть? меняй смелее. нет? сначала добавь на то, что трогаешь.
   • "характеризующий тест" — на легаси без тестов: зафиксируй ТЕКУЩЕЕ поведение тестом,
     потом меняй (см. рефакторинг).
   • не трогай то, что не понимаешь, без нужды — у странного кода бывает причина.
```

⚠️ В чужом коде «странное» решение часто не глупость, а зашитый багфикс/крайний случай. Прежде
чем «исправить очевидно лишнее» — выясни через `git blame`/тесты, зачем оно. См. [техдолг](../01-craft/07-tech-debt.md)
и [правило бойскаута](../01-craft/03-clean-code.md), но осторожно.

---

## ⚠️ Ловушки

- ❌ Пытаться понять всё перед началом — паралич. Понимай по мере необходимости.
- ❌ Читать подряд вместо «следования за сценарием».
- ❌ Игнорировать тесты и историю (`git log/blame`) — это лучшая документация.
- ❌ «Исправлять» непонятное странное место без выяснения причины.
- ❌ Большие изменения в незнакомом коде без страховки тестами.
- ❌ Стесняться спросить автора — 5 минут вопроса экономят день раскопок.

---

## ✅ Упражнения на размышление

1. **Метро-карта.** Возьми незнакомый опенсорс-проект. За 30 минут составь верхнеуровневую карту:
   точки входа, крупные модули, как связаны. Не ныряя в детали.
2. **Один поток.** Выбери один сценарий (например, обработка запроса) и проследи его end-to-end
   отладчиком/поиском. Нарисуй путь.
3. **Тесты как дока.** Найди модуль, прочти только его тесты. Понял, что он делает, не читая
   реализацию?
4. **Blame.** Найди «странную» строку в проекте. Через `git blame`/лог выясни, зачем она.

---

## ❓ Проверь себя

1. Почему навык чтения кода важнее навыка письма для Senior?
2. Опиши стратегию «сверху вниз» для незнакомой кодовой базы.
3. Чем полезны тесты и `git blame` при изучении кода?
4. Как безопасно менять незнакомый код без тестов?
