# 03 · Чистый код: пишем для людей 🖼️⭐

> 🎯 **Цель блока:** усвоить главную истину ремесла — код пишется для **людей**, а не для
> компьютера. Читаемость важнее «умности».

---

## 📖 Код читают в 10 раз чаще, чем пишут

```
   ты пишешь функцию ОДИН раз.
   её читают ДЕСЯТКИ раз: ты через месяц, коллеги, ревьюер, тот, кто чинит баг.
   → оптимизируй под ЧТЕНИЕ, а не под скорость написания.
```

💡 ⭐ Компилятору всё равно, как назван твой `x`. А человеку — нет. «Умный» однострочник, который
никто не понимает, — это **долг**, а не достижение. Senior гордится не тем, что код сложный, а
тем, что он **очевидный**.

> 🧭 Тот же дух, что в [ООП-проектировании](../../OOP/03-design/17-principles.md) (KISS) — но
> здесь шире: про любой код.

---

## ⭐ Признаки чистого кода

```
   ✅ читается как текст: понятно БЕЗ комментариев, что и зачем
   ✅ функция делает ОДНО и названа по тому, что делает
   ✅ нет сюрпризов: имя обещает = код делает
   ✅ короткие функции, мало вложенности, ранний выход
   ✅ нет дублирования логики (но и не абстрагируй преждевременно)
```

Грязный код наоборот: длинные функции «обо всём», имена `data`, `tmp`, `flag2`, вложенность на
5 уровней, копипаста, комментарии-костыли «не трогать, работает».

🖼️
```
   ГРЯЗНО                          ЧИСТО
   def p(d):                       def calculate_total(order):
       t = 0                           return sum(item.price * item.qty
       for x in d:                                 for item in order.items)
           t += x[0]*x[1]
       return t                    ← имя говорит ЧТО; тело говорит КАК
```

## ⭐⭐ Комментарии: код говорит «как», комментарий — «почему»

```
   ❌ # увеличиваем i на 1
      i += 1                          (комментарий пересказывает код — мусор)

   ✅ # API лимит 100 req/min, поэтому батчим по 50 с паузой
      batch_size = 50                 (объясняет НЕОЧЕВИДНОЕ "почему")
```

💡 ⭐⭐ Хороший код **самодокументируется** через имена и структуру. Комментарий нужен там, где
из кода не видно **причину**: бизнес-правило, обход бага, неочевидный выбор. Если тянет писать
комментарий «что делает код» — лучше переименуй/раздели функцию.

⚠️ Комментарии **врут со временем**: код меняют, комментарий забывают. Меньше комментариев-«что»,
больше понятного кода.

---

## 📖 Принцип бойскаута

> «Оставь код чище, чем нашёл.»

Не нужен героический рефакторинг всего. Тронул файл по задаче — заодно подправь имя, разбей
длинную функцию, убери дублирование рядом. Маленькие улучшения накапливаются; код **деградирует**
без этого.

---

## ⚠️ Ловушки

- ❌ Считать «умный» код хорошим. Умный код — тот, что понятен, а не тот, что сложен.
- ❌ Преждевременная абстракция «на будущее» — усложняет сейчас ради гипотетического потом (YAGNI).
- ❌ Комментировать каждую строку вместо понятных имён.
- ❌ Большие функции «чтобы всё рядом». Одна функция — одна задача.
- ❌ Чистить весь проект разом вместо принципа бойскаута по ходу.

---

## ✅ Упражнения

1. **Переименование.** Возьми свой старый код, найди 5 невнятных имён (`d`, `tmp`, `flag`),
   переименуй так, чтобы стало понятно без комментариев.
2. **Разбей функцию.** Найди функцию длиннее 30 строк или с 3+ уровнями вложенности. Раздели на
   мелкие с говорящими именами.
3. **Комментарий-аудит.** Найди 5 комментариев. Какие — «что» (убрать/переименовать), какие —
   «почему» (оставить)?
4. **Бойскаут.** На следующей задаче улучши 2 мелочи в тронутом файле, не относящиеся к задаче.

---

## ❓ Проверь себя

1. Почему код оптимизируют под чтение, а не под написание?
2. Чем «умный» код отличается от «хорошего»?
3. Когда комментарий полезен, а когда вреден?
4. В чём принцип бойскаута?

---

## ✅ Чек-лист

- [ ] Пишу код, понятный без комментариев
- [ ] Имена обещают ровно то, что код делает
- [ ] Комментирую «почему», а не «что»
- [ ] Держу функции короткими и про одно
- [ ] Оставляю тронутый код чуть чище
