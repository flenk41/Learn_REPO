# 04 · Регистры 🖼️⭐

> 🎯 **Цель блока:** понять регистры — крошечную сверхбыструю память внутри CPU, где реально
> происходят вычисления.

---

## 📖 Где CPU «держит» данные прямо сейчас

```
   CPU не считает «в RAM». он загружает данные в РЕГИСТРЫ (внутри ядра), считает там, потом
   сохраняет обратно. регистр — самая быстрая память: доступ за доли такта.

   иерархия скорости (модуль 07-08):
   РЕГИСТРЫ (быстрее всего, но их единицы-десятки) > кэш L1 > L2 > L3 > RAM (медленно)
```

💡 ⭐ Регистры — «рабочий стол» процессора: маленький, но мгновенный. Их **очень мало** (десятки),
поэтому компилятор тщательно решает, что в них держать (это называется «распределение регистров»).
Всё, что не влезло, живёт в памяти и подгружается.

🖼️
```
   RAM (гигабайты, медленно)  ──load──►  РЕГИСТРЫ (байты, мгновенно)  ──считаем здесь──►
                              ◄─store──                               сохраняем результат
   вычисления идут в регистрах; память — это «склад», регистры — «руки».
```

---

## ⭐ Какие бывают регистры (x86-64, упрощённо)

```
   • РЕГИСТРЫ ОБЩЕГО НАЗНАЧЕНИЯ — для данных и адресов: rax, rbx, rcx, rdx, rsi, rdi, r8–r15.
     (в ассемблере увидишь именно их). 64-битные; есть «части»: eax (32), ax (16), al (8).
   • УКАЗАТЕЛЬ СТЕКА (rsp) — вершина стека (модуль 17). критичен для функций.
   • УКАЗАТЕЛЬ БАЗЫ (rbp) — база кадра стека (иногда).
   • СЧЁТЧИК КОМАНД (rip) — адрес следующей инструкции (тот самый PC).
   • ФЛАГИ (rflags) — результаты сравнений (равно/больше/перенос) → для условных переходов.
   • ВЕКТОРНЫЕ (xmm/ymm/zmm) — для SIMD, обработки нескольких данных разом (модуль 24).
```

💡 ⭐ Имена регистров ты постоянно встретишь в ассемблере (Уровень 2). Не нужно зубрить все —
важно понимать роль: данные/адреса (общие), вершина стека (rsp), следующая инструкция (rip), флаги
(для if).

## ⭐⭐ Почему регистров мало и почему это важно

```
   регистры ОЧЕНЬ быстрые, но физически дорогие → их единицы-десятки.
   следствие для производительности:
   • компилятор старается держать «горячие» переменные в регистрах (быстро).
   • когда регистров не хватает — «выгрузка» (spill) переменных в стек/память (медленнее).
   • плотные циклы с малым числом переменных → влезают в регистры → быстро.
   • много переменных/сложные выражения → spill → обращения к памяти.
```

💡 ⭐⭐ Связь с твоим кодом: компилятор сам распределяет регистры (тебе не надо), но понимание, что
«регистров мало, память дальше», объясняет, почему простые плотные циклы быстры, а почему
оптимизатор так старается их использовать. В godbolt видно `mov` между регистрами и памятью —
это и есть эта работа.

## 📖 Регистры в ассемблере (превью Уровня 2)

```asm
   ; int add(int a, int b) { return a + b; }   (упрощённо, System V ABI)
   ; аргументы приходят в регистрах edi, esi (модуль 18 — calling convention)
   add  eax, edi, esi     ; концептуально: eax = a + b
   ret                    ; вернуть (результат в eax)
```

```
   увидишь: данные двигаются между регистрами (mov), считаются (add/mul), сравниваются (cmp),
   результат функции — в rax/eax. это «язык» Уровня 2.
```

## ⚠️ Ловушки

- ❌ Думать, что вычисления идут «в памяти» — они идут в регистрах.
- ❌ Пытаться вручную управлять регистрами на высоком уровне — это работа компилятора.
- ❌ Зубрить все имена регистров — важнее роли (данные, rsp, rip, флаги).
- ❌ Игнорировать, что регистров мало → spill в память при сложном коде.

## ✅ Упражнения

1. **godbolt.** Скомпилируй функцию с 2–3 аргументами на `-O0`. Найди регистры (edi, esi, eax).
   Куда кладётся результат?
2. **Spill.** Сделай функцию с МНОГО локальных переменных. Видно ли обращения к стеку (`[rbp-…]`/
   `[rsp-…]`)? Это и есть выгрузка.
3. **Флаги.** Сделай `if (a > b)`. Найди `cmp` и условный переход (`jg`/`jle`). Это работа флагов.
4. **Роли.** Выпиши: какой регистр — следующая инструкция? вершина стека? результат функции?
