# 05 · Машинный код и инструкции 🖼️⭐

> 🎯 **Цель блока:** понять, что машинный код — это просто байты, которые CPU читает как
> инструкции, и как ассемблер соотносится с ними.

---

## 📖 Программа — это байты в памяти

```
   на диске и в памяти твоя программа — это БАЙТЫ. часть из них (секция .text) — машинный код:
   последовательность инструкций, закодированных числами. CPU читает эти байты и исполняет.

   три представления ОДНОГО И ТОГО ЖЕ:
   • МАШИННЫЙ КОД:  55 48 89 e5 ...        (байты — то, что реально в файле)
   • АССЕМБЛЕР:     push rbp; mov rbp,rsp  (читаемая запись тех же байтов, 1:1)
   • ИСХОДНИК:      int main() { ... }      (высокоуровневый код, компилируется в верхнее)
```

💡 ⭐ Ассемблер и машинный код — **одно и то же**, разная запись: ассемблер для людей, байты для
CPU. Ассемблер транслируется в машинный код почти один-в-один (модуль 09). Поэтому, читая
ассемблер, ты фактически читаешь то, что исполняет процессор.

🖼️
```
   исходник (C)  ──компилятор──►  ассемблер  ──ассемблер(as)──►  машинный код (байты)
   int a+b           add eax,...            55 48 89 e5...
   человеку         человеку (низко)       процессору
   ↑ много-к-одному    ↑ почти один-к-одному
```

## ⭐ Как закодирована инструкция

```
   инструкция в машинном коде — это байты, кодирующие:
   • ЧТО делать (опкод: add/mov/jump…)
   • НАД ЧЕМ (операнды: регистры, константы, адреса памяти)
   разные инструкции — разной длины (x86: от 1 до 15 байт; переменная длина).
   (в RISC, например ARM, инструкции фиксированной длины — проще декодировать.)
```

```
   пример (концептуально):
   байты:  48 01 f0
   декод:  add rax, rsi      ; rax = rax + rsi
   CPU читает 48 01 f0 → понимает «сложить rsi в rax» → делает.
```

💡 ⭐ Тебе не нужно кодировать инструкции вручную (это делает ассемблер). Но важно понимать: **код
— это данные-байты**, лежащие в памяти, и CPU интерпретирует их как команды. Отсюда, кстати, и
класс уязвимостей: если заставить CPU исполнить «не те» байты как код (трек безопасности).

## ⭐⭐ ISA: контракт между кодом и процессором

```
   ISA (Instruction Set Architecture) — НАБОР инструкций, который понимает семейство процессоров:
   • x86-64 (Intel/AMD, десктопы/серверы) — сложный, переменная длина, историческое наследие.
   • ARM (телефоны, Apple Silicon, серверы) — RISC, фикс. длина, проще, энергоэффективнее.
   • RISC-V — открытая ISA, растёт.

   ISA — это «контракт»: компилятор генерирует инструкции ISA, любой процессор этой ISA их исполнит.
   код под x86 НЕ запустится на ARM напрямую — другой набор инструкций (нужна перекомпиляция/эмуляция).
```

💡 ⭐⭐ Поэтому программы компилируют под конкретную архитектуру. Один и тот же C-исходник компилятор
превратит в РАЗНЫЙ машинный код для x86 и ARM. ISA — стабильный интерфейс ([граница из модуля 01](../00-intro/01-abstraction-layers.md)),
на который опирается весь софт. В godbolt можно переключать целевую ISA и видеть разный ассемблер.

## 📖 Endianness и размеры (коротко)

```
   • данные занимают несколько байт (int = 4 байта, и т.д.) — связь с [памятью в C].
   • ПОРЯДОК байтов (endianness): little-endian (x86, ARM обычно) — младший байт первым.
     важно при работе с бинарными данными/сетью/файлами (где big-endian — «сетевой порядок»).
   • выравнивание (alignment) — данные лежат по «круглым» адресам для скорости (трек C: padding).
```

> 🧭 Это смыкается с [работой с памятью в C](../../C/02-memory/08-memory-model.md) и
> [границей памяти в Interop](../../Interop/02-boundary/08-abi-layout.md): представление данных в
> байтах одинаково важно и там.

## ✅ Упражнения

1. **Три представления.** В godbolt: исходник слева, ассемблер справа. Скомпилируй локально и
   `objdump -d` — увидишь и байты, и ассемблер рядом. Сопоставь.
2. **ISA.** В godbolt переключи компилятор/архитектуру на ARM. Как изменился ассемблер той же функции?
3. **Длина инструкций.** В `objdump -d` посмотри на адреса инструкций — на сколько байт «прыгают»?
   Все ли одинаковой длины (x86 — нет)?
4. **Endianness.** Запиши число 0x12345678 как 4 байта в little-endian. В каком порядке?
