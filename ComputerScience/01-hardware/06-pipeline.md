# 06 · Конвейер и предсказание ветвлений 🖼️⭐⭐

> 🎯 **Цель блока:** понять, как CPU ускоряется через конвейер и спекуляцию — и почему «непредсказуемые»
> ветвления (if) могут стоить дорого.

---

## 📖 Конвейер: несколько инструкций «в работе» одновременно

```
   если бы CPU делал инструкцию ПОЛНОСТЬЮ перед следующей — простаивал бы.
   КОНВЕЙЕР (pipeline): инструкция проходит стадии (fetch→decode→execute→…), и пока одна на
   стадии execute, СЛЕДУЮЩАЯ уже на decode, а ещё одна — на fetch. как конвейер на заводе.
   → за каждый такт «выходит» готовая инструкция, хотя каждая проходит несколько стадий.
```

🖼️
```
   без конвейера:  [F D E][F D E][F D E]        — медленно, стадии простаивают
   с конвейером:   [F D E]                       такт 1: F инстр.1
                    [F D E]                      такт 2: D инстр.1 + F инстр.2
                     [F D E]                     такт 3: E инстр.1 + D инстр.2 + F инстр.3
                   → ~1 инструкция за такт, стадии заняты параллельно
```

💡 ⭐ Конвейер — главная причина, почему CPU быстр: он держит все «станки» занятыми. Современные
CPU имеют глубокие конвейеры (десятки стадий) и исполняют несколько инструкций за такт
(суперскаляр). Но конвейер уязвим к «затыкам».

---

## ⭐ Что ломает конвейер: ветвления

```
   проблема: ВЕТВЛЕНИЕ (if, цикл, вызов) — CPU не знает, какая инструкция следующая, пока не
   вычислит условие. а конвейер уже хочет грузить следующую СЕЙЧАС. что грузить?

   if (x > 0)       ← пока не вычислили x>0, неизвестно, идти в then или else
       A();
   else
       B();
   → если ждать вычисления — конвейер встанет (пузырь). дорого.
```

---

## ⭐⭐ Предсказание ветвлений и спекуляция

```
   решение CPU: ПРЕДСКАЗАТЬ, куда пойдёт ветвление, и спекулятивно исполнять наперёд.
   • ПРЕДСКАЗАТЕЛЬ ВЕТВЛЕНИЙ — по истории угадывает (этот if обычно true → грузим then).
   • СПЕКУЛЯЦИЯ — исполняем предсказанную ветку, не дожидаясь условия.
   • УГАДАЛ (обычно >90%) → конвейер не простаивал, всё быстро.
   • ОШИБСЯ (misprediction) → сбросить спекулятивную работу, начать заново = ШТРАФ (десятки тактов!).
```

🖼️
```
   предсказание верное:  [▓▓▓▓▓▓▓▓] конвейер полон, летим
   ошибка предсказания:  [▓▓▓✗.....] сброс, перезаполнение → потеря ~15-20 тактов
   → ПРЕДСКАЗУЕМЫЕ ветвления быстрые, НЕПРЕДСКАЗУЕМЫЕ (50/50) — дорогие.
```

💡 ⭐⭐ Практический вывод для быстрого кода: **предсказуемые ветвления почти бесплатны, случайные —
дороги**. Классический пример: обработка ОТСОРТИРОВАННОГО массива быстрее неотсортированного при
условии внутри цикла — потому что в отсортированном ветвление предсказуемо. Иногда выгоднее код
БЕЗ ветвлений (branchless) для горячих участков. (Уровень 4.)

---

## 📖 Другие «затыки» конвейера (зависимости)

```
   • ЗАВИСИМОСТЬ ПО ДАННЫМ — инструкция ждёт результат предыдущей (b = a+1; c = b+1; — c ждёт b).
     CPU частично решает out-of-order исполнением независимых инструкций.
   • ПРОМАХ КЭША — данных нет в кэше, ждём RAM (~100 тактов!) — главный «убийца» скорости (модуль 08).
   • ДЛИННЫЕ операции — деление и т.п. долгие.
   → CPU старается «прятать» задержки, переставляя независимую работу. Но если работа зависимая
     и непредсказуемая — простаивает.
```

⚠️ Важный нюанс безопасности: спекулятивное исполнение породило атаки (Spectre/Meltdown) — CPU
спекулятивно трогает данные, оставляя следы в кэше. Упоминаем для полноты (трек безопасности).

- ❌ Игнорировать стоимость непредсказуемых ветвлений в горячем коде.

## ✅ Упражнения

1. **Сортированный vs нет.** (Классика) Сделай цикл с `if (data[i] > threshold)` по массиву.
   Замерь на отсортированном и случайном массиве. Разница? Почему (предсказание)?
2. **godbolt.** Посмотри ассемблер цикла с `if`. Найди условный переход (`jg`/`jle`). Это точка
   предсказания.
3. **Зависимость.** Сравни цикл с зависимой цепочкой (`x = x*a+b`) и независимыми операциями.
   Какой быстрее и почему?
4. **Branchless.** Подумай, как заменить простой `if` на арифметику без ветвления (напр., max
   через сравнение). Зачем это в горячем коде?
