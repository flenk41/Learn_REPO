# 07 · RAM, шина и иерархия памяти 🖼️⭐

> 🎯 **Цель блока:** понять, почему память — это «иерархия» от быстрой и маленькой к медленной и
> большой, и почему расстояние до данных определяет скорость.

---

## 📖 Память: компромисс скорость ↔ объём ↔ цена

```
   нельзя сделать память одновременно быстрой, большой и дешёвой — это [trade-off].
   решение: ИЕРАРХИЯ — несколько уровней, от крошечного-мгновенного до огромного-медленного:

   РЕГИСТРЫ   — байты,      ~0 тактов,   в ядре CPU
   КЭШ L1     — ~32-64 КБ,  ~4 такта,    в ядре
   КЭШ L2     — ~256КБ-1МБ, ~12 тактов,  рядом с ядром
   КЭШ L3     — ~МБ-десятки,~40 тактов,  общий на CPU
   RAM        — ГБ,         ~100-300 тактов,  отдельные чипы по шине
   ДИСК/SSD   — ТБ,         ~миллионы тактов, далеко
```

💡 ⭐ Ключевая интуиция: **чем дальше данные от ядра, тем дольше их ждать** — и разрыв огромный.
RAM в ~100 раз медленнее L1. Диск — в миллионы раз. Поэтому «где лежат данные» часто важнее, чем
«сколько инструкций».

🖼️
```
        быстро, мало          ←──────────────────→         медленно, много
   [регистры][L1][L2][L3]────шина────[ RAM ]────────[ SSD/диск ]
      ~такты      ~десятки тактов     ~100+ тактов    ~миллионы тактов
   данные «поднимаются» вверх по мере использования (кэширование, модуль 08).
```

---

## ⭐ Почему RAM «медленная» (относительно CPU)

```
   • RAM — отдельные чипы, связаны с CPU ШИНОЙ (физическое расстояние, электрические задержки).
   • CPU работает на ГГц (наносекунды на такт), а доступ к RAM — десятки-сотни наносекунд.
   • «стена памяти» (memory wall): CPU ускорялись быстрее памяти → разрыв рос → кэши стали критичны.
   → большую часть «ожидания» в реальном коде составляет ожидание ДАННЫХ из памяти, не вычисления.
```

💡 ⭐ Контринтуитивно: современный CPU чаще **ждёт данные**, чем «считает». Поэтому оптимизация
часто = «приблизить данные» (уместить в кэш, улучшить локальность), а не «убрать инструкции».

---

## ⭐⭐ Локальность — почему иерархия работает

```
   иерархия эффективна благодаря ЛОКАЛЬНОСТИ — программы обращаются к памяти не случайно:
   • ВРЕМЕННА́Я локальность — недавно использованное скоро понадобится снова (переменная цикла).
   • ПРОСТРАНСТВЕННАЯ локальность — рядом с использованным скоро понадобится соседнее (элементы массива).

   → CPU держит «горячие» и «соседние» данные в быстром кэше. пока код локален — почти всё из кэша
     (быстро). как только обращения «прыгают» по памяти случайно — постоянные промахи (медленно).
```

🖼️
```
   ЛОКАЛЬНЫЙ доступ (массив подряд):  [▓▓▓▓▓▓▓▓] → почти всё из кэша → быстро
   СЛУЧАЙНЫЙ доступ (по указателям):  ▓..▓...▓.▓ → постоянные промахи → медленно
   та же «работа», но разница в разы из-за расположения данных.
```

💡 ⭐⭐ Это фундамент быстрого кода: **структуры данных, дружащие с локальностью, в разы быстрее**.
Массив (данные подряд) обычно быстрее связного списка (данные разбросаны по памяти), даже при той
же Big-O — потому что массив локален, а список «прыгает» (промахи кэша). Алгоритмическая сложность
не видит этого; железо — видит. (Подробно — модуль 08 и 23.)

---

## 📖 Связь с Big-O и реальностью

```
   Big-O (трек Алгоритмы) считает ОПЕРАЦИИ, считая все «одинаковыми по цене». но на железе
   доступ к памяти стоит по-разному (кэш vs RAM). поэтому:
   • два O(n) алгоритма могут отличаться в РАЗЫ из-за локальности.
   • иногда «худший» по Big-O, но локальный алгоритм быстрее «лучшего», но прыгающего по памяти.
   → Big-O — необходимая теория, иерархия памяти — почему практика иногда расходится с ней.
```

> 🧭 Это дополняет [Big-O из Алгоритмов](../../Algorithms/02-complexity/12-tradeoffs.md): теория
> сложности + реальность железа = полное понимание производительности.

## ⚠️ Ловушки

- ❌ Считать все обращения к памяти одинаково быстрыми (разница до ~100×).
- ❌ Думать, что CPU всегда «считает» — чаще ждёт данные.
- ❌ Игнорировать локальность при выборе структур данных.
- ❌ Полагаться только на Big-O, забыв про кэш (практика может отличаться в разы).

## ✅ Упражнения

1. **Числа.** Запомни порядки: L1 ~4 такта, RAM ~100+ тактов. Во сколько раз RAM дальше L1?
2. **Массив vs список.** Подумай (и при желании замерь): обход массива в 1млн int vs связного
   списка из 1млн узлов. Что быстрее и почему (локальность)?
3. **Локальность.** Приведи пример временно́й и пространственной локальности из своего кода.
4. **Прыжки.** Где в твоём коде доступ к памяти «прыгает» случайно (указатели, хеши)? Это
   кандидаты на промахи кэша.
