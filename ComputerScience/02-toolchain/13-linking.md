# 13 · Линковка: статическая и динамическая 🖼️⭐⭐

> 🎯 **Цель блока:** понять, как линковщик собирает финальную программу и в чём разница между
> статической и динамической линковкой библиотек.

---

## 📖 Что делает линковщик

```
   ЛИНКОВЩИК (ld) превращает набор .o + библиотеки в ИСПОЛНЯЕМЫЙ файл:
   1. СОБИРАЕТ секции всех .o вместе (.text к .text, .data к .data…).
   2. РАЗРЕШАЕТ СИМВОЛЫ — связывает каждый «нужен» (U) с «определён» (T) (модуль 12).
   3. ВЫПОЛНЯЕТ РЕЛОКАЦИИ — подставляет финальные адреса.
   4. ПОДКЛЮЧАЕТ БИБЛИОТЕКИ — статически или динамически (ниже).
   5. формирует исполняемый ELF/PE с точкой входа.
```

🖼️
```
   main.o  +  utils.o  +  libc  ──ld──►  program
   [code][data]  [code][data]   [printf,...]      [объединённый код+данные, адреса расставлены,
   символы связаны: call printf → реальный printf  символы разрешены]
```

---

## ⭐ Библиотеки: статические vs динамические

```
   СТАТИЧЕСКАЯ библиотека (.a / .lib) — архив объектных файлов. линковщик КОПИРУЕТ нужный код
   ПРЯМО в твой исполняемый. библиотека «вшита».

   ДИНАМИЧЕСКАЯ библиотека (.so / .dll / .dylib) — отдельный файл. в исполняемый кладётся только
   ССЫЛКА; реальный код подгружается при ЗАПУСКЕ (динамическим загрузчиком, модуль 20).
```

🖼️
```
   СТАТИЧЕСКАЯ:  program = [твой код + КОПИЯ кода библиотеки]  → большой, самодостаточный
   ДИНАМИЧЕСКАЯ: program = [твой код + ссылка на libX.so]  +  libX.so (отдельно, общая для всех)
```

---

## ⭐⭐ Trade-off: что выбрать

```
   СТАТИЧЕСКАЯ:
   ✅ самодостаточный бинарник (всё внутри) — проще распространять, нет «отсутствующих .so/.dll».
   ✅ чуть быстрее старт (нечего подгружать), нет накладных на разрешение при запуске.
   ❌ больше размер каждого бинарника (код библиотеки в каждом).
   ❌ обновление библиотеки = пересборка всех программ (баг/уязвимость в libX → пересобрать всё).

   ДИНАМИЧЕСКАЯ:
   ✅ одна копия библиотеки в памяти/на диске на много программ (экономия).
   ✅ обновил libX.so (фикс уязвимости) → все программы получают фикс БЕЗ пересборки.
   ✅ меньше размер бинарников.
   ❌ «dependency hell»: нужная версия .so/.dll должна быть на машине («DLL not found»).
   ❌ чуть медленнее старт (разрешение символов при запуске, PLT/GOT — модуль 20).
```

💡 ⭐⭐ Это классический [trade-off (Senior)](../../Senior/02-decisions/08-tradeoffs.md): размер/
независимость против гибкости/обновляемости. Системные библиотеки обычно динамические (общие, фиксят
централизованно); для переносимости/контейнеров иногда статическая (всё внутри — «работает везде»).
Понимая разницу, ты осознанно выбираешь и понимаешь ошибки вида «library not found».

## 📖 Как это выглядит на практике

```
   gcc main.o -o prog -lm            ; слинковать с библиотекой m (math), обычно динамически
   gcc main.o -static -o prog -lm    ; статически (всё вшить)
   ldd prog                          ; показать, какие ДИНАМИЧЕСКИЕ библиотеки нужны программе
   → увидишь libc.so, libm.so и т.д. — их подгрузит загрузчик при запуске (модуль 20).

   типичная ошибка рантайма: "error while loading shared libraries: libX.so: cannot open"
   = динамическая библиотека не найдена на машине (есть ссылка, нет файла/версии).
```

> 🧭 Связь с [зависимостями/supply chain (Безопасность)](../../Security/03-defensive-code/19-secrets-dependencies.md):
> динамические библиотеки — это зависимости, которые надо обновлять/контролировать; статическая
> «вшивка» означает, что фиксы не приедут сами.

---

## ⚠️ Ловушки

- ❌ Не понимать разницу .a (статич.) и .so/.dll (динамич.).
- ❌ «cannot open shared library» при запуске — это динамическая зависимость не найдена (не ошибка кода).
- ❌ Думать, что статическая «всегда лучше» — теряешь централизованные обновления безопасности.
- ❌ Думать, что динамическая «всегда лучше» — dependency hell, проблемы переносимости.
- ❌ Забыть `-lИмя` для нужной библиотеки → undefined reference при линковке.

---

## ✅ Упражнения

1. **ldd.** Скомпилируй программу, запусти `ldd ./prog`. Какие .so ей нужны? Это динамические зависимости.
2. **Статика vs динамика.** Слинкуй одну программу обычно и с `-static`. Сравни размер бинарников.
   `ldd` на статической — что показывает?
3. **Trade-off.** Для трёх сценариев (контейнер/портативная утилита; системный демон; библиотека с
   частыми security-фиксами) реши: статическая или динамическая? Аргументируй.
4. **Ошибка.** Воспроизведи `undefined reference`, забыв `-lm` для math-функции. Добавь — пропало.
