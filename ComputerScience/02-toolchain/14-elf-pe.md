# 14 · ELF и PE 🖼️⭐

> 🎯 **Цель блока:** понять формат исполняемого файла — ELF (Linux) и PE (Windows) — структуру,
> которая говорит ОС, как загрузить и запустить программу.

---

## 📖 Исполняемый файл — это структурированный контейнер

```
   исполняемый файл — не просто «машинный код». это контейнер по СТАНДАРТНОМУ ФОРМАТУ, который
   говорит операционной системе:
   • где код, где данные (секции)
   • с какого адреса начинать (точка входа, entry point)
   • какие динамические библиотеки нужны
   • как разложить всё в память процесса при запуске

   форматы по ОС:
   • ELF (Executable and Linkable Format) — Linux, Unix, Android, многие embedded.
   • PE (Portable Executable) — Windows (.exe, .dll).
   • Mach-O — macOS.
   разные форматы, одна ИДЕЯ: описать программу так, чтобы ОС её загрузила.
```

💡 ⭐ Поэтому Linux-бинарник не запустится на Windows и наоборот — не только из-за ISA, но и из-за
формата файла и системных вызовов. Формат — «упаковка», которую понимает конкретная ОС.

---

## ⭐ Структура ELF (упрощённо)

```
   ┌─────────────────────────────┐
   │ ELF Header                   │  — «паспорт»: ELF? 32/64 бит? архитектура? ТОЧКА ВХОДА (адрес)?
   ├─────────────────────────────┤
   │ Program Headers (сегменты)   │  — для ЗАГРУЗЧИКА: какие куски куда грузить в память, права (rwx).
   ├─────────────────────────────┤
   │ Секции:                      │
   │   .text   (код)              │
   │   .rodata (константы)        │
   │   .data   (иниц. данные)     │
   │   .bss    (нулевые — размер) │
   │   .symtab (символы) ...      │
   ├─────────────────────────────┤
   │ Section Headers (для линкера)│  — описание секций (для линковки/отладки).
   └─────────────────────────────┘
```

💡 ⭐ Два «взгляда» на ELF: **секции** (для линковщика — модуль 12) и **сегменты/program headers**
(для загрузчика — что грузить в память при запуске, модуль 15). ELF Header сверху содержит главное:
магическое число (это ELF), битность, архитектуру и **точку входа** — адрес, с которого начнётся
исполнение.

---

## ⭐⭐ Смотрим ELF своими глазами (readelf)

```
   readelf -h prog      ; ELF Header: тип, архитектура, точка входа (Entry point address)
   readelf -l prog      ; Program Headers (сегменты для загрузки): LOAD-сегменты, права
   readelf -S prog      ; Section Headers: .text, .data, .bss, .rodata, размеры, адреса
   readelf -d prog      ; динамическая секция: нужные .so (NEEDED), интерпретатор (ld-linux)
   readelf -s prog      ; таблица символов

   file prog            ; кратко: ELF 64-bit, x86-64, динамически слинкован, и т.д.
```

💡 ⭐⭐ Это превращает «бинарник — чёрный ящик» в читаемую структуру. `readelf -h` покажет точку
входа; `readelf -d` — какие библиотеки подгрузятся и какой динамический загрузчик (`ld-linux.so`).
Связав это с предыдущими модулями, ты видишь весь путь: исходник → секции → ELF → (дальше) загрузка
в память (модуль 15).

## 📖 PE (Windows) — та же идея, другие имена

```
   PE (.exe/.dll) структурно похож:
   • DOS-заголовок (исторический) + PE-заголовок (сигнатура "PE").
   • секции: .text, .data, .rdata (≈rodata), .bss, .idata (импорты), .edata (экспорты).
   • таблица ИМПОРТА — какие функции из каких DLL нужны (аналог NEEDED в ELF).
   • точка входа, базовый адрес, и т.д.
   инструменты Windows: dumpbin, и сторонние (PE-explorer'ы).
   суть та же: контейнер, описывающий программу для загрузчика Windows.
```

> 🧭 Форматы файлов и их разбор — важны и в [безопасности](../../Security/01-recon/06-recon-scanning.md)
> (анализ бинарей, вредоносов) и [Interop](../../Interop/02-boundary/08-abi-layout.md) (бинарная совместимость).

## ⚠️ Ловушки

- ❌ Думать, что исполняемый файл — «просто код» (это структурированный контейнер).
- ❌ Путать секции (для линкера) и сегменты/program headers (для загрузчика).
- ❌ Ожидать, что ELF запустится на Windows (формат + syscalls + ISA разные).
- ❌ Не знать про точку входа (entry point) — откуда начинается исполнение.
- ❌ Бояться `readelf` — это просто «прочитать структуру файла».

---

## ✅ Упражнения

1. **Паспорт.** `file prog` и `readelf -h prog`. Какая архитектура, битность, где точка входа?
2. **Секции.** `readelf -S prog`. Найди .text/.data/.bss/.rodata. Сопоставь с модулем 12.
3. **Зависимости.** `readelf -d prog` (или `ldd prog`). Какие .so нужны? Какой интерпретатор
   (ld-linux)?
4. **Сегменты.** `readelf -l prog`. Найди LOAD-сегменты и их права (R/W/X). Что грузится в память?
