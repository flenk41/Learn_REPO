# 20 · Динамические библиотеки, PLT/GOT 🖼️⭐

> 🎯 **Цель блока:** понять, как именно вызов функции из динамической библиотеки находит её адрес
> во время выполнения — механизм PLT/GOT и ленивое связывание.

---

## 📖 Проблема: адрес функции известен только при запуске

```
   при динамической линковке (модуль 13) код printf лежит в libc.so, который грузится при ЗАПУСКЕ,
   возможно по РАЗНОМУ адресу каждый раз (ASLR). значит:
   • на этапе компиляции/линковки реальный адрес printf НЕИЗВЕСТЕН.
   • как же call printf узнает, куда прыгать?

   решение: УРОВЕНЬ КОСВЕННОСТИ — вызов идёт не напрямую, а через таблицу, которую заполняет
   динамический загрузчик при запуске. эта пара таблиц — PLT и GOT.
```

💡 ⭐ Идея проста: вместо «прыгни на printf по фиксированному адресу» (адрес неизвестен заранее) —
«прыгни туда, где в таблице записан адрес printf» (таблицу заполнит загрузчик, когда узнает адрес).
Косвенность через таблицу = гибкость размещения библиотек.

---

## ⭐ GOT и PLT

```
   GOT (Global Offset Table) — таблица АДРЕСОВ. для каждой внешней функции/данных — ячейка, куда
      загрузчик впишет реальный адрес (когда узнает). код обращается к функции ЧЕРЕЗ ячейку GOT.

   PLT (Procedure Linkage Table) — таблица «трамплинов». код вызывает не printf напрямую, а
      printf@plt — маленький кусочек, который смотрит в GOT и прыгает по записанному там адресу.

   связка: call printf  →  на самом деле  call printf@plt  →  jmp [GOT-ячейка printf]  →  реальный printf
```

🖼️
```
   твой код:   call printf@plt
                    │
   PLT:        printf@plt: jmp [GOT[printf]]   ← косвенный прыжок через GOT
                    │
   GOT[printf]: [ адрес настоящего printf ]    ← заполняет загрузчик при запуске
                    │
                    ▼
              libc.so: ...настоящий printf...
```

## ⭐⭐ Ленивое связывание (lazy binding)

```
   разрешать адреса ВСЕХ функций при старте — дорого (программа может вызвать не все). поэтому
   обычно ЛЕНИВО:
   1. при ПЕРВОМ вызове printf — GOT-ячейка ещё указывает обратно в PLT/резолвер.
   2. PLT зовёт ДИНАМИЧЕСКИЙ РЕЗОЛВЕР (часть ld.so): найди printf в libc, узнай адрес.
   3. резолвер ВПИСЫВАЕТ адрес в GOT-ячейку и прыгает в printf.
   4. при СЛЕДУЮЩИХ вызовах GOT уже содержит адрес → прыжок напрямую, без резолвера. быстро.

   → платим за разрешение символа ОДИН раз, при первом вызове. экономим старт.
```

💡 ⭐⭐ Это объясняет: почему динамические программы стартуют чуть медленнее (косвенность +
ленивое разрешение), почему первый вызов библиотечной функции чуть дороже последующих, и как ASLR
совместим с динамической линковкой (адреса вписываются в GOT в рантайме, где бы библиотека ни легла).
GOT/PLT — «телефонная книга», которую загрузчик заполняет на лету.

## 📖 Безопасность: GOT — цель атак, отсюда защиты

```
   GOT хранит адреса функций → если атакующий перезапишет GOT-ячейку (через уязвимость записи),
   он перенаправит вызов (например, free → system) — захват исполнения.
   защиты: RELRO (сделать GOT read-only после связывания), full RELRO + bind-now (разрешить всё
   при старте и закрыть GOT), ASLR, и т.д.
   → понимая механизм, понимаешь и класс атак, и защиты (трек безопасности).
```

> 🧭 Связь: [безопасность памяти](../../Security/02-vulnerabilities/09-injection-sqli.md) (общий
> принцип «не дай данным стать управлением») и hardening.

## ⭐ Посмотреть на практике

```
   objdump -d prog | grep '@plt'     ; вызовы через PLT (printf@plt, и т.д.)
   readelf -r prog                   ; релокации, в т.ч. для GOT
   readelf -d prog                   ; динамическая секция (NEEDED библиотеки, FLAGS типа BIND_NOW)
   LD_DEBUG=bindings ./prog          ; (Linux) увидеть ленивое связывание символов в действии
   gdb: на первом вызове функции — GOT-ячейка меняется после разрешения.
```

## ✅ Упражнения
