# 19 · Виртуальная память глазами программы 🖼️⭐⭐

> 🎯 **Цель блока:** понять, почему адреса в твоей программе «ненастоящие» — это виртуальные адреса,
> которые MMU отображает на физическую RAM. Это объясняет изоляцию, segfault и многое.

> 🧭 Глубоко тема раскрыта в [треке ОС](../../OS/02-memory/08-why-virtual-memory.md); здесь —
> взгляд со стороны программы и связь с тулчейном/железом.

---

## 📖 Адреса в программе — виртуальные

```
   когда ты печатаешь &x и видишь адрес — это ВИРТУАЛЬНЫЙ адрес, не физический в RAM.
   каждый процесс думает, что у него ВСЯ память «в одиночку», с адреса 0 и далее. на самом деле:
   • у каждого процесса — СВОЁ виртуальное адресное пространство (изолированное).
   • ОС + MMU (аппаратный блок в CPU) переводят виртуальные адреса в физические на лету.
   • два процесса могут иметь «один и тот же» адрес 0x1000 — он указывает на РАЗНУЮ физ. память.
```

🖼️
```
   программа A: вирт. адрес 0x1000 ─┐
                                    ├─MMU (таблицы страниц)─► разная физ. RAM
   программа B: вирт. адрес 0x1000 ─┘
   каждый процесс изолирован: «видит» своё пространство, не может залезть в чужое.
```

💡 ⭐⭐ Это объясняет многое: почему процессы изолированы (не портят друг друга), почему один и тот
же бинарник можно запускать многократно (каждому — своё пространство), почему адреса «одинаковые»
в каждом запуске (если без ASLR). Программа живёт в **иллюзии** собственной непрерывной памяти,
которую поддерживают ОС и MMU.

---

## ⭐ Страницы и таблицы страниц

```
   память управляется СТРАНИЦАМИ (обычно 4 КБ):
   • виртуальное пространство разбито на страницы; физическая RAM — на кадры (frames).
   • ТАБЛИЦА СТРАНИЦ отображает «виртуальная страница → физический кадр».
   • MMU использует её на КАЖДОЕ обращение к памяти (с кэшем переводов — TLB, для скорости).
   • не вся виртуальная память отображена: только нужное. обращение к неотображённой → PAGE FAULT.
```

💡 ⭐ Страничная организация даёт гибкость: память выделяется страницами по мере нужды; разные
страницы имеют разные права (код — r-x, данные — rw-, .rodata — r--); страницы можно «выгружать» на
диск (swap). Это механика под «картой памяти процесса» (модуль 16).

---

## ⭐⭐ Что это объясняет на практике

```
   • ИЗОЛЯЦИЯ: процесс не может прочитать/испортить память другого (разные пространства) → безопасность,
     стабильность (упал один — другие живы).
   • SEGFAULT (снова): обращение к виртуальному адресу, который НЕ отображён или без прав
     (NULL, чужое, запись в r-x) → MMU сигналит → ОС шлёт SIGSEGV → крэш. теперь причина видна на
     уровне страниц/прав.
   • LAZY-выделение: malloc «большого» может не дать физ. память сразу — страницы выделятся при
     первом обращении (page fault → ОС даёт кадр). поэтому «выделил, но не трогал» дёшево.
   • ОБЩИЕ страницы: одна копия .so в физ. RAM отображается в пространства МНОГИХ процессов (экономия,
     модуль 13/20). код read-only можно безопасно делить.
   • ASLR: ОС рандомизирует адреса (стек/куча/библиотеки) при запуске → атакующему труднее (защита).
```

💡 ⭐⭐ Связав это с предыдущим: твои виртуальные адреса (модуль 16) → MMU через таблицы страниц →
физическая RAM. Права страниц объясняют, почему запись в код/константы падает. Lazy-выделение
объясняет, почему `malloc(1GB)` мгновенный, а «потрогать» его — нет. Это глубокое понимание памяти,
которым обладает системный разработчик.

## 📖 Цена виртуальной памяти

```
   перевод адресов не бесплатен:
   • каждое обращение к памяти требует перевода (виртуальный→физический) через таблицы.
   • ускоряет TLB (кэш недавних переводов в MMU). промах TLB → дороже (идти по таблицам страниц).
   • page fault (нет страницы) → ОС обрабатывает (выделить/подгрузить со swap) → дорого.
   → плотный, локальный доступ дружит и с кэшем данных, и с TLB. разбросанный — бьёт по обоим.
```

> 🧭 Полная механика (paging, MMU, TLB, page fault, swap) — [трек ОС, Уровень 2](../../OS/02-memory/09-paging.md).

## ⚠️ Ловушки

- ❌ Думать, что адреса в программе — физические (они виртуальные).
- ❌ Считать, что процессы делят одно адресное пространство (изолированы).
- ❌ Не понимать, что права страниц вызывают segfault (запись в r-x/r--).
- ❌ Думать, что `malloc` сразу резервирует физ. RAM (часто lazy).
- ❌ Игнорировать TLB/локальность при оптимизации доступа к памяти.

## ✅ Упражнения

1. **Одинаковые адреса.** Запусти программу, печатающую `&global`, дважды. Адреса одинаковые (без
   ASLR) или разные? Почему процессы не мешают друг другу?
2. **maps.** `cat /proc/<pid>/maps` — права страниц (r-x для кода, rw- для данных, r-- для rodata).
   Сопоставь с картой памяти (модуль 16).
3. **Lazy malloc.** Выдели большой буфер (`malloc(1GB)`) и НЕ трогай vs трогай каждую страницу.
   Сравни поведение/память (RSS).
4. **Segfault и права.** Запись в строковый литерал (r--) → segfault. Объясни через права страниц.
