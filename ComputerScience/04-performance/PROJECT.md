# 🚀 Финальный проект · «Ускорь программу, понимая железо»

> Капстоун трека: возьми реальную медленную задачу и ускори её в разы, **доказывая каждый шаг
> измерениями и объясняя через железо**. Это синтез всего трека.

> Linux/WSL для `perf`; godbolt для ассемблера.

---

## 🎯 Что делаем

Берёшь задачу, обрабатывающую данные (где есть что ускорять), и проводишь её через полный цикл
оптимизации **с замерами и объяснениями**. Результат — ускоренная программа + отчёт «почему стало
быстрее».

```
   медленная версия → профиль → причина → оптимизация → замер → повтор → быстрая версия + отчёт
```

---

## 📋 Подходящие задачи (выбери одну)

- Обработка большого массива/матрицы (суммы, фильтры, преобразования).
- Простой парсер/обработчик текста на больших данных.
- Подсчёт статистики по большому набору записей (структуры).
- Симуляция (частицы, клеточный автомат) на большой сетке.
- Реализация алгоритма (сортировка/поиск/граф) с акцентом на скорость.

Главное — чтобы было что измерять и ускорять (достаточно данных, заметное время).

---

## 📋 Этапы (каждый — замер + объяснение)

### 1. Базовая версия и замер
- Напиши «наивную» рабочую версию. Замерь время (на -O2, с прогревом/усреднением).
- `perf stat`: IPC, cache-misses, branch-misses. Это твоя точка отсчёта.

### 2. Найди узкое место и причину
- `perf record`/`report`: где время? `perf stat`: тип (memory/compute/branch-bound)?
- Сформулируй гипотезу: почему медленно.

### 3. Алгоритм и структуры данных
- Прежде всего: оптимальна ли сложность (Big-O)? Правильные ли структуры (с учётом локальности)?
- Если нет — исправь. Замерь.

### 4. Локальность (cache-friendly)
- Сделай доступ к данным последовательным/компактным (массивы, SoA, правильный порядок обхода).
- Замерь, сверь cache-misses до/после.

### 5. Ветвления / вычисления (если актуально)
- Если branch-bound — сделай ветвления предсказуемыми/branchless. Если compute-bound — проверь
  автовекторизацию (godbolt: есть SIMD?), помоги компилятору.

### 6. Финальный замер и отчёт
- Сравни базовую и финальную версии: во сколько раз быстрее? Что дало основной выигрыш?
- Объясни КАЖДУЮ оптимизацию через железо (кэш/ветвления/конвейер/SIMD), подкрепи числами perf.

---

## 📊 Отчёт должен показать

```
   • базовая версия: время, perf-метрики.
   • для каждой оптимизации: что изменил, зачем (гипотеза), результат (числа), объяснение через железо.
   • итог: общий ускорение в N раз; что дало больше всего (обычно алгоритм + локальность).
   • вывод: подтвердилась ли методология (90% выигрыша на первых шагах)?
```

---

## 🌟 Бонусы

- 🥚 График время-vs-размер данных до/после — видно изменение поведения на кэше.
- 🐣 Покажи ассемблер горячего цикла до/после (godbolt) — что изменилось.
- 🦅 Найди размеры кэшей своего CPU по «ступенькам» и объясни, где данные перестают влезать.
- 🚀 Сравни с «наивной микрооптимизацией инструкций» — докажи, что алгоритм+локальность дали больше.

---

## ✅ Критерии готовности

- [ ] Есть базовая и финальная версии с реальным ускорением (желательно в разы)
- [ ] Каждый шаг подкреплён замерами (время + perf-метрики)
- [ ] Узкое место найдено профайлером, не угадано
- [ ] Применены правильные лечения под тип узкого места
- [ ] Каждая оптимизация объяснена через железо
- [ ] Вывод про методологию (что дало основной выигрыш)
