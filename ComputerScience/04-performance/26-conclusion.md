```

   • ТУЛЧЕЙН (Ур.2) — как исходник становится программой: компиляция, ассемблер, линковка, ELF.

   • ПРОИЗВОДИТЕЛЬНОСТЬ (Ур.4) — как превратить это знание в быстрый код.

```

## ⭐⭐ Методология быстрого кода

```
   собранный из трека рецепт (порядок важен):

   1. ПРАВИЛЬНЫЙ АЛГОРИТМ И СТРУКТУРЫ ДАННЫХ — сначала Big-O (трек Алгоритмы). никакая
      микрооптимизация не спасёт O(n²) там, где нужен O(n log n).
   2. ЯСНЫЙ ПРОСТОЙ КОД — дай компилятору оптимизировать (модуль 21). не мешай ему трюками.
   3. ИЗМЕРЬ (профайлер) — найди РЕАЛЬНОЕ узкое место и его ПРИЧИНУ (модуль 25). не угадывай.
   4. ОПРЕДЕЛИ ТИП — memory/compute/branch-bound (модуль 22).
   5. ЛЕЧИ ПРАВИЛЬНО:
      • memory-bound → локальность, cache-friendly данные (модуль 23). ЧАЩЕ ВСЕГО ЭТО.
      • branch-bound → предсказуемость/branchless (модуль 6).
      • compute-bound → лучше алгоритм, SIMD/автовекторизация (модуль 24).
   6. ИЗМЕРЬ СНОВА — стало ли быстрее? повтори для следующего узкого места.
   7. ОСТАНОВИСЬ — когда «достаточно быстро» (не оптимизируй бесконечно — trade-off со сложностью).
```

🖼️
```
   алгоритм/структуры → ясный код → ИЗМЕРЬ → тип узкого места → правильное лечение → ИЗМЕРЬ → стоп
        (Big-O)         (компилятор)  (perf)    (memory/compute)   (кэш/ветвл/SIMD)
   порядок важен: 90% выигрыша — на первых шагах (алгоритм + локальность), не в микротрюках.
```

💡 ⭐⭐ Главный вывод трека: **быстрый код — это правильный алгоритм + дружба с памятью + измерение**,
а не «магические трюки» и не «меньше инструкций». Большинство реального ускорения — это лучшая
сложность и локальность данных. Знание железа говорит тебе, ПОЧЕМУ это работает и КУДА смотреть.

## ⭐ Чего НЕ делать (антипаттерны производительности)

```
   ❌ преждевременная оптимизация — усложнять до измерения (чаще вредит).
   ❌ микрооптимизация инструкций при узком месте в кэше/алгоритме/I-O.
   ❌ оптимизация по интуиции без профайлера.
   ❌ жертвовать читаемостью ради скорости, которая не нужна (trade-off, модуль Senior).
   ❌ игнорировать, что узкое место часто ВНЕ CPU — БД, сеть, диск, syscalls (на порядки дороже).
```

> 🧭 [Senior-производительность](../../Senior/03-practices/17-performance-mindset.md): «достаточно
> быстро» — валидная цель; не плати сложностью за ненужную скорость.

## ⚠️ Ловушки

- ❌ Оптимизировать не в том порядке (микротрюки до алгоритма/локальности/замера).
- ❌ Забыть, что узкое место часто вне CPU (БД/сеть/I-O).
- ❌ Оптимизировать бесконечно вместо «достаточно быстро».
- ❌ Применять знание железа как «трюки» вместо методологии (измерь → пойми → лечи).

## ✅ Упражнения на размышление

1. **Методология.** Возьми свой «медленный» код. Пройди рецепт: алгоритм? замерил? тип узкого
   места? правильное лечение?

   меняет приоритеты?
