# 25 · Профилирование на уровне инструкций 🖼️⭐⭐

> 🎯 **Цель блока:** научиться измерять, а не угадывать, — находить реальные узкие места и их
> причину (кэш? ветвления? алгоритм?) инструментами.

---

## 📖 Измеряй, не угадывай

```
   главное правило производительности (повтор из Senior): интуиция о скорости ВРЁТ. тормозит
   обычно НЕ там, где думаешь. поэтому:
   ИЗМЕРЬ → найди реальное узкое место → пойми ПРИЧИНУ → оптимизируй ЕГО → измерь снова.
   оптимизация без замеров = трата сил на не то место.
```

💡 ⭐⭐ Это финал трека: всё знание о железе (кэш, ветвления, конвейер) применяется через
**измерение**. Профайлер показывает не только ГДЕ медленно, но и ПОЧЕМУ (промахи кэша? ошибки
предсказания?), а значит — как лечить правильно (модули 22-24).

---

## ⭐ Уровни профилирования

```
   1. ТАЙМЕРЫ (грубо) — замер времени участков (std::chrono, time). «что вообще медленно».
   2. ПРОФАЙЛЕР по сэмплам (perf, профайлеры IDE) — где программа проводит время (горячие функции/
      строки). «куда уходит время».
   3. СЧЁТЧИКИ CPU (perf stat / perf record) — почему: cache-misses, branch-misses, IPC, TLB-misses.
      «в чём ПРИЧИНА медленности».
   → спускайся по уровням: что медленно → где → почему.
```

🖼️
```
   таймер:    "функция X занимает 70% времени"      (ГДЕ)
   perf record: "строка с обходом списка — горячая"  (точнее ГДЕ)
   perf stat: "cache-misses зашкаливают"             (ПОЧЕМУ → memory-bound → локальность)
   → лечишь причину, а не симптом.
```

---

## ⭐⭐ perf — окно в железо (Linux)

```
   perf stat ./prog          — сводка: время, инструкции, IPC, cache-misses, branch-misses.
      • IPC (инструкций за такт) низкий → CPU простаивает (ждёт что-то).
      • высокий cache-misses → memory-bound → работай с данными/локальностью (модуль 23).
      • высокий branch-misses → непредсказуемые ветвления (модуль 6) → предсказуемость/branchless.

   perf record ./prog ; perf report   — где (функции/строки) тратится время (сэмплы).
   perf annotate              — на уровне ИНСТРУКЦИЙ: какие инструкции «горячие».
   perf list                  — какие события можно считать (кэш, TLB, ветвления...).
```

💡 ⭐⭐ `perf stat` за секунды говорит ТИП узкого места (модуль 22): низкий IPC + высокие
cache-misses = ждёшь память; высокие branch-misses = страдаешь от ветвлений. Это превращает
оптимизацию из гадания в диагностику: измерил причину → применил правильное лечение.

## ⭐ Правильное измерение

```
   ✅ МЕРЯЙ НА -O2/-O3 (релизный режим), не -O0.
   ✅ ПРОГРЕВ — первый прогон «холодный» (кэш, страницы); усредняй несколько прогонов.
   ✅ ДОСТАТОЧНЫЙ объём данных/итераций (чтобы эффект был стабилен, больше шума).
   ✅ НЕ ДАЙ компилятору выкинуть «бесполезный» код — используй результат (печать/volatile/benchmark-библиотеки).
   ✅ МЕНЯЙ ОДНО за раз, меряй до/после (как в [отладке](../../Senior/03-practices/13-debugging-method.md)).
   ✅ микробенчмарки коварны — лучше мерить на реалистичной нагрузке, где возможно.
```

## 📖 Закон Амдала — что вообще стоит ускорять

```
   ускорять имеет смысл только то, что ЗАНИМАЕТ ВРЕМЯ. ускорил кусок на 5% времени → максимум
   выигрыш 5%. ускорил доминирующие 80% → вот рычаг.
   → профайлер показывает, ГДЕ эти доминирующие проценты. оптимизируй их, остальное не трогай.
   (правило: 80% времени обычно в ~20% кода — найди их.)
```

> 🧭 Это [Senior-производительность (закон Амдала, профилируй)](../../Senior/03-practices/17-performance-mindset.md):
> работай только с узким местом, измеряя.

## ⚠️ Ловушки

- ❌ Оптимизировать по интуиции без профайлера.
- ❌ Мерить на `-O0` (нереалистично).
- ❌ Без прогрева/усреднения (шумные числа).
- ❌ Дать компилятору выкинуть измеряемый код.
- ❌ Оптимизировать не-узкое место (закон Амдала: эффект мал).
- ❌ Смотреть только «где», не выясняя «почему» (тип узкого места).

---

## ✅ Упражнения (Linux/WSL)

1. **perf stat.** Запусти `perf stat` на своей программе. Какой IPC? cache-misses? branch-misses?
   Какой тип узкого места?
2. **Где время.** `perf record` + `perf report` — найди горячую функцию/строку. Совпала с
   ожиданием?
3. **До/после.** Оптимизируй узкое место (например, список→массив). Перемерь `perf stat` —
   улучшились ли cache-misses и время?
4. **Амдал.** Найди функцию, занимающую <5% времени. Оптимизируй её. Изменилось ли общее время?
   Вывод?
