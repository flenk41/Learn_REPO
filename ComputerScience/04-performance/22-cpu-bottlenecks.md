# 22 · Что мешает CPU работать быстро 🖼️⭐⭐

> 🎯 **Цель блока:** собрать главные причины, почему «современный быстрый CPU» простаивает, — чтобы
> писать код, который не создаёт эти затыки.

---

## 📖 CPU быстр, но часто ждёт

```
   современное ядро может исполнять несколько инструкций за такт (суперскаляр, конвейер). но в
   реальном коде оно часто простаивает, ОЖИДАЯ. главные «убийцы» скорости:

   1. ПРОМАХИ КЭША — ждём данные из RAM (~100+ тактов). ЧАЩЕ ВСЕГО ГЛАВНОЕ.
   2. ОШИБКИ ПРЕДСКАЗАНИЯ ВЕТВЛЕНИЙ — сброс конвейера (~15-20 тактов).
   3. ЗАВИСИМОСТИ ПО ДАННЫМ — инструкция ждёт результат предыдущей.
   4. ПРОМАХИ TLB — ждём перевод адреса.
   5. ДЛИННЫЕ операции (деление), сериализация, контеншн в многопотоке.
   → «быстрый код» = код, который НЕ создаёт эти ожидания.
```

💡 ⭐⭐ Ключевое осознание: на практике производительность чаще ограничена **доступом к памяти и
ветвлениями**, а не «числом инструкций». Поэтому оптимизация инструкций (что любит делать новичок)
часто бесполезна — узкое место в другом. Сначала пойми, ЧЕМ ограничен код.

---

## ⭐ Чем ограничен код: memory-bound vs compute-bound

```
   MEMORY-BOUND — упирается в ПАМЯТЬ (ждёт данные). признак: много промахов кэша, мало вычислений
      на байт данных. лечение: локальность, компактность, меньше данных (модуль 23).
      ← БОЛЬШИНСТВО реального кода с данными memory-bound.

   COMPUTE-BOUND — упирается в ВЫЧИСЛЕНИЯ (ALU занят). признак: данные в кэше, но много счёта.
      лечение: лучше алгоритм, SIMD (модуль 24), меньше работы.

   BRANCH-BOUND — упирается в непредсказуемые ветвления. лечение: предсказуемость, branchless.

   → определи ТИП узкого места (профайлером, модуль 25), потом лечи правильно.
```

🖼️
```
   профиль показывает:
   высокий cache-miss   → memory-bound  → работай с данными/локальностью
   высокий branch-miss  → branch-bound  → работай с ветвлениями
   ALU занят, кэш ок     → compute-bound → алгоритм/SIMD
   лечить НЕ ТОТ тип = тратить силы зря.
```

## ⭐⭐ Промахи кэша — обычно враг №1

```
   почему кэш доминирует: разрыв CPU↔RAM огромен (~100×). один промах = сотня «потерянных»
   инструкций. в коде, перебирающем данные, расположение данных решает больше, чем арифметика.

   что создаёт промахи:
   • случайный доступ / прыжки по указателям (списки, деревья, хеши с плохой локальностью).
   • большие «прыжки» (страйд) по памяти.
   • раздутые структуры (тянут лишнее, меньше полезного в кэше).
   • обход данных «не в том порядке» (матрица по столбцам).
   → лечение — модуль 23 (cache-friendly код): последовательность, компактность, локальность.
```

💡 ⭐⭐ Поэтому массив часто бьёт связный список, плотная структура — раздутую, последовательный
обход — случайный, даже при одинаковой Big-O. **Где и как лежат данные** — главный рычаг скорости
для большинства задач. Это то, чего Big-O не видит, а железо — определяет.

## 📖 Зависимости и параллелизм на уровне инструкций

```
   CPU умеет исполнять независимые инструкции параллельно (ILP, out-of-order). но ЗАВИСИМАЯ цепочка
   мешает:
      x = x*a + b;  (каждая итерация ждёт предыдущий x) — сериализуется.
   vs независимые накопители (x1, x2, x3, x4 параллельно) — CPU занимает все исполнительные блоки.
   приём: разбивать длинные зависимые цепочки на независимые (например, несколько частичных сумм).
   (но сначала — профилируй; это для горячих участков.)
```

## ⚠️ Ловушки

- ❌ Оптимизировать число инструкций, когда узкое место — память/ветвления.
- ❌ Не определить ТИП узкого места (memory/compute/branch) перед лечением.
- ❌ Игнорировать промахи кэша — обычно главный враг.
- ❌ Длинные зависимые цепочки в горячем цикле.
- ❌ Оптимизировать «на глаз» без профайлера (модуль 25).

   Почему (ILP)?
