# 23 · Cache-friendly код 🖼️⭐⭐

> 🎯 **Цель блока:** освоить конкретные приёмы написания кода, дружащего с кэшем, — самый
> практичный рычаг скорости для большинства задач с данными.

---

## 📖 Расположение данных решает

```
   из модулей 8 и 22: промахи кэша — обычно главный тормоз. значит, оптимизация = сделать так,
   чтобы данные были «под рукой» (в кэше), когда нужны. это достигается ТЕМ, КАК ты раскладываешь и
   обходишь данные — на уровне дизайна, а не трюков.
```

💡 ⭐⭐ Это «data-oriented» взгляд: думай не «какие объекты», а «как данные лежат и как я по ним
хожу». Часто переписать структуры данных под кэш даёт ускорение в РАЗЫ — больше, чем любая
микрооптимизация инструкций.

---

## ⭐ Приёмы cache-friendly кода

```
   1. ПОСЛЕДОВАТЕЛЬНЫЙ ДОСТУП — иди по памяти подряд. массивы > связные структуры.
      обход массива по порядку использует целые кэш-линии; прыжки — тянут линию ради одного элемента.

   2. ЛОКАЛЬНОСТЬ — держи вместе то, что используется вместе по времени и в пространстве.

   3. КОМПАКТНОСТЬ — меньше данных → больше влезает в кэш:
      • убирай ненужные поля; упаковывай структуры (порядок полей влияет на padding, трек C).
      • меньшие типы где уместно (int32 вместо int64, если хватает).

   4. ПРАВИЛЬНЫЙ ПОРЯДОК ОБХОДА — матрицы по строкам (как лежат), не по столбцам (модуль 8).

   5. SoA vs AoS — «структура массивов» vs «массив структур»:
      обрабатываешь ОДНО поле у многих объектов → SoA (поля подряд) кэш-дружнее.

   6. БЛОКИНГ (tiling) — обрабатывай данные блоками, влезающими в кэш (для матриц/больших проходов).

   7. ИЗБЕГАЙ погони за указателями в горячем коде (списки/деревья из разбросанных узлов).
```

🖼️
```
   AoS (массив структур):  [x0 y0 z0][x1 y1 z1]...  — если нужны только x: тянем и y,z зря.
   SoA (структура массивов): [x0 x1 x2...][y0 y1...]  — нужны x → все x подряд, кэш полон. ✅
   выбор зависит от паттерна доступа.
```

## ⭐⭐ Пример: массив vs список (та же O(n))

```
   суммирование 10 млн чисел:
   • МАССИВ:  данные подряд → последовательный доступ → почти всё из кэша → быстро.
   • СПИСОК:  узлы разбросаны по куче → каждый next = прыжок → промах за промахом → в РАЗЫ медленнее.
   обе O(n)! разница — только локальность. на практике массив может быть в 5-10× быстрее.

   вывод: выбирай структуры данных НЕ только по Big-O, но и по локальности доступа.
   связный список хорош для частых вставок в середину; для обхода/перебора — массив/вектор.
```

💡 ⭐⭐ Это главный практический урок трека о скорости. Big-O (трек Алгоритмы) необходим, но
**локальность — второе измерение**, которое определяет реальную скорость. Senior-разработчик
держит в голове оба: алгоритмическую сложность И поведение на кэше.

## 📖 Многопоток: false sharing

```
   в параллельном коде — ещё один кэш-эффект:
   FALSE SHARING — два потока пишут в РАЗНЫЕ переменные, лежащие в ОДНОЙ кэш-линии → ядра постоянно
   синхронизируют линию → резкое замедление.
   лечение: выравнивай/разноси данные разных потоков по разным кэш-линиям (padding до 64 байт).
   (связь с многопоточностью — C++/Rust/ОС.)
```

## ⚠️ Ловушки

- ❌ Выбирать структуры только по Big-O, игнорируя локальность.
- ❌ Связные списки/деревья указателей в горячем коде перебора.
- ❌ Раздутые структуры (лишние поля тянутся в кэш).
- ❌ Обход матриц по столбцам.
- ❌ False sharing в многопотоке.
- ❌ Оптимизировать локальность БЕЗ профилирования (сначала найди узкое место).

4. **Компактность.** Уменьши структуру (убери/упакуй поля), замерь обход массива таких структур до
   и после.
