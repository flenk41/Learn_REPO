# 02 · Инструменты: godbolt, objdump, readelf, gdb 🖼️⭐

> 🎯 **Цель блока:** собрать инструменты, которые делают «невидимое» видимым — ассемблер,
> содержимое бинарника, что происходит при исполнении. Без них трек был бы абстракцией.

---

## 📖 Видеть, а не верить

```
   главный принцип трека: не верь на слово — ПОСМОТРИ. эти инструменты показывают, что РЕАЛЬНО
   происходит: в какой ассемблер компилируется твой код, что внутри бинарника, как идёт исполнение.
```

💡 ⭐ С этими инструментами каждая тема трека становится экспериментом: написал C → увидел
ассемблер → понял. Это в разы сильнее, чем читать теорию.

---

## ⭐ Главные инструменты

```
   🔭 COMPILER EXPLORER (godbolt.org) — пишешь C/C++/Rust, СРАЗУ видишь ассемблер рядом.
      • меняешь код/флаги оптимизации (-O0/-O2) → видишь, как меняется ассемблер.
      • лучший способ «пощупать» компиляцию. Работает в браузере, ничего не ставить.

   🔍 objdump -d program — ДИЗАССЕМБЛИРОВАТЬ готовый бинарник (машинный код → ассемблер).
      • увидеть реальные инструкции скомпилированной программы.

   📦 readelf -a program  /  nm program — заглянуть в СТРУКТУРУ ELF-файла:
      • секции (.text/.data/.bss), символы, заголовки, зависимости (динамические библиотеки).

   🐛 gdb — отладчик: пошаговое исполнение, регистры, стек, память в реальном времени.
      • `disassemble`, `info registers`, `x` (examine memory), breakpoints.

   📊 perf (Linux) — профилирование на уровне CPU: где время, промахи кэша, ветвления (Уровень 4).

   🔬 strace / ltrace — какие системные вызовы / вызовы библиотек делает программа (связь с ОС).
```

🖼️
```
   твой C-код  ──godbolt──►  ассемблер (видишь сразу)
   твой бинарник ──objdump──► реальные инструкции
                 ──readelf──► секции, символы, структура ELF
                 ──gdb──────► исполнение по шагам: регистры, стек, память
   "невидимое" становится видимым.
```

---

## ⭐⭐ Compiler Explorer — твой главный учебный инструмент

```
   почему godbolt идеален для этого трека:
   ✅ мгновенная обратная связь: код слева → ассемблер справа, цветом связаны строки.
   ✅ эксперименты: добавь оптимизацию (-O2) → увидь, как компилятор переписал твой код.
   ✅ сравнение: разные компиляторы, языки, архитектуры (x86 vs ARM).
   ✅ ничего не настраивать — в браузере.

   рабочий цикл трека:
   напиши маленький C → посмотри ассемблер → пойми, что каждая инструкция делает →
   измени код/флаг → увидь разницу → сделай вывод.
```

💡 ⭐⭐ Совет: начинай с **простейшего** (функция `int add(int a, int b){ return a+b; }`) на `-O0`
(без оптимизации — ассемблер близок к коду), потом усложняй и включай `-O2` (увидишь магию
оптимизатора). Маленькие примеры — лучший учитель ассемблера.

## 📖 Минимальная установка

```
   • godbolt — НИЧЕГО не нужно, открой godbolt.org.
   • локально (Linux/WSL/Mac): обычно уже есть gcc/clang, objdump, readelf, nm, gdb (в binutils).
     проверь: gcc --version, objdump --version, gdb --version.
   • Windows: лучше WSL (Linux-инструменты работают как родные) — ELF, objdump, readelf «из коробки».
   • perf — Linux (и WSL2 частично); для глубокого профилирования.
   совет: для этого трека удобнее всего Linux/WSL — мир системного программирования «родной» там.
```

> 🧭 WSL/Linux-окружение — то же, что используется в треках [ОС](../../OS/00-setup/02-tools.md) и
> [Безопасность](../../Security/00-foundations/03-lab-setup.md). Один раз настроил — пригодится везде.

## ⚠️ Ловушки

- ❌ Читать про ассемблер, не открыв godbolt — останется абстракцией.
- ❌ Сразу смотреть `-O2` — оптимизированный ассемблер запутает новичка; начни с `-O0`.
- ❌ Брать огромную программу для разбора — начни с одной маленькой функции.
- ❌ На Windows мучиться без WSL — ELF/objdump/readelf там «родные» в Linux-окружении.
- ❌ Бояться gdb — это просто «пошагово смотреть, что происходит».

## ✅ Упражнения

1. **godbolt.** Открой godbolt.org, напиши `int add(int a,int b){return a+b;}`. Посмотри ассемблер
   на `-O0`. Сколько инструкций? Что они делают?
2. **Оптимизация.** Включи `-O2` для той же функции. Изменилось? Попробуй функцию с циклом.
3. **objdump.** Скомпилируй маленькую программу локально и дизассемблируй (`objdump -d`). Найди
   свою функцию `main`.
4. **readelf.** Запусти `readelf -h` (заголовок) и `readelf -S` (секции) на своём бинарнике. Что видишь?
