# 🎙️ Voice Changer — архитектура (обзор)

> Проект: десктоп-приложение, которое в **реальном времени** меняет твой голос (питч, тембр,
> «другой человек», эффекты) и отдаёт результат в Discord/игры/OBS через **виртуальный микрофон** —
> с **интерфейсом**, где крутишь параметры вживую.

Здесь — **общий обзор и требования**, единые для любого языка. Дальше — две конкретные архитектуры:
- 🟦 **[C/C++ архитектура](ARCHITECTURE-CPP.md)** — на JUCE/PortAudio + DSP-библиотеки.
- 🦀 **[Rust архитектура](ARCHITECTURE-RUST.md)** — на cpal + egui + DSP-крейты.

> ⚠️ **Этика и закон:** меняй СВОЙ голос для развлечения/конфиденциальности/творчества. Не выдавай
> себя за конкретного реального человека ради обмана, мошенничества, травли или обхода верификации —
> это незаконно и вредно. Имитация чужого голоса (voice cloning) требует согласия. Соблюдай правила
> платформ (Discord/игры) и закон. Связь с [этикой влияния](../SocialEng/02-psychology/12-manipulation-vs-persuasion.md).

## 🎯 Что значит «реалистично менять голос»

```
   наивный подход (только питч-шифт) звучит как «бурундук» или «монстр» — НЕ реалистично.
   человеческий голос = ВЫСОТА (pitch, частота связок) + ТЕМБР (форманты, резонансы голосового тракта).

   реалистичная смена = менять ОБА независимо:
   • PITCH SHIFT — поднять/опустить высоту (выше/ниже голос).
   • FORMANT SHIFT — сдвинуть форманты (меняет «размер» говорящего: мужской↔женский↔детский).
   • сохранять естественность (без металлического артефакта) → нужен хороший алгоритм.
```

💡 Ключ к «реалистичности» — **раздельное управление питчем и формантами** (формант-сохраняющий
питч-шифт). Просто ускорить/замедлить сэмпл = неестественно. Поэтому в обеих архитектурах ядро DSP —
это качественный pitch+formant процессор (phase vocoder / PSOLA / WORLD-вокодер), а не примитивный resample.

## 🧱 Общие компоненты (есть в обеих архитектурах)

```
   ┌──────────────┐   ┌─────────────────────────┐   ┌──────────────┐   ┌──────────────┐
   │ 1. ЗАХВАТ    │──►│ 2. DSP-ЯДРО (обработка)  │──►│ 3. ВЫВОД/    │──►│ 4. UI        │
   │ микрофон     │   │ pitch+formant+эффекты    │   │ РОУТИНГ      │   │ ползунки,    │
   │ (audio in)   │   │ (реальное время, кадрами)│   │ вирт. микрофон│  │ пресеты,     │
   └──────────────┘   └─────────────────────────┘   │ + мониторинг │   │ метры        │
                            ▲ параметры                └──────────────┘   └──────┬───────┘
                            └───────────────────────── thread-safe ──────────────┘
   + 5. ПОТОКИ: аудио-callback (реалтайм, НЕ блокировать) ⟂ UI-поток; параметры — без локов.
```

1. **Захват (capture)** — читать микрофон малыми блоками (буферами) через аудио-API.
2. **DSP-ядро (processing)** — кольцевой буфер → кадры → pitch shift + formant shift + цепочка эффектов
   (реверб, эквалайзер, «робот», шумодав) → выходные сэмплы.
3. **Вывод/роутинг (output)** — отдать результат в **виртуальное аудиоустройство**, чтобы другие
   приложения видели его как «микрофон»; плюс мониторинг себе в наушники.
4. **UI** — ползунки (pitch, formant, эффекты), пресеты («женский», «робот», «глубокий»), индикаторы
   уровня; параметры передаются в DSP потокобезопасно.
5. **Потоки** — аудио-callback работает в **реальном времени** и не должен блокироваться (никаких
   аллокаций/локов/файлов внутри); UI — отдельный поток; параметры — через атомики/lock-free.

## ⏱️ Жёсткие требования реального времени

```
   • НИЗКАЯ ЗАДЕРЖКА — цель < ~30-50 мс (иначе эхо/рассинхрон с губами). буфер 256-512 сэмплов @ 48кГц.
   • БЕЗ ПРОПУСКОВ (glitches) — аудио-поток не успел → щелчки. callback обязан укладываться в дедлайн.
   • В АУДИО-ПОТОКЕ НЕЛЬЗЯ: malloc/free, локи (mutex), файловый/сетевой I/O, исключения. всё заранее.
   • буферизация — ring buffer между захватом, обработкой и выводом.
```

💡 Это «hard real-time audio»: главная инженерная сложность не в DSP-формуле, а в том, чтобы
обработка **укладывалась в дедлайн каждый блок** и не аллоцировала/не блокировалась в аудио-потоке.
Эта дисциплина одинакова в C++ и Rust — отличаются только инструменты.

## 🔀 Роутинг в «микрофон» (одинаково для обоих)

```
   чтобы Discord/игра услышали изменённый голос, выход должен прикинуться микрофоном:
   • ПРОСТО: установить готовый виртуальный кабель (VB-CABLE / VB-Audio) — пишешь в него,
     в Discord выбираешь его как «микрофон». быстрый путь для MVP.
   • СЛОЖНО (свой драйвер): написать виртуальное аудиоустройство (Windows: APO/драйвер; сложно).
   • АЛЬТЕРНАТИВА: оформить как VST-плагин и грузить в хост (OBS/voicemeeter) — переиспользует DSP-ядро.

   рекомендация для старта: пиши в существующий виртуальный кабель, не пиши свой драйвер.
```

## 🧠 Два уровня «реализма»

```
   УРОВЕНЬ A — DSP (классика): pitch + formant shift + эффекты.
     управляемо, низкая задержка, без GPU. даёт «другой тембр/пол/персонаж». ← начни отсюда.

   УРОВЕНЬ B — нейросетевая конверсия голоса (voice conversion, напр. RVC):
     превращает твой голос в ГОЛОС КОНКРЕТНОЙ модели — максимально «реалистично».
     требует ML-инференса (часто GPU), выше задержка, тяжелее. ← продвинутый путь.
     ⚠️ клонирование реального человека — только с согласия; иначе этика/закон (см. дисклеймер).
```

## 🗺️ С чего начать (обе архитектуры)

```
   1. сквозной звук: микрофон → (ничего) → колонки. убедись, что аудио-поток работает без щелчков.
   2. добавь ОДИН эффект (громкость/простой питч) — проверь задержку и стабильность.
   3. сделай качественный pitch+formant (библиотека) — это «сердце реализма».
   4. подключи UI с ползунками к параметрам DSP (потокобезопасно).
   5. выведи в виртуальный кабель → проверь в Discord.
   6. пресеты, эффекты, полировка. (опц.) уровень B — нейроконверсия.
```
