## ⭐⭐ SQL vs NoSQL: что выбрать

```
   SQL (реляционные: PostgreSQL, MySQL) — ПО УМОЛЧАНИЮ для большинства:
   • связанные данные, JOIN-ы, транзакции (ACID), сильная консистентность.
   • когда: финансы, заказы, пользователи, всё со связями и где важна целостность.

   NoSQL (документные/key-value/колоночные/графовые) — под конкретный паттерн:
   • документные (MongoDB) — гибкая схема, вложенные документы.
   • key-value (Redis/DynamoDB) — простой доступ по ключу, огромный масштаб/скорость.
   • колоночные (Cassandra) — экстремальная запись/аналитика, временные ряды.
   • графовые (Neo4j) — данные про связи (соцграф, рекомендации).

   правило: по умолчанию реляционная; NoSQL — когда есть ДОКАЗАННАЯ причина (масштаб/паттерн/схема).
```

💡 ⭐⭐ Дефолт — **реляционная БД** (покрывает 90%, даёт связи/транзакции/SQL; Postgres ещё и JSONB для
гибкости). NoSQL бери под доказанную потребность: «ключ-значение на огромном масштабе» → DynamoDB,
«граф друзей» → графовая, «лог событий/временные ряды» → колоночная. «Возьмём Mongo, потому что модно»
без причины — частая дорогая ошибка.

## ⭐ Масштабирование чтения: реплики

```
   read-heavy система (читают чаще, чем пишут — большинство):
   • PRIMARY принимает ЗАПИСИ. REPLICA (копии) обслуживают ЧТЕНИЯ.
   • направляешь SELECT на реплики → primary разгружен → выдерживаешь больше чтений.
   • бонус: отказоустойчивость (primary упал → реплика становится primary).

   ⚠️ REPLICATION LAG — реплики отстают (асинхронно) → можно прочитать чуть устаревшее
   (записал в primary, сразу прочитал с реплики — ещё нет). «read-your-writes» проблема.
```

💡 ⭐ Реплики чтения — стандартный шаг после кэша: один primary пишет, много реплик читают. Решает
масштаб **чтения** и даёт запас надёжности. Помни про **lag**: критичные «прочитать сразу после
записи» чтения направляй на primary.

---

## ⭐⭐ Масштабирование записи/объёма: шардинг

```
   когда данные/запись НЕ влезают в одну машину → ШАРДИНГ (разбить данные по серверам-шардам):
   • по ХЕШУ ключа (hash(user_id) % N) — равномерно.
   • по ДИАПАЗОНУ (A-M, N-Z) или по региону.

   ⚠️ ШАРДИНГ ДОРОГО УСЛОЖНЯЕТ:
   • запросы/JOIN-ы между шардами — тяжелы.
   • неравномерность (горячие шарды), выбор shard key критичен.
   • решардинг (добавить шард) болезнен; транзакции между шардами сложны.
   → шардинг — КРАЙНЕЕ средство, когда кэш + реплики + оптимизация исчерпаны.
```

💡 ⭐⭐ **Лестница масштабирования БД** (иди по порядку, не прыгай!):
**оптимизация (индексы/запросы) → кэш → реплики чтения → шардинг**. Шардинг масштабирует запись и
объём, но дорого усложняет всё (кросс-шардовые запросы, решардинг). Большинству систем хватает «одна
реляционная + реплики + кэш». Преждевременный шардинг — классическая ошибка оверинжиниринга.

## 📖 Денормализация и полиглот-персистентность

```
   • ДЕНОРМАЛИЗАЦИЯ — сознательно дублировать данные ради скорости чтения (меньше JOIN-ов) ценой
     сложности обновления. частый приём на масштабе (особенно в NoSQL).
   • POLYGLOT PERSISTENCE — разные хранилища под разные задачи в одной системе:
     PostgreSQL (основные данные) + Redis (кэш/сессии) + Elasticsearch (поиск) + S3 (файлы/медиа).
     каждое — под свою сильную сторону. норма для крупных систем.
```

## ⚠️ Ловушки

- ❌ Брать NoSQL «по моде» без доказанной причины (теряешь связи/транзакции/SQL).
- ❌ Прыгать к шардингу до кэша/реплик (огромная сложность раньше времени).
- ❌ Игнорировать replication lag (чтение с реплики сразу после записи).
- ❌ Считать реплики решением проблемы ЗАПИСИ (они для чтения; запись — на одном primary).
- ❌ Плохой shard key → перекос (горячие шарды) и частые кросс-шардовые запросы.
- ❌ Хранить файлы/медиа в реляционной БД (им место в объектном хранилище/S3 + CDN).
