# 04 · Балансировщики нагрузки 🖼️⭐⭐

> 🎯 **Цель блока:** понять балансировщик нагрузки — блок, который распределяет трафик по серверам и
> делает возможным горизонтальное масштабирование и отказоустойчивость.

---

## ⭐⭐ Зачем балансировщик

```
   один сервер имеет предел (CPU, память, соединения). когда трафика больше — ставят НЕСКОЛЬКО
   серверов и БАЛАНСИРОВЩИК (load balancer, LB) перед ними:

   • РАСПРЕДЕЛЯЕТ запросы по серверам (никто не перегружен).
   • ОТКАЗОУСТОЙЧИВОСТЬ — упал сервер → LB перестаёт слать на него (health-чек), трафик к живым.
   • МАСШТАБ — добавил сервер → LB включил его в ротацию. убирать так же.
   • точка входа — клиенты знают один адрес (LB), а не каждый сервер.
```

🖼️
```
   запросы ──► [ БАЛАНСИРОВЩИК ] ──┬──► [ Сервер 1 ] ✓
                                   ├──► [ Сервер 2 ] ✓
                                   ├──► [ Сервер 3 ] ✗ (упал → LB исключил)
                                   └──► [ Сервер 4 ] ✓ (добавлен → включён)
   health-чеки следят, кто жив. трафик — только на здоровые серверы.
```

💡 ⭐⭐ Балансировщик — **ключ к горизонтальному масштабу и отказоустойчивости**. Он превращает «один
сервер = точка отказа и предел» в «пул серверов, которые можно добавлять/убирать/терять без падения
сервиса». Почти любая масштабируемая система начинается с «поставить LB перед серверами».

---

## ⭐ Алгоритмы распределения

```
   как LB выбирает сервер для запроса:
   • ROUND ROBIN — по кругу (1,2,3,1,2,3...). просто, равномерно при одинаковых серверах.
   • LEAST CONNECTIONS — на наименее загруженный (по числу активных соединений). для разной нагрузки.
   • IP HASH / sticky — один клиент → всегда тот же сервер (для сессий — но лучше stateless, модуль 09).
   • WEIGHTED — мощным серверам больше трафика.

   выбор зависит от того, одинаковы ли серверы и нужна ли «липкость» к серверу.
```

💡 ⭐ По умолчанию — **round robin** (просто и честно при равных серверах) или **least connections**
(если запросы разной тяжести). Sticky-сессии (привязка клиента к серверу) старайся НЕ использовать —
лучше сделать серверы stateless (модуль 09), тогда любой запрос идёт на любой сервер.

---

## 📖 L4 vs L7 и резерв самого LB

```
   • L4 (транспортный) — балансирует по IP/портам, не смотрит в содержимое. быстро.
   • L7 (прикладной) — понимает HTTP: может маршрутизировать по URL/заголовкам (/api → одни серверы,
     /images → другие), терминировать TLS. гибче, чуть дороже.

   ⚠️ сам балансировщик — тоже точка отказа! в проде их минимум ДВА (active-passive/active-active),
   often + DNS/anycast перед ними. «кто балансирует балансировщик» — резервирование на каждом уровне.
```

💡 L7-балансировщик (понимает HTTP) гибче: маршрут по пути, A/B, TLS-терминация. И помни: LB сам не
должен быть единственной точкой отказа — резервируй и его. Принцип «убери единые точки отказа» — сквозной
для надёжности (модуль 16).

---

## ⚠️ Ловушки

- ❌ Один сервер без LB → предел мощности + единая точка отказа.
- ❌ Sticky-сессии вместо stateless-серверов (ломается при падении сервера, мешает масштабу).
- ❌ Балансировщик без health-чеков (шлёт трафик на мёртвый сервер).
- ❌ Сам LB как единственная точка отказа (нужно резервирование).
- ❌ Считать, что LB решает всё — за ним всё равно общая БД/кэш (свои узкие места).
