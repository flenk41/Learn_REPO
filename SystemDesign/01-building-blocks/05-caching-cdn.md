# 05 · Кэширование и CDN 🖼️⭐⭐

> 🎯 **Цель блока:** понять кэш и CDN — самые мощные рычаги скорости и разгрузки системы. Чаще всего
> именно кэш превращает «тормозит под нагрузкой» в «летает».

> 🧭 Углубляет [кэширование в БД](../../Database/04-scale/21-caching.md) на уровне всей системы.

---

## ⭐⭐ Кэш: частые данные в быстром месте

```
   КЭШ — хранение результатов в БЫСТРОМ хранилище (память/Redis), чтобы не пересчитывать/не ходить в БД.
   • первый запрос → БД (медленно) → положить в кэш.
   • следующие → из кэша (быстро), БД не трогаем.

   зачем: БД и вычисления — дорогие; память — в ~100 000× быстрее диска (модуль 01).
   высокий hit rate (доля попаданий) → БД разгружена, ответы быстры.
```

🖼️
```
   запрос → [ КЭШ? ] ─попал (hit)──► быстрый ответ (микросекунды)
               └─промах (miss)─► [ БД ] → положить в кэш → ответ
   read-heavy системы (читают чаще, чем пишут) выигрывают от кэша больше всего.
```

💡 ⭐⭐ Кэш — **первый инструмент масштабирования чтения**: частые данные отдаются из памяти, а не из
БД. Когда прикидка показывает «4000 чтений/с» (модуль 01) — ответ почти всегда «кэш». Но за скорость
платишь **инвалидацией** (когда данные в кэше устарели) — это главная сложность.

---

## ⭐ Где кэшировать: уровни

```
   кэш есть на КАЖДОМ уровне пути запроса:
   • БРАУЗЕР/КЛИЕНТ — кэш на устройстве (HTTP-кэш, локальное хранилище).
   • CDN — кэш статики ближе к пользователю (см. ниже).
   • БАЛАНСИРОВЩИК/обратный прокси — кэш ответов (Varnish/Nginx).
   • КЭШ ПРИЛОЖЕНИЯ — общий Redis/Memcached между серверами. ⭐ главный для динамики.
   • КЭШ БД — буферный пул СУБД (страницы в RAM, из коробки).

   чем ближе к пользователю кэш — тем быстрее и тем сильнее разгрузка.
```

💡 ⭐ Стандарт для динамических данных — **распределённый кэш (Redis)**, общий для всех серверов
приложения (поэтому серверы остаются stateless, модуль 09). Кэшируй: тяжёлые запросы, профили,
каталог, конфиги, сессии, счётчики.

---

## ⭐⭐ Инвалидация: главная сложность

```
   ПРОБЛЕМА: данные изменились в БД, а в кэше — старая копия → пользователь видит устаревшее (stale).

   СТРАТЕГИИ:
   • TTL (время жизни) — запись истекает через N секунд → перечитается. простейшее; устаревание ≤ TTL.
   • ИНВАЛИДАЦИЯ ПРИ ЗАПИСИ — при UPDATE удаляешь/обновляешь кэш. точнее, но сложнее.
   • CACHE-ASIDE (самый частый) — приложение само: miss → читает БД → кладёт в кэш; при записи — сбрасывает.

   подводные камни:
   • STALE DATA — устаревшее (баланс свежести через TTL).
   • CACHE STAMPEDE — TTL истёк → лавина запросов разом бьёт в БД. решают: блокировка пересчёта,
     рандомизация TTL, фоновое обновление.
   • вытеснение (eviction) — кэш ограничен; политика LRU (выкинуть наименее используемое).
```

💡 ⭐⭐ «В программировании две сложные вещи: инвалидация кэша и придумывание имён». Прагматично:
**TTL для большинства** (устаревание на N секунд часто терпимо), точечная инвалидация при записи — для
важного. Это [trade-off](../../Senior/02-decisions/08-tradeoffs.md): свежесть данных против нагрузки и
сложности.

## ⭐ CDN: контент ближе к пользователю

```
   CDN (сеть доставки контента) — серверы по всему миру, кэширующие СТАТИКУ (картинки, видео, JS/CSS,
   иногда ответы API) близко к пользователю.
   • пользователь из Японии получает файл с японского узла CDN, а не с твоего сервера в США.
   → меньше задержка, меньше нагрузка на твои серверы, устойчивость к всплескам.

   что класть в CDN: статические ассеты, медиа, скачивания. кэш по TTL/версионированию имён (cache busting).
```

💡 ⭐ CDN — это «географический кэш статики»: разгружает твои серверы и режет задержку для далёких
пользователей. Почти любая веб-система ставит статику за CDN. Для динамики CDN тоже умеет (edge-кэш),
но осторожно с приватными/персональными данными.

## ⚠️ Ловушки

- ❌ Кэшировать до оптимизации запросов/индексов (кэш прячет, но не лечит плохие запросы).
- ❌ Игнорировать инвалидацию → показываешь устаревшее как актуальное.
- ❌ Слишком долгий TTL для часто меняемых данных (несвежесть) / слишком короткий (нет смысла).
- ❌ Cache stampede при истечении TTL (лавина в БД).
- ❌ Кэшировать персональные данные в общем/публичном кэше/CDN (утечка между пользователями).
- ❌ Кэшировать всё подряд (низкий hit rate — тратит память зря).
