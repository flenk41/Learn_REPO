# 20 · Проектируем чат-мессенджер 🖼️⭐⭐

> 🎯 **Цель блока:** спроектировать систему обмена сообщениями в реальном времени. Главное новое —
> **постоянные соединения** (WebSocket), доставка онлайн/офлайн и статусы.

---

## ① Требования · ② Масштаб

```
   ФУНКЦИОНАЛЬНО: 1-на-1 и групповые чаты; доставка в РЕАЛЬНОМ ВРЕМЕНИ; история сообщений; статусы
   (доставлено/прочитано, онлайн); доставка офлайн-получателю (когда вернётся).
   НЕФУНКЦИОНАЛЬНО: НИЗКАЯ ЗАДЕРЖКА (мгновенно), надёжность (не терять сообщения), порядок сообщений,
   масштаб на миллионы одновременных соединений.

   ключевая особенность: не «запрос-ответ», а ПОСТОЯННОЕ соединение для мгновенной доставки.
```

---

## ③ Связь: почему WebSocket

```
   обычный HTTP (запрос-ответ) плох для чата: сервер не может сам «толкнуть» сообщение клиенту.
   • поллинг (клиент спрашивает «есть новое?») — задержка + нагрузка. ❌
   • WEBSOCKET — ПОСТОЯННОЕ двунаправленное соединение: сервер ПУШИТ сообщение клиенту мгновенно. ✅
     (альтернативы: SSE — только сервер→клиент; long-polling — запасной вариант.)

   значит: нужен слой серверов, ДЕРЖАЩИХ тысячи живых WebSocket-соединений (gateway/connection servers).
```

💡 ⭐⭐ Сердце чата — **постоянные WebSocket-соединения**: сервер сам толкает сообщение получателю, без
опроса. Это меняет архитектуру: появляется слой **connection-серверов**, удерживающих живые соединения,
и задача — найти, на каком сервере сидит получатель, чтобы доставить ему сообщение.

---

## ④⑤ Высокоуровневая схема и доставка

🖼️
```
   Алиса (WS) ─► [ Connection-сервер 1 ] ─► [ сервис сообщений ] ─► найти, где Боб
                                                    │                     │
                                            [ сохранить в БД ]    [ Боб онлайн? ]
                                                                   ├─да→ [Conn-сервер 3]─► push Бобу (WS)
                                                                   └─нет→ сохранить как недоставленное
                                                                          (+ push-уведомление на телефон)
   реестр присутствия: user → на каком connection-сервере сидит (в Redis).
```

```
   ДОСТАВКА:
   1. Алиса шлёт сообщение по своему WS → connection-сервер → сервис сообщений.
   2. сохранить сообщение (БД) — надёжность/история.
   3. узнать в РЕЕСТРЕ ПРИСУТСТВИЯ (Redis), онлайн ли Боб и на каком connection-сервере.
   4. онлайн → доставить на тот сервер → он пушит Бобу по его WS.
   5. офлайн → пометить недоставленным; послать PUSH-уведомление; доставить, когда Боб подключится.
```

💡 ⭐ Два ключевых хранилища: **БД сообщений** (история, надёжность, порядок) и **реестр присутствия**
(кто где онлайн — в Redis, быстро). Связь между connection-серверами — через общую шину/очередь (Kafka/
Redis pub-sub): сервис сообщений «публикует» сообщение нужному серверу получателя.

---

## ⑥ Детали и компромиссы

```
   • ПОРЯДОК сообщений — sequence id/timestamp на чат; клиент сортирует. (часы серверов не идеальны → логические id.)
   • ДОСТАВКА: at-least-once + дедупликация по message_id на клиенте (идемпотентность, модуль 07).
   • ХРАНЕНИЕ истории — шардинг по chat_id (сообщения одного чата вместе); часто колоночная БД (Cassandra)
     под огромную запись/временной ряд.
   • ГРУППЫ — fan-out сообщения участникам (как лента, модуль 19; для огромных групп — свои приёмы).
   • СТАТУСЫ (печатает/прочитано) — эфемерны, через ту же WS-шину; не обязаны быть надёжными.
   • ШИФРОВАНИЕ — транспорт TLS; для приватности — end-to-end (ключи у клиентов, сервер не читает).
   • масштаб соединений — много connection-серверов за L4-балансировщиком; присутствие в Redis.
```

💡 ⭐⭐ Чат сочетает почти всё из трека: WebSocket (реальное время), Redis (присутствие), БД/Cassandra
(история, шардинг по chat_id), очередь/pub-sub (связь серверов), at-least-once + идемпотентность
(надёжная доставка), push-уведомления (офлайн). Это «экзамен» по строительным блокам.

---

## ⚠️ Ловушки

- ❌ Поллинг вместо WebSocket (задержка + нагрузка).
- ❌ Не сохранять сообщения до доставки (потеря при сбое/офлайне).
- ❌ Полагаться на时чные метки серверов для порядка (используй логические id/sequence).
- ❌ Игнорировать офлайн-доставку и push-уведомления.
- ❌ Доставка без дедупликации (дубли сообщений у клиента).
- ❌ Реляционная БД с JOIN для истории на огромном масштабе (нужен шардинг/колоночная).
