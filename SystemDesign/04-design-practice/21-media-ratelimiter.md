# 21 · Хранилище медиа + rate limiter 🖼️⭐⭐

> 🎯 **Цель блока:** две частые подзадачи дизайна — хранение/раздача больших файлов (фото/видео) и
> ограничение частоты запросов (rate limiting). Оба всплывают почти в каждой системе.

---

## 🖼️ Часть A — Хранилище и раздача медиа

```
   ЗАДАЧА: загрузка и показ фото/видео (как Instagram/YouTube) на масштабе.

   ① ключевое: файлы — НЕ в реляционной БД! им место в ОБЪЕКТНОМ ХРАНИЛИЩЕ (S3/GCS):
      • в БД — только МЕТАДАННЫЕ (id, владелец, URL, размер) + ссылка на объект.
   ② раздача — через CDN (модуль 05): файл кэшируется близко к пользователю → быстро + разгрузка.
   ③ загрузка — PRESIGNED URL: клиент грузит файл ПРЯМО в S3 по временной подписанной ссылке,
      минуя твои серверы (не гоняешь гигабайты через приложение). сервер только выдаёт ссылку и метаданные.
```

🖼️
```
   загрузка:  клиент → сервер: «хочу загрузить» → выдаёт presigned URL → клиент грузит ПРЯМО в S3
   показ:     клиент → CDN (есть копия?) ─hit─► отдаёт. ─miss─► тянет из S3, кэширует, отдаёт.
   твои серверы НЕ гоняют сами файлы — только метаданные и ссылки.
```

```
   ④ обработка — АСИНХРОННО (очередь, модуль 07): загрузил видео → перекодировать в разные качества/
      форматы (480p/720p/1080p), сгенерить превью → готово → обновить статус. пользователь не ждёт.
   ⑤ видео — стриминг по частям (HLS/DASH): дробится на сегменты, плеер тянет по мере просмотра + адаптив качества.
```

💡 ⭐⭐ Три правила медиа: **файлы → объектное хранилище (S3), не БД** (в БД только метаданные);
**раздача → CDN** (близко, быстро, разгрузка); **загрузка → presigned URL** (клиент грузит прямо в S3,
минуя твои серверы). Обработку (перекодирование) — асинхронно через очередь. Это снимает гигабайты с
твоего приложения.

---

## 🚦 Часть B — Rate limiter (ограничение частоты)

```
   ЗАДАЧА: не дать клиенту/IP слать слишком много запросов (защита от злоупотреблений, DDoS, перебора,
   справедливое распределение). напр. «100 запросов/мин на пользователя».

   АЛГОРИТМЫ:
   • TOKEN BUCKET ⭐ — ведро токенов пополняется со скоростью R, запрос берёт токен; нет токенов → 429.
     допускает короткие всплески (накопленные токены). самый популярный.
   • LEAKY BUCKET — запросы «вытекают» с фиксированной скоростью (сглаживает в ровный поток).
   • FIXED / SLIDING WINDOW — считать запросы в окне времени (фикс. окно проще, но «скачок на границе»;
     sliding точнее).

   где хранить счётчики: в Redis (быстро, общий для всех серверов — важно при горизонтальном масштабе!).
   ответ при превышении: HTTP 429 Too Many Requests (+ заголовок Retry-After).
```

💡 ⭐⭐ Стандарт rate limiting — **token bucket** в **Redis** (общий счётчик для всех stateless-серверов).
Почему Redis: серверов много, а лимит — общий на пользователя, значит счётчик должен быть в общем
быстром хранилище, а не в памяти каждого сервера. Превысил → **429** с `Retry-After`. Часто rate
limiter ставят на уровне API-gateway/балансировщика.

## 📖 Связь и применение

```
   • rate limiting — часть [безопасности на масштабе (модуль 22)](22-security-at-scale.md): защита от
     перебора паролей, скрейпинга, DDoS, дорогих операций.
   • presigned URL и метаданные-в-БД — применимы к любым «файловым» фичам (аватары, документы, бэкапы).
   • CDN + объектное хранилище — стандарт для статики/медиа в любой системе.
```

---

## ⚠️ Ловушки

- ❌ Хранить файлы в реляционной БД (раздувает, медленно) — нужны S3 + метаданные в БД.
- ❌ Гонять загрузки/раздачу через свои серверы вместо presigned URL + CDN.
- ❌ Синхронное перекодирование в запросе (пользователь ждёт минуты) — нужна очередь.
- ❌ Rate limiter в памяти сервера (при нескольких серверах лимит «размножается») — нужен Redis.
- ❌ Fixed window без понимания «скачка на границе» (2× лимита на стыке окон).
- ❌ Не возвращать 429/Retry-After (клиент не понимает, что притормозить).
