# 18 · Проектируем сокращатель ссылок (TinyURL) 🖼️⭐⭐

> 🎯 **Цель блока:** пройти первую полноценную задачу дизайна по каркасу из 6 шагов. Сокращатель
> ссылок — классика: простая суть, но всплывают кэш, масштаб чтения, генерация ключей.

> 🧭 Применяем [каркас из 6 шагов (модуль 02)](../00-intro/02-approach-framework.md). Дальше задачи
> усложняются — но метод тот же.

---

## ① Требования

```
   ФУНКЦИОНАЛЬНЫЕ:
   • сократить длинный URL → короткий (tinyurl.com/abc123).
   • переход по короткому → редирект на оригинал.
   • (опц.) кастомный алиас, срок жизни, аналитика кликов.

   НЕФУНКЦИОНАЛЬНЫЕ:
   • МАСШТАБ: read-heavy сильно (переходов >> создания, ~100:1).
   • НИЗКАЯ ЗАДЕРЖКА редиректа (мгновенно).
   • высокая ДОСТУПНОСТЬ (ссылки не должны «отваливаться»).
```

---

## ② Оценка масштаба

```
   допустим: 100М новых ссылок/месяц.
   • запись: 100М / 2.6М с ≈ ~40 ссылок/сек (мало!).
   • чтение (100:1): ~4000 редиректов/сек (вот нагрузка → КЭШ).
   • хранилище: 100М/мес × 12 × 5 лет × ~500 байт ≈ ~3 ТБ (помещается).
   ВЫВОД: запись лёгкая, ЧТЕНИЕ — главное → кэш + реплики чтения.
```

---

## ③ API

```
   POST /urls        { "long_url": "..." }        → { "short": "abc123" }
   GET  /{short}                                   → 301/302 редирект на long_url
   (опц.) POST с { "alias": "...", "ttl": ... }
```

---

## ④ Высокоуровневая схема

🖼️
```
   создать:  клиент → LB → сервер → сгенерить ключ → сохранить (short → long) в БД
   перейти:  клиент → LB → сервер → [КЭШ short→long?] ─hit─► редирект
                                          └─miss─► БД → в кэш → редирект
   [ CDN/LB ] → [ stateless серверы ] → [ Redis кэш ] + [ БД (key-value) + реплики чтения ]
```

---

## ⑤ Детализация: как генерировать короткий ключ

```
   короткий код = 6-7 символов из [a-z A-Z 0-9] (62 символа). 62⁷ ≈ 3.5 триллиона — хватит надолго.

   ВАРИАНТЫ генерации:
   • СЧЁТЧИК + base62 — глобально растущий id (1,2,3...) → закодировать в base62 (короткая строка).
     проблема: единый счётчик — узкое место/SPOF → раздавать ДИАПАЗОНЫ id серверам (батчами).
   • СЛУЧАЙНЫЙ ключ — сгенерить случайные 7 символов, проверить уникальность (коллизия → повтор).
     просто, но нужна проверка существования (индекс по ключу).
   • ХЕШ от URL (+соль), взять первые N — но возможны коллизии, надо разрешать.

   хранилище: key-value по short_code (быстрый поиск по ключу) — отлично шардируется (модуль 10).
```

💡 ⭐⭐ Ключевое решение задачи — **генерация уникального короткого ключа без узкого места**. Единый
счётчик прост, но это SPOF → раздавай серверам диапазоны id блоками (каждый кодирует в base62
локально). Поиск `short → long` идеален для **key-value** хранилища и легко шардируется по ключу.

---

## ⑥ Масштаб и компромиссы

```
   • ЧТЕНИЕ (редиректы) — главная нагрузка → агрессивный КЭШ (Redis): популярные ссылки в памяти,
     БД почти не трогается. + реплики чтения.
   • запись лёгкая → один primary справится; шардинг по short_code, если объём вырастет.
   • редирект 301 (постоянный) кэшируется браузером/CDN → ещё меньше нагрузки (НО теряешь часть
     аналитики кликов); 302 (временный) — считаешь все клики, но больше запросов. ← компромисс.
   • аналитика кликов — асинхронно (очередь), чтобы не тормозить редирект.
```

💡 ⭐ Компромисс 301 vs 302 — типичный «глубинный» момент: 301 разгружает (кэшируется), но прячет часть
кликов от аналитики; 302 считает всё, но нагружает. Выбор — по тому, что важнее (производительность vs
полная аналитика). Аналитику считай асинхронно, чтобы не замедлять редирект.

---

## ⚠️ Ловушки

- ❌ Единый счётчик без раздачи диапазонов (узкое место/SPOF).
- ❌ Игнорировать, что система read-heavy (без кэша редиректы задушат БД).
- ❌ Реляционная БД с тяжёлыми JOIN там, где нужен простой key-value поиск.
- ❌ Синхронная запись аналитики в редирект (замедляет главный путь).
- ❌ Не подумать про коллизии (для случайных/хеш-ключей).
