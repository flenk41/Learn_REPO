# 02 · Как подходить к задаче дизайна 🖼️⭐⭐

> 🎯 **Цель блока:** получить пошаговый каркас, по которому решается любая задача системного дизайна —
> на собеседовании и в реальной работе. Структура важнее эрудиции.

---

## ⭐⭐ Каркас из 6 шагов

```
   1. ТРЕБОВАНИЯ — уточни функциональные + нефункциональные, зафиксируй scope (модуль 01).
   2. ОЦЕНКА МАСШТАБА — прикинь нагрузку (записи/чтения/хранилище), найди, где будет больно.
   3. API / интерфейс — определи основные операции (что система предоставляет: эндпоинты/методы).
   4. ВЫСОКОУРОВНЕВАЯ СХЕМА — нарисуй блоки: клиент → балансировщик → серверы → БД/кэш/очереди.
   5. ДЕТАЛИЗАЦИЯ — углубись в 1-2 ключевых места (хранение данных, узкое место, сложная фича).
   6. МАСШТАБ И КОМПРОМИССЫ — как это растёт, где падает, что чем платим. назови trade-offs.
```

🖼️
```
   поток решения (всегда в этом порядке):

   требования → оценка → API → высокоуровневая схема → детали → масштаб/компромиссы
       │           │       │           │                  │            │
   что строим?  насколько  что даёт  как блоки      где сложно?   как растёт,
   приоритеты?  большое?   наружу?   связаны?       углубиться    чем платим?
```

💡 ⭐⭐ Этот каркас — твоя страховка от паники: на любой вопрос «спроектируй X» идёшь по шагам
1→6. Структурированный подход важнее, чем знать готовый ответ: интервьюер оценивает **ход мысли**, а
не зазубренную схему. И в реальной работе тот же порядок: понять → оценить → набросать → углубить →
обсудить компромиссы.

---

## ⭐ Думай вслух и итеративно

```
   • НАЧНИ ПРОСТО — сначала рабочая схема для умеренного масштаба, потом масштабируй.
     (не вываливай сразу «шарды + кэш + очереди» — добавляй по мере роста нагрузки.)
   • ПРОГОВАРИВАЙ — озвучивай решения и АЛЬТЕРНАТИВЫ: «можно так или так, выберу X, потому что...».
   • РЕАГИРУЙ на узкие места — «здесь будет 4000 чтений/с → добавлю кэш».
   • НЕ молчи и не пиши идеал в голове — дизайн это диалог.
```

💡 ⭐ Лучшая стратегия — **итеративная**: простая работающая схема → находишь узкое место → добавляешь
блок (кэш/реплику/очередь) → объясняешь, что это дало и чем заплатил. Так ты показываешь инженерное
мышление, а не фокус «сразу идеальная архитектура». Начинать с переусложнённого — частая ошибка.

---

## 📖 Чего ждёт интервьюер (и реальность)

```
   оценивают НЕ «знал ли ты ответ», а:
   • структуру мышления (идёшь ли по шагам).
   • уточняешь ли требования (не строишь вслепую).
   • знаешь ли строительные блоки и КОГДА их применять.
   • называешь ли компромиссы (видишь ли обе стороны).
   • как реагируешь на «а если ×10 нагрузки?» (умеешь ли масштабировать решение).

   в реальной работе те же навыки = спроектировать сервис, который не развалится.
```

> 🧭 Это перекликается с [Senior-коммуникацией](../../Senior/04-leadership/19-communication.md):
> ясно объяснить решение и его цену важнее, чем знать «секретный» лучший ответ.

---

## ⚠️ Ловушки

- ❌ Прыгать к схеме, минуя требования и оценку.
- ❌ Сразу переусложнять (шарды/микросервисы для тысячи пользователей).
- ❌ Молчать и проектировать «в голове» (дизайн — диалог, думай вслух).
- ❌ Давать «зазубренный» ответ без адаптации под конкретную задачу.
- ❌ Игнорировать вопрос «а если нагрузка ×10?» (не уметь масштабировать своё решение).
