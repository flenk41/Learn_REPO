# 10 · Репликация и шардинг 🖼️⭐

> 🎯 **Цель блока:** масштабировать слой данных — копиями (репликация) и разбиением (шардинг). Серверы
> приложения масштабируются легко; настоящий вызов масштаба — это данные.

> 🧭 Подробно — в [🗄️ БД: репликация/шардинг](../../Database/04-scale/19-replication-sharding.md).
> Здесь — как архитектор применяет это в системе.

---

## ⭐ Репликация: масштаб чтения + надёжность

```
   РЕПЛИКАЦИЯ — копии данных на нескольких серверах.
   PRIMARY принимает ЗАПИСИ → реплицирует на REPLICA (копии), которые обслуживают ЧТЕНИЯ.

   даёт:
   ✅ масштаб ЧТЕНИЯ — SELECT-ы распределяешь по репликам (большинство систем read-heavy).
   ✅ отказоустойчивость — primary упал → реплика становится primary (failover).
   ✅ гео-близость — реплики ближе к пользователям (меньше задержка).

   ⚠️ replication lag — реплики отстают (асинхронно) → read-your-writes проблема (модуль 06).
```

💡 ⭐ Репликация — первый шаг масштаба данных после кэша: один primary пишет, много реплик читают.
Решает чтение и даёт HA. Но это **не решает масштаб записи** (все записи всё равно на одном primary) —
для этого нужен шардинг.

---

## ⭐⭐ Шардинг: масштаб записи и объёма

```
   ШАРДИНГ — разбить ДАННЫЕ по нескольким серверам (шардам), каждый хранит ЧАСТЬ.
   нужен, когда данные/запись НЕ влезают в одну машину даже с репликами.

   shard key (по чему делим):
   • по ХЕШУ ключа — hash(user_id) % N → равномерно.
   • по ДИАПАЗОНУ — A-M / N-Z.   • по РЕГИОНУ/зоне.

   ⚠️ цена шардинга (дорого!):
   • кросс-шардовые запросы/JOIN-ы — тяжелы (надо опросить много шардов).
   • горячие шарды (плохой ключ → перекос).   • решардинг (добавить шард) болезнен.
   • транзакции между шардами — сложно.
```

🖼️
```
   репликация (масштаб ЧТЕНИЯ):           шардинг (масштаб ЗАПИСИ/ОБЪЁМА):
   [PRIMARY] ──► [реплика][реплика]        [шард 1: 0-33%][шард 2: 34-66%][шард 3: 67-100%]
   запись→primary, чтения→реплики          каждый шард — своя часть данных (+ свои реплики)
```

💡 ⭐⭐ Запомни разницу: **репликация = копии (масштаб ЧТЕНИЯ + надёжность)**, **шардинг = разбиение
(масштаб ЗАПИСИ + объём)**. Шардинг мощный, но дорого усложняет всё — поэтому он **в конце лестницы**:
оптимизация → кэш → реплики → и только потом шардинг. Многие NoSQL шардируют автоматически (отсюда их
«масштабируемость»).

---

## 📖 Лестница масштаба данных (повтор — это важно)

```
   1. ОПТИМИЗАЦИЯ — индексы, запросы, убрать N+1. часто хватает!
   2. КЭШ — снять нагрузку чтения (модуль 05).
   3. РЕПЛИКИ — масштаб чтения, HA.
   4. ШАРДИНГ — когда упёрся в запись/объём. крайнее средство.
   прыгать через ступени (сразу шардинг) — частая дорогая ошибка.
```

> 🧭 Шардинг тесно связан с консистентностью (данные распределены) — следующий модуль 11 (CAP).

---

## ⚠️ Ловушки

- ❌ Считать репликацию решением масштаба ЗАПИСИ (она для чтения; запись на одном primary).
- ❌ Прыгать к шардингу до кэша/реплик/оптимизации.
- ❌ Плохой shard key → горячие шарды (перекос) и частые кросс-шардовые запросы.
- ❌ Игнорировать replication lag (чтение с реплики сразу после записи).
- ❌ Недооценивать боль решардинга и кросс-шардовых транзакций.
