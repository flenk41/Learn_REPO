# 09 · Stateless и сессии ⭐⭐ 🖼️

> 🎯 **Цель блока:** понять, почему серверы делают «без состояния» (stateless) и куда тогда девать
> сессии/данные пользователя. Это условие, без которого горизонтальный масштаб не работает.

---

## ⭐⭐ Stateless: сервер не помнит

```
   STATELESS-СЕРВЕР — не хранит у СЕБЯ состояние между запросами. любой запрос любого пользователя
   может обработать ЛЮБОЙ сервер, потому что всё нужное либо в запросе, либо в ОБЩЕМ хранилище.

   STATEFUL (плохо для масштаба) — сервер помнит у себя (сессия в памяти, корзина в локальной переменной):
   • запрос пользователя ДОЛЖЕН попасть на ТОТ ЖЕ сервер (sticky) — хрупко.
   • сервер упал → состояние потеряно (разлогинило, корзина пропала).
   • не добавить/не убрать сервер свободно.
```

🖼️
```
   STATEFUL (состояние на сервере) ❌          STATELESS (состояние в общем сторе) ✅
   ┌─────────┐ помнит сессию A                 ┌─────────┐   ┌─────────┐
   │ Сервер1 │ ← пользователь A привязан       │ Сервер1 │   │ Сервер2 │  любой обслужит любого
   └─────────┘   (упал → потеряли A)           └────┬────┘   └────┬────┘
   ┌─────────┐ помнит сессию B                      └─────┬───────┘
   │ Сервер2 │                                            ▼
   └─────────┘                                   [ Redis / БД: сессии, данные ]  ← общее состояние
```

💡 ⭐⭐ Правило горизонтального масштаба: **серверы приложения — stateless**, всё состояние (сессии,
данные, корзины) — в **общих хранилищах** (Redis/БД). Тогда балансировщик шлёт запрос на любой сервер,
серверы можно свободно добавлять/убирать/терять. Это тот «секрет», что делает scale-out простым
(модуль 08).

## ⭐⭐ Куда девать сессии

```
   пользователь залогинился — где хранить «кто он»? три подхода:

   1. ОБЩЕЕ ХРАНИЛИЩЕ СЕССИЙ (session store в Redis/БД):
      сервер кладёт сессию в Redis по session_id (из cookie). любой сервер прочитает. ⭐ частый выбор.

   2. ТОКЕНЫ (JWT) — состояние в самом токене у клиента:
      сервер выдаёт подписанный токен с данными (user_id, роль). клиент шлёт его в каждом запросе.
      сервер ПРОВЕРЯЕТ подпись — не нужно хранилище. масштабируется отлично.
      ⚠️ минус: токен трудно отозвать до истечения (нужны короткий срок + refresh).

   3. STICKY-СЕССИИ (привязка к серверу) — антипаттерн, избегай (ломает масштаб/отказоустойчивость).
```

💡 ⭐⭐ Два рабочих способа: **общий session store (Redis)** — просто, легко отозвать; или **JWT-токены**
— без хранилища, но сложнее отзыв. Оба держат серверы stateless. **Sticky-сессии** (запомнить
пользователя на конкретном сервере) — то, чего избегают: упал сервер → разлогинило, масштаб ломается.

## 📖 Где состояние жить ОБЯЗАНО

```
   полностью без состояния система не бывает — оно просто ВЫНЕСЕНО из серверов приложения в
   специализированные хранилища:
   • данные → БД.   • сессии/кэш → Redis.   • файлы/медиа → объектное хранилище (S3).
   • очереди задач → брокер (модуль 07).
   серверы приложения — «вычислительный слой» без памяти; состояние — в слое данных.
   → вычисления масштабируешь горизонтально легко, а состояние — отдельными способами (реплики/шарды).
```

> 🧭 Это та же идея, что в [Roblox: сервер — источник правды, а не клиент](../../Roblox/02-scripting/11-services-script-types.md)
> и [stateless из ОС/процессов](../../OS/01-processes/03-process.md): отделяй вычисление от состояния.

## ⚠️ Ловушки

- ❌ Хранить сессию/корзину в памяти сервера → sticky-сессии, потеря при падении, нет масштаба.
- ❌ Sticky-сессии как «решение» (хрупко: упал сервер — потеряли пользователей).
- ❌ JWT без срока/отзыва → украденный токен живёт долго (нужны короткий TTL + refresh).
- ❌ Локальный кэш на сервере как источник правды (у каждого свой → рассинхрон; общий Redis).
- ❌ Думать, что «stateless» = «нет состояния вообще» (оно вынесено в слой данных).
