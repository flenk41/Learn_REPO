# 17 · Наблюдаемость: логи, метрики, трейсинг 🖼️⭐

> 🎯 **Цель блока:** сделать систему «видимой» — чтобы понимать, что в ней происходит, и быстро
> находить проблемы. Нельзя чинить и масштабировать то, что не измеряешь.

---

## ⭐ Три столпа наблюдаемости

```
   ЛОГИ (logs) — записи событий: «что произошло» (ошибка, запрос, действие). текст с контекстом.
   • структурированные (JSON) с уровнями (debug/info/warn/error) и trace_id. централизованный сбор (ELK/Loki).

   МЕТРИКИ (metrics) — числа во времени: «сколько/как быстро» (rps, задержка, CPU, ошибки, длина очереди).
   • агрегируемы, дёшевы, для дашбордов и алертов (Prometheus + Grafana).

   ТРАССИРОВКА (tracing) — путь ОДНОГО запроса через все сервисы: «где ушло время».
   • распределённый трейсинг (OpenTelemetry/Jaeger) — видно, какой сервис/шаг тормозит.
```

🖼️
```
   ЛОГИ      → что случилось (детали события, ошибки).        «error: timeout to PaymentService»
   МЕТРИКИ   → сколько/как быстро (тренды, алерты).           p99 latency = 800ms, errors = 2%
   ТРАССЫ    → где в цепочке сервисов ушло время.             запрос: API 5ms → Orders 12ms → DB 740ms ⚠️
```

💡 ⭐ Три столпа дополняют друг друга: **метрики** говорят «что-то не так» (алерт по росту ошибок/
задержки), **трассы** — «ГДЕ именно» (какой сервис/шаг), **логи** — «ПОЧЕМУ» (детали ошибки). Без них
ты чинишь вслепую. Это инфраструктура для всего трека: масштаб (модуль 12) и надёжность (16)
невозможны без измерения.

---

## ⭐⭐ Что мерить: «золотые сигналы»

```
   четыре ключевые метрики для любого сервиса (Google SRE «golden signals»):
   • LATENCY (задержка) — как быстро отвечает. смотри ПЕРЦЕНТИЛИ (p50/p95/p99), не среднее!
   • TRAFFIC (нагрузка) — сколько запросов (rps).
   • ERRORS (ошибки) — доля неуспешных ответов.
   • SATURATION (насыщение) — насколько забиты ресурсы (CPU/память/диск/соединения/очередь).

   ⚠️ СРЕДНЕЕ ВРЁТ: средняя задержка 100мс может прятать, что 1% пользователей ждёт 5 секунд.
   смотри p95/p99 — «как живёт худшим пользователям».
```

💡 ⭐⭐ Мерь **перцентили, а не среднее**: p99 latency показывает боль реальных пользователей, которую
среднее скрывает. Четыре golden signals (задержка/нагрузка/ошибки/насыщение) — минимум для каждого
сервиса. По ним строишь дашборды и алерты.

---

## 📖 Алерты и культура

```
   • АЛЕРТЫ — автоматическое оповещение при проблеме (ошибки > X%, p99 > Y, диск > 90%).
     алертуй на СИМПТОМЫ для пользователя (медленно/ошибки), а не на каждую мелочь (шум → игнор).
   • SLI/SLO — целевые показатели (напр. «99.9% запросов < 300мс»); по ним понимаешь, здорова ли система.
   • correlation id / trace_id — сквозной идентификатор запроса во всех логах/сервисах (связать картину).
   • дашборды — общая картина здоровья системы с одного взгляда.
```

> 🧭 Это инженерная основа [методичной отладки](../../Senior/03-practices/13-debugging-method.md) и
> [«измеряй, потом оптимизируй»](../02-scaling/12-bottlenecks.md): наблюдаемость даёт данные для решений.

## ⚠️ Ловушки

- ❌ Смотреть среднее вместо перцентилей (p99 прячется за средним).
- ❌ Логи без структуры/trace_id (невозможно связать и искать).
- ❌ Алерты на всё подряд → шум → их игнорируют (алертуй на симптомы для пользователя).
- ❌ Нет распределённого трейсинга в микросервисах (не найти, где тормозит).
- ❌ Логировать секреты/персональные данные (утечка через логи).
- ❌ Чинить/масштабировать по интуиции без метрик.
