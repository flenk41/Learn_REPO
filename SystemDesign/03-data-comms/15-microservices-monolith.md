# 15 · Микросервисы vs монолит 🖼️⭐⭐

> 🎯 **Цель блока:** разобрать главный архитектурный спор — один большой сервис (монолит) против
> множества маленьких (микросервисы) — и научиться выбирать осознанно, а не по моде.

---

## ⭐⭐ Монолит vs микросервисы

```
   МОНОЛИТ — всё приложение как ОДНА развёртываемая единица (один кодбейс, один деплой, одна БД).
   • плюсы: ПРОСТО (один проект, локальные вызовы, одна транзакция, легко отлаживать/деплоить).
   • минусы: на масштабе — большой кодбейс, любая правка → деплой всего, команды мешают друг другу,
     нельзя масштабировать части отдельно.

   МИКРОСЕРВИСЫ — система из МНОГИХ маленьких независимых сервисов (каждый — своя зона, свой деплой, часто своя БД).
   • плюсы: независимый деплой/масштаб частей, команды автономны, разные технологии, изоляция отказов.
   • минусы: ОГРОМНАЯ операционная сложность (сеть, распределённость, мониторинг, согласованность,
     распределённые транзакции, версионирование контрактов).
```

🖼️
```
   МОНОЛИТ:                          МИКРОСЕРВИСЫ:
   ┌─────────────────────┐          [Users] [Orders] [Payments] [Notifs]
   │ users orders payments│          каждый: свой деплой, своя БД, общается по сети (API/события)
   │  всё в одном, одна БД │          ▲ гибко и независимо, но сеть/распределённость/сложность
   └─────────────────────┘
```

💡 ⭐⭐ Это **не «микросервисы лучше»**, а компромисс: монолит — простота, микросервисы — независимость
ценой огромной операционной сложности. Микросервисы решают проблему **МАСШТАБА КОМАНД и частей**, а не
«делают код лучше». Маленькой команде они чаще вредят (распределённый монолит — худшее из обоих миров).

---

## ⭐⭐ Начинай с монолита

```
   почти всегда правильный старт — МОНОЛИТ (часто «модульный монолит»: чёткие модули внутри одного деплоя):
   • быстрее разрабатывать, проще деплой/отладка, одна транзакция, нет сетевой боли.
   • когда вырастешь и заболит (большая команда, части нужно масштабировать/деплоить отдельно) —
     ВЫДЕЛЯЙ микросервисы по границам, которые УЖЕ проявились.

   «monolith first» — рекомендация даже от сторонников микросервисов. преждевременные микросервисы —
   классическая дорогая ошибка (распределённость без нужды).
```

💡 ⭐⭐ Правило: **начни с (модульного) монолита, выделяй микросервисы по мере боли**. Ты не знаешь
правильных границ сервисов в начале — узнаёшь их, когда система и команда выросли. Дробить рано =
получить распределённый монолит (сервисы связаны, но ещё и по сети — мучительно).

---

## ⭐ Как резать на сервисы (если уже надо)

```
   границы — по БИЗНЕС-ВОЗМОЖНОСТЯМ (bounded context), а не по техническим слоям:
   ✅ сервис «Платежи», «Заказы», «Каталог» (каждый владеет своими данными).
   ❌ сервис «контроллеры», «база» (технические слои — порождают болтовню по сети).

   принципы:
   • каждый сервис ВЛАДЕЕТ своими данными (своя БД) — не лезут в чужую напрямую, только через API/события.
   • слабая связанность, общение по API (gRPC внутри) или событиями (модуль 14).
   • избегай распределённых транзакций — используй saga/события и eventual consistency.
```

> 🧭 Это прямое применение [границ и зацепления из ООП/проектирования](../../OOP/03-design/16-composition-over-inheritance.md):
> высокая связность внутри сервиса, слабая связанность между. Плохие границы → «распределённый монолит».

---

## ⚠️ Ловушки

- ❌ Начинать с микросервисов «потому что модно/масштабируемо» (преждевременная распределённость).
- ❌ Резать по техническим слоям → болтовня по сети, связанные сервисы.
- ❌ Общая БД у нескольких сервисов (скрытая связанность — это распределённый монолит).
- ❌ Недооценивать операционную цену (мониторинг, трейсинг, деплой, сеть, согласованность).
- ❌ Распределённые транзакции между сервисами (очень сложно; используй saga/события).
- ❌ Думать, что микросервисы «улучшают код» (они решают масштаб команд/частей, не качество).
