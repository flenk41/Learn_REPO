# 16 · Надёжность и отказоустойчивость 🖼️⭐⭐

> 🎯 **Цель блока:** проектировать систему, которая продолжает работать, когда части отказывают. В
> большой системе что-то ломается ВСЕГДА — вопрос в том, переживёт ли это пользователь.

---

## ⭐⭐ Главный принцип: отказы неизбежны

```
   в распределённой системе сбои — НОРМА, а не исключение: серверы падают, сеть рвётся, диски мрут,
   зависимости тормозят. чем больше компонентов — тем чаще что-то не так.

   → проектируй НЕ «чтобы не падало», а «чтобы переживало падения»:
   • убери ЕДИНЫЕ ТОЧКИ ОТКАЗА (SPOF) — дублируй каждый критичный компонент.
   • изолируй сбои — отказ одной части не должен валить всю систему.
   • деградируй мягко — лучше урезанный сервис, чем полное падение.
```

💡 ⭐⭐ Сдвиг мышления: не «как сделать, чтобы не ломалось» (невозможно), а **«как продолжить работать,
когда ломается»**. Это отличает надёжную систему. Каждый критичный компонент спрашивай: «а если он
упадёт — что будет?». Если ответ «всё умрёт» — это SPOF, его надо дублировать.

---

## ⭐ Избыточность и устранение SPOF

```
   ЕДИНАЯ ТОЧКА ОТКАЗА (SPOF) — компонент, чьё падение валит всю систему. находи и устраняй:
   • один сервер → пул за балансировщиком.
   • один балансировщик → два (active-active/passive) + DNS/anycast.
   • одна БД → primary + реплики + failover.
   • один дата-центр → несколько зон/регионов (для высокой доступности).

   REDUNDANCY (избыточность) — иметь запас: упал один — подхватил другой.
   FAILOVER — автоматическое переключение на резерв при сбое.
```

💡 ⭐ Рецепт высокой доступности — **дублирование на каждом уровне** + автоматический failover. «Сколько
девяток» (99.9% = ~9ч простоя/год, 99.99% = ~52мин) определяется тем, есть ли запас на каждом
критичном звене. Один экземпляр чего-либо критичного = потолок надёжности.

---

## ⭐⭐ Паттерны устойчивости

```
   когда зависимость (другой сервис/БД) тормозит или падает — не дай ей утянуть тебя:

   • TIMEOUT — не жди вечно ответа; оборви по таймауту (иначе зависнешь, исчерпаешь потоки).
   • RETRY с backoff — повтори при временном сбое, но с НАРАСТАЮЩЕЙ паузой + джиттер (не добивай).
     ⚠️ retry безопасен только для ИДЕМПОТЕНТНЫХ операций (иначе дубли).
   • CIRCUIT BREAKER («предохранитель») — если зависимость стабильно падает, ПЕРЕСТАНЬ её звать на время
     (быстро отдавай ошибку/заглушку), дай ей восстановиться, не засыпай запросами.
   • BULKHEAD («переборки») — изолируй ресурсы (пулы потоков/соединений) по зависимостям, чтобы одна
     медленная не съела все ресурсы и не уронила остальное.
   • GRACEFUL DEGRADATION — отkey функцию урезай, а не падай (нет рекомендаций → покажи популярное).
```

🖼️
```
   circuit breaker:  CLOSED (зовём) ──много ошибок──► OPEN (не зовём, сразу ошибка)
                          ▲                                    │ через время
                          └────── HALF-OPEN (пробный запрос) ◄─┘
   защищает и тебя (не висишь), и упавшую зависимость (не добиваешь).
```

💡 ⭐⭐ Связка **timeout + retry(backoff) + circuit breaker** — базовая защита межсервисного вызова.
Без таймаута один зависший вызов исчерпает потоки и уронит твой сервис (каскадный отказ). Без circuit
breaker ты добиваешь падающую зависимость ретраями. Эти паттерны — то, что отличает «учебную» систему
от продакшен-надёжной.

---

## 📖 Резервные копии и восстановление

```
   • БЭКАПЫ данных — регулярные, ПРОВЕРЯЕМЫЕ (умеешь восстановиться!), хранить отдельно.
   • RPO (сколько данных потерять допустимо) и RTO (за сколько восстановиться) — цели восстановления.
   • health-чеки + мониторинг (модуль 17) — заметить сбой быстро.
   • chaos-инженерия — НАРОЧНО ломать в проде (Netflix Chaos Monkey), чтобы проверить устойчивость.
```

> 🧭 Бэкап без проверки восстановления — иллюзия (как в [безопасности БД](../../Database/04-scale/23-security.md)).

---

## ⚠️ Ловушки

- ❌ Единые точки отказа (один LB/БД/сервер) — потолок надёжности.
- ❌ Межсервисные вызовы без таймаутов → зависший вызов исчерпывает потоки → каскадный отказ.
- ❌ Retry без backoff/джиттера → добиваешь падающую зависимость (retry storm).
- ❌ Retry неидемпотентной операции → дубли (двойное списание).
- ❌ Падать целиком вместо мягкой деградации.
- ❌ Бэкапы без проверки восстановления / без понимания RPO/RTO.
