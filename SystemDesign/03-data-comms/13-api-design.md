## ⭐ API — это контракт

```
   API определяет, ЧТО система предоставляет наружу: операции, форматы запросов/ответов, ошибки.
   это КОНТРАКТ между клиентом и сервером (и между сервисами).

   хороший API:
   • ПОНЯТНЫЙ — имена операций/полей говорят сами за себя.
   • ПОСЛЕДОВАТЕЛЬНЫЙ — единый стиль (множественное число, формат дат, коды ошибок).
   • СТАБИЛЬНЫЙ — не ломает клиентов при изменениях (версионирование).
   • МИНИМАЛЬНЫЙ — отдаёт нужное, скрывает внутреннее.
```

💡 ⭐ API переживает реализацию: внутри можно переписать всё, но контракт менять больно (клиенты
зависят). Поэтому проектируй интерфейс осознанно и думай о совместимости с первого дня. Это
[«проектируй под изменения»](../../Senior/03-practices/15-design-for-change.md) на уровне системы.

## ⭐⭐ Три стиля: REST, gRPC, GraphQL

```
   REST (HTTP + JSON) — ресурсы и методы. стандарт для публичных/веб-API.
   • GET /users/42 , POST /orders , DELETE /carts/7 . коды 200/404/500.
   • плюсы: простой, кэшируемый, везде поддержан. минусы: over/under-fetching (лишние/недостающие данные).

   gRPC (HTTP/2 + protobuf) — бинарный RPC «вызов функции по сети». для СЕРВИС↔СЕРВИС внутри.
   • плюсы: быстрый, типизированный контракт (.proto), стриминг. минусы: не для браузера напрямую, бинарь.

   GraphQL — клиент САМ запрашивает нужные поля одним запросом.
   • плюсы: нет over/under-fetching, гибко для сложных UI. минусы: сложнее кэш/безопасность, N+1 на сервере.
```

🖼️
```
   REST     → публичные API, веб, простота, кэш.            GET /users/42 → {json}
   gRPC     → межсервисное общение внутри, скорость, типы.  UserService.Get(42) → User
   GraphQL  → гибкие клиенты (мобайл/сложный UI), один запрос за всё нужное.
```

💡 ⭐⭐ Правило выбора: **REST** — по умолчанию для публичных и веб-API (просто, кэшируемо, везде).
**gRPC** — для общения сервисов внутри (быстро, типизированно). **GraphQL** — когда клиентам нужна
гибкая выборка и мучает over/under-fetching. Часто комбинируют: REST/GraphQL наружу, gRPC внутри.

## ⭐ Версионирование и совместимость

```
   API меняется, но клиенты не должны ломаться:
   • ВЕРСИЯ в пути или заголовке: /v1/users , /v2/users . старые клиенты — на v1.
   • ОБРАТНО СОВМЕСТИМЫЕ изменения (безопасны): добавить НОВОЕ поле/эндпоинт.
   • ЛОМАЮЩИЕ (опасны): удалить/переименовать поле, изменить тип, сделать поле обязательным → новая версия.
   • депрекация: помечай старое, давай время мигрировать, потом убирай.
```

💡 ⭐ Золотое правило API: **добавлять можно, ломать — нельзя** (без новой версии). Новое поле —
безопасно; убрать/переименовать — сломаешь клиентов. Это та же дисциплина, что в
[миграциях БД](../../Database/04-scale/22-migrations.md) и [semver](../../Git/04-pro/20-tags-releases.md).

## 📖 Хорошие практики REST

```
   • ресурсы — существительные во множественном: /users , /orders (не /getUser).
   • методы HTTP по смыслу: GET (читать), POST (создать), PUT/PATCH (изменить), DELETE (удалить).
   • коды статуса: 200 ок, 201 создано, 400 плохой запрос, 401/403 авторизация, 404 нет, 429 too many, 5xx сервер.
   • пагинация для списков (?page/?cursor) — не отдавай миллион записей.
   • идемпотентность: GET/PUT/DELETE идемпотентны; POST — нет (защищай от двойных созданий).
   • понятные ошибки: код + машиночитаемое тело { "error": "...", "code": "..." }.
```

## ⚠️ Ловушки

- ❌ Ломающие изменения без версии (рушишь клиентов).
- ❌ Глаголы в путях REST (/getUser) вместо ресурсов (/users/42).
- ❌ Отдавать списки без пагинации (огромные ответы, нагрузка).
- ❌ Невнятные ошибки (всё 200 с «ok:false» / голый 500 без деталей).
- ❌ GraphQL без защиты от тяжёлых запросов (глубина/сложность) и N+1.
- ❌ gRPC наружу в браузер напрямую (нужен прокси/gateway).
