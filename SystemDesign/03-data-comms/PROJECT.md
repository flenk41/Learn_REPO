# 🏗️ Проект уровня 3 — Спроектируй устойчивый сервис целиком

> 🎯 **Цель:** довести систему до «production-готовности по архитектуре»: продуманный API, осознанная
> синхронность/асинхронность, границы компонентов, устойчивость к сбоям и наблюдаемость.

---

## Что делаем

Возьми систему из уровней 1-2 и спроектируй её «нервную систему» — интерфейсы, связь, надёжность,
видимость. Это превращает «коробочки и стрелочки» в систему, которая не падает и понятна в эксплуатации.

---

## Этапы

- [ ] **API.** Спроектируй основной API (REST/gRPC/GraphQL — обоснуй). Эндпоинты, форматы, коды ошибок,
      пагинация, версионирование. Что наружу, что между сервисами.
- [ ] **Sync/async.** Раздели операции: что синхронно (нужно сразу), что асинхронно (очередь/события).
      Опиши хотя бы один гибрид (быстрый ответ + фоновая работа + уведомление).
- [ ] **Границы.** Реши: монолит или микросервисы? Обоснуй. Если сервисы — нарежь по бизнес-границам,
      укажи, кто чем владеет и как общается (API/события).
- [ ] **Надёжность.** Найди и устрани SPOF. На критичных межсервисных вызовах — timeout + retry(backoff)
      + circuit breaker. Опиши graceful degradation для одной фичи. Укажи бэкапы и RPO/RTO.
- [ ] **Наблюдаемость.** Какие метрики (golden signals), логи (с trace_id), трассы. 3-4 алерта на
      симптомы для пользователя. 2 SLO.

---

## 📦 Результат

1. **Схема** системы с компонентами, API и потоками (sync/async обозначены).
2. **Архитектурный документ:**
   - дизайн API (стиль + ключевые эндпоинты + версионирование);
   - что синхронно/асинхронно и почему;
   - границы (монолит/сервисы) с обоснованием;
   - устойчивость (устранённые SPOF + паттерны на критичных вызовах + деградация);
   - наблюдаемость (метрики/логи/трассы + алерты + SLO).
3. 3-4 явно названных компромисса.

---

## 🌟 Усложнения

- Нарисуй распределённый трейс одного запроса через сервисы (где время уходит).
- Опиши сценарий отказа (упал сервис платежей) и как система переживает.
- Спроектируй saga для операции, охватывающей несколько сервисов (без распределённой транзакции).

---

## ✅ Критерии

- [ ] API спроектирован как стабильный контракт (версионирование, ошибки, пагинация)
- [ ] Осознанно разделены sync/async, обоснованы границы (монолит/сервисы)
- [ ] Устойчивость: нет SPOF, есть паттерны на критичных вызовах, деградация
- [ ] Наблюдаемость: метрики/логи/трассы, алерты на симптомы, SLO
