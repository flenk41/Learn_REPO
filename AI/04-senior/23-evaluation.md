# 23 · Оценка и тестирование промптов 🛠️

> 🎯 **Цель блока:** научиться **измерять** качество промптов, а не оценивать «на глаз».
> Это отличает профессионала: он улучшает промпты по данным.

---

## 📖 Зачем измерять

Новичок меняет промпт и думает «вроде стало лучше». Профессионал **проверяет** на наборе
случаев и **знает**, стало ли лучше. Особенно важно, когда промпт используется регулярно
или в продукте.

🖼️
```
   Новичок:  промпт A → "вроде ок" → промпт B → "вроде лучше?"  (субъективно)
   Эксперт:  промпт A vs B на 10 тестах → A: 6/10, B: 9/10 → B лучше (объективно)
```

---

## ⭐ Тестовый набор (test set)

Собери набор **разных** входных данных, включая сложные и краевые случаи:

```
Для промпта-классификатора отзывов собери тесты:
- явно позитивный
- явно негативный
- смешанный ("хорошо, но дорого")
- нейтральный
- с сарказмом ("ну просто супер, сломалось за день")  ← краевой случай
- очень короткий
- очень длинный
```

💡 Хороший тестовый набор покрывает реальное разнообразие. Промпт, который работает на
простых случаях, может проваливаться на сложных — тесты это выявляют.

---

## ⭐ Критерии оценки

Определи **заранее**, что значит «хороший ответ»:

```
- Точность     — правильный ли результат?
- Полнота      — всё ли учтено?
- Формат       — соблюдена ли структура?
- Тон          — подходящий ли стиль?
- Отсутствие галлюцинаций — нет ли выдумок?
- Стабильность — одинаково ли хорош на разных входах?
```

💡 Без явных критериев оценка субъективна. С критериями — ты можешь сравнивать промпты
честно.

---

## ⭐ A/B-сравнение промптов

```
1. Возьми промпт A и промпт B (например с few-shot и без).
2. Прогони оба на одном тестовом наборе.
3. Оцени каждый ответ по критериям.
4. Сравни: какой промпт надёжнее?
```

> 💡 Меняй **по одному** за раз: если изменил сразу роль, формат и примеры — не поймёшь, что
> именно помогло. Это как отладка в программировании: изолируй переменную.

---

## ⭐ ИИ как оценщик (LLM-as-judge)

Можно поручить **оценку ответов другому ИИ** (или тому же в отдельном запросе):

```
"Оцени этот ответ по шкале 1–10 по критериям: точность, полнота, ясность.
По каждому критерию — балл и обоснование.

ЗАДАЧА: [исходная задача]
ОТВЕТ: [ответ для оценки]"
```

💡 Полезно для массовой оценки: прогнал 50 ответов через ИИ-судью, получил баллы. Но
ИИ-судья тоже не идеален — для критичного перепроверяй сам.

---

## 📖 Признаки, что промпт надо улучшать

```
⚠️ нестабильность — на похожих входах разное качество
⚠️ провалы на краевых случаях
⚠️ галлюцинации
⚠️ игнорирование части инструкций
⚠️ неверный формат время от времени
```

Лечение — добавить недостающий паттерн (few-shot для стабильности, заземление против
галлюцинаций, чёткий формат, явные правила).

---

## 📖 Итеративное улучшение по данным

```
1. Базовый промпт → прогон на тестах → замер.
2. Найди частую ошибку.
3. Измени ОДНО (добавь пример / правило / формат).
4. Прогон → замер → стало лучше? оставь : откати.
5. Повтори, пока качество не устроит.
```

💡 Это цикл инженера: гипотеза → изменение → измерение → вывод. Так промпты доводят до
продакшен-качества.

---

## ✅ Задачи

1. **Тестовый набор** — собери 7–10 разных входов для своего промпта, включая краевые.
2. **Критерии** — определи 3–4 критерия «хорошего ответа» для своей задачи.
3. **A/B-тест** — сравни два варианта промпта на наборе, выбери лучший по критериям.
4. **Изменение по одному** — улучшай промпт, меняя по одному элементу, фиксируй эффект.
5. **ИИ-судья** — попроси ИИ оценить набор ответов по твоим критериям.
6. ⭐ **Доведи промпт** — возьми важный для себя промпт и доведи его до стабильного качества
   по тестам.

---

## ❓ Проверь себя

1. Почему оценку «на глаз» стоит заменить измерением?
2. Что такое тестовый набор и зачем краевые случаи?
3. Зачем заранее определять критерии оценки?
4. Почему менять промпт нужно по одному изменению?
5. Что такое LLM-as-judge? В чём его ограничение?
6. Опиши цикл улучшения промпта по данным.
