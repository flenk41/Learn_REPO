# 11 · Защита памяти и адресное пространство 🖼️⭐

> 🎯 **Цель блока:** понять, как ОС защищает память — через права страниц и изоляцию адресных
> пространств — и как устроена раскладка памяти процесса в деталях.

---

## ⭐ Защита через права страниц

Из модуля 09: у каждой страницы есть флаги прав (R/W/X, user/kernel). ОС использует их как
**защиту**:

```
   код программы      → r-x  (читать, исполнять; ПИСАТЬ нельзя)
   данные/куча/стек   → rw-  (читать, писать; ИСПОЛНЯТЬ нельзя)
   чужая память       → недоступна вообще (нет в твоей таблице)
```

💡 Нарушил права — ОС вмешивается:

```
   запись в код (r-x)        → fault → процесс падает
   исполнение данных (rw-)   → fault (защита от атак, см. ниже)
   доступ к чужой странице   → segfault (модуль 08)
```

---

## ⭐ Изоляция адресных пространств

Главная защита: у каждого процесса **своя таблица страниц** (модуль 09), значит — своя карта
памяти. Чужие страницы просто **не существуют** в его карте.

🖼️
```
   процесс A: таблица A → видит только свои фреймы
   процесс B: таблица B → видит только свои фреймы
        (никакого пересечения, если ОС явно не сделала общую страницу)
```

💡 Поэтому процесс **не может** прочитать или испортить память другого, даже случайно, даже
вредоносно. Это аппаратно-программная гарантия (MMU + таблицы + флаги). Фундамент стабильности:
краш одной программы не трогает остальные.

---

## ⭐ Защита от атак: W^X, ASLR

ОС использует защиту памяти и против взлома:

```
   W^X (Write XOR eXecute) — страница либо записываемая, либо исполняемая, но не оба
        → нельзя записать вредоносный код и тут же его исполнить

   ASLR — случайное размещение областей памяти при каждом запуске
        → атакующий не знает заранее адреса стека/библиотек
```

💡 Эти механизмы (флаг «не исполнять» для данных, рандомизация адресов) сильно усложняют
эксплойты вроде переполнения буфера. Связь с [безопасностью в треке Сети](../../Network/04-advanced/20-security.md)
и [UB в C](../../C/04-senior/20-alignment-ub.md): многие атаки бьют по памяти, и ОС обороняется.

## 📖 Раскладка адресного пространства (детально)

Теперь подробная карта процесса (расширение модуля 03):

```
   высокие адреса
   ┌──────────────────────┐
   │ ядро (для процесса    │  недоступно из user space
   │   недоступно)         │
   ├──────────────────────┤
   │ Стек ↓                │  rw-, растёт вниз; локальные переменные, кадры вызовов
   │   ...                 │
   │ (mmap: библиотеки,    │  разделяемые библиотеки, файлы в память
   │  отображённые файлы)  │
   │   ...                 │
   │ Куча ↑                │  rw-, растёт вверх; malloc/new (через brk/mmap → syscalls!)
   ├──────────────────────┤
   │ BSS / Data            │  rw-, глобальные переменные
   │ Text (код)            │  r-x, инструкции (только чтение + исполнение)
   └──────────────────────┘
   низкие адреса (нулевая страница недоступна → разыменование NULL = segfault)
```

💡 Заметь: нулевая страница (адрес 0) **намеренно** недоступна — поэтому разыменование `NULL`
сразу даёт segfault, а не тихо портит данные. Куча растёт через системные вызовы `brk`/`mmap`
(вот что делает `malloc` под капотом — связь с модулем 01).

---

## ⚠️ Ловушки

- ❌ Думать, что защита памяти — «дополнительная опция». Это базовый механизм ОС.
- ❌ Считать, что процесс может «случайно» прочитать чужую память. Изоляция это исключает.
- ❌ Игнорировать W^X/ASLR — они реально мешают эксплойтам, не «бюрократия».
- ❌ Удивляться, что `NULL`-разыменование = segfault: нулевая страница специально закрыта.

---

## 🛠️ Практика

1. `cat /proc/<PID>/maps` — найди области с правами `r-x` (код), `rw-` (данные/куча/стек),
   `r--` (только чтение). Сопоставь со схемой.
2. Запусти программу дважды и сравни адреса в `maps` — увидишь ASLR (адреса разные).
3. Объясни, почему запись по `NULL` падает сразу, а не «когда-нибудь потом».
