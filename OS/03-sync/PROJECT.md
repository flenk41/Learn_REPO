# 🚀 Пет-проект уровня 3 — «Синхронизация и файлы вживую»

> 🎯 **Цель:** прочувствовать гонки и их лечение, увидеть deadlock, разобрать файловую систему и
> буферизацию — собрать практику синхронизации и ресурсов.

---

## 📋 Что делаем

Серия небольших экспериментов по синхронизации и файловым системам с отчётом. Часть требует
немного кода (любой язык курса с потоками: C/C++/Python/Rust).

### Часть A. Гонка и её лечение (с кодом)

1. Напиши программу: два (или больше) потока по 100000 раз увеличивают общий счётчик **без**
   синхронизации. Запусти несколько раз — итог почти всегда меньше ожидаемого. Объясни почему.
2. Добавь **мьютекс** вокруг инкремента. Теперь итог корректный. Объясни, что изменилось.
3. (Опц.) Сравни время с мьютексом и без — увидь цену синхронизации.

### Часть B. Deadlock

4. На бумаге (или в коде, осторожно) смоделируй deadlock: два потока, два замка, разный порядок
   захвата. Опиши, кто что держит и ждёт.
5. Покажи, как единый порядок захвата устраняет deadlock. Сопоставь с четырьмя условиями Коффмана.

### Часть C. Файловая система

6. Через `ls -i`, `stat`, `df -i` исследуй inode: посмотри метаданные файла, число свободных inode.
7. Создай жёсткую ссылку (`ln file link`) — покажи через `ls -i`, что inode общий. Переименуй
   файл — объясни, почему данные не двигаются.

### Часть D. Буферизация

8. Прочитай большой файл дважды, замерь оба времени — объясни разницу (page cache).

### Отчёт

`report.md`: код экспериментов A/B, наблюдения, разбор deadlock с условиями Коффмана, находки по
inode и page cache. Главный вывод: «почему общая память требует синхронизации и какой ценой».
