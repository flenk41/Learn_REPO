# 16 · Файловые системы 🖼️⭐

> 🎯 **Цель блока:** понять, как ОС превращает «голый» диск (поток блоков) в удобную абстракцию
> файлов и папок — inode, блоки, директории.

---

## 📖 Диск — это просто блоки

Физически диск (HDD/SSD) — это огромный массив одинаковых **блоков** (например, по 4 КБ),
пронумерованных. Никаких «файлов» на нём нет — это придумала ОС.

🖼️
```
   диск глазами железа:  [блок0][блок1][блок2][блок3]...[блокN]
                          голые пронумерованные кусочки

   диск глазами тебя:    /home/user/notes.txt   (файл с именем!)
                          ▲ эту иллюзию создаёт ФАЙЛОВАЯ СИСТЕМА
```

💡 **Файловая система** (ФС) — способ организации блоков так, чтобы из них получались файлы и
папки с именами. Это абстракция, как «процесс» поверх CPU и «виртуальная память» поверх RAM.

---

## ⭐ inode — «паспорт файла»

В Unix-ФС каждый файл описывается структурой **inode** (индексный узел). Inode хранит **всё о
файле, кроме имени**:

```
   inode:
   ├── размер
   ├── права (rwx), владелец
   ├── времена (создан/изменён)
   └── УКАЗАТЕЛИ НА БЛОКИ данных на диске  ← где реально лежит содержимое
```

🖼️
```
   имя файла  →  номер inode  →  inode  →  блоки данных на диске
   notes.txt       1234          [метаданные + список блоков]   [блок 7][блок 19][блок 42]
```

💡 Имя файла и сам файл — **разные** вещи! Имя живёт в директории и лишь ссылается на inode.
Поэтому один файл может иметь несколько имён (жёсткие ссылки), а переименование — это просто
смена записи в директории, без трогания данных.

---

## ⭐ Директория — это файл со списком имён

```
   директория /home/user/ — это файл, содержащий пары:
        "notes.txt"  → inode 1234
        "photo.jpg"  → inode 5678
        "docs"       → inode 9012 (тоже директория)
```

💡 Папки — не «контейнеры с файлами внутри», а **списки соответствий имя→inode**. Путь
`/home/user/notes.txt` ОС разбирает по шагам: корень → найти `home` → найти `user` → найти
`notes.txt` → получить inode. Дерево каталогов — это связи между директориями.

---

## 📖 Фрагментация и журналирование

```
   ФРАГМЕНТАЦИЯ — блоки файла разбросаны по диску → чтение медленнее (особенно HDD)
   ЖУРНАЛ (journaling) — ФС записывает намерения в журнал ПЕРЕД изменением
        → при сбое питания можно восстановиться, не потеряв ФС
```

💡 Журналируемые ФС (ext4, NTFS, APFS) переживают внезапное отключение: незавершённую операцию
либо докатывают, либо откатывают по журналу. Без журнала сбой мог оставить ФС в «битом»
состоянии. Это та же идея надёжности через лог, что у транзакций БД.

## ⚠️ Ловушки

- ❌ Думать, что «файл» — это что-то физическое на диске. Это абстракция ОС поверх блоков.
- ❌ Путать имя файла и сам файл (inode). Имя — лишь ссылка в директории.
- ❌ Считать директорию «папкой с файлами внутри». Это список имя→inode.
- ❌ Игнорировать журналирование — без него сбой питания грозит потерей ФС.

---

## 🛠️ Практика

1. На Linux: `ls -i` — увидишь номера inode файлов. `df -i` — сколько inode свободно.
2. `stat файл` — посмотри метаданные из inode (размер, права, времена, число ссылок).
3. Создай жёсткую ссылку (`ln file link`) и увидь через `ls -i`, что у них **один** inode.
4. Узнай тип своей ФС: `df -T` (Linux) / в свойствах диска (Windows: NTFS).
