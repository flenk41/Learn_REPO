# 14 · Мьютексы, семафоры, блокировки 🖼️⭐

> 🎯 **Цель блока:** освоить инструменты синхронизации — мьютекс, семафор, и понять, как они
> защищают критическую секцию от гонок.

---

## ⭐ Мьютекс — «один ключ»

**Мьютекс** (mutex, от *mutual exclusion*) — замок, который держит **один** поток за раз. Хочешь
в критическую секцию — возьми ключ; кто-то уже взял — жди.

🖼️
```
   поток A: lock() 🔒 ──► в критической секции ──► unlock() 🔓
   поток B: lock() ... ждёт, пока A не сделает unlock() ... 🔒 ──► своя очередь
```

```
   mutex.lock()      # захватить (если занят — ждать)
   # ... критическая секция: безопасно работаем с общими данными ...
   mutex.unlock()    # освободить
```

💡 Мьютекс превращает «прочитать-изменить-записать» в неделимую с точки зрения других операцию:
пока держишь замок, никто не вклинится. Гонка (модуль 13) исчезает. Это тот самый `Mutex` из
[Rust](../../Rust/04-senior/20-concurrency.md)/[C++](../../Cpp/04-senior/21-concurrency.md).

⚠️ Главное правило: **обязательно** делай `unlock` (и на путях ошибок!). Забыл — другие зависнут
навсегда. Поэтому в C++/Rust используют RAII-обёртки (замок освобождается автоматически).

---

## ⭐ Семафор — «N пропусков»

**Семафор** — счётчик, разрешающий доступ **N** потокам одновременно (мьютекс — частный случай
с N=1).

```
   семафор = 3 (например, 3 свободных соединения к БД)
   поток берёт пропуск → счётчик 3→2→1→0
   счётчик 0 → следующие ждут, пока кто-то не вернёт пропуск (→ счётчик растёт)
```

💡 Семафор хорош, когда ресурс ограничен числом, а не «один-единственный»: пул соединений, N
параллельных загрузок, ограничение числа рабочих. Бинарный семафор (0/1) ≈ мьютекс.

---

## 📖 Другие примитивы

```
   условная переменная (condition variable) — «ждать, пока не наступит условие»
        (например, поток-потребитель спит, пока производитель не положит данные)
   read-write lock — много читателей ИЛИ один писатель (быстрее, когда чтений много)
   spinlock        — «активное ожидание» в цикле (для очень коротких секций)
   барьер (barrier) — все потоки ждут друг друга в точке встречи
```

💡 Выбор примитива — под задачу. Часто хватает мьютекса. Read-write lock — когда читают намного
чаще, чем пишут (как [shared XOR mutable в Rust](../../Rust/02-memory/10-borrow-checker.md):
много `&` ИЛИ один `&mut`).

---

## ⭐ Цена синхронизации

```
   👍 убирает гонки → корректность
   👎 потоки ЖДУТ друг друга → меньше параллелизма
   👎 захват/освобождение замка — накладные расходы
   👎 неаккуратность → deadlock (модуль 15)
```

💡 Синхронизация — компромисс: безопасность ценой части скорости. Поэтому критические секции
делают **короткими** (держи замок минимально), а где можно — избегают общих изменяемых данных
вовсе (тогда и синхронизация не нужна).

---

## ⚠️ Ловушки

- ❌ Забыть `unlock` (особенно при ошибке/исключении) → вечная блокировка других. Используй RAII.
- ❌ Держать замок слишком долго → убивает параллелизм (все ждут).
- ❌ Слишком много мелких замков → накладные и риск deadlock.
- ❌ Думать, что мьютекс делает данные «быстрее». Он делает их **корректными**, но добавляет ожидание.
