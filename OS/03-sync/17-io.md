# 17 · Ввод-вывод и буферизация 🖼️⭐

> 🎯 **Цель блока:** понять, как ОС управляет вводом-выводом (диск, сеть, устройства), зачем
> нужна буферизация и чем блокирующий I/O отличается от асинхронного.

---

## 📖 ОС — посредник между программой и устройствами

Программа не общается с диском или сетевой картой напрямую (модуль 01). Она просит ОС через
системные вызовы (`read`/`write`), а ОС работает с устройством через **драйвер**.

🖼️
```
   программа: read(file)
        ▼ syscall
   ЯДРО → драйвер → КОНТРОЛЛЕР устройства → железо (диск/сеть)
        ▲                                        │
        └──────────── данные обратно ◄───────────┘
```

💡 Драйвер (вспомни [трек Интеграция](../../Interop/04-drivers/19-what-is-driver.md)) знает
конкретное железо и переводит общие команды ОС в его «язык». Программе же всё равно, какой у
тебя диск — она видит единый интерфейс «файл».

---

## ⭐ Зачем буферизация

Устройства **медленные** относительно CPU (модуль 12, иерархия памяти). Дёргать диск на каждый
байт — катастрофа. Поэтому ОС **буферизует**: копит данные в памяти и обменивается с устройством
крупными порциями.

```
   без буфера: write('a') → диск; write('b') → диск; ... (тысячи медленных обращений)
   с буфером:  write копит в буфер RAM → когда накопилось ~4КБ → один обмен с диском
```

💡 ⭐ Та же идея, что «не делай частых мелких syscalls» (модуль 01) и «не шли мелких пакетов»
(трек Сети). **Кэш страниц** (page cache) — ОС держит недавно прочитанные/записанные блоки диска
в RAM: повторное чтение берётся из памяти, мгновенно. Поэтому второе открытие файла быстрее.

⚠️ Обратная сторона: записанное может ещё **не быть** на диске (лежит в буфере). При сбое
питания потеряется. Поэтому есть `flush`/`fsync` — «точно запиши на диск сейчас» (важно для БД).

---

## ⭐ Блокирующий vs неблокирующий I/O

```
   БЛОКИРУЮЩИЙ (blocking):  read() ждёт, пока данные не придут → поток «спит» (состояние
                            ожидания, модуль 04). Просто, но поток простаивает.

   НЕБЛОКИРУЮЩИЙ/АСИНХРОННЫЙ: «начни читать и сообщи, когда готово» → поток не ждёт,
                            делает другое. Сложнее, но эффективнее под нагрузкой.
```

🖼️
```
   блокирующий:  [read ждёт диск.................] потом работаем
   асинхронный:  [запустил read][работаем над другим][пришло — обработали]
```

💡 Сервер на тысячи соединений не может держать поток на каждое ожидание (дорого). Поэтому
используют асинхронный I/O / мультиплексирование (`epoll`/`select`, а в языках — `async`/`await`,
вспомни [asyncio в Python](../../Python/04-senior/23-concurrency.md) и
[async в Rust](../../Rust/04-senior/24-async-ecosystem.md)). Один поток обслуживает многих,
не простаивая на ожидании.

---

## 📖 DMA — устройства пишут в память сами

```
   без DMA: CPU вручную копирует каждый байт устройство↔память (занят, медленно)
   с DMA (Direct Memory Access): устройство пишет данные прямо в RAM само,
        а CPU занимается полезной работой; в конце — прерывание «готово»
```

💡 **DMA** разгружает процессор: пока диск/сетевая карта переливают данные в память, CPU
свободен. Прерывание (interrupt) сообщает, когда закончили. Прерывания — способ устройств
«постучаться» к ядру (как сигналы между процессами, модуль 07, но от железа).

---

## ⚠️ Ловушки

- ❌ Думать, что `write` сразу пишет на диск. Часто — в буфер; на диск позже (или по `fsync`).
- ❌ Делать ввод-вывод по одному байту без буферизации — катастрофа по скорости.
- ❌ Держать поток на каждом блокирующем ожидании при тысячах соединений — не масштабируется.
- ❌ Полагать, что CPU копирует все данные устройств вручную — для этого есть DMA.

## 🛠️ Практика

1. Прочитай большой файл дважды подряд и сравни время — второе чтение быстрее (page cache).
2. На Linux: `vmstat 1` — колонки `bi`/`bo` (блоки ввода-вывода) под нагрузкой на диск.
3. Объясни, почему сервер на async обслуживает больше соединений, чем «поток на соединение».
