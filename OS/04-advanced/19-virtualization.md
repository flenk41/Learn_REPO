# 19 · Виртуализация и контейнеры 🖼️⭐

> 🎯 **Цель блока:** понять, как ОС изолирует целые окружения — виртуальные машины и контейнеры —
> и какие механизмы ядра за этим стоят (namespaces, cgroups).

---

## 📖 Две идеи изоляции: ВМ и контейнеры

Мы видели изоляцию **процессов** (виртуальная память, модуль 08). Можно изолировать и **целые
окружения** — двумя способами:

🖼️
```
   ВИРТУАЛЬНАЯ МАШИНА (VM)              КОНТЕЙНЕР (Docker)
   ┌────────────────────┐              ┌────────────────────┐
   │ приложение          │             │ приложение          │
   │ ГОСТЕВАЯ ОС (своя!) │             │ (использует ХОЗЯЙСКОЕ│
   │ виртуальное железо  │             │  ядро напрямую)     │
   ├────────────────────┤              ├────────────────────┤
   │ гипервизор          │             │ движок контейнеров  │
   │ ОС хоста + железо   │             │ ОС хоста + железо   │
   └────────────────────┘              └────────────────────┘
   тяжёлая, полная ОС внутри            лёгкий, делит ядро хоста
```

💡 **ВМ** эмулирует целый компьютер — внутри своя ОС (тяжело, но полная изоляция). **Контейнер**
изолирует процесс, но **делит ядро** с хостом (легко и быстро, изоляция слабее). Контейнеры —
причина «работает на моей машине» больше не проблема: упаковал окружение целиком.

---

## ⭐ Как ядро делает контейнеры: namespaces

Контейнер — это **обычный процесс**, которому ядро показывает «свой мир» через **namespaces**:

```
   PID namespace    — контейнер видит свои процессы с PID 1, не видит чужих
   mount namespace  — своя файловая система (свой «корень» /)
   network namespace— свой сетевой стек (свои интерфейсы, IP)
   user namespace   — свои пользователи (root в контейнере ≠ root хоста)
```

💡 ⭐ Namespaces — это **виртуализация имён/представлений**: процесс думает, что он один в
системе, хотя делит ядро с сотней других. Прямое развитие идеи изоляции из уровня 2, но не
только для памяти — для всего: процессов, сети, файлов. (Связь с пробросом портов контейнеров =
NAT из [трека Сети](../../Network/04-advanced/19-nat-firewall-vpn.md).)

---

## ⭐ Лимиты ресурсов: cgroups

Namespaces дают **изоляцию видимости**, а **cgroups** (control groups) — **ограничение
ресурсов**:

```
   cgroup для контейнера:
   ├── не больше 2 ядер CPU
   ├── не больше 512 МБ памяти
   └── не больше N МБ/с дискового I-O
```

💡 cgroups (модуль 18, лимиты) не дают контейнеру «съесть» всю машину. Связка **namespaces
(что вижу) + cgroups (сколько могу взять)** = контейнер. Docker/Kubernetes — это удобные
обёртки над этими механизмами ядра Linux.

## 📖 Когда что

```
   ВМ          — нужна полная изоляция, другая ОС, недоверенный код → безопаснее, но тяжелее
   КОНТЕЙНЕР   — нужна лёгкость, быстрый старт, та же ОС, плотная упаковка → быстро, но делит ядро
```

💡 Часто их **сочетают**: контейнеры запускают внутри ВМ (изоляция ВМ + удобство контейнеров).
В облаке (модуль 24) так и устроено: твои контейнеры крутятся на ВМ провайдера.

## ⚠️ Ловушки

- ❌ Путать ВМ (своя ОС внутри) и контейнер (делит ядро хоста).
- ❌ Считать контейнер «полностью изолированным как ВМ». Он делит ядро — изоляция слабее.
- ❌ Думать, что Docker — «магия». Это namespaces + cgroups ядра под удобной обёрткой.
- ❌ Считать root в контейнере безопасным по умолчанию (user namespace важен).

## 🛠️ Практика

1. Запусти контейнер: `docker run -it --rm alpine sh`. Внутри `ps aux` — увидишь только свои
   процессы (PID namespace), свой `/` (mount namespace).
2. Сравни: на хосте этот контейнер — обычный процесс (`ps` на хосте его покажет).
3. Поставь лимит памяти: `docker run --memory=256m ...` — это cgroups в действии.
