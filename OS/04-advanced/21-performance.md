# 21 · Производительность ОС 🖼️⭐

> 🎯 **Цель блока:** понять, что замедляет систему на уровне ОС — контекстные переключения,
> промахи кэша/TLB, swap, блокировки — и как находить узкие места.

---

## ⭐ Где «утекает» производительность

Собери воедино «дорогие» вещи из всего трека:

```
   контекстные переключения (модуль 06)  — переключение процессов не бесплатно
   переходы user↔kernel (модуль 01)       — частые мелкие syscalls дороги
   промахи кэша/TLB (модуль 12)           — обращение к RAM в ~100 раз медленнее кэша
   page fault / swap (модуль 10)          — диск в тысячи раз медленнее RAM; thrashing = смерть
   ожидание блокировок (модуль 14)        — потоки спят на мьютексах вместо работы
   блокирующий I/O (модуль 17)            — поток простаивает на ожидании
```

💡 «Тормозит» почти всегда сводится к одному из этих. Диагностика = понять, **на чём** система
теряет время: считает (CPU-bound), ждёт диск/сеть (I-O-bound), или жонглирует памятью (swap).

---

## ⭐ Три типа «занятости»

```
   CPU-bound   — упирается в процессор (CPU ~100%); ускорят: алгоритмы, параллелизм, кэш
   I-O-bound   — ждёт диск/сеть (CPU простаивает, диск занят); ускорят: буферы, async, SSD, кэш
   memory-bound— не хватает RAM → swap/thrashing; лечит: больше RAM, меньше памяти, локальность
```

💡 ⭐ Первый шаг оптимизации — **определить тип**. Бессмысленно докупать ядра, если ты I-O-bound
(ждёшь диск), или оптимизировать алгоритм, если ты в swap. Сначала измерь, потом чини — как в
[профилировании C](../../C/04-senior/24-debugging-profiling.md) и
[Python](../../Python/04-senior/22-profiling.md).

---

## 📖 Метрики, на которые смотреть

```
   load average     — средняя длина очереди готовых процессов (>число ядер = перегрузка)
   CPU %            — user / system (ядро) / iowait (ждём диск) / idle
   context switches — vmstat cs: слишком много = накладные на переключения
   swap si/so       — активность подкачки = нехватка памяти
   iowait высокий   — система ждёт диск (I-O-bound)
```

💡 Высокий **iowait** при низком CPU = ты ждёшь диск, а не считаешь. Высокий **load** при
свободном CPU = что-то блокирует (I-O или замки). Эти метрики сразу указывают направление.

---

## ⭐ Цена параллелизма

```
   👍 больше потоков → больше работы одновременно (на многих ядрах)
   👎 но: переключения, синхронизация, борьба за общие данные (cache contention)
   → после некоторого числа потоков прирост падает или становится отрицательным
```

💡 «Добавлю потоков — будет быстрее» работает до предела. Слишком много потоков → они мешают
друг другу (замки, кэш, переключения). Оптимум часто ≈ числу ядер для CPU-bound задач.

---

## ⚠️ Ловушки

- ❌ Оптимизировать вслепую без измерения типа нагрузки (CPU/IO/memory-bound).
- ❌ Докупать ядра при I-O-bound или память-bound проблеме.
- ❌ Бесконечно наращивать потоки — после предела становится хуже.
- ❌ Игнорировать iowait и swap-активность как индикаторы.

## 🛠️ Практика

1. `top`/`htop` — посмотри разбивку CPU (us/sy/id/wa). Найди, чем занята система сейчас.
2. `vmstat 1` — следи за `cs` (переключения), `wa` (iowait), `si/so` (swap) под нагрузкой.
3. Определи для знакомой программы её тип: CPU-bound, I-O-bound или memory-bound? Обоснуй.
