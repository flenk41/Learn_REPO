## ⭐ Методика: от симптома к причине

Когда «всё тормозит/виснет», не гадай — иди по слоям ресурсов (CPU → память → диск → сеть):

```
   1. CPU?     top/htop: какой процесс ест CPU? us высокий = считает, sy = много syscalls
   2. ПАМЯТЬ?  free: мало свободной? swap активен? → memory-bound, thrashing (модуль 10)
   3. ДИСК?    iostat/vmstat: высокий iowait/bi-bo? → I-O-bound (модуль 17)
   4. СЕТЬ?    ss/iftop: упёрлись в сеть? (трек Сети)
   5. КОНКРЕТНЫЙ ПРОЦЕСС? strace/perf: на чём он висит?
```

🖼️
```
   «система тормозит»
        ▼
   CPU 100%? ──да──► кто? (top) → CPU-bound: алгоритм/параллелизм
        │нет
   swap активен? ──да──► memory-bound: больше RAM / меньше памяти
        │нет
   iowait высокий? ──да──► I-O-bound: диск/сеть, буферы, async
        │нет
   что-то блокирует → strace/perf на подозрительном процессе
```

💡 Эта лестница за минуту локализует, **на чём** теряется время. Большинство проблем — один из
четырёх ресурсов. Сначала найди ресурс, потом причину.

## 📖 Инструменты по задачам

```
   обзор:        top, htop, glances, Диспетчер задач
   память:       free, vmstat, /proc/meminfo
   диск I-O:     iostat, iotop, vmstat (bi/bo/wa)
   сеть:         ss, netstat, iftop, nload
   процесс:      strace (syscalls), ltrace (библиотеки), lsof (файлы), /proc/<PID>/
   профилировка: perf (Linux) — где процесс жжёт CPU, по функциям
   ядро/события: dmesg, journalctl
```

💡 Связка от общего к частному: `top` (кто виноват) → `vmstat`/`free`/`iostat` (какой ресурс) →
`strace`/`perf` (что именно делает процесс). От симптома системы к строке кода.

---

## ⭐ strace и perf — рентген процесса

```
   strace -p <PID>  — какие системные вызовы делает процесс прямо сейчас
        «висит на read()?» → ждёт I-O. «куча мелких write?» → нет буферизации.
   perf top         — какие функции жгут CPU в реальном времени (профиль)
```

💡 ⭐ Это вершина диагностики: ты видишь, **на каком syscall завис** процесс или **в какой
функции** жжёт CPU. Связь с профилированием в языках ([C](../../C/04-senior/24-debugging-profiling.md),
[Python](../../Python/04-senior/22-profiling.md)) — но теперь с точки зрения ОС.

## 📖 Инженерная картина: весь трек вместе

```
   программа становится ПРОЦЕССОМ (уровень 1)
     → получает ВИРТУАЛЬНУЮ ПАМЯТЬ: стек/куча/код на страницах (уровень 2 ⭐)
     → ПЛАНИРОВЩИК даёт ей CPU по кусочкам, переключая (уровень 1)
     → она делает SYSCALLS к ядру за файлами/сетью/памятью (уровень 0)
     → при многопоточности СИНХРОНИЗИРУЕТ доступ к общим данным (уровень 3)
     → ОС защищает, изолирует, ограничивает и УЧИТЫВАЕТ ресурсы (уровни 2-4)
     → а ты всё это ВИДИШЬ инструментами наблюдения
```

💡 Это итог: ОС — невидимый дирижёр, дающий твоему коду память, процессор и устройства, и
защищающий программы друг от друга. Теперь невидимое стало видимым.

## ⚠️ Ловушки

- ❌ Чинить наугад («перезагружу») вместо лестницы CPU→память→диск→сеть.
- ❌ Смотреть только на CPU, игнорируя iowait/swap (часто виноваты они).
- ❌ Лезть в strace/perf сразу — сначала общий обзор (top, vmstat), потом частности.
- ❌ Делать вывод по одному снимку — для динамики нужны `top`/`vmstat 1` во времени.
