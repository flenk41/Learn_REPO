# 04 · Ключи и связи 🖼️⭐⭐

> 🎯 **Цель блока:** освоить первичные и внешние ключи и три типа связей — то, как таблицы
> соединяются в осмысленную модель данных.

---

## 📖 Первичный ключ (PRIMARY KEY)

```
   ПЕРВИЧНЫЙ КЛЮЧ — столбец(ы), УНИКАЛЬНО идентифицирующий каждую строку.
   • уникален и НЕ NULL.
   • часто суррогатный: id (автоинкремент/UUID) — искусственный, стабильный.
   • либо естественный: email, ИНН — реальный атрибут (но осторожно: может меняться).
   рекомендация: обычно суррогатный id (не меняется, прост, эффективен для связей).

   clients (id PK, name, email)   ← id уникально идентифицирует клиента.
```

💡 ⭐ Первичный ключ — «удостоверение личности» строки. По нему ссылаются другие таблицы. Обычно
берут суррогатный `id` (автоинкремент или UUID): он стабилен (email клиента может смениться, id —
нет) и эффективен. СУБД автоматически индексирует PK (быстрый поиск по нему).

---

## ⭐⭐ Внешний ключ (FOREIGN KEY) и связи

```
   ВНЕШНИЙ КЛЮЧ — столбец, ссылающийся на первичный ключ ДРУГОЙ таблицы. так выражаются СВЯЗИ.
   orders (id PK, client_id FK → clients.id, ...)
   client_id в orders ССЫЛАЕТСЯ на клиента → «этот заказ принадлежит клиенту с таким id».

   ЦЕЛОСТНОСТЬ: СУБД ПРИНУЖДАЕТ — нельзя заказ с client_id, которого нет в clients (ссылочная
   целостность). нельзя удалить клиента, у которого есть заказы (или каскадно — по настройке).
```

🖼️
```
   clients               orders
   ┌────┬──────┐         ┌────┬───────────┬──────┐
   │ id │ name │         │ id │ client_id │ sum  │
   ├────┼──────┤         ├────┼───────────┼──────┤
   │ 1  │ Анна │◄────────┤ 10 │     1     │ 500  │  ← заказ 10 принадлежит Анне (client_id=1)
   │ 2  │ Иван │◄────────┤ 11 │     2     │ 300  │
   └────┴──────┘         └────┴───────────┴──────┘
        PK                          FK ─────┘
```

💡 ⭐⭐ Внешние ключи — это «клей» реляционной модели: связи между таблицами + гарантия
целостности (нет «висячих» ссылок на несуществующих клиентов). Именно по этим связям делаются
JOIN-ы (Уровень 2) — соединение данных из таблиц.

---

## ⭐ Три типа связей

```
   1:1 (один-к-одному) — одна строка A ↔ одна строка B. редко; обычно объединяют в одну таблицу
       или разделяют для безопасности/опциональности (user ↔ user_profile).

   1:N (один-ко-многим) — одна строка A ↔ МНОГО строк B. самая частая!
       один клиент → много заказов. реализация: FK на стороне «многих» (orders.client_id).

   N:M (многие-ко-многим) — много A ↔ много B. нужна ПРОМЕЖУТОЧНАЯ таблица (junction/link).
       заказы ↔ товары (заказ содержит много товаров, товар в многих заказах):
       order_items (order_id FK, product_id FK, qty)  ← связующая таблица.
```

🖼️
```
   1:N:  [клиент] ──< [заказы]        FK в «много» (orders.client_id)
   N:M:  [заказы] >──[order_items]──< [товары]   связующая таблица с двумя FK
   распознать N:M: «и то много, и это много» → нужна третья таблица.
```

💡 ⭐ Самый важный навык моделирования: **распознать тип связи**. «У клиента много заказов, но
заказ — у одного клиента» = 1:N (FK в заказах). «Заказ содержит много товаров, товар — в многих
заказах» = N:M (связующая таблица order_items). N:M ВСЕГДА требует промежуточную таблицу — частая
ошибка новичков забыть её.

## 📖 ON DELETE / ON UPDATE

```
   что делать при удалении строки, на которую ссылаются?
   • RESTRICT/NO ACTION — запретить удаление (есть заказы → клиента не удалить). безопасно по умолчанию.
   • CASCADE — удалить и связанные (удалил заказ → удалились его order_items). для «составных частей».
   • SET NULL — обнулить ссылку (удалил категорию → у товаров category_id = NULL).
   выбирай осознанно: CASCADE для «частей целого», RESTRICT для важных связей.
```

---

## ⚠️ Ловушки

- ❌ Забыть промежуточную таблицу для связи N:M (нельзя выразить двумя FK напрямую).
- ❌ Естественный ключ, который может меняться (email как PK → проблемы при смене).
- ❌ Нет внешних ключей → «висячие» ссылки, нарушение целостности.
- ❌ Неосторожный CASCADE (удалил клиента → каскадом снёс все заказы случайно).
- ❌ Хранить связь списком id в одном поле вместо FK/связующей таблицы.
