# 07 · Типы данных и схема 🖼️

> 🎯 **Цель блока:** научиться выбирать правильные типы данных и ограничения — основа корректной и
> эффективной схемы.

---

## 📖 Выбор типа — это важно

```
   правильный тип = корректность + эффективность + защита от грязных данных.
   неправильный тип = баги, потеря данных, медленные запросы, лишняя память.
   пример: хранить дату как TEXT → нельзя сортировать/сравнивать корректно, нет валидации.
           хранить деньги как FLOAT → ошибки округления (0.1+0.2 ≠ 0.3!). нужен DECIMAL.
```

---

## ⭐ Основные типы (на примере PostgreSQL)

```
   ЧИСЛА:
   • INTEGER / BIGINT — целые (id, количества). BIGINT для больших.
   • DECIMAL(p,s) / NUMERIC — точные дробные → ДЕНЬГИ (не FLOAT!), точность важна.
   • REAL / DOUBLE — приближённые дробные (научные расчёты, где округление ок).

   ТЕКСТ:
   • VARCHAR(n) — строка с лимитом; TEXT — без лимита. (в Postgres разница мала; TEXT удобен).
   • CHAR(n) — фиксированная длина (редко).

   ДАТА/ВРЕМЯ:
   • DATE, TIME, TIMESTAMP — храни даты как ДАТЫ (не текст!) → сортировка, арифметика, валидация.
   • TIMESTAMPTZ — с часовым поясом (важно для распределённых/глобальных приложений).

   ПРОЧЕЕ:
   • BOOLEAN — true/false.
   • UUID — глобально уникальные id.
   • JSON/JSONB — полуструктурированные данные (Postgres: JSONB индексируемый) — мост к NoSQL-гибкости.
   • массивы, ENUM, и др. (Postgres богат типами).
```

💡 ⭐ Правила: **деньги → DECIMAL** (не float!), **даты → DATE/TIMESTAMP** (не текст!), **id →
INTEGER/BIGINT/UUID**. Выбирай наименьший подходящий тип (экономия памяти → больше в кэш → быстрее).
JSONB в Postgres даёт гибкость NoSQL внутри реляционной БД (для редко-запрашиваемых гибких данных).

---

## ⭐⭐ Ограничения (constraints) — защита данных

```
   ОГРАНИЧЕНИЯ заставляют СУБД гарантировать валидность данных:
   • PRIMARY KEY — уникальный идентификатор (модуль 04).
   • FOREIGN KEY — ссылочная целостность (модуль 04).
   • NOT NULL — обязательное поле.
   • UNIQUE — уникальное значение (email, username).
   • CHECK — произвольная проверка (age >= 0, price > 0, status IN ('new','paid','shipped')).
   • DEFAULT — значение по умолчанию (created_at DEFAULT now()).

   CREATE TABLE orders (
       id SERIAL PRIMARY KEY,
       client_id INTEGER NOT NULL REFERENCES clients(id),
       amount DECIMAL(10,2) NOT NULL CHECK (amount > 0),
       status TEXT NOT NULL DEFAULT 'new' CHECK (status IN ('new','paid','shipped')),
       created_at TIMESTAMPTZ NOT NULL DEFAULT now()
   );
```

💡 ⭐⭐ Ограничения — **защита целостности на уровне БД**, а не «надеемся, что приложение проверит».
СУБД ГАРАНТИРУЕТ: не будет заказа без клиента, с отрицательной суммой, с невалидным статусом. Это
надёжнее проверок в коде (которые можно обойти/забыть). Закладывай ограничения в схему — это
«типы и инварианты» для данных.

## 📖 Эволюция схемы (превью миграций)

```
   схема НЕ статична — приложение растёт, нужно добавлять столбцы/таблицы, менять типы.
   изменения схемы (ALTER TABLE) на работающей БД с данными — деликатны (модуль 22 «миграции»):
   • добавить столбец с DEFAULT, сделать обязательным позже.
   • не ломать существующие данные/запросы.
   проектируй схему с запасом на изменения, но не переусложняй заранее (YAGNI).
```

---

## ⚠️ Ловушки

- ❌ Деньги во FLOAT/REAL → ошибки округления. Используй DECIMAL.
- ❌ Даты/числа как TEXT → нет сортировки/валидации/арифметики.
- ❌ Нет ограничений (NOT NULL/CHECK/FK) → грязные данные пролезут.
- ❌ Полагаться только на проверки в приложении (БД — последний рубеж целостности).
- ❌ Игнорировать часовые пояса (TIMESTAMP vs TIMESTAMPTZ) в глобальных приложениях.
- ❌ Слишком большие типы «на всякий случай» (память, кэш).
