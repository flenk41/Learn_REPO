# 17 · Изоляция и блокировки 🖼️⭐

> 🎯 **Цель блока:** понять, как БД управляет параллельными транзакциями — уровни изоляции,
> аномалии, блокировки — чтобы избегать гонок и дедлоков.

---

## 📖 Параллельные транзакции мешают друг другу

```
   тысячи транзакций выполняются ОДНОВРЕМЕННО над одними данными. без управления — АНОМАЛИИ:
   • DIRTY READ — прочитал НЕзафиксированные (ещё откатываемые) данные другой транзакции.
   • NON-REPEATABLE READ — прочитал дважды, между чтениями другой изменил → разные значения.
   • PHANTOM READ — повторный запрос вернул НОВЫЕ строки (другой вставил).
   • LOST UPDATE — два обновили одно, одно изменение потерялось.
   ИЗОЛЯЦИЯ (I в ACID) защищает от этих аномалий.
```

---

## ⭐ Уровни изоляции: trade-off строгость ↔ скорость

```
   уровни (от слабого/быстрого к строгому/медленному):
   READ UNCOMMITTED — почти нет изоляции (грязное чтение возможно). редко.
   READ COMMITTED   — видишь только зафиксированное (нет dirty read). ДЕФОЛТ в PostgreSQL.
   REPEATABLE READ  — + повторное чтение даёт то же (нет non-repeatable read).
   SERIALIZABLE     — как будто транзакции выполнялись ПО ОЧЕРЕДИ (нет всех аномалий). строжайший, медленнее.

   ↑ выше уровень = меньше аномалий, но больше блокировок/откатов (меньше параллелизма).
```

🖼️
```
   уровень изоляции →  слабее ─────────────────────────► строже
                       (быстро, аномалии)          (медленно, безопасно)
   READ UNCOMMITTED   READ COMMITTED   REPEATABLE READ   SERIALIZABLE
                       ↑ дефолт                          ↑ для критичной корректности
   выбираешь под задачу: финансы → строже; аналитика → можно слабее.
```

💡 ⭐ Уровень изоляции — это [trade-off (Senior)](../../Senior/02-decisions/08-tradeoffs.md):
строгость (корректность) против параллелизма (скорость). По умолчанию READ COMMITTED (баланс). Для
критичных операций (финансовые инварианты) — выше. Знай, от каких аномалий защищает каждый уровень,
и выбирай осознанно.

## ⭐⭐ Блокировки и MVCC

```
   как БД реализует изоляцию (два подхода):

   БЛОКИРОВКИ (locking) — транзакция блокирует данные, которые меняет; другие ждут.
   • row-level (одна строка) vs table-level (вся таблица). тонкие блокировки → больше параллелизма.
   • shared (на чтение, много можно) vs exclusive (на запись, эксклюзивно).
   ⚠️ DEADLOCK — две транзакции ждут блокировки ДРУГ ДРУГА вечно → БД обнаруживает и откатывает одну.

   MVCC (Multi-Version Concurrency Control) — как в PostgreSQL:
   • хранит НЕСКОЛЬКО версий строк; читатели видят согласованный СНИМОК, НЕ блокируя писателей.
   • «читатели не ждут писателей, писатели не ждут читателей» → высокий параллелизм.
   • цена: версии накапливаются → нужна очистка (VACUUM в Postgres).
```

💡 ⭐⭐ Postgres использует **MVCC**: читатели видят снимок данных на момент начала, не блокируя
писателей — отсюда высокая конкурентность. Но обновления создают версии строк → периодический
VACUUM их чистит. Понимание MVCC объясняет поведение Postgres (почему чтение не блокируется,
зачем VACUUM). Блокировки всё равно есть для конфликтующих записей — отсюда возможны дедлоки.

## 📖 Практика: избегать проблем

```
   ✅ держи транзакции КОРОТКИМИ (меньше блокировок, меньше конфликтов).
   ✅ единый ПОРЯДОК доступа к данным во всех транзакциях → меньше дедлоков (классика).
   ✅ для конкурентных обновлений: SELECT ... FOR UPDATE (явная блокировка строки) или
      оптимистичные блокировки (версия/timestamp, проверка при апдейте).
   ✅ выбирай уровень изоляции под задачу (не SERIALIZABLE везде — медленно).
   ✅ обрабатывай ошибки сериализации/дедлока (повторить транзакцию).
   ✅ в Postgres — следи за VACUUM (раздувание версий).
```

> 🧭 Это [конкурентность из ОС/капстоуна](../../Capstone/03-storage/17-transactions.md): гонки,
> блокировки, дедлоки — те же, но управляет ими СУБД. Минимизируй конфликты (короткие транзакции).

## ⚠️ Ловушки

- ❌ Игнорировать уровень изоляции (полагаться на дефолт, не понимая аномалий).
- ❌ Lost update (два читают-меняют-пишут одно) без FOR UPDATE/оптимистичной блокировки.
- ❌ Дедлоки от разного порядка захвата (используй единый порядок).
- ❌ SERIALIZABLE везде (медленно) или READ UNCOMMITTED для критичного (аномалии).
- ❌ Долгие транзакции (много блокировок, конфликты, раздувание версий).
- ❌ Не обрабатывать ошибки дедлока/сериализации (нужен retry).
