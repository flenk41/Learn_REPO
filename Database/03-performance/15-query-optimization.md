# 15 · Оптимизация запросов 🖼️⭐⭐

> 🎯 **Цель блока:** научиться писать и чинить запросы так, чтобы они были быстрыми — включая
> главную беду приложений: проблему N+1.

---

## 📖 Методология: измеряй → пойми → чини

```
   как в [Senior-производительности]: НЕ оптимизируй вслепую.
   1. НАЙДИ медленные запросы (логи медленных запросов, мониторинг).
   2. EXPLAIN ANALYZE (модуль 14) — пойми, почему медленно.
   3. ПОЧИНИ: индекс / переписать запрос / изменить схему.
   4. ЗАМЕРЬ снова.
   обычно 80% проблем — это отсутствие индекса или N+1.
```

---

## ⭐⭐ N+1 — главная беда приложений

```
   N+1 — антипаттерн, когда вместо ОДНОГО запроса делается 1 + N запросов:
   ❌ ПЛОХО:
      orders = SELECT * FROM orders;           -- 1 запрос: получили 100 заказов
      for each order:
          client = SELECT * FROM clients WHERE id = order.client_id;  -- +100 запросов!
      → 101 запрос вместо 1. на 1000 заказов = 1001 запрос → катастрофа по скорости.

   ✅ ХОРОШО — один запрос с JOIN:
      SELECT orders.*, clients.name FROM orders JOIN clients ON clients.id = orders.client_id;
      → 1 запрос, все данные сразу.
```

🖼️
```
   N+1:  [запрос заказов] → [запрос клиента][запрос клиента]...×N  → N+1 обращений к БД (медленно!)
   JOIN: [один запрос с JOIN] → все данные разом → 1 обращение (быстро)
   каждое обращение к БД = сетевой round-trip + работа СУБД. N+1 убивает производительность.
```

💡 ⭐⭐ N+1 — **самая частая причина тормозов** в приложениях с ORM (ORM незаметно делает запрос на
каждый объект). Решение: загружать связанные данные ОДНИМ запросом (JOIN / eager loading / `JOIN
FETCH` / `prefetch_related`). Всегда проверяй, сколько запросов делает страница — N+1 прячется
везде. (Связь: [⚙️ цена обращений к БД](../../ComputerScience/00-intro/01-abstraction-layers.md) —
сеть в миллионы раз дороже инструкции.)

## ⭐ Приёмы оптимизации запросов

```
   ✅ ИНДЕКСЫ на столбцы фильтров/join'ов/сортировок (модуль 13).
   ✅ ИЗБЕГАЙ N+1 — загружай связанное одним JOIN.
   ✅ SELECT только НУЖНЫЕ столбцы (не SELECT * — меньше данных, шанс index-only scan).
   ✅ ФИЛЬТРУЙ рано и эффективно (WHERE по индексированному столбцу до join'ов где можно).
   ✅ LIMIT для больших результатов (не тяни миллион строк, если нужно 20).
   ✅ избегай функций над столбцами в WHERE (ломают индекс — модуль 13).
   ✅ для пагинации больших данных — keyset (WHERE id > last_id LIMIT n) вместо больших OFFSET.
   ✅ батчинг — обрабатывай данные пачками, не по одной строке.
   ✅ агрегируй в БД (GROUP BY), а не тяни всё в приложение и считай там.
```

💡 ⭐ Перенос работы В БД: пусть СУБД фильтрует, сортирует, агрегирует (она оптимизирована для
этого), а не тяни сырые данные в приложение и обрабатывай там. «Дай мне сумму» (`SUM` в SQL)
лучше, чем «дай все строки» + суммирование в коде.

## 📖 Когда оптимизировать схему/архитектуру

```
   иногда запрос не ускорить — проблема глубже:
   • ДЕНОРМАЛИЗАЦИЯ (модуль 05) — если постоянные тяжёлые join'ы.
   • КЭШИРОВАНИЕ (модуль 21) — частые одинаковые запросы → кэш (Redis).
   • МАТЕРИАЛИЗОВАННЫЕ ПРЕДСТАВЛЕНИЯ — предвычисленные тяжёлые агрегаты.
   • РЕПЛИКИ ДЛЯ ЧТЕНИЯ (модуль 19) — разгрузить основную БД.
   • ПЕРЕСМОТР СХЕМЫ/ИНДЕКСОВ.
   но СНАЧАЛА — индексы и устранение N+1 (90% проблем).
```

> 🧭 Это [Senior-методология](../../Senior/03-practices/17-performance-mindset.md): профилируй,
> найди узкое место (обычно БД и в ней — N+1/нет индекса), чини прицельно, не преждевременно.

## ⚠️ Ловушки

- ❌ N+1 (запрос на каждый объект вместо JOIN) — главный убийца.
- ❌ Оптимизировать без EXPLAIN/измерений (гадание).
- ❌ SELECT * когда нужны 2 столбца.
- ❌ Тянуть данные в приложение и считать там вместо агрегации в БД.
- ❌ Большие OFFSET для пагинации (медленно — используй keyset).
- ❌ Преждевременная денормализация/кэш до устранения базовых проблем (индексы, N+1).

---

## ✅ Задачи

1. **Найди N+1.** В коде/сценарии найди N+1 (запрос в цикле). Перепиши одним JOIN. Замерь разницу
   в числе запросов и времени.
2. **Индекс + переписать.** Возьми медленный запрос, по EXPLAIN добавь индекс И/ИЛИ перепиши.
   Измерь до/после.
3. ⭐ **SELECT *.** Сравни `SELECT *` и `SELECT нужные_столбцы` по EXPLAIN (есть ли index-only scan?).
4. ⭐ **Агрегация в БД.** Перенеси подсчёт из приложения (тянуть строки + считать) в SQL (GROUP BY/SUM).
5. **Keyset-пагинация.** Замени большой OFFSET на keyset (WHERE id > last). Сравни на большой таблице.
