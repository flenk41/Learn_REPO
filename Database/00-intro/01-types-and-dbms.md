# 01 · Типы БД и как устроена СУБД 🖼️⭐

> 🎯 **Цель блока:** получить карту видов баз данных (реляционные vs NoSQL) и понять, из чего
> состоит СУБД внутри.

---

## 📖 Две большие семьи: реляционные и NoSQL

```
   РЕЛЯЦИОННЫЕ (SQL) — данные в ТАБЛИЦАХ со строгой схемой, связи между таблицами, язык SQL.
     PostgreSQL, MySQL, SQLite, Oracle, SQL Server.
     ✅ строгая структура, связи, транзакции ACID, мощные запросы. ← дефолт для большинства задач.

   NoSQL — «не только SQL», разные модели данных под разные задачи (модуль 18):
     • документные (MongoDB) — JSON-документы, гибкая схема.
     • key-value (Redis) — простые пары ключ→значение, очень быстро.
     • колоночные (Cassandra) — для огромных объёмов/аналитики.
     • графовые (Neo4j) — для связей (соцсети, рекомендации).
     ✅ гибкость/масштаб под специфику; ❌ обычно слабее по связям/транзакциям/запросам.
```

💡 ⭐ Главное правило: **по умолчанию — реляционная БД** (PostgreSQL). Она покрывает 90% задач:
структура, связи, транзакции, мощный SQL. NoSQL — когда есть конкретная причина (специфичная
модель данных, экстремальный масштаб, гибкая схема). Этот трек — в основном про реляционные
(фундамент), NoSQL — на Уровне 4.

## ⭐ OLTP vs OLAP

```
   ещё одно деление — по НАГРУЗКЕ:
   OLTP (транзакционные) — много мелких операций (заказы, профили): запись/чтение отдельных записей.
     оптимизированы под быстрые транзакции. ← обычные приложения.
   OLAP (аналитические) — тяжёлые запросы по огромным данным (отчёты, аналитика): агрегация миллионов строк.
     отдельные системы/хранилища (data warehouse), колоночное хранение.
   → не смешивай: тяжёлая аналитика на боевой OLTP-базе её положит. для аналитики — отдельное хранилище.
```

## ⭐⭐ Как устроена СУБД внутри

```
   СУБД — сложная система. упрощённо, путь запроса:

   1. КЛИЕНТ шлёт SQL по сети (порт, протокол).
   2. ПАРСЕР — разбирает SQL, проверяет синтаксис/права.
   3. ОПТИМИЗАТОР (планировщик) — строит ПЛАН выполнения: какие индексы, в каком порядке join'ы,
      как искать. выбирает дешёвый план (стоимостная оптимизация). ← «мозг» СУБД.
   4. ИСПОЛНИТЕЛЬ — выполняет план: читает данные (через буферный кэш/с диска), применяет операции.
   5. STORAGE ENGINE — как данные лежат на диске (страницы, B-tree индексы), буферный пул (кэш в RAM).
   6. ЖУРНАЛ (WAL) — пишет изменения для надёжности/транзакций (durability).
   7. МЕНЕДЖЕР ТРАНЗАКЦИЙ И БЛОКИРОВОК — управляет параллельным доступом (ACID, изоляция).
```

🖼️
```
   SQL → [парсер] → [ОПТИМИЗАТОР: лучший план] → [исполнитель] → [storage: страницы/индексы/кэш]
                                                                  [WAL: журнал] [транзакции/блокировки]
   ключевое: ОПТИМИЗАТОР сам решает, КАК выполнить твой декларативный запрос эффективно.
```

💡 ⭐⭐ Самое важное для понимания — **оптимизатор/планировщик**: он превращает твой «что хочу»
(SQL) в «как искать» (план с индексами и порядком операций). Поэтому один и тот же результат можно
получить быстро или медленно — зависит от плана (и наличия индексов). Уровень 3 научит читать план
(EXPLAIN) и помогать оптимизатору. Storage engine (страницы, B-tree, WAL) — это [капстоун-БД](../../Capstone/03-storage/16-storage-engine.md).

## 📖 Какую СУБД выбрать для учёбы

```
   • PostgreSQL — мощная, открытая, «правильная» реляционная БД. лучший выбор для серьёзного изучения.
     богатый SQL, индексы, транзакции, расширения. индустриальный стандарт.
   • SQLite — встроенная (один файл, без сервера). идеально для НАЧАЛА: ноль настройки, учишь SQL.
   • MySQL — популярна (веб), похожа на Postgres по основам.
   рекомендация: начни с SQLite (мгновенно), перейди на PostgreSQL для глубины.
```

## ⚠️ Ловушки

- ❌ Брать NoSQL «потому что модно», когда реляционная подошла бы лучше (теряешь связи/транзакции/SQL).
- ❌ Гонять тяжёлую аналитику (OLAP) на боевой транзакционной (OLTP) базе.
- ❌ Считать все БД одинаковыми — модель данных и нагрузка определяют выбор.
- ❌ Игнорировать роль оптимизатора (он определяет скорость запроса).
