# 21 · Кэширование 🖼️⭐⭐

> 🎯 **Цель блока:** освоить кэширование — хранение частых данных в быстром хранилище, чтобы
> разгрузить БД. Один из главных приёмов масштабирования.

---

## 📖 Зачем кэш: БД — узкое место

```
   частые одинаковые запросы бьют по БД (диск, вычисления). КЭШ хранит результаты в БЫСТРОМ
   хранилище (память, Redis) → повторный запрос берётся из кэша, не трогая БД.
   • первый запрос → БД (медленно) → сохранить в кэш.
   • следующие → из кэша (быстро), БД не трогается.
   читаемые часто, меняемые редко данные (профиль, каталог, настройки) — идеальны для кэша.
```

🖼️
```
   без кэша:  запрос → [ БД ] каждый раз (медленно, нагрузка)
   с кэшем:   запрос → [ КЭШ? ] ─есть(hit)──► быстрый ответ
                          └─нет(miss)─► [ БД ] → сохранить в кэш → ответ
   высокий hit rate (попадания) → БД разгружена, ответы быстры.
```

💡 ⭐⭐ Кэш — один из самых эффективных приёмов масштабирования чтения: частые данные отдаются из
памяти (наносекунды-микросекунды), а не из БД (миллисекунды). Redis — стандартный кэш (key-value в
памяти, очень быстрый). Но кэш добавляет проблему — **инвалидацию** (когда данные устарели).

---

## ⭐ Где и как кэшировать

```
   УРОВНИ КЭША:
   • в приложении (in-memory) — самый быстрый, но не разделяется между серверами.
   • РАСПРЕДЕЛЁННЫЙ (Redis/Memcached) — общий для всех серверов приложения. стандарт.
   • кэш самой БД (буферный пул) — СУБД кэширует страницы в RAM (есть из коробки).
   • CDN — кэш статики/ответов ближе к пользователю (для веба).

   ЧТО кэшировать:
   • результаты тяжёлых запросов (агрегаты, отчёты).
   • часто читаемые, редко меняемые данные (профили, каталог, конфиги).
   • сессии, токены, счётчики.
```

---

## ⭐⭐ Главная проблема: инвалидация кэша

```
   «в программировании две сложные вещи: инвалидация кэша и придумывание имён».
   ПРОБЛЕМА: данные изменились в БД, но в кэше — СТАРАЯ копия → пользователь видит устаревшее.

   СТРАТЕГИИ:
   • TTL (time-to-live) — кэш истекает через N секунд → перечитается из БД. простейшее. данные
     устаревают максимум на TTL. (профиль на 5 мин — терпимо.)
   • ИНВАЛИДАЦИЯ ПРИ ЗАПИСИ — при UPDATE данных удаляешь/обновляешь кэш. точнее, но сложнее
     (надо помнить все места кэша этих данных).
   • WRITE-THROUGH — пишешь и в БД, и в кэш одновременно.
   • CACHE-ASIDE (самый частый) — приложение проверяет кэш, при miss читает БД и кладёт в кэш;
     при записи — инвалидирует кэш.
```

💡 ⭐⭐ Инвалидация — самая сложная часть: легко получить **stale data** (показываешь устаревшее) или
сложную логику сброса. Прагматично: **TTL** для большинства (данные устаревают максимум на TTL —
часто приемлемо), точечная инвалидация при записи — для важного. Это [trade-off](../../Senior/02-decisions/08-tradeoffs.md):
свежесть данных против сложности/нагрузки. Чем короче TTL — свежее, но больше нагрузки на БД.

---

## 📖 Подводные камни кэша

```
   ⚠️ STALE DATA — устаревшие данные в кэше (баланс свежести и нагрузки через TTL).
   ⚠️ CACHE STAMPEDE — TTL истёк → МНОГО запросов одновременно бьют в БД (промахи разом) → пик нагрузки.
      решения: блокировка на пересчёт, рандомизация TTL, фоновое обновление.
   ⚠️ КЭШИРОВАТЬ НЕ ВСЁ — редкие/уникальные запросы (низкий hit rate) кэшировать бессмысленно.
   ⚠️ КОНСИСТЕНТНОСТЬ — кэш = ещё одна копия данных → eventual consistency (модуль 20).
   ⚠️ ПАМЯТЬ — кэш ограничен; нужна политика вытеснения (LRU — наименее используемое).
```

> 🧭 Кэш = key-value хранилище (Redis) — это твой [капстоун-KV](../../Capstone/03-storage/13-kv-store.md)
> с TTL/LRU. Сначала оптимизируй запросы/индексы (модули 13-15), потом кэшируй (не наоборот).

---

## ⚠️ Ловушки

- ❌ Кэшировать до оптимизации запросов/индексов (кэш прячет, но не решает плохие запросы).
- ❌ Игнорировать инвалидацию → stale data (показываешь устаревшее как актуальное).
- ❌ Слишком долгий TTL для часто меняемых данных (несвежесть) / слишком короткий (нет смысла).
- ❌ Cache stampede (массовые промахи при истечении TTL).
- ❌ Кэшировать всё подряд (низкий hit rate — бессмысленно, тратит память).
- ❌ Забыть, что кэш — копия → данные могут расходиться с БД.

## ✅ Задачи

1. **Cache-aside.** Реализуй: проверь кэш → miss → читай БД → положи в кэш с TTL. Замерь, ускорился ли
   повторный запрос.
2. **TTL.** Подбери TTL для разных данных (профиль, каталог, цена товара). Trade-off свежесть/нагрузка?
3. ⭐ **Инвалидация.** При UPDATE данных инвалидируй кэш. Проверь, что после изменения показывается свежее.
4. ⭐ **Redis.** Используй Redis как кэш (Docker). SET с TTL, GET. Где применил бы в реальном приложении?
5. **Что не кэшировать.** Приведи примеры данных/запросов, которые кэшировать НЕ стоит. Почему?
