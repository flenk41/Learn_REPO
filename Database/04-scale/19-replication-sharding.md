# 19 · Репликация и шардинг 🖼️⭐⭐

> 🎯 **Цель блока:** понять, как масштабировать БД за пределы одной машины — репликация (копии) и
> шардинг (разбиение данных).

---

## 📖 Одна машина — предел

```
   одна БД-машина имеет пределы: CPU, память, диск, число соединений. при росте:
   • слишком много ЧТЕНИЙ → одна машина не справляется.
   • слишком много ДАННЫХ → не влезают / медленно.
   • нужна ОТКАЗОУСТОЙЧИВОСТЬ → одна машина = единая точка отказа.
   решения: РЕПЛИКАЦИЯ (копии для чтения/надёжности) и ШАРДИНГ (разбить данные по машинам).
```

---

## ⭐ Репликация: копии данных

```
   РЕПЛИКАЦИЯ — поддержание КОПИЙ базы на нескольких серверах.
   PRIMARY (master) — принимает ЗАПИСИ. REPLICA (slave) — копии, обновляются за primary (через WAL).

   зачем:
   ✅ МАСШТАБ ЧТЕНИЯ — направляй SELECT на реплики (разгрузить primary). записи → на primary.
   ✅ ОТКАЗОУСТОЙЧИВОСТЬ — primary упал → реплика становится primary (failover).
   ✅ ГЕО-распределение — реплики ближе к пользователям (меньше задержка).

   ⚠️ REPLICATION LAG — реплики отстают от primary (асинхронно) → можешь прочитать УСТАРЕВШИЕ данные
      (записал на primary, сразу прочитал с реплики — ещё нет). это «eventual consistency».
```

🖼️
```
   запись ──► [ PRIMARY ] ──реплицирует (WAL)──► [ REPLICA 1 ][ REPLICA 2 ]
   чтение ◄────────────────────────────────────  ◄─── чтения распределены по репликам
   primary один (записи), реплик много (чтения). lag → реплики чуть отстают.
```

💡 ⭐ Репликация решает масштаб ЧТЕНИЯ и надёжность: одна primary для записи, много реплик для
чтения. Большинство приложений read-heavy (читают чаще, чем пишут) → реплики сильно помогают. Но
**replication lag**: реплика отстаёт → «read-your-writes» проблема (записал, читаешь с реплики —
устарело). Решают: критичные чтения → с primary, или синхронная репликация (медленнее).

---

## ⭐⭐ Шардинг: разбиение данных

```
   ШАРДИНГ (горизонтальное партиционирование) — разбить ДАННЫЕ по нескольким серверам (шардам).
   каждый шард хранит ЧАСТЬ данных. зачем: когда данные/запись НЕ влезают в одну машину.

   КАК делить (shard key — ключ шардирования):
   • по ДИАПАЗОНУ — клиенты A-M на шард 1, N-Z на шард 2.
   • по ХЕШУ ключа — hash(user_id) % число_шардов → равномерное распределение.
   • по зоне/региону.

   ⚠️ СЛОЖНОСТИ шардинга:
   • запросы ПО НЕСКОЛЬКИМ шардам (join'ы между шардами) — дорого/сложно.
   • неравномерность (горячие шарды).
   • решардинг (добавить шард) — болезненно.
   • транзакции между шардами — сложно.
   → шардинг — КРАЙНЕЕ средство, когда репликация/оптимизация не хватает. усложняет всё.
```

🖼️
```
   шардинг по hash(user_id):
   [ШАРД 1: users 0-33%][ШАРД 2: users 34-66%][ШАРД 3: users 67-100%]
   каждый шард — часть данных + свои реплики. запрос к конкретному user → на нужный шард.
   ⚠️ запрос «все users из города X» → опросить ВСЕ шарды (дорого).
```

💡 ⭐⭐ Шардинг масштабирует ЗАПИСЬ и ОБЪЁМ (данные не на одной машине), но **дорого усложняет**:
кросс-шардовые запросы/join'ы/транзакции тяжелы, выбор shard key критичен (плохой → перекос),
решардинг болезнен. Поэтому: **сначала оптимизация + вертикальный рост + репликация (для чтения),
шардинг — в последнюю очередь** (когда реально упёрся в объём/запись). Многие NoSQL шардируют
автоматически (отсюда их «масштабируемость»).

---

## 📖 Вертикальное vs горизонтальное масштабирование

```
   ВЕРТИКАЛЬНОЕ (scale up) — мощнее одна машина (больше CPU/RAM/диск). просто, но есть потолок и дорого.
   ГОРИЗОНТАЛЬНОЕ (scale out) — больше машин (реплики, шарды). масштабируется дальше, но сложнее.

   порядок масштабирования БД (от простого к сложному):
   1. ОПТИМИЗАЦИЯ (индексы, запросы, N+1) — модули 13-15. часто этого хватает!
   2. ВЕРТИКАЛЬНО (мощнее машина).
   3. КЭШИРОВАНИЕ (модуль 21) — снять нагрузку чтения.
   4. РЕПЛИКИ ДЛЯ ЧТЕНИЯ.
   5. ШАРДИНГ (если упёрся в запись/объём) — крайнее средство.
```

> 🧭 Это [системный дизайн] (масштабирование) и [Senior-trade-offs](../../Senior/02-decisions/08-tradeoffs.md):
> каждый шаг масштаба добавляет сложность — иди по лестнице, не прыгай к шардингу преждевременно.

## ⚠️ Ловушки

- ❌ Шардировать преждевременно (огромная сложность; сначала оптимизация/реплики).
- ❌ Игнорировать replication lag (читать с реплики сразу после записи на primary).
- ❌ Плохой shard key → перекос (горячие шарды) или частые кросс-шардовые запросы.
- ❌ Считать репликацию решением проблемы ЗАПИСИ (она для чтения; запись — на одном primary).
- ❌ Прыгать к горизонтальному масштабу до исчерпания оптимизации/вертикального.
- ❌ Кросс-шардовые транзакции/join'ы без понимания их цены.
