# 🏗️ Проект уровня 4 — Спроектируй и нагрузь БД реального сервиса

> 🎯 **Цель:** пройти полный цикл — от модели данных до масштабирования и защиты — на одном
> сквозном проекте. Это финальный проект трека: ты собираешь всё вместе.

---

## Что строим

Выбери сервис (на твой вкус): **мини-маркетплейс**, **соцсеть**, **трекер задач** или **бронирование**.
Спроектируй для него БД целиком и доведи до «production-ready» решений.

---

## Этап 1 — Модель данных (уровни 1-2)

- [ ] Спроектируй схему: таблицы, ключи (PK/FK), связи (1:N, M:N), типы столбцов.
- [ ] Нормализуй до 3НФ; отметь, где сознательно денормализуешь (и почему).
- [ ] Нарисуй ER-диаграмму (хоть в тексте/ASCII).
- [ ] Напиши DDL (CREATE TABLE с ограничениями: NOT NULL, UNIQUE, FK, CHECK).

## Этап 2 — Запросы (уровень 2)

- [ ] 8-10 реальных запросов сервиса (списки, фильтры, JOIN-ы, агрегаты, подзапросы/CTE).
- [ ] Хотя бы один сложный отчёт (GROUP BY + HAVING + несколько JOIN).
- [ ] INSERT/UPDATE/DELETE для основных операций.

## Этап 3 — Производительность (уровень 3)

- [ ] Залей тестовые данные (десятки-сотни тысяч строк — скриптом/генератором).
- [ ] Прогони ключевые запросы через **EXPLAIN**. Найди seq scan там, где нужен индекс.
- [ ] Добавь индексы под запросы. Замерь «до/после». Покажи, что ускорилось.
- [ ] Найди и устрани N+1 (если есть ORM/код доступа).
- [ ] Оберни критичную операцию (например, оформление заказа) в **транзакцию**. Объясни ACID-гарантии.

## Этап 4 — Масштаб и надёжность (уровень 4)

- [ ] Опиши стратегию масштабирования сервиса (лестница: оптимизация → кэш → реплики → шардинг).
- [ ] Добавь **кэш** для частого тяжёлого запроса (cache-aside + TTL). Опиши инвалидацию.
- [ ] Реши, где нужна **сильная** консистентность, где терпима **eventual** (CAP).
- [ ] Напиши **миграцию** (добавление поля) — UP/DOWN, обратно совместимую.
- [ ] Закрой **безопасность**: параметризованные запросы, роли с минимумом привилегий, хеш паролей,
      план бэкапа.

---

## 📦 Результат

Документ (README проекта) + SQL-скрипты:
1. **Схема** (DDL + ER-диаграмма + обоснование нормализации).
2. **Запросы** (10+ с комментариями зачем).
3. **Производительность** (EXPLAIN до/после, индексы, замеры).
4. **Транзакции** (критичная операция + разбор ACID).
5. **Масштаб/надёжность** (стратегия, кэш, CAP-решения, миграция, безопасность).

---

## 🌟 Усложнения (по желанию)

- Подними реальную СУБД в Docker (PostgreSQL), залей данные, прогони всё «по-настоящему».
- Настрой реплику (read replica) и направь чтения на неё.
- Подключи Redis как кэш и замерь эффект на нагрузке.
- Сравни план одного запроса в PostgreSQL и в другой СУБД.
- Добавь NoSQL-компонент (например, кэш/лог событий) и обоснуй гибридность.
