```
   утечка БД = катастрофа: пароли, персональные данные, финансы пользователей. БД — приоритетная
```

## ⭐⭐ SQL-инъекции — угроза №1

```
   SQL-ИНЪЕКЦИЯ — атакующий внедряет свой SQL через пользовательский ввод (модуль 12).
   ❌ УЯЗВИМО (склейка строк):
      query = "SELECT * FROM users WHERE name = '" + input + "'"
      ввод: ' OR '1'='1  →  вернёт ВСЕХ; хитрее → читать/менять/удалять данные.

   ✅ ЗАЩИТА — ПАРАМЕТРИЗОВАННЫЕ ЗАПРОСЫ (prepared statements):
      query = "SELECT * FROM users WHERE name = ?"  + передать input ОТДЕЛЬНО
      → ввод воспринимается как ДАННЫЕ, не код. инъекция невозможна.
   + ORM обычно параметризует сам (но не давай ему «сырые» куски с вводом).
   + валидация ввода, наименьшие привилегии (доп. слои).
```

💡 ⭐⭐ SQL-инъекция — [уязвимость №3 OWASP](../../Security/02-vulnerabilities/09-injection-sqli.md),
причина множества утечек. **Единственная надёжная защита — параметризованные запросы** (разделять
код SQL и данные пользователя). НИКОГДА не склеивай пользовательский ввод в текст запроса. Это
правило №1 безопасности при работе с БД.

## ⭐ Доступ и привилегии

```
   ✅ НАИМЕНЬШИЕ ПРИВИЛЕГИИ — пользователь БД приложения имеет ТОЛЬКО нужные права:
      • приложению обычно не нужны DROP TABLE, создание пользователей, доступ к чужим базам.
      • разные роли: приложение (CRUD на свои таблицы), миграции (DDL), аналитика (только чтение).
      → взломали приложение → ущерб ограничен (не снесут всю БД).
   ✅ СИЛЬНЫЕ ПАРОЛИ/КЛЮЧИ для доступа к БД; разные на окружения.
   ✅ НЕ хранить креды БД в коде (переменные окружения / менеджер секретов — модуль безопасности).
   ✅ ОГРАНИЧИТЬ СЕТЕВОЙ ДОСТУП — БД НЕ должна торчать в интернет; только приложение/нужные хосты
      (firewall, VPC). частая дыра — открытая наружу БД с дефолтным паролем (боты находят).
```

💡 ⭐ Принцип наименьших привилегий + закрытый сетевой доступ — фундамент. Открытая в интернет БД
с дефолтными кредами — классика взломов (боты сканируют постоянно). Приложение должно ходить в БД
с минимальными правами, по закрытой сети.

## ⭐⭐ Шифрование, бэкапы, аудит

```
   🔐 ШИФРОВАНИЕ:
   • В ПОКОЕ (at rest) — диск/файлы БД зашифрованы (если украдут диск/бэкап — данные защищены).
   • В ПЕРЕДАЧЕ (in transit) — соединение приложение↔БД по TLS (не открытым текстом).
   • ЧУВСТВИТЕЛЬНОЕ — пароли НЕ хранить (хешировать: bcrypt/argon2!); карты/секреты — шифровать/токенизировать.

   💾 БЭКАПЫ:
   • регулярные, ПРОВЕРЯЕМЫЕ (умеешь восстановиться!), хранить отдельно/офлайн (против шифровальщиков).
   • point-in-time recovery (WAL) — восстановиться на любой момент.
   • бэкап без проверки восстановления = иллюзия защиты.

   📋 АУДИТ:
   • логировать доступ/изменения чувствительных данных (кто что когда).
   • мониторинг аномалий (массовое чтение, доступ в нерабочее время).
```

💡 ⭐⭐ Полная защита БД — это слои: **параметризация** (инъекции) + **минимум привилегий** + **закрытая
сеть** + **шифрование** (диск/TLS/пароли-хеши) + **бэкапы** (проверенные!) + **аудит**. Пароли
пользователей — ОБЯЗАТЕЛЬНО хешировать (bcrypt/argon2, [не хранить открыто](../../Security/03-defensive-code/17-password-storage.md)).
Бэкапы без проверки восстановления — самая частая ложная защита.

## 📖 Связь с безопасностью приложения

```
   защита БД — часть общей безопасности (трек AppSec):
   • инъекции (модуль 09 трека безопасности).
   • контроль доступа к данным (IDOR — не отдать чужие данные, модуль 11 там).
   • защита секретов/кредов (модуль 19 там).
   • не раскрывать структуру БД в ошибках (общие сообщения пользователю).
```

## ⚠️ Ловушки

- ❌ Склейка ввода в SQL (инъекция) — используй параметризацию ВСЕГДА.
- ❌ БД с избыточными привилегиями приложения / дефолтными кредами.
- ❌ БД, открытая в интернет (сканируется ботами).
- ❌ Пароли пользователей открытым текстом/слабым хешем (нужен bcrypt/argon2).
- ❌ Креды БД в коде/репозитории.
- ❌ Бэкапы без проверки восстановления (или их отсутствие).

## ✅ Задачи на размышление

1. **Инъекция → защита.** Покажи уязвимый запрос (склейка) и параметризованный. Почему второй безопасен?
2. **Привилегии.** Спроектируй роли БД для приложения (CRUD), миграций (DDL), аналитики (чтение).
   Зачем разделять?
3. ⭐ **Сеть и секреты.** Как ограничить доступ к БД (сеть) и где хранить креды (не в коде)?
4. ⭐ **Бэкап-тест.** Опиши процесс бэкапа И восстановления. Почему важно тестировать восстановление?
5. **Пароли.** Как хранить пароли пользователей в БД? Почему не открытым текстом?
