# 22 · Миграции и эволюция схемы 🖼️⭐

> 🎯 **Цель блока:** научиться безопасно менять схему работающей БД с данными — через миграции,
> версионируемые и обратимые.

---

## 📖 Схема меняется, данные — нет

```
   приложение развивается → схема БД меняется (новые столбцы, таблицы, изменения типов).
   но в БД уже ЕСТЬ ДАННЫЕ и она РАБОТАЕТ (production). нельзя просто «пересоздать».
   МИГРАЦИИ — управляемый, версионируемый способ изменять схему пошагово, безопасно, обратимо.
```

💡 ⭐ Миграция — это как [git для схемы БД](../../Senior/03-practices/13-debugging-method.md):
последовательность версионированных изменений, применяемых по порядку, фиксируемых, обратимых. Без
миграций изменения схемы хаотичны (кто что поменял? как откатить? как синхронизировать dev/prod?).

---

## ⭐ Что такое миграция

```
   МИГРАЦИЯ — файл с изменением схемы (и/или данных), версионированный, в системе контроля версий.
   обычно две части:
   • UP — применить изменение (ALTER TABLE ADD COLUMN ...).
   • DOWN — откатить (ALTER TABLE DROP COLUMN ...) — для отмены.

   инструменты миграций: Flyway, Liquibase, Alembic (Python), встроенные в ORM (Django/Rails migrations).
   они ведут таблицу версий (какие миграции применены) → применяют новые по порядку, на всех окружениях
   (dev → stage → prod) одинаково.
```

🖼️
```
   миграции версионированы и применяются по порядку:
   v1: создать users  →  v2: добавить email  →  v3: создать orders  →  v4: добавить индекс
   каждое окружение (dev/stage/prod) применяет ТЕ ЖЕ миграции → схемы синхронны.
   таблица schema_migrations помнит, что уже применено.
```

---

## ⭐⭐ Безопасные миграции на production

```
   изменения схемы на РАБОТАЮЩЕЙ БД с данными — опасны (блокировки, поломка приложения). правила:

   ✅ ОБРАТНО СОВМЕСТИМО — миграция не должна ломать СТАРУЮ версию приложения (во время деплоя
      работают обе версии). добавляй, не ломай.
   ✅ ДОБАВЛЕНИЕ СТОЛБЦА: добавь NULLABLE или с DEFAULT (не сразу NOT NULL на большой таблице —
      блокировка/долго). потом заполни, потом сделай NOT NULL отдельной миграцией.
   ✅ ПЕРЕИМЕНОВАНИЕ/УДАЛЕНИЕ — многошагово: добавь новое → перенеси данные → переключи код → удали
      старое (не одним махом — сломаешь работающее приложение).
   ✅ БОЛЬШИЕ ТАБЛИЦЫ — ALTER может блокировать надолго (миллионы строк). используй приёмы без
      долгой блокировки (CONCURRENTLY для индексов в Postgres, батчами).
   ✅ ВСЕГДА БЭКАП перед миграцией; тестируй на копииprod-данных.
   ✅ DOWN-миграция или план отката.
```

💡 ⭐⭐ Главный принцип: **миграции обратно совместимы и многошаговы**. На production старая и новая
версии кода работают одновременно (rolling deploy) → миграция не должна сломать старую. «Удалить
столбец» = сначала перестать использовать в коде (деплой) → потом удалить (следующая миграция).
Резкие изменения на большой таблице блокируют БД → пользователи видят падение. Это [Senior-доведение
без простоев](../../Senior/03-practices/15-design-for-change.md).

## 📖 Миграции данных

```
   миграции меняют не только СТРУКТУРУ, но иногда и ДАННЫЕ (backfill):
   • добавил столбец full_name → заполни из first_name + last_name для существующих строк.
   • большие backfill'ы — БАТЧАМИ (по 1000 строк), не одним UPDATE на миллионы (блокировка/нагрузка).
   • отдели миграцию СХЕМЫ от миграции ДАННЫХ (структура быстро, данные постепенно).
```

## ⚠️ Ловушки

- ❌ Менять схему вручную на production (без миграций — не отследить/откатить/синхронизировать).
- ❌ Несовместимая миграция, ломающая работающую версию приложения во время деплоя.
- ❌ NOT NULL/изменение типа на большой таблице одним махом (блокировка → простой).
- ❌ Удалить/переименовать столбец одним шагом (сломать старый код).
- ❌ Backfill миллионов строк одним UPDATE (блокировка, нагрузка) — батчами.
- ❌ Без бэкапа / без тестирования на копии данных / без плана отката.
