# 01 · Как языки понимают друг друга: ABI 🖼️⭐

> 🎯 **Цель блока:** понять ABI — «физический контракт», по которому код на разных языках
> вызывает друг друга. И почему C стал универсальным переходником.

---

## 📖 Проблема: языки не знают друг о друге

Python ничего не знает о том, как устроены функции C. Rust не знает о Python. Чтобы
вызвать функцию из другого языка, нужен **общий низкоуровневый контракт** — как именно
передавать аргументы, возвращать значения, укладывать данные в памяти.

Этот контракт называется **ABI**.

---

## ⭐ ABI vs API

Ты уже знаешь **API** (из разделов «Проекты и API») — это контракт на уровне **исходного
кода**: какие функции есть, что принимают. **ABI** — это контракт на уровне **машинного
кода и байтов**:

| | **API** | **ABI** |
|--|---------|---------|
| Уровень | исходный код | машинный код / байты |
| Что определяет | имена функций, типы | как передаются аргументы, раскладка в памяти |
| Кто использует | программист | компилятор, линковщик, рантайм |
| Пример | `int add(int, int)` | аргументы в регистрах, результат в EAX |

🖼️
```
   API:  "функция add принимает два int и возвращает int"   (читает человек)
   ABI:  "первый int — в регистр EDI, второй — в ESI,
          результат вернётся в EAX, стек выровнен по 16 байт"  (исполняет процессор)
```

💡 **ABI** (Application Binary Interface) описывает: как передаются аргументы (регистры/
стек), как выглядит возврат, как уложены структуры в памяти, как именуются символы. Два
кода могут вызывать друг друга, только если **согласны об ABI**.

---

## ⭐⭐ Почему C — лингва-франка интеграции

У C очень **простой и стабильный ABI**, который стал **стандартом де-факто**. Почти каждый
язык умеет:
- вызывать функции с C-ABI;
- предоставлять свои функции с C-ABI наружу.

🖼️
```
        Python ──┐
        Rust   ──┤
        C++    ──┼──► C-ABI ◄──┼── (общий «переходник»)
        JS/Node──┤              │
        Go     ──┘              │
                  все говорят на «языке C» для связи
```

💡 C-ABI — это «английский язык» мира программирования: не всегда родной, но его знают все.
Поэтому интеграция языков почти всегда идёт **через C-ABI**, даже если ни одной строчки на
C ты не пишешь.

```c
// Это "C-ABI функция" — её сможет вызвать кто угодно:
int add(int a, int b) { return a + b; }
```

```rust
// Rust тоже умеет давать C-ABI наружу:
#[no_mangle]
pub extern "C" fn add(a: i32, b: i32) -> i32 { a + b }
//          └─ extern "C" = "используй C-ABI"
```

---

## 📖 Name mangling — почему нужен `extern "C"`

C++ и Rust «искажают» имена функций (name mangling) — кодируют типы в имя символа для
перегрузки. Из-за этого другие языки не находят функцию по простому имени.

```
   C++:   void add(int,int)  →  символ "_Z3addii"   (искажённое имя)
   C:     void add(int,int)  →  символ "add"        (как написано)
```

`extern "C"` отключает искажение → функция получает простое имя и C-ABI:

```cpp
extern "C" int add(int a, int b) { return a + b; }   // C++ отдаёт C-совместимо
```

💡 Поэтому при интеграции почти всегда пишут `extern "C"` (в C++/Rust) и `#[no_mangle]`
(в Rust) — чтобы функцию можно было найти и вызвать снаружи.

---

## 📖 Разные виды ABI (для общего понимания)

```
- Calling convention — как передавать аргументы (cdecl, stdcall…). Обычно решается сама.
- Платформенный ABI — отличается на Windows/Linux/macOS и x86/ARM.
- ABI рантайма — у Python/Java/.NET свой внутренний ABI (поэтому им нужен FFI-мост).
```

> 💡 Тебе редко нужно знать ABI в деталях — инструменты (компилятор, FFI-библиотеки)
> берут это на себя. Важно понимать **идею**: связь возможна, потому что есть общий
> бинарный контракт, и этот контракт — обычно C-ABI.
