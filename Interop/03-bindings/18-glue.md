# 18 · Клей: процессы, IPC, REST/gRPC 🖼️

> 🎯 **Цель блока:** освоить «свободную связь» — когда языки работают как отдельные
> программы и общаются через процессы и сеть. Часто это проще и надёжнее, чем FFI.

---

## 📖 Когда не нужен FFI

FFI (тесная связь в одном процессе) мощен, но сложен и опасен (память!). Часто **достаточно
свободной связи**: языки — отдельные программы, общаются сообщениями.

🖼️
```
   FFI:           один процесс, общая память  →  быстро, но риск ошибок памяти
   Свободная связь: разные процессы, обмен данными  →  медленнее, но ИЗОЛИРОВАНО и просто
```

💡 При свободной связи нет вопросов раскладки, владения, UB на границе — каждый язык
управляет **своей** памятью, а обмениваются они **копиями данных** (через текст/байты). Это
устраняет весь Уровень 2 проблем ценой скорости.

---

## ⭐ Способ 1: подпроцесс

Одна программа запускает другую и общается через ввод-вывод:

```python
import subprocess, json

# Python запускает Rust/C/любую программу, передаёт и получает данные
result = subprocess.run(
    ["./my_fast_program", "input_data"],
    capture_output=True, text=True
)
output = json.loads(result.stdout)   # программа вернула JSON
```

💡 Просто и надёжно: краш дочерней программы не валит родителя. Обмен через stdin/stdout,
аргументы, файлы. Часто данные сериализуют в **JSON** (универсальный формат).

```
✅ просто, изолированно, любые языки
⚠️ накладные расходы на запуск процесса (плохо для частых мелких вызовов)
```

---

## ⭐ Способ 2: REST API (HTTP)

Сервисы на разных языках общаются по HTTP (из разделов «Проекты и API» языковых треков):

🖼️
```
   [Python-сервис]  ◄──HTTP/JSON──►  [Rust-сервис]  ◄──HTTP──►  [JS-фронтенд]
   каждый — независимая программа, возможно на своей машине
```

💡 Это основа **микросервисов**: систему делят на сервисы, каждый на удобном для задачи
языке, общаются по REST. Так устроены крупные системы (разные команды, разные языки).

```
✅ языки полностью независимы, легко масштабировать, разные машины
⚠️ медленнее (сеть + сериализация), нужна инфраструктура
```

---

## ⭐ Способ 3: gRPC (быстрый типизированный RPC)

Когда REST медленноват, используют **gRPC** — быстрый бинарный протокол с описанием
контракта:

```protobuf
// contract.proto — описание сервиса (язык-независимое)
service Calculator {
    rpc Square(Number) returns (Number);
}
message Number { int64 value = 1; }
```

💡 Из этого `.proto` генерируется код для **многих языков** — Python, Rust, Go, JS, C++.
Каждый получает типобезопасный клиент/сервер. gRPC быстрее REST (бинарный, HTTP/2) и даёт
строгий контракт. Используется в больших распределённых системах.

---

## ⭐ Способ 4: IPC и очереди сообщений

```
   - Pipes / именованные каналы — поток данных между процессами на одной машине.
   - Сокеты (TCP/Unix) — обмен между процессами/машинами.
   - Очереди сообщений (RabbitMQ, Kafka, Redis) — асинхронный обмен, буферизация.
   - Разделяемая память — быстрый обмен большими данными (продвинуто).
```

💡 Очереди сообщений особенно хороши для асинхронных систем: один сервис кладёт задачу,
другой (на другом языке) её забирает. Полная развязка во времени и пространстве.

---

## 📋 Как выбрать: FFI или «клей»

```
   Нужна максимальная скорость, частые вызовы, общие данные  →  FFI (Уровни 1-2)
   Изоляция, надёжность, разные команды/машины              →  REST / gRPC / очереди
   Разовый запуск программы на другом языке                 →  подпроцесс
   Асинхронные потоки задач                                 →  очереди сообщений
```

🖼️ Шкала ещё раз:
```
   FFI ──── биндинги ──── WASM ──── встраивание ──── подпроцесс ──── REST/gRPC ──── очереди
   тесно, быстро, рискованно                          свободно, изолированно, надёжно
```

> 💡 **Важный вывод трека:** не всегда нужен FFI. Часто «скучная» связь через REST/процессы
> — правильный выбор: проще, надёжнее, без багов памяти. FFI бери, когда скорость и тесная
> интеграция действительно критичны.
