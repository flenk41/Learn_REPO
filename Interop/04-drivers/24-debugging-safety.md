# 24 · Отладка и безопасность системного кода 🛠️⭐

> 🎯 **Цель блока:** освоить инструменты и принципы безопасной разработки системного кода.
> Здесь цена ошибки максимальна, поэтому дисциплина важнее всего.

---

## ⚠️ Почему системный код — особый

```
   Обычная программа: ошибка → краш ОДНОЙ программы, перезапустил, поправил.
   Драйвер/ядро:      ошибка → крах ВСЕЙ системы (kernel panic / BSOD), потеря данных,
                      уязвимость в безопасности, «кирпич» из устройства.
```

💡 Ставки максимальны. Поэтому системные разработчики работают особенно осторожно: тестируют
в изоляции, используют специальные инструменты, проверяют каждую строчку с памятью.

---

## 🛡️ Принцип 1: изоляция — тестируй в виртуалке

> ⚠️ **Никогда не тестируй сырой kernel-код на рабочей машине.** Используй:
> - **виртуальную машину** (QEMU, VirtualBox) — упадёт VM, не твой ПК;
> - **контейнеры** где применимо;
> - **снапшоты** — откатиться после краха за секунды.

🖼️
```
   Твоя машина  →  [виртуалка с тестовым ядром]  →  тут можно безопасно крашить
   крах VM ≠ крах твоего ПК. Откатил снапшот — продолжил.
```

---

## 🛠️ Принцип 2: инструменты отладки ядра

```
   - dmesg / printk     — лог ядра (твой основной «вывод», модуль 21).
   - KASAN              — AddressSanitizer ДЛЯ ЯДРА: ловит ошибки памяти в kernel-коде!
   - KGDB               — отладчик ядра (пошагово, точки останова).
   - ftrace / perf      — трассировка и профилирование ядра.
   - panic logs / crash — анализ дампов после краха.
```

💡 **KASAN** — особенно ценен: как ASan ловит ошибки памяти в обычном C
([C-курс](../../C/02-memory/12-memory-errors-tools.md)), KASAN ловит их **в ядре**.
Собирай тестовое ядро с KASAN — он найдёт use-after-free и overflow до того, как они уронят
систему в продакшене.

---

## 🛡️ Принцип 3: дисциплина памяти (всё из курса сходится)

Системный код = ручная память без страховки. Применяй **всё**, что учил:

```
   ✅ kmalloc парен с kfree (как malloc/free, C-трек).
   ✅ Проверяй каждый указатель на NULL.
   ✅ copy_to/from_user для границы user↔kernel (модуль 22) — никогда не разыменовывай
      userspace-указатель напрямую.
   ✅ volatile/ioread/iowrite для регистров устройств.
   ✅ Следи за временем жизни и владением (как в Rust/interop).
   ✅ Проверяй размеры буферов (overflow в ядре = уязвимость).
   ✅ Освобождай ВСЕ ресурсы в exit-функции (утечка живёт до перезагрузки).
```

💡 Здесь окупается **весь курс**: память из C, RAII из C++, владение из Rust, граница из
interop — всё применяется на самом опасном уровне.

---

## 🛡️ Принцип 4: Rust как защита

Как мы видели (модуль 23), **Rust устраняет целые классы ошибок на этапе компиляции**.
Где можно — используй Rust для системного кода: borrow checker не даст написать
use-after-free, который в C прошёл бы до рантайма.

```
   C:    надейся на дисциплину + KASAN (поймает в рантайме).
   Rust: компилятор НЕ ДАЁТ написать небезопасный код (поймано до запуска).
```

---

## 🛡️ Принцип 5: безопасность (security)

Драйверы — мишень для атак (полный доступ к системе). Учитывай:
```
   ⚠️ Проверяй ВСЕ данные из userspace (не доверяй размерам, указателям, значениям).
   ⚠️ Buffer overflow в драйвере = захват системы атакующим.
   ⚠️ Integer overflow при расчёте размеров → переполнение буфера.
   ⚠️ Гонки данных (race conditions) в ядре → уязвимости.
   ⚠️ Минимизируй привилегии и поверхность атаки.
```

💡 Системный код должен быть не только корректным, но и **устойчивым к злонамеренным
данным**. Каждое значение из userspace — потенциально враждебное.

   ✅ Освобождаю все ресурсы при выгрузке.
