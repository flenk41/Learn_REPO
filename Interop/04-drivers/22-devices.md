# 22 · Работа с устройствами 🖼️⭐

> 🎯 **Цель блока:** понять, как драйвер реально общается с железом — через память
> устройства, файлы `/dev` и команды. Здесь тема памяти курса встречает физическое железо.

---

## ⭐ Устройства как файлы (Unix-философия)

В Linux/Unix действует принцип **«всё — это файл»**. Устройства представлены файлами в `/dev`:

```bash
/dev/sda        # диск
/dev/random     # генератор случайных чисел
/dev/null       # «чёрная дыра»
/dev/ttyUSB0    # последовательный порт
/dev/mydevice   # твоё устройство (из драйвера, модуль 21)
```

🖼️
```
   Пользователь:  cat /dev/mydevice     →  ОС  →  драйвер.read()  →  устройство
                  echo "x" > /dev/mydevice → ОС → драйвер.write() → устройство
```

💡 Гениально: пользователь работает с устройством обычными файловыми операциями (read/
write/open), а драйвер переводит их в работу с железом. Программам не нужно знать детали.

---

## ⭐ file_operations — сердце символьного драйвера

Драйвер регистрирует функции, вызываемые при операциях с его файлом:

```c
static ssize_t my_read(struct file* f, char __user* buf, size_t len, loff_t* off) {
    // отдать данные пользователю
    copy_to_user(buf, kernel_data, n);   // КОПИРОВАТЬ из ядра в userspace
    return n;
}
static ssize_t my_write(struct file* f, const char __user* buf, size_t len, loff_t* off) {
    copy_from_user(kernel_buf, buf, len); // КОПИРОВАТЬ из userspace в ядро
    return len;
}

static struct file_operations fops = {
    .read = my_read,
    .write = my_write,
    .open = my_open,
    .release = my_release,
};
```

> ⚠️ Нельзя просто разыменовать указатель из userspace в ядре! Память user space и kernel
> space разделена (модуль 20). Поэтому **обязательно** `copy_to_user`/`copy_from_user` —
> они безопасно переносят данные через границу user↔kernel. Это та же «граница памяти»
> (Уровень 2), но между процессом и ядром.

## ⭐ Memory-mapped I/O — разговор с железом через адреса

Как драйвер управляет физическим устройством? Часто через **memory-mapped I/O (MMIO)**:
регистры устройства «отображены» на адреса памяти. Запись по адресу = команда устройству.

🖼️
```
   Адресное пространство:
   ...
   [0xFE000000] ← регистр устройства "включить"     запись 1 сюда → устройство включается
   [0xFE000004] ← регистр "данные"                  чтение отсюда → данные от устройства
   ...
   Драйвер пишет/читает эти адреса = управляет железом
```

```c
void __iomem* regs = ioremap(device_phys_addr, size);  // отобразить регистры
iowrite32(1, regs + ENABLE_OFFSET);                     // включить устройство
uint32_t status = ioread32(regs + STATUS_OFFSET);       // прочитать статус
iounmap(regs);
```

💡 Память железа выглядит как обычные адреса, но за ними — **физическое устройство**.
Запись числа в «регистр» физически меняет состояние железа (зажигает светодиод, отправляет
байт). Здесь указатели из [C-курса](../../C/02-memory/09-pointers.md) управляют реальным
миром.

> ⚠️ MMIO требует `volatile`/специальных функций (`ioread/iowrite`): компилятор не должен
> «оптимизировать» обращения к регистрам, ведь их значение меняет железо, а не программа.
> Помнишь `volatile` из [C-курса](../../C/04-senior/20-alignment-ub.md)? Вот его настоящее
> применение.

## 📖 ioctl — команды устройству

Read/write подходят для потока данных, но устройствам нужны и **команды** («перемотай»,
«установи скорость»). Для этого — `ioctl`:

```c
// userspace
ioctl(fd, MYDEVICE_SET_SPEED, 9600);   // отправить команду драйверу
```
```c
// драйвер
static long my_ioctl(struct file* f, unsigned int cmd, unsigned long arg) {
    switch (cmd) {
        case MYDEVICE_SET_SPEED: /* настроить устройство */ break;
    }
}
```

💡 `ioctl` — «универсальная команда» для всего, что не вписывается в read/write. Так
настраивают порты, управляют GPU, конфигурируют устройства.

## 📖 Прерывания — устройство зовёт драйвер

Устройство сообщает о событии (пришли данные, нажата клавиша) через **прерывание** —
сигнал процессору. Драйвер регистрирует **обработчик прерывания**:

```c
request_irq(irq_number, my_handler, ...);   // зарегистрировать обработчик

irqreturn_t my_handler(int irq, void* dev) {
    // быстро обработать событие от устройства
    return IRQ_HANDLED;
}
```

💡 Прерывания позволяют не «опрашивать» устройство постоянно, а реагировать на события.
Обработчик должен быть **очень быстрым** (он прерывает всё остальное).
