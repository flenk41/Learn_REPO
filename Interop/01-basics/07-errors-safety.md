# 07 · Ошибки и безопасность на границе 🖼️

> 🎯 **Цель блока:** научиться обрабатывать ошибки через границу и писать безопасные FFI-
> вызовы. На стыке языков пропадают привычные защиты — нужно компенсировать вручную.

---

## ⚠️ На границе теряются защиты языка

```
   Python: исключения, проверки границ, безопасные типы
        ▼ ГРАНИЦА ▼
   C: ничего этого нет — сырые указатели, никаких проверок, краш вместо исключения
```

> ⚠️ Когда Python зовёт C, ты выходишь из «безопасной зоны». Ошибка в C **обрушит весь
> процесс** (включая Python) — не будет аккуратного исключения, будет segfault. Защиту
> нужно строить самому.

---

## ⭐ Как C сообщает об ошибках (у него нет исключений)

C не умеет бросать исключения. Способы сообщить об ошибке:

```c
// 1. Код возврата
int do_work(int x);           // 0 = успех, иначе код ошибки

// 2. Специальное значение
void* make_thing();           // NULL = ошибка

// 3. Через выходной параметр
int parse(const char* s, int* out);   // return = успех, out = результат
```

В Python проверяй явно:
```python
result = lib.do_work(5)
if result != 0:
    raise RuntimeError(f"C-функция вернула ошибку {result}")

ptr = lib.make_thing()
if not ptr:                   # NULL → ошибка
    raise MemoryError("не удалось создать объект")
```

💡 C-ошибки = значения (как `Result` в [Rust](../../Rust/03-middle/15-error-handling.md), но
вручную). На границе ты **переводишь** C-коды ошибок в исключения своего языка.

---

## ⚠️ Главные опасности FFI

```
⚠️ Неверная сигнатура  →  данные искажаются, краш
⚠️ Выход за границы массива в C  →  порча памяти, краш
⚠️ Разыменование NULL  →  segfault (рушит весь процесс)
⚠️ Неверное владение памятью  →  утечки, use-after-free, double-free (модуль 06)
⚠️ Несовпадение раскладки структур  →  чтение мусора (модуль 11)
⚠️ Передача неверного размера  →  чтение/запись за пределы
⚠️ C хранит указатель дольше жизни данных  →  висячий указатель
```

💡 Все эти ошибки — про **память** и **типы** на стыке. Защита: точные сигнатуры, передача
размеров, проверка NULL и кодов возврата, дисциплина владения.

---

## ⭐ Принципы безопасного FFI

```
1. ✅ Объявляй ТОЧНЫЕ сигнатуры (argtypes/restype).
2. ✅ Передавай размеры буферов, никогда не доверяй «C сам разберётся».
3. ✅ Проверяй каждый код возврата и NULL.
4. ✅ Чёткое владение: кто выделил — тот освобождает (модуль 06).
5. ✅ Оборачивай сырой FFI в безопасный слой (см. ниже).
6. ✅ Тестируй C-часть отдельно (с ASan), прежде чем звать из Python.
```

---

## ⭐ Безопасная обёртка — главный приём

Не разбрасывай сырые FFI-вызовы по коду. Спрячь их за **безопасным интерфейсом** (как
[API-дизайн](../../Python/03b-projects-api/02-designing-api.md) из языковых треков):

```python
import ctypes

class SafeMath:
    """Безопасная Python-обёртка над C-библиотекой."""
    def __init__(self, lib_path="./mathlib.so"):
        self._lib = ctypes.CDLL(lib_path)
        self._lib.divide.argtypes = [ctypes.c_double, ctypes.c_double,
                                     ctypes.POINTER(ctypes.c_double)]
        self._lib.divide.restype = ctypes.c_int

    def divide(self, a: float, b: float) -> float:
        out = ctypes.c_double()
        code = self._lib.divide(a, b, ctypes.byref(out))
        if code != 0:                          # C сообщил об ошибке
            raise ZeroDivisionError("деление на ноль в C")
        return out.value                       # переводим в нормальный Python
```

🖼️
```
   Пользователь  →  SafeMath.divide(10, 0)  →  [обёртка проверяет, бросает исключение]
                    видит привычный Python: чистые типы, исключения, никаких сырых указателей
```

💡 Внутри — опасный FFI; снаружи — чистый безопасный Python. Так устроены **все** хорошие
обёртки над C (NumPy, requests-зависимости и т.д.). Это объединяет interop с дизайном API.

## 📖 Отладка через границу

> ⚠️ Краш в FFI трудно отлаживать: Python-трейсбек обрывается на границе, а где упало в C —
> непонятно. Приёмы:
> - тестируй C-функции отдельной C-программой под **AddressSanitizer**
>   (`-fsanitize=address`) — поймает ошибки памяти;
> - проверяй сигнатуры дважды;
> - добавляй отладочный вывод в C;
> - используй `gdb` для нативного краша (см. [C-курс, инструменты](../../C/02-memory/12-memory-errors-tools.md)).
