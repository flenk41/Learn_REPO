# 06 · Кто владеет памятью 🖼️⭐⭐

> 🎯 **Цель блока:** разобраться с самым опасным вопросом интеграции — **кто выделяет и кто
> освобождает память** через границу языков. Это причина 90% багов в FFI.

---

## ⚠️ Главная опасность: память, выделенную в одном языке, нельзя освобождать в другом

```
   C выделил память через malloc  →  освобождать ТОЛЬКО через C-free
   Python создал объект           →  освобождает ТОЛЬКО сборщик мусора Python
   Rust владеет Box               →  освобождает ТОЛЬКО Rust
```

> ⚠️ Смешаешь — катастрофа: `free()` чужой памяти = краш или порча кучи. Каждый язык
> использует **свой аллокатор** (вспомни аллокаторы из [C-курса](../../C/04-senior/19-allocators.md)),
> и они несовместимы.

🖼️
```
   C malloc   ──► [память от C-аллокатора]
   Python GC  ✗ пытается освободить → ☠️ краш (чужой аллокатор!)
```

---

## ⭐ Главное правило: кто выделил — тот и освобождает

```
   ✅ C выделил  →  C предоставляет функцию free_*() для освобождения
   ✅ Python выделил буфер  →  Python и освобождает (GC)
   ✅ Передаёшь данные C на время вызова  →  C НЕ должен их хранить дольше
```

🖼️
```
   Правильно:
   char* s = make_string();    // C выделил
   use_string(s);
   free_string(s);             // C ЖЕ и освободил (своей функцией)
```

💡 Хорошая C-библиотека для каждого «создать» даёт парный «освободить» (как
create/destroy из [C++ RAII](../../Cpp/02-memory/10-raii.md)). Пользователь обязан вызвать
именно эту функцию, а не `free` своего языка.

---

## ⭐ Три стратегии владения на границе

### 1. Вызывающий выделяет буфер (caller-allocated)
Самый безопасный: **Python выделяет** буфер, C только заполняет.
```python
buf = ctypes.create_string_buffer(256)   # Python владеет буфером
lib.fill_buffer(buf, 256)                 # C пишет в него
result = buf.value                        # Python читает
# Python сам освободит буфер (GC) — никакой путаницы
```
💡 Память от начала до конца принадлежит Python. C только пишет в чужой буфер. Нет вопроса
«кто освобождает».

### 2. Библиотека выделяет + даёт функцию освобождения
```python
lib.make_data.restype = ctypes.c_void_p
ptr = lib.make_data()        # C выделил
# ... используем ...
lib.free_data(ptr)           # ОБЯЗАТЕЛЬНО освободить функцией C, не Python!
```
> ⚠️ Забудешь `lib.free_data` → утечка. Освободишь через Python → краш.

### 3. Передача владения (ownership transfer)
Один язык **отдаёт** владение другому. Сложно и опасно — нужно, чтобы принимающая сторона
знала, как правильно освободить. Часто избегают.

---

## 🖼️ Жизненный цикл памяти через границу

```
   Кто выделил? ──┬── вызывающий (Python)  →  вызывающий и освобождает  ✅ просто
                  │
                  └── библиотека (C)        →  библиотека даёт free-функцию  ⚠️ не забудь вызвать
```

💡 Предпочитай стратегию 1 (вызывающий выделяет) — она убирает путаницу. Если библиотека
выделяет — всегда ищи и вызывай её функцию освобождения.

---

## ⚠️ Висячие указатели через границу

```python
lib.make_data.restype = ctypes.c_void_p
ptr = lib.make_data()
lib.free_data(ptr)
lib.use_data(ptr)        # ⚠️ use-after-free! память уже освобождена → краш
```

💡 Те же ошибки, что в [C](../../C/02-memory/11-dynamic-memory.md) (use-after-free,
double-free), но теперь **через границу языков** — их ещё труднее отлаживать, потому что
краш в C, а причина — в Python-коде.

---

## 📖 Время жизни и «не храни дольше вызова»

```python
data = [1.0, 2.0, 3.0]
arr = (ctypes.c_double * 3)(*data)
lib.start_processing(arr)    # ⚠️ если C СОХРАНИТ указатель на arr и будет
                             #    использовать ПОЗЖЕ — а Python соберёт arr мусором → краш
```

> ⚠️ Если передал C указатель «на время вызова», C не должен хранить его и использовать
> после. Иначе Python-сборщик удалит данные, а C будет работать с висячим указателем.
> Держи переданные данные живыми, пока C ими пользуется.

## ✅ Задачи

1. **Caller-allocated.** Сделай C-функцию, заполняющую переданный буфер; выдели буфер в
   Python, прочитай результат. Убедись, что освобождать вручную не нужно.
2. **Library-allocated.** Сделай C-функцию `make_data`/`free_data`; вызови, используй,
   освободи правильной функцией.
3. **Поймай use-after-free.** Освободи данные, потом используй — увидь краш (под ASan, если
   соберёшь C с `-fsanitize=address`).
4. **Время жизни.** Воспроизведи ситуацию, где Python собирает данные, нужные C (намеренно).
5. ⭐ **Парный API.** Спроектируй мини-C-библиотеку с чётким владением: на каждый `*_new`
   есть `*_free`, документируй, кто за что отвечает.
