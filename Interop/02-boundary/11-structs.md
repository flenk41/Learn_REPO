# 11 · Структуры, выравнивание, указатели 🖼️⭐

> 🎯 **Цель блока:** научиться безопасно передавать структуры через границу — самое частое
> место рассогласования раскладки памяти.

---

## 📖 Структура должна совпадать байт-в-байт

Чтобы оба языка читали структуру одинаково, их описания должны давать **идентичную
раскладку**: те же типы, тот же порядок, то же выравнивание.

🖼️
```
   C struct:      [a: int][b: double][c: char]
   Python должен описать ТОЧНО ТАК ЖЕ, иначе поля прочитаются не с тех байтов
```

```c
// C
typedef struct {
    int32_t  id;
    double   value;
    uint8_t  flag;
} Record;
```
```python
# Python — точная копия!
class Record(ctypes.Structure):
    _fields_ = [
        ("id",    ctypes.c_int32),
        ("value", ctypes.c_double),
        ("flag",  ctypes.c_uint8),
    ]
```

> ⚠️ Порядок полей и типы должны **совпадать**. Перепутал порядок или тип — прочитаешь
> мусор. Используй типы фиксированного размера (модуль 08).

## ⭐ Выравнивание и padding в структурах

Помнишь padding ([C-курс](../../C/04-senior/20-alignment-ub.md))? Компилятор вставляет
«дырки», чтобы поля были выровнены. Оба языка должны вставить **те же** дырки.

🖼️
```
   typedef struct { uint8_t a; int32_t b; } S;
   адрес:  0    1  2  3   4   5   6   7
          [a][ padding ][    b      ]      ← 3 байта дырки после a (b выровнен по 4)
   sizeof = 8, не 5
```

💡 ctypes по умолчанию применяет **стандартное C-выравнивание** — поэтому обычно совпадает
автоматически. Проверь: `ctypes.sizeof(S)` должно равняться `sizeof(S)` в C.

## ⚠️ Упакованные структуры (#pragma pack)

Если C-код **упаковывает** структуру (убирает padding) — нужно сказать об этом и Python:

```c
#pragma pack(1)              // без padding! поля впритык
typedef struct { uint8_t a; int32_t b; } Packed;   // sizeof = 5
#pragma pack()
```
```python
class Packed(ctypes.Structure):
    _pack_ = 1               # тоже без padding!
    _fields_ = [("a", ctypes.c_uint8), ("b", ctypes.c_int32)]
```

> ⚠️ Если C упаковал, а Python нет (или наоборот) — раскладка разойдётся, поля прочитаются
> не там. **Упаковка должна совпадать.** Поэтому `#pragma pack` на границе — частый
> источник багов; используй его осознанно.

## ⭐ Структуры по значению vs по указателю

```c
double area_by_value(Rectangle r);          // копия структуры
double area_by_pointer(const Rectangle* r); // указатель (без копии)
```
```python
rect = Rectangle(3, 4)
lib.area_by_pointer.argtypes = [ctypes.POINTER(Rectangle)]
lib.area_by_pointer(ctypes.byref(rect))     # передаём указатель
```

🖼️
```
   По значению:   структура КОПИРУЕТСЯ через границу (безопасно, но дороже для больших)
   По указателю:  передаётся адрес (быстро; C может ИЗМЕНИТЬ структуру; следи за временем жизни)
```

💡 Маленькие структуры — по значению (просто). Большие или изменяемые — по указателю. Если
по указателю и C изменяет — читаешь изменения обратно из того же объекта.

## ⚠️ Указатели ВНУТРИ структур — особая осторожность

```c
typedef struct {
    char*  name;     // указатель внутри структуры!
    int    age;
} Person;
```

> ⚠️ Структура с указателем внутри — двойная сложность владения: кто владеет памятью, на
> которую указывает `name`? Когда структуру передают через границу, нужно решить судьбу и
> самой структуры, и того, на что она ссылается. Это частый источник утечек и висячих
> указателей. По возможности избегай указателей внутри структур на границе — или строго
> документируй владение (модуль 09).

## ⭐ Вложенные структуры и массивы в структурах

```c
typedef struct {
    int32_t x, y;
} Point;
typedef struct {
    Point    corners[4];     // массив структур внутри структуры
    int32_t  count;
} Polygon;
```
```python
class Point(ctypes.Structure):
    _fields_ = [("x", ctypes.c_int32), ("y", ctypes.c_int32)]
class Polygon(ctypes.Structure):
    _fields_ = [("corners", Point * 4), ("count", ctypes.c_int32)]
```

💡 Вложенность работает, но усложняет раскладку. Сверяй `sizeof` обеих сторон.
