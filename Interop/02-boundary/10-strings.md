# 10 · Строки и кодировки между языками 🖼️⭐

> 🎯 **Цель блока:** освоить самую коварную тему интеграции — строки. У разных языков они
> устроены совершенно по-разному, и на границе это рождает множество багов.

---

## 📖 Почему строки так сложны на границе

Каждый язык хранит строки по-своему:

| Язык | Как хранит строку |
|------|-------------------|
| **C** | массив байт + `\0` в конце (нуль-терминатор), длина не хранится |
| **Python** | объект Unicode + длина (знает свою длину) |
| **Rust** | `String`/`&str` — байты UTF-8 + длина, **без** `\0` |
| **JS** | UTF-16 объект |

🖼️
```
   C:      [п][р][и][в][е][т][\0]        ← длина = «ищи \0»
   Python: объект{ "привет", len=6 }     ← длина известна, Unicode
   Rust:   [байты UTF-8] + длина          ← без \0
```

💡 Три разных мира! На границе строку нужно **перевести** из одного представления в другое.

---

## ⚠️ Три ловушки строк

### 1. Кодировка: байты vs Unicode
C работает с **байтами**, Python — с **Unicode-символами**. Перевод обязателен:
```python
# Python str → C (байты)
s = "привет"
c_bytes = s.encode("utf-8")        # str → bytes (обязательно!)
lib.process.argtypes = [ctypes.c_char_p]
lib.process(c_bytes)

# C → Python str
lib.get_text.restype = ctypes.c_char_p
result = lib.get_text().decode("utf-8")   # bytes → str
```

> ⚠️ Забыл `encode`/`decode` → ошибка типов или искажённый текст. Кириллица/эмодзи без
> правильной кодировки превратятся в мусор.

### 2. Нуль-терминатор
```
   C ищет конец строки по \0. Если \0 нет — C читает ЗА пределами строки → мусор/краш.
   Если в данных есть \0 в середине — C обрежет строку на нём.
```
💡 `c_char_p` в ctypes сам добавляет `\0` при передаче из bytes. Но при работе с «сырыми»
данными (где могут быть нулевые байты) используй буфер с явной длиной, а не строку.

### 3. Кто владеет строкой
```python
lib.make_string.restype = ctypes.c_char_p
s = lib.make_string()          # C вернул строку. Кто её освободит?
```
> ⚠️ Если C **выделил** строку — кто её освобождает? (модуль 09). Часто C-функции
> возвращают указатель на **внутренний** буфер (нельзя освобождать, нельзя хранить долго)
> или на **выделенную** память (нужно освободить функцией C). Читай документацию функции!

## ⭐ Безопасный приём: вызывающий даёт буфер

Лучший способ получить строку из C — **Python выделяет буфер**, C заполняет:

```c
// C: пишет в переданный буфер, возвращает реальную длину
int get_message(char* buf, int buf_size);
```
```python
buf = ctypes.create_string_buffer(256)    # Python владеет буфером
lib.get_message.argtypes = [ctypes.c_char_p, ctypes.c_int]
length = lib.get_message(buf, 256)
text = buf.value.decode("utf-8")          # читаем результат
# Python сам освободит буфер — никаких вопросов владения
```

💡 Это модель «вызывающий владеет» (модуль 09) для строк. Нет путаницы с освобождением,
нет висячих указателей. C **обязан** уважать размер буфера (не писать больше `buf_size`).

> ⚠️ C должен проверять размер буфера, иначе — переполнение (классическая уязвимость).
> Безопасные C-функции работы со строками (`snprintf`, не `sprintf`) важны вдвойне на границе.

## 📖 UTF-8 как общий язык строк

```
   ✅ Договорись использовать UTF-8 ВЕЗДЕ на границе:
      - Python: .encode("utf-8") / .decode("utf-8")
      - Rust:   String уже UTF-8 (удобно!)
      - C:      работай с байтами UTF-8
```

💡 UTF-8 — фактический стандарт. Если все стороны используют UTF-8, остаётся следить только
за `\0` и владением. Rust здесь удобен — его строки **уже** UTF-8.

## 📖 Передача массива строк

Массив строк — это «указатель на указатели» (`char**`):
```c
void process_lines(const char** lines, int count);
```
```python
lines = [b"first", b"second", b"third"]
arr = (ctypes.c_char_p * len(lines))(*lines)
lib.process_lines(arr, len(lines))
```
💡 Сложнее обычного массива (два уровня указателей). Следи за временем жизни всех строк.
