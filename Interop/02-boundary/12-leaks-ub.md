# 12 · Утечки и UB на границе 🛠️⭐

> 🎯 **Цель блока:** научиться находить и предотвращать ошибки памяти на стыке языков.
> Здесь они особенно коварны: причина в одном языке, краш в другом.

---

## ⚠️ Почему баги на границе так трудно ловить

```
   Обычный баг:  ошибка и краш в ОДНОМ языке → понятный трейсбек.
   Баг на границе: ошибка в Python-коде → краш в C → НЕТ трейсбека Python,
                   только "Segmentation fault". Где причина — неясно.
```

🖼️
```
   Python: lib.process(arr, wrong_size)   ← причина здесь (неверный размер)
        ▼ граница ▼
   C: читает за пределами arr  →  💥 segfault   ← краш здесь, но виноват Python
```

💡 Граница «глотает» информацию об ошибке. Поэтому нужны специальные инструменты и
дисциплина.

---

## ⚠️ Каталог ошибок памяти на границе

```
💧 Утечка:           C/библиотека выделила, никто не освободил
☠️ Double-free:      освободили дважды (часто из-за обёрток)
💀 Use-after-free:   использовали после освобождения (через границу!)
🔫 Buffer overflow:  C вышел за переданный массив/буфер
🎭 Type mismatch:    неверная сигнатура → данные искажены
📐 Layout mismatch:  раскладка структур разошлась (модуль 11)
⏳ Lifetime bug:     язык-владелец удалил данные, пока другой ими пользуется
🚫 Wrong allocator:  освободили чужим аллокатором
```

💡 Все они — комбинация ошибок памяти из [C-курса](../../C/02-memory/12-memory-errors-tools.md)
и рассогласования между языками. Знание памяти из языковых треков здесь окупается полностью.

---

## 🛠️ Инструмент 1: AddressSanitizer для C-части

Главное правило: **тестируй C-код отдельно** под ASan, **до** интеграции.

```bash
# собрать C-библиотеку/тест с ASan
gcc -fsanitize=address -g test_lib.c lib.c -o test_lib
./test_lib
```

💡 Напиши маленькую C-программу, которая вызывает твои функции напрямую, прогони под ASan.
Он поймает выход за границы, утечки, use-after-free **в C** — там, где их проще исправить.
Только потом подключай к Python.

> ⚠️ Запуск Python с C-библиотекой под ASan возможен, но сложнее (нужно подгрузить ASan-
> рантайм). Проще изолировать и проверить C-часть отдельно.

---

## 🛠️ Инструмент 2: Valgrind

```bash
valgrind --leak-check=full python main.py
```

💡 Valgrind может прогнать **весь** процесс (Python + C) и найти утечки/ошибки в нативной
части. Выдаёт много «шума» от самого Python — фильтруй, ищи свою библиотеку в стеках.

---

## 🛠️ Инструмент 3: дисциплина и обёртки

Лучший «инструмент» — не допускать ошибок:
```
✅ Точные сигнатуры (argtypes/restype) — против type mismatch.
✅ Передача размеров везде — против buffer overflow.
✅ Сверка sizeof структур — против layout mismatch.
✅ Чёткое владение + RAII-обёртки (модуль 09) — против утечек/double-free/UAF.
✅ Проверка NULL и кодов возврата (модуль 07).
✅ Держи переданные данные живыми, пока C ими пользуется — против lifetime-багов.
```

---

## ⭐ Стратегия «безопасное ядро, опасная граница тонкая»

🖼️
```
   ┌─ Python (безопасно) ─┐  ┌─ тонкий FFI-слой ─┐  ┌─ C (тестирован ASan) ─┐
   │  обычный код          │  │ ВСЕ опасные вызовы │  │  проверенный нативный  │
   │                       │  │ собраны здесь      │  │  код                   │
   └───────────────────────┘  └────────────────────┘  └────────────────────────┘
                               ▲ единственное место, где можно ошибиться с памятью
```

💡 Сделай **тонкий, хорошо протестированный FFI-слой**, и держи всю опасную работу с
указателями **только в нём**. Остальной код — безопасный. Меньше места для ошибок = легче
отлаживать.

## 🧪 Эксперимент: воспроизведи и поймай

Собери C с ASan и воспроизведи каждую ошибку, наблюдая отчёт:
```c
// 1. выход за границы
void bug1(int* arr) { arr[100] = 0; }       // передай маленький массив
// 2. утечка
void* bug2() { return malloc(100); }         // не освобождай
// 3. use-after-free
void bug3(int* p) { free(p); *p = 5; }
```

## ✅ Задачи

1. **Изолированный тест.** Напиши C-программу, тестирующую твои FFI-функции, прогони под ASan.
2. **Поймай overflow.** Передай из Python неверный размер, воспроизведи выход за границы,
   поймай ASan-ом.
3. **Поймай утечку.** Создай C-память без освобождения, найди через Valgrind на Python-скрипте.
4. **Type mismatch.** Сделай неверную сигнатуру, увидь искажение/краш, объясни.
5. **Тонкий FFI-слой.** Перепиши проект так, чтобы ВСЕ сырые вызовы были в одном модуле-
   обёртке.
6. ⭐ **Полный аудит.** Возьми свою связку Python+C, прогони C под ASan, процесс под
   Valgrind, исправь всё найденное.
