# 🦀 Драйверы на Rust

> Rust приходит в драйверы ради **безопасности памяти** там, где баг = kernel panic или дыра. Три
> направления: **Rust for Linux** (модули ядра), **embedded Rust** (сильнейшая сторона), userspace.

> 🧭 Концепции — [Интеграция · Rust/embedded](../Interop/04-drivers/23-rust-embedded.md).
> ⚠️ Те же правила: ядро — в виртуалке, железо — на учебной плате.

---

## ⭐⭐ Зачем Rust в драйверах

```
   драйверы (особенно ядра) — это код с ПОЛНЫМ доступом, где баг катастрофичен:
   • use-after-free, переполнение буфера, гонки данных в ядре = kernel panic + уязвимость.
   • большая доля CVE в ядрах — именно эти классы багов памяти.

   Rust ловит их НА КОМПИЛЯЦИИ (владение/заимствование, без GC):
   • нет use-after-free, нет двойного free, нет гонок данных — гарантировано компилятором.
   • та же скорость, что C (zero-cost), без рантайма/сборщика мусора.
   → идеально для кода, где «упасть нельзя».
```

💡 ⭐⭐ Главная причина Rust в драйверах — **безопасность памяти без потери скорости**. В ядре и на
железе цена ошибки максимальна, а Rust устраняет целые классы багов на этапе компиляции. Это та же
тема [владения/заимствования](../Rust/02-memory/10-borrow-checker.md), но применённая там, где она
ценнее всего.

## ⭐ Rust for Linux (модули ядра)

```
   Rust for Linux — официальная поддержка написания модулей ядра Linux на Rust (вошла в mainline,
   ядро 6.1+, активно растёт). пишешь .ko на Rust с безопасными обёртками над ядровыми API.
```

```rust
   // упрощённый скелет модуля ядра на Rust (kernel crate):
   use kernel::prelude::*;

   module! {
       type: MyDriver,
       name: "my_driver",
       author: "Ты",
       description: "Учебный драйвер на Rust",
       license: "GPL",
   }

   struct MyDriver;

   impl kernel::Module for MyDriver {
       fn init(_module: &'static ThisModule) -> Result<Self> {
           pr_info!("my_driver: загружен\n");   // аналог printk
           Ok(MyDriver)
       }
   }
   impl Drop for MyDriver {                       // вызовется при выгрузке (как exit)
       fn drop(&mut self) { pr_info!("my_driver: выгружен\n"); }
   }
```

💡 ⭐ Rust for Linux даёт **безопасные обёртки** над ядровыми API: то, что в C делаешь руками
(`copy_to_user`, освобождение ресурсов), в Rust обёрнуто так, что нарушения ловятся компилятором, а
очистка — через `Drop` (RAII). ⚠️ Экосистема **молодая**: поддержано не всё, API меняется, но
направление официальное и растущее. Для продакшена ядро всё ещё преимущественно C — но Rust наступает.

## ⭐⭐ Embedded Rust (сильнейшая сторона)

```
   embedded Rust — зрелая, мощная экосистема для микроконтроллеров (Cortex-M, RISC-V, ESP32):
   • no_std — без стандартной библиотеки/ОС, прямо на железе.
   • PAC (Peripheral Access Crate) — типобезопасный доступ к регистрам, СГЕНЕРИРОВАННЫЙ из SVD (svd2rust).
   • embedded-hal — общие ТРЕЙТЫ (GPIO, UART, SPI, I2C): драйвер датчика работает на ЛЮБОЙ плате,
     реализующей трейты. переносимость драйверов!
   • RTIC / Embassy — конкурентность/async на МК без ОС.
```

```rust
   #![no_std]
   #![no_main]
   use embedded_hal::digital::OutputPin;

   // мигнуть светодиодом — через трейт, не через сырой регистр:
   fn blink<P: OutputPin>(led: &mut P) {
       led.set_high().ok();      // типобезопасно; компилятор гарантирует корректность
       // задержка...
       led.set_low().ok();
   }
   // доступ к регистру (PAC) тоже типобезопасен:
   // dp.GPIOA.odr.modify(|_, w| w.odr5().set_bit());  ← нельзя записать в чужой бит по ошибке
```

💡 ⭐⭐ Embedded Rust — **флагман**: `embedded-hal`-трейты делают драйверы устройств **переносимыми**
между платами (драйвер датчика не зависит от конкретного МК), а PAC даёт **типобезопасный доступ к
регистрам** (компилятор не даст записать не тот бит). Это уровень безопасности, недостижимый в «голом»
C, при той же скорости. Многие выбирают Rust именно ради embedded.

## 📖 Userspace-драйверы на Rust

```
   на ПК (без модуля ядра) — обычные крейты:
   • rusb / nusb — общение с USB-устройствами (как libusb, но безопасно).
   • vfio / uio — PCI-устройства в userspace.
   • RAII + Result — ресурсы освобождаются сами, ошибки явны (не молчаливые коды).
   простой и безопасный способ «поговорить с железом» без риска для ядра.
```

## ⭐ Реальность и зрелость (честно)

```
   ✅ EMBEDDED Rust — зрелый, продакшен-готовый, растущая индустрия. ← начинай отсюда.
   🆕 Rust for Linux — официально в ядре, но молод: ограниченный набор драйверов, меняющийся API.
   ✅ USERSPACE — стабильно (rusb и т.п.).
   ⚠️ unsafe — при доступе к сырому железу/FFI всё равно нужен `unsafe` (но локализованный, обёрнутый
      в безопасный API). Rust не «отменяет» опасность железа, а ИЗОЛИРУЕТ её в маленькие проверенные блоки.
```

> 🧭 `unsafe` и FFI к C-драйверам — это [Interop/unsafe-FFI](../Rust/04-senior/22-unsafe-ffi.md):
> Rust оборачивает небезопасное в безопасный интерфейс.

## ⚠️ Ловушки (Rust-драйверы)

- ❌ Ждать, что Rust for Linux уже покрывает всё (молодо; многое ещё на C).
- ❌ Думать, что Rust «убирает» `unsafe` для железа (нужен, но локализуй и оборачивай).
- ❌ Тянуть `std` в embedded (там `no_std`; нет кучи/ОС по умолчанию).
- ❌ Игнорировать `embedded-hal`-трейты и писать к конкретному МК (теряешь переносимость).
- ❌ Тестировать kernel-модуль на рабочей ОС вместо ВМ.
