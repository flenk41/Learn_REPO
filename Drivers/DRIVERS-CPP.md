# 🟦 Драйверы на C++

> Честно: C++ — **НЕ для ядра Linux**, но **отличен для embedded** и userspace-драйверов. Разберём,
> где C++ уместен, где нет и почему, с конкретными приёмами.

> 🧭 Базовые концепции — [Интеграция · Драйверы](../Interop/04-drivers/20-userspace-kernel.md).
> ⚠️ Те же правила безопасности: ядро/железо — в виртуалке/на учебной плате.

---

## ⭐⭐ Где C++ можно, где нельзя

```
   ❌ ЯДРО LINUX — НЕТ. ядро написано на C, в нём НЕТ C++-рантайма:
      • нет исключений, нет RTTI, нет STL, нет new/delete по умолчанию.
      • name mangling, статические конструкторы, размотка стека — всё это ядру чуждо.
      → драйверы ядра Linux пишут на C (или теперь на Rust). C++ — мимо.

   ⚠️ ЯДРО WINDOWS — ОГРАНИЧЕННО. KMDF допускает ПОДМНОЖЕСТВО C++ (классы, без исключений/STL),
      но многие пишут на C. C++ в ядре Windows — «осторожно и частично».

   ✅ EMBEDDED / BARE-METAL — ДА, и отлично. C++ без рантайма (-fno-exceptions -fno-rtti) даёт
      zero-cost абстракции: классы периферии, шаблоны для регистров, типобезопасность. Arduino = C++!

   ✅ USERSPACE — ДА. драйверы поверх libusb/VFIO удобно обернуть в C++ (RAII, классы устройств).
```

💡 ⭐⭐ Запомни границу: **в ядре Linux C++ не используют** (нет рантайма; ядро — это C), а вот в
**embedded и userspace C++ — сильный выбор**. Если кто-то говорит «напишу драйвер ядра Linux на C++» —
это красный флаг непонимания. Зато на микроконтроллере C++ часто лучше C.

## ⚙️ Embedded на C++ (сильная сторона)

```cpp
   // безопасный, zero-cost доступ к пину через класс (компилируется в то же, что и C-биты):
   class Pin {
       volatile uint32_t& odr;
       uint32_t mask;
   public:
       Pin(volatile uint32_t& reg, int bit) : odr(reg), mask(1u << bit) {}
       void high() { odr |=  mask; }     // встроится (inline), накладных расходов 0
       void low()  { odr &= ~mask; }
       void toggle(){ odr ^=  mask; }
   };

   Pin led(GPIOA_ODR, 5);
   led.high();                            // читается лучше, чем GPIOA_ODR |= (1<<5)
```

```
   приёмы embedded-C++:
   • -fno-exceptions -fno-rtti — убрать рантайм-зависимости (как в ядре микроконтроллера).
   • классы периферии (Pin, Uart, Spi) — читаемо и типобезопасно, без накладных расходов.
   • шаблоны для регистров/адресов (проверка на этапе компиляции).
   • constexpr для адресов/конфигов. НЕТ динамической памяти (или свой аллокатор).
   • Arduino-фреймворк — это C++ (классы Serial, digitalWrite и т.п.).
```

💡 ⭐⭐ C++ в embedded даёт **читаемость и типобезопасность БЕЗ накладных расходов**: `led.high()`
компилируется ровно в `GPIOA_ODR |= (1<<5)`, но код понятнее и ошибиться сложнее. Это «zero-cost
abstractions» — главный козырь C++ для драйверов периферии. Отключаешь исключения/RTTI — и C++
работает даже без ОС.

## ⭐ Userspace-драйверы на C++

```cpp
   // обёртка над libusb с RAII (ресурс освобождается автоматически):
   class UsbDevice {
       libusb_device_handle* h = nullptr;
   public:
       UsbDevice(uint16_t vid, uint16_t pid) {
           h = libusb_open_device_with_vid_pid(ctx, vid, pid);
           if (!h) throw std::runtime_error("устройство не найдено");
       }
       ~UsbDevice() { if (h) libusb_close(h); }    // RAII: закрыли при выходе из области
       int transfer(...) { /* libusb_bulk_transfer */ }
   };
```

💡 ⭐ В userspace C++ блистает: **RAII** (деструктор сам закрывает устройство/освобождает буферы),
классы устройств, исключения для ошибок — всё это делает драйвер чище и безопаснее, чем «голый C».
Здесь исключения/STL доступны (это обычная программа, не ядро).

## 📖 Почему ядро Linux — это C

```
   • ядро задумано как C-проект: стабильный C-ABI, простая модель без скрытых вызовов.
   • C++-фичи привносят НЕЯВНОЕ: размотку стека (исключения), статические конструкторы,
     RTTI-таблицы, аллокации STL — в ядре это либо запрещено, либо опасно/непредсказуемо.
   • альтернатива безопасности в ядре теперь — Rust (Rust for Linux), а не C++.
   → не пытайся тащить C++ в kernel-драйвер Linux. для ядра: C или Rust.
```

> 🧭 Если нужна **безопасность памяти в ядре** — это путь [🦀 Rust](DRIVERS-RUST.md), а не C++.

## ⚠️ Ловушки (C++-драйверы)

- ❌ Писать kernel-драйвер Linux на C++ (нет рантайма; используй C/Rust).
- ❌ Исключения/STL/RTTI в embedded без понимания (включают рантайм/раздувают; `-fno-exceptions`).
- ❌ Динамическая память (`new`) в bare-metal без своего аллокатора (фрагментация/отказ).
- ❌ Думать, что классы «медленнее» битов — в embedded-C++ они zero-cost (инлайнятся).
- ❌ Забыть `volatile` для регистров (как и в C — обязательно).
- ❌ Тяжёлые абстракции/виртуальные вызовы на горячем пути микроконтроллера без нужды.
