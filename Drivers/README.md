# 🔌 Драйверы на C / C++ / Rust — практический гайд (обзор)

> Как написать драйвер — программу, через которую ОС и приложения общаются с **железом** (или
> виртуальным устройством). Здесь — честная картина по языкам и три практических разбора:
> - 🟨 **[Драйверы на C](DRIVERS-C.md)** — рабочая лошадь (ядро Linux, Windows WDK, embedded).
> - 🟦 **[Драйверы на C++](DRIVERS-CPP.md)** — честно: embedded — да, ядро — почти нет.
> - 🦀 **[Драйверы на Rust](DRIVERS-RUST.md)** — Rust for Linux + сильный embedded, безопасность.

> 🧭 Концепции (что такое драйвер, userspace vs kernel) глубоко разобраны в
> [треке Интеграция · Драйверы](../Interop/04-drivers/19-what-is-driver.md). Здесь — практика по языкам.

> ⚠️ **Безопасность и закон:** драйверы ядра работают с ПОЛНЫМИ привилегиями — баг роняет всю
> систему или открывает дыру. Учись и тестируй в **виртуальной машине** (не на рабочей ОС). На
> Windows ядровые драйверы требуют **подписи** (и Secure Boot). Не ставь чужие непроверенные драйверы.

## 🎯 Что такое драйвер

```
   ДРАЙВЕР — прослойка между ОС/приложениями и КОНКРЕТНЫМ устройством:
   • знает «язык» железа (регистры, протоколы, прерывания) и переводит его в понятный ОС интерфейс.
   • приложение пишет в /dev/mydevice или вызывает API ОС → драйвер дёргает железо.

   приложение → [ ОС / системный вызов ] → [ ДРАЙВЕР ] → [ ЖЕЛЕЗО ]
   (read/write/ioctl)                       (твой код)     (регистры, память, прерывания)
```

💡 Драйвер — это «переводчик» между софтом и конкретным устройством. Без него ОС не знает, как
говорить с твоей видеокартой/USB-гаджетом/датчиком. Драйвер бывает не только для физического железа,
но и для виртуальных устройств (например, виртуальный микрофон из [VoiceChanger](../VoiceChanger/README.md)).

## ⭐⭐ Три вида драйверов (важно для выбора языка)

```
   1. ЯДРО (kernel-mode) — драйвер работает ВНУТРИ ядра ОС, с полным доступом.
      • Linux: модуль ядра (.ko), Windows: .sys (WDM/KMDF).
      • МОЩНО, но баг = крах всей системы (kernel panic / BSOD) + дыра в безопасности.
      • язык: C (доминирует), Rust (растёт в Linux). C++ — почти нет.

   2. USERSPACE (user-mode) — драйвер как обычная программа, общается с железом через интерфейсы ядра.
      • Linux: libusb (USB), UIO / VFIO (PCI), FUSE (ФС). Windows: UMDF.
      • БЕЗОПАСНЕЕ (упал — упал только процесс, не система). медленнее (переходы в ядро).
      • язык: любой — C, C++, Rust, даже Python (для прототипов).

   3. EMBEDDED / BARE-METAL — нет ОС (или RTOS): пишешь ПРЯМО в регистры микроконтроллера.
      • STM32, AVR/Arduino, ESP32, RISC-V. «драйвер» = код доступа к периферии (GPIO, UART, SPI...).
      • язык: C (классика), C++ (отлично), Rust (сильный embedded-экосистема).
```

🖼️
```
   где работает драйвер:
   ┌─────────────────────────────────────────┐
   │  приложения (userspace)                  │  ← userspace-драйверы тут (безопасно)
   ├─────────────────────────────────────────┤
   │  ЯДРО ОС  ← kernel-драйверы тут (опасно, │
   │            полный доступ, .ko / .sys)    │
   ├─────────────────────────────────────────┤
   │  ЖЕЛЕЗО (регистры, прерывания, DMA)      │
   └─────────────────────────────────────────┘
   embedded: ОС нет вообще — твой код = и «приложение», и «драйвер», прямо поверх железа.
```

💡 ⭐⭐ Выбор вида определяет почти всё: **ядро** — максимум возможностей, максимум риска (только C/Rust);
**userspace** — безопаснее и проще (любой язык), для USB/PCI многого хватает; **embedded** — без ОС,
прямой доступ к регистрам (C/C++/Rust одинаково уместны). Начинай с **userspace или embedded** — там
учиться безопаснее, чем в ядре.

## ⭐ Реальность по языкам (честно)

```
                ЯДРО (Linux/Windows)        USERSPACE            EMBEDDED / bare-metal
   ─────────    ───────────────────────     ─────────────        ──────────────────────
   🟨 C         ✅ ДОМИНИРУЕТ (ядра на C)    ✅ да (libusb...)    ✅ классика (HAL, регистры)
   🟦 C++       ⚠️ почти НЕТ (ядро = C;      ✅ да (обёртки)      ✅ отлично (классы периферии,
                нет исключений/STL/RTTI)                          zero-cost, Arduino = C++)
   🦀 Rust      🆕 Rust for Linux (растёт)   ✅ да (rusb, vfio)   ✅ сильно (no_std, embedded-hal)
```

💡 ⭐ Ключевые выводы:
- **C** — универсален и доминирует в ядре. Если делаешь kernel-драйвер под Linux/Windows — почти всегда C.
- **C++** — **НЕ для ядра Linux** (ядро на C, без C++-рантайма), но **отличен для embedded** и userspace.
- **Rust** — **Rust for Linux** позволяет писать модули ядра на Rust (безопаснее C, но экосистема молода);
  очень силён в **embedded** (memory-safe доступ к регистрам).

## 🗺️ С чего начать (безопасно)

```
   1. НЕ начинай с kernel-драйвера на железной машине — только в виртуалке (QEMU/VirtualBox) или на учебной плате.
   2. ПРОЩЕ ВСЕГО — embedded: возьми плату (STM32/Arduino/ESP32) → помигай светодиодом (GPIO) →
      UART → SPI/I2C-датчик. Это «драйверы» в чистом виде, без риска уронить ОС.
   3. USERSPACE на ПК: через libusb поговори с USB-устройством — драйвер без ядра.
   4. ЯДРО (продвинуто): собери учебный char-driver в Linux-ВМ (insmod/rmmod/dmesg) — см. гайд по C.
   5. RUST: embedded Rust (blink на embedded-hal) или Rust for Linux модуль.
```

👉 Выбирай язык по задаче: **[🟨 C](DRIVERS-C.md)** (ядро/везде), **[🟦 C++](DRIVERS-CPP.md)**
(embedded/userspace), **[🦀 Rust](DRIVERS-RUST.md)** (безопасность, embedded, Rust-for-Linux).
