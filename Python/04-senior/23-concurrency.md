## ⭐ GIL — Global Interpreter Lock

В CPython есть **GIL** — глобальная блокировка, которая разрешает выполнять Python-байткод
**только одному потоку за раз**.

![GIL: один поток за раз](../assets/gil.svg)

> ⚠️ Следствие: для **CPU-нагрузки** (вычисления) потоки в Python **не дают ускорения** —
> GIL не пускает их на разные ядра одновременно. Это шокирует приходящих из C/Java.

**Но** GIL отпускается во время **ожидания** (сеть, диск, `sleep`). Поэтому для
**I/O-задач** потоки полезны.

---

## 📋 Три инструмента — когда что

| Инструмент | Для чего | Память |
|-----------|----------|--------|
| **threading** (потоки) | I/O-ожидание (сеть, файлы) | общая (один процесс) |
| **multiprocessing** (процессы) | CPU-вычисления (обход GIL) | у каждого своя! |
| **asyncio** | много I/O-ожиданий в одном потоке | общая, очень лёгкая |

## ⭐ threading — для ввода-вывода

Потоки делят **общую память** (как в модуле про память C). Хороши, когда задачи **ждут**
(скачивание, запросы):

```python
import threading
import time

def download(name):
    print(f"{name}: старт")
    time.sleep(2)              # имитация ожидания сети (GIL отпущен!)
    print(f"{name}: готово")

threads = [threading.Thread(target=download, args=(f"файл{i}",)) for i in range(3)]
for t in threads: t.start()    # запустить
for t in threads: t.join()     # дождаться
# 3 «загрузки» по 2 сек завершатся за ~2 сек, а не 6 — ожидание идёт параллельно
```

### ⚠️ Общая память → гонки данных (как в C!)

```python
counter = 0
def add():
    global counter
    for _ in range(1000000):
        counter += 1           # ⚠️ гонка! (как в C-курсе)

# нужна блокировка:
lock = threading.Lock()
def add_safe():
    global counter
    for _ in range(1000000):
        with lock:             # критическая секция
            counter += 1
```

💡 Если ты прошёл C-курс — это та же проблема гонок данных и мьютексов. Идея одинакова.

## ⭐ multiprocessing — для вычислений (обход GIL)

Каждый процесс — **отдельный интерпретатор со своей памятью и своим GIL**. Они реально
работают на разных ядрах параллельно.

```python
from multiprocessing import Pool

def heavy(n):
    return sum(i*i for i in range(n))

if __name__ == "__main__":          # обязательно на Windows!
    with Pool(4) as pool:           # 4 процесса на 4 ядра
        results = pool.map(heavy, [10**7, 10**7, 10**7, 10**7])
    print(sum(results))             # реально параллельно → быстрее
```

🖼️
```
   Процесс1 (своя память, своё ядро) ─┐
   Процесс2 (своя память, своё ядро) ─┼─► настоящий параллелизм
   Процесс3 ...                       ─┘
```

> ⚠️ Цена: у каждого процесса **своя память** — данные между ними нужно **копировать**
> (сериализовать через pickle). Создание процессов дороже потоков. Для мелких задач
> накладные расходы съедят выигрыш.

## ⭐ asyncio — тысячи I/O-задач в одном потоке

`asyncio` — кооперативная многозадачность: один поток переключается между задачами в
точках `await` (где идёт ожидание). Очень лёгкая по памяти.

```python
import asyncio

async def download(name):
    print(f"{name}: старт")
    await asyncio.sleep(2)         # await = "пока жду, дай поработать другим"
    print(f"{name}: готово")
    return name

async def main():
    results = await asyncio.gather(
        download("a"), download("b"), download("c")
    )
    print(results)

asyncio.run(main())
# 3 задачи завершатся за ~2 сек, в ОДНОМ потоке, почти без памяти на задачу
```

🖼️
```
   один поток, переключается на await:
   a: старт─await──────────готово
   b:   старт─await────────готово   все ждут "одновременно"
   c:     старт─await──────готово
```

💡 asyncio идеален для тысяч сетевых соединений (веб-серверы, скрейперы, боты). В отличие
от потоков — почти нет накладной памяти на задачу.

## 📋 Шпаргалка выбора

```
   Задача ждёт (сеть/диск), их немного   → threading
   Задача ждёт, их ТЫСЯЧИ                 → asyncio
   Задача считает (CPU), нужно ускорить   → multiprocessing
   Задача считает и её много              → multiprocessing / numpy / C
```
