## 📖 Инкапсуляция — прячем детали за интерфейсом

**Инкапсуляция** — объект скрывает свои данные и даёт доступ к ним только через методы/
свойства. Снаружи видно «что можно делать», а не «как устроено внутри».

В Python нет настоящих `private`-полей (как в C++/Java) — есть **соглашения**:

```python
class Account:
    def __init__(self, balance):
        self.owner = "Аня"      # публичный — часть API
        self._balance = balance  # _одно подчёркивание = "не трогай снаружи" (соглашение)
        self.__pin = "1234"      # __два = name mangling → _Account__pin (защита от случайностей)
```

```
   self.x      публичный    — часть интерфейса, меняй свободно
   self._x     защищённый   — "внутреннее, не трогай" (но технически доступно)
   self.__x    приватное    — Python переименует в _Класс__x (труднее задеть случайно)
```

💡 ⭐ Python верит программисту: `_x` — это **договорённость**, а не запрет. Зато код
честный и простой. `__x` (name mangling) защищает от случайного переопределения в наследниках,
но это **не** безопасность — данные всё ещё достижимы.

## ⭐⭐ `@property` — метод, который выглядит как атрибут

Проблема: хочется проверять значение при присваивании, но не ломать `obj.x = 5`-синтаксис.
Решение — **свойство**: снаружи это атрибут, внутри — методы.

```python
class Account:
    def __init__(self, balance):
        self._balance = balance

    @property
    def balance(self):              # геттер: вызывается при чтении obj.balance
        return self._balance

    @balance.setter
    def balance(self, value):       # сеттер: вызывается при obj.balance = ...
        if value < 0:
            raise ValueError("Баланс не может быть отрицательным")
        self._balance = value

acc = Account(100)
print(acc.balance)    # 100  — выглядит как атрибут, а вызывается геттер
acc.balance = 50      # ок   — вызывается сеттер с проверкой
# acc.balance = -10   # ValueError!
```

🖼️
```
   acc.balance        ──►  вызывает balance()      [геттер]
   acc.balance = 50   ──►  вызывает balance(50)    [сеттер с проверкой]
   снаружи — обычный атрибут, внутри — контроль
```

💡 ⭐⭐ `@property` — главный инструмент инкапсуляции в Python. Начинаешь с публичного
`self.x`; когда понадобился контроль — превращаешь в свойство, **не меняя код снаружи**.
Это «питоничная» замена геттерам/сеттерам Java (там их пишут сразу, в Python — по необходимости).

## ⭐ Вычисляемые свойства (только для чтения)

Свойство без сеттера = read-only вычисляемый атрибут:

```python
class Rectangle:
    def __init__(self, w, h):
        self.w = w
        self.h = h

    @property
    def area(self):           # считается на лету, не хранится
        return self.w * self.h

r = Rectangle(3, 4)
print(r.area)     # 12 — как атрибут, но вычисляется
# r.area = 100    # AttributeError: нет сеттера
```

💡 Вычисляемое свойство всегда **актуально** (меняешь `w` — `area` пересчитается) и не
хранит лишних данных. Если вычисление дорогое — кешируй через `functools.cached_property`.

## 📖 Дескрипторы — механика под капотом свойств

`@property` — это удобная обёртка над **дескриптором**: объектом с методами `__get__`/`__set__`.
Свой дескриптор полезен, когда одну и ту же логику доступа нужно переиспользовать в многих
классах:

```python
class Positive:                          # переиспользуемая проверка "число > 0"
    def __set_name__(self, owner, name):
        self._name = "_" + name
    def __get__(self, obj, objtype=None):
        return getattr(obj, self._name)
    def __set__(self, obj, value):
        if value <= 0:
            raise ValueError("Должно быть > 0")
        setattr(obj, self._name, value)

class Product:
    price = Positive()      # дескриптор как атрибут класса
    qty   = Positive()

p = Product()
p.price = 100     # ок
# p.qty = -5      # ValueError
```

💡 Дескрипторы — продвинутая тема, но именно на них держатся `@property`, методы, `@classmethod`,
`@staticmethod` и ORM-поля (Django/SQLAlchemy). Понимать их полезно; писать вручную — редко.

## ⚠️ Ловушки

- ❌ Думать, что `_x`/`__x` дают безопасность. Это соглашение и защита от случайностей, **не**
  защита данных. Кто захочет — достанет.
- ❌ Писать геттеры/сеттеры «на всякий случай» для каждого поля (как в Java). В Python начинай
  с публичного атрибута, переходи на `@property` **только когда нужен контроль**.
- ❌ Прятать в сеттер тяжёлую логику или побочные эффекты — присваивание должно быть «дешёвым».
- ❌ Путать `@property` (свойство экземпляра) с атрибутом класса.

## ✅ Задачи

1. **Temperature.** Класс с `celsius` (свойство) и `fahrenheit` (вычисляемое свойство,
   пересчёт из celsius). Сеттер `celsius` запрещает значения ниже −273.15.
2. **Account.** Поле `_balance` через `@property` с проверкой (≥0) в сеттере + метод `deposit`.
3. **Circle.** Свойство `radius` (с проверкой >0) и read-only `area`, `circumference`.
4. ⭐ **Дескриптор `NonEmptyString`.** Переиспользуй его для полей `name` и `email` класса `User`.
5. **Рефакторинг.** Возьми класс с публичными полями, добавь проверку через `@property`, не
   меняя код, который этот класс использует.
