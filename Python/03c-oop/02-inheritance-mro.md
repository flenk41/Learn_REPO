## 📖 super() — правильный вызов родителя

В модуле 16 ты видел переопределение методов. Часто нужно **дополнить** родителя, а не заменить —
для этого `super()`:

```python
class Animal:
    def __init__(self, name):
        self.name = name

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)     # вызвать конструктор родителя (задаст self.name)
        self.breed = breed         # потом добавить своё

d = Dog("Рекс", "хаски")
print(d.name, d.breed)             # Рекс хаски
```

💡 ⭐ `super().__init__(...)` — не «магия», а вызов следующего по цепочке метода. Без него
`self.name` не установится. Правило: в переопределённом `__init__` почти всегда зови
`super().__init__(...)`.

> ⚠️ В модуле 16 в примере было `super().__init__` без скобок — это **ничего не вызывает**
> (просто ссылка на метод). Всегда со скобками и аргументами: `super().__init__(name)`.

## ⭐⭐ MRO — порядок разрешения методов

Когда зовёшь `obj.method()`, Python ищет `method` по цепочке классов в строгом порядке —
**MRO** (Method Resolution Order). Для одиночного наследования это просто «вверх по дереву»:

```python
class A:
    def hello(self): print("A")
class B(A):
    pass

B().hello()        # "A" — в B нет, нашли в A
print(B.__mro__)   # (B, A, object) — порядок поиска
```

🖼️
```
   B().hello()  →  ищем в B → нет → в A → есть! → "A"
   MRO: B → A → object   (object — общий предок всех классов)
```

💡 `ClassName.__mro__` (или `.mro()`) показывает точный порядок поиска. Это твой инструмент
отладки «откуда взялся этот метод».

---

## ⭐⭐ Множественное наследование и проблема ромба

Python разрешает наследовать **от нескольких** классов. Тут MRO становится важен:

```python
class A:
    def who(self): print("A")
class B(A):
    def who(self): print("B")
class C(A):
    def who(self): print("C")
class D(B, C):       # ромб: D → B и C → оба от A
    pass

D().who()            # "B" — по MRO
print([c.__name__ for c in D.__mro__])   # ['D', 'B', 'C', 'A', 'object']
```

🖼️
```
        A           "ромб" наследования
       / \          D наследует и B, и C, оба от A
      B   C         MRO (алгоритм C3): D → B → C → A → object
       \ /          каждый класс встречается ОДИН раз, дети раньше родителей
        D
```

💡 ⭐⭐ Python строит MRO по алгоритму **C3-линеаризации**: слева направо, потомки раньше
предков, каждый класс один раз. Поэтому `A` вызовется единожды, даже если путей к нему два.
`super()` идёт **по MRO**, а не «к прямому родителю» — это и делает кооперативное множественное
наследование возможным.

> 🏛️ Множественное наследование — мощно, но опасно. Трек ООП объясняет, почему обычно лучше
> [композиция вместо наследования](../../OOP/03-design/16-composition-over-inheritance.md).

## ⭐ Миксины — наследование ради «добавки поведения»

**Миксин** — маленький класс, который не используется сам по себе, а «подмешивает» поведение:

```python
class JsonMixin:
    def to_json(self):
        import json
        return json.dumps(self.__dict__)

class ReprMixin:
    def __repr__(self):
        return f"{type(self).__name__}({self.__dict__})"

class User(JsonMixin, ReprMixin):   # подмешали обе способности
    def __init__(self, name):
        self.name = name

u = User("Аня")
print(u.to_json())   # {"name": "Аня"}
print(u)             # User({'name': 'Аня'})
```

💡 Миксины — «питоничный» способ переиспользовать поведение без глубоких иерархий. Называй их
с суффиксом `...Mixin`, держи маленькими и без своего состояния.

---

## ⚠️ Ловушки

- ❌ Забыть `super().__init__(...)` → атрибуты родителя не инициализируются.
- ❌ Писать `super().__init__` без скобок — это не вызов, а ссылка на метод (ничего не делает).
- ❌ Строить глубокие иерархии (5+ уровней) — хрупко, трудно читать. Чаще нужна композиция.
- ❌ Полагаться на «прямого родителя» в множественном наследовании — `super()` идёт по MRO,
  порядок может удивить. Всегда проверяй `__mro__`.
- ❌ Наследоваться только ради переиспользования кода (это «is-a»-связь, а не «у меня есть код»).

## 🛠️ Практика

```python
# выведи MRO любого класса и проследи, откуда берётся метод
class A:
    def f(self): print("A.f")
class B(A): pass
class C(A):
    def f(self): print("C.f")
class D(B, C): pass

print([c.__name__ for c in D.__mro__])
D().f()    # предскажи результат ДО запуска, потом проверь
```
