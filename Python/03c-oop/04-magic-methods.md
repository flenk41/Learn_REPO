## 📖 Что такое dunder-методы

**Dunder** (double underscore) — методы вида `__name__`, которыми Python управляет поведением
объекта в операциях языка. В модуле 16 ты видел `__init__`, `__repr__`, `__add__`, `__eq__`.
Здесь — глубже и системно.

```
   ты пишешь          Python зовёт
   a + b          →   a.__add__(b)
   len(a)         →   a.__len__()
   a[i]           →   a.__getitem__(i)
   for x in a     →   a.__iter__()
   with a as x    →   a.__enter__() / a.__exit__()
   print(a)       →   a.__str__()  (или __repr__)
```

💡 Перегрузка операторов — это **не** «хитрость», а способ дать своим типам естественный
синтаксис (как у `int`, `list`, `str`). Делай это, когда операция **очевидна** для типа
(вектор + вектор), и не делай, когда смысл неясен.

## ⭐ Операторы сравнения и арифметики

```python
from functools import total_ordering

@total_ordering                 # допишет >, >=, <=, != из __eq__ и __lt__
class Money:
    def __init__(self, amount): self.amount = amount
    def __repr__(self):  return f"Money({self.amount})"
    def __eq__(self, o): return self.amount == o.amount
    def __lt__(self, o): return self.amount <  o.amount
    def __add__(self, o): return Money(self.amount + o.amount)

a, b = Money(100), Money(50)
print(a + b)          # Money(150)
print(a > b)          # True  — выведено из __lt__ через @total_ordering
print(sorted([a, b])) # [Money(50), Money(100)] — сортировка использует __lt__
```

💡 ⭐ `@total_ordering` экономит код: реализуешь `__eq__` и `__lt__`, остальные сравнения Python
достроит. Объект с `__lt__` автоматически **сортируется** через `sorted()`/`.sort()`.

## ⭐⭐ Контекстные менеджеры — свой `with`

`with` гарантирует освобождение ресурса даже при ошибке. Свой класс делает это через
`__enter__`/`__exit__`:

```python
class Timer:
    def __enter__(self):
        import time
        self.start = time.perf_counter()
        return self                       # то, что попадёт в "as t"
    def __exit__(self, exc_type, exc_val, tb):
        import time
        self.elapsed = time.perf_counter() - self.start
        print(f"Заняло {self.elapsed:.3f} c")
        return False                      # False = не глушить исключение

with Timer() as t:                        # __enter__ при входе
    sum(range(10_000_000))
# __exit__ при выходе — даже если внутри была ошибка
```

🖼️
```
   with Timer() as t:        ──►  __enter__()   (захват ресурса)
       ...тело...
   (выход из блока)          ──►  __exit__()    (освобождение — ВСЕГДА)
   ошибка внутри → __exit__ всё равно вызовется (гарантия очистки)
```

💡 ⭐⭐ `with` — питоничный RAII (из курса C++!): ресурс освобождается детерминированно. Для
простых случаев есть `@contextlib.contextmanager` — генератор вместо класса. Применяй для
файлов, соединений, блокировок, «временно поменять состояние и вернуть назад».

> 🏛️ Это инкапсуляция «жизненного цикла ресурса» — близко к идее RAII из
> [C++](../../Cpp/02-memory/10-raii.md).

## ⭐ Контейнерные протоколы: len, индексация, итерация

Чтобы свой объект вёл себя как список/коллекция:

```python
class Playlist:
    def __init__(self): self._songs = []
    def add(self, s): self._songs.append(s)

    def __len__(self):            return len(self._songs)   # len(pl)
    def __getitem__(self, i):     return self._songs[i]     # pl[0], а ещё for и in!
    def __iter__(self):           return iter(self._songs)  # for s in pl

pl = Playlist(); pl.add("A"); pl.add("B")
print(len(pl))          # 2
print(pl[0])            # A
for song in pl:         # работает благодаря __iter__ (или даже __getitem__)
    print(song)
print("A" in pl)        # True — Python использует итерацию/__contains__
```

💡 Реализовав `__len__` + `__getitem__` + `__iter__`, ты получаешь объект, который ведёт себя
как встроенная коллекция: индексация, `for`, `in`, распаковка. Это и есть «питоничность».

## 📖 `__str__` vs `__repr__` и `__call__`

```python
class Point:
    def __init__(self, x, y): self.x, self.y = x, y
    def __repr__(self): return f"Point({self.x}, {self.y})"   # для разработчика (отладка)
    def __str__(self):  return f"({self.x}, {self.y})"         # для пользователя (print)

class Adder:
    def __init__(self, n): self.n = n
    def __call__(self, x): return x + self.n      # объект можно ВЫЗВАТЬ как функцию

add5 = Adder(5)
print(add5(10))    # 15 — add5(...) вызвал __call__
```

💡 Правило: `__repr__` — однозначное «как создать объект» (для отладки), `__str__` — красиво для
людей. Если есть только один — делай `__repr__`. `__call__` превращает объект в «функцию с
состоянием» (полезно для декораторов и фабрик).

## ⚠️ Ловушки

- ❌ Перегружать операторы там, где смысл неочевиден (`user1 + user2` — что это?). Читаемость важнее.
- ❌ В `__exit__` вернуть `True` по ошибке — это **проглотит** исключение. Возвращай `False`/`None`,
  если не хочешь подавлять ошибку.
- ❌ Реализовать `__eq__`, но забыть `__hash__` → объект станет нехешируемым (нельзя в `set`/`dict`
  ключом). Если объект неизменяемый — задай `__hash__` согласованно с `__eq__`.
- ❌ Делать только `__str__` без `__repr__` — в списках/отладке объект покажется невнятно.
