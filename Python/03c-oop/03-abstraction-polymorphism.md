# 3 · Абстракция и полиморфизм 🖼️⭐⭐

> 🎯 **Цель блока:** освоить абстракцию (ABC, Protocol) и полиморфизм в Python — включая
> «утиную типизацию», которая отличает Python от C++/Java.

---

## ⭐⭐ Полиморфизм — один вызов, разное поведение

**Полиморфизм** — код работает с объектами через **общий интерфейс**, не зная их точного типа:

```python
class Circle:
    def area(self): return 3.14 * self.r ** 2
class Square:
    def area(self): return self.side ** 2

def total_area(shapes):          # не знает, какие именно фигуры
    return sum(s.area() for s in shapes)   # зовёт area() у каждой

# работает с любыми объектами, у которых есть area()
```

💡 ⭐⭐ Полиморфизм **убирает `if по типу`**. Вместо `if isinstance(s, Circle): ... elif Square:`
ты зовёшь `s.area()` — каждый объект знает, как считать себя сам. Добавил новую фигуру — функция
`total_area` менять не нужно.

## ⭐ Duck typing — «если крякает как утка»

В Python полиморфизм не требует общего базового класса. Важно **поведение**, а не тип:

```python
class Duck:
    def quack(self): print("Кря!")
class Person:
    def quack(self): print("Я изображаю утку")

def make_it_quack(thing):
    thing.quack()         # неважно, какой класс — лишь бы был quack()

make_it_quack(Duck())     # Кря!
make_it_quack(Person())   # Я изображаю утку
```

🖼️
```
   "Если оно ходит как утка и крякает как утка — это утка."
   Python не проверяет ТИП, он проверяет НАЛИЧИЕ метода во время вызова.
   → гибко, но ошибки вылезают в рантайме, а не при компиляции
```

💡 ⭐ Duck typing — «питоничный» полиморфизм. В C++/Java объекты обязаны наследовать общий
интерфейс; в Python достаточно «иметь нужные методы». Гибко, но за гибкость платишь: опечатка в
имени метода всплывёт только при вызове.

## ⭐⭐ Абстрактные классы (ABC) — обязательный контракт

Когда хочешь **заставить** наследников реализовать методы — используй `abc`:

```python
from abc import ABC, abstractmethod

class Shape(ABC):                    # абстрактный базовый класс
    @abstractmethod
    def area(self): ...              # без реализации — обязателен у наследников

    def describe(self):              # обычный метод — общий для всех
        return f"Фигура с площадью {self.area()}"

class Circle(Shape):
    def __init__(self, r): self.r = r
    def area(self): return 3.14 * self.r ** 2

# Shape()           # ❌ TypeError: нельзя создать абстрактный класс
# class Bad(Shape): pass; Bad()   # ❌ TypeError: area() не реализован
print(Circle(2).describe())        # Фигура с площадью 12.56
```

💡 ⭐⭐ `ABC` + `@abstractmethod` превращают «договорённость» в **проверяемый контракт**: Python
не даст создать объект, пока все абстрактные методы не реализованы. Это аналог `interface`/
`abstract class` из Java/C++.

## ⭐ Protocol — «структурный» интерфейс (без наследования)

Иногда хочется контракт, но **без** обязательного наследования (в духе duck typing, но с
проверкой типов). Для этого `typing.Protocol`:

```python
from typing import Protocol

class Drawable(Protocol):        # контракт по структуре
    def draw(self) -> None: ...

def render(obj: Drawable):       # принимает ЛЮБОЙ объект с методом draw()
    obj.draw()

class Button:                    # НЕ наследует Drawable — но подходит!
    def draw(self): print("[Кнопка]")

render(Button())                 # ок: у Button есть draw()
```

💡 `Protocol` = «утиная типизация, которую понимает проверяльщик типов (mypy/pyright)». Объект
подходит, если у него есть нужные методы — наследование не требуется. Лучшее из двух миров:
гибкость duck typing + статическая проверка.

| Подход | Связь | Проверка |
|---|---|---|
| `ABC` (наследование) | явная (`class C(Base)`) | в рантайме при создании |
| `Protocol` (структурная) | неявная (по методам) | статически (mypy) |
| duck typing | никакой | только при вызове |

## ⚠️ Ловушки

- ❌ Городить `if isinstance(x, ...)` по типам вместо полиморфного вызова метода.
- ❌ Забывать, что duck typing откладывает ошибку до рантайма — покрывай тестами.
- ❌ Делать базовый класс абстрактным «на всякий случай» — `ABC` нужен, когда реально хочешь
  запретить создание и обязать методы.
- ❌ Путать `Protocol` (статический контракт по структуре) с `ABC` (наследование + рантайм-запрет).

## ✅ Задачи

1. **Полиморфизм.** Классы `Cat`, `Dog`, `Cow` с методом `sound()`. Функция, печатающая звуки
   списка животных без `if по типу`.
2. **ABC.** Абстрактный `PaymentMethod` с `@abstractmethod pay(amount)`. Реализации `Card`,
   `Cash`. Проверь, что нельзя создать `PaymentMethod()` и неполную реализацию.
3. ⭐ **Protocol.** Опиши `Comparable` Protocol и функцию `maximum(items)`, работающую с любым
   типом, у которого есть нужные методы.
4. **Duck typing.** Функция `serialize(obj)`, зовущая `obj.to_dict()` — работает с разными
   классами без общего предка.
5. **Сравнение.** Реши одну задачу через `ABC` и через `Protocol`. Сравни, когда что удобнее.
