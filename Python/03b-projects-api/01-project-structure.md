# 1 · Структура проекта: пакеты и модули 🖼️

> 🎯 **Цель блока:** научиться раскладывать Python-проект так, как это делают в реальных
> командах. Опираемся на [18 · Модули, пакеты, venv](../03-middle/18-modules-venv.md),
> но идём к полноценной структуре проекта.

---

## 📖 От скрипта к проекту

Один файл `main.py` на 1000 строк — это боль: не найти ничего, страшно менять, нельзя
переиспользовать. Решение — **разделение ответственности**: каждый модуль отвечает за
одну тему.

---

## ⭐ Стандартная раскладка проекта

🖼️ Так выглядит типичный современный Python-проект (layout `src/`):

```
myproject/
├── src/
│   └── myapp/                ← пакет (папка с кодом)
│       ├── __init__.py       ← делает папку пакетом, задаёт публичный API
│       ├── __main__.py       ← точка входа: python -m myapp
│       ├── client.py         ← один модуль = одна тема
│       ├── models.py
│       └── utils.py
├── tests/                    ← тесты
│   └── test_client.py
├── pyproject.toml            ← описание проекта и зависимостей
├── requirements.txt          ← зависимости (или внутри pyproject)
├── README.md
└── .gitignore                ← venv/, __pycache__/ и т.п.
```

| Элемент | Что это |
|---------|---------|
| `src/myapp/` | сам пакет (импортируется как `import myapp`) |
| `__init__.py` | помечает папку пакетом; здесь задают, что «видно снаружи» |
| `__main__.py` | запуск пакета как программы (`python -m myapp`) |
| `tests/` | тесты (pytest) |
| `pyproject.toml` | метаданные проекта, зависимости, настройки инструментов |

💡 Раскладка `src/` — современный стандарт: код пакета лежит в `src/`, что защищает от
случайных «неправильных» импортов и делает проект устанавливаемым.

## ⭐ Пакет и `__init__.py`

**Пакет** — это папка с файлом `__init__.py`. Модули внутри импортируются через точку:

```python
# src/myapp/client.py
def get_user(login): ...

# использование
from myapp.client import get_user
import myapp.client
```

### `__init__.py` задаёт «лицо» пакета

```python
# src/myapp/__init__.py — что увидят те, кто пишет "from myapp import ..."
from .client import get_user, get_repos
from .models import User

__version__ = "1.0.0"
__all__ = ["get_user", "get_repos", "User"]   # публичный API пакета
```

🖼️
```
   Пользователь:  from myapp import get_user     ← удобно, видит "лицо"
                  # не нужно знать, что это в client.py

   Внутри:        свободно переставляешь функции между модулями,
                  публичный импорт остаётся прежним
```

💡 Так пользователь пишет `from myapp import get_user`, не зная внутренней структуры. Это
уже проектирование API (следующий модуль).

## ⭐ Публичное и приватное: соглашение про `_`

В Python нет `private`, но есть **соглашение**: имена с подчёркивания `_` — внутренние,
их не трогают снаружи.

```python
def get_user(login):        # ✅ публичное — часть API
    return _fetch(login)    # _fetch — приватное, деталь реализации

def _fetch(url):            # подчёркивание = "не трогай, я внутренний"
    ...
```

| Имя | Смысл |
|-----|-------|
| `name` | публичное (часть API) |
| `_name` | внутреннее (деталь реализации, можно менять) |
| `__name` | очень внутреннее (искажается интерпретатором в классах) |

💡 `__all__` в `__init__.py` + соглашение `_` — это и есть «инкапсуляция по-питоновски».
Сравни с opaque-типами из [C-курса](../../C/03b-projects-api/02-designing-api.md).

## ⭐ pyproject.toml — паспорт проекта

Современный способ описать проект (заменяет старый `setup.py`):

```toml
[project]
name = "myapp"
version = "1.0.0"
description = "Мой первый нормальный проект"
requires-python = ">=3.10"
dependencies = [
    "requests>=2.31",
]

[project.scripts]
myapp = "myapp.__main__:main"     # создаёт команду "myapp" в терминале

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"
```

Установка проекта в режиме разработки (изменения видны сразу):
```bash
pip install -e .          # -e = editable, ставит из текущей папки
```

## ⭐ Точка входа: `__main__.py` и `if __name__`

```python
# src/myapp/__main__.py
def main():
    print("Запуск myapp")

if __name__ == "__main__":
    main()
```
```bash
python -m myapp           # запускает __main__.py
```

💡 Помни `if __name__ == "__main__"` из модуля 18 — он отделяет «код для запуска» от «кода
для импорта».

## 📖 Направление зависимостей

Как и в C: модули зависят **в одну сторону**, без циклов.

```
   __main__.py / cli ──► сервисы (client) ──► утилиты, модели
   (верх)                (середина)            (низ, самодостаточны)
```

> ⚠️ Если `utils.py` импортирует `cli.py` — это цикл и тревожный сигнал. Низкоуровневое не
> должно зависеть от высокоуровневого. Циклический импорт в Python ещё и **падает** с
> ошибкой `ImportError`.

## ✅ Задачи

1. **Скрипт → пакет.** Возьми свой проект из Middle (например библиотеку книг), преврати
   в пакет `src/myapp/` с несколькими модулями.
2. **__init__ как API.** Настрой `__init__.py` так, чтобы работало
   `from myapp import <главное>`, задай `__all__` и `__version__`.
3. **Приватность.** Раздели функции на публичные и `_приватные`, проверь, что снаружи
   зовутся только публичные.
4. **pyproject.** Напиши `pyproject.toml`, установи проект через `pip install -e .`,
   импортни из другого места.
5. **Точка входа.** Сделай `__main__.py`, запусти через `python -m myapp`.
6. ⭐ **Цикл импорта.** Намеренно создай циклический импорт, поймай ошибку, исправь
   реструктуризацией.
